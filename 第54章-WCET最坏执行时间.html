本章目标:第52章可调度性分析需要每个任务的执行时间 C——这个 C 就是 WCET(Worst-Case Execution Time,最坏执行时间)。你怎么知道一段代码"最慢"要跑多久?这个看似简单的问题,是实时系统里最难的之一(现代 CPU 的 Cache、流水线、分支预测让它几乎无法精确求出)。本章讲 WCET 为何难、怎么估、以及如何"为可预测性而编程"让代码的最坏时间可控。这是实时分析的最后一块拼图。

54.1 WCET：实时分析的基石

回顾第52章:可调度性分析(RMS/EDF 判据)要用每个任务的执行时间 C。但这个 C 不能用"平均执行时间",必须用"最坏执行时间"——因为实时要保证的是最坏情况不误点。

为什么必须用WCET(最坏), 不能用平均?
  可调度性分析要证明"即使最坏情况也不误点"
  → 如果用平均执行时间算, 万一某次执行恰好是最坏(慢),
    实际可能就误点了 → 证明失效!
  → 必须用WCET: 假设每个任务都花最坏时间, 若这样都不误点,
    那实际(通常更快)一定不误点 → 安全的保证

  WCET 是整个实时分析的【输入】:
   WCET → 可调度性分析(52) → "保证不误点"的结论
   WCET不准(低估) → 分析结论不可信 → 系统可能实际误点!

WCET 是实时可调度性分析的输入基石:整个"证明不误点"的推理,都建立在"每个任务的执行时间不超过其 WCET"这个前提上。用平均时间分析是危险的——万一实际执行慢于平均,证明就失效了。只有用 WCET(最坏)分析,"最坏都不误点"才能推出"实际一定不误点"。反过来,如果 WCET 估低了(低估),整个可调度性证明就不可信,系统可能在实际运行中误点——所以 WCET 必须是安全的上界(宁可高估不可低估)。这就是为什么求 WCET 如此重要又如此讲究。

54.2 生活例子：最坏通勤时间

你要赶 9 点的重要会议,住处到公司平均 30 分钟。你会 8:30 出门吗?不会——因为"平均 30 分钟"意味着经常超过 30 分钟(堵车、等红灯、地铁延误)。你会算最坏通勤时间:"最堵的时候要 50 分钟",于是 8:10 出门,留足余量。

实时任务也一样:要保证 deadline 前完成,必须按"最坏执行时间"(WCET,最堵的 50 分钟)来规划,而不是平均(30 分钟)。如果你按平均 30 分钟规划(8:30 出门),遇到堵车就迟到(误 deadline)。而 WCET 难就难在:通勤时间受太多因素影响(路况、天气、意外),很难精确知道"最坏到底多少"——只能保守估个上界(宁可估 60 分钟早到,也不能估 40 分钟迟到)。代码的 WCET 也面临同样的困境:受 Cache、分支、输入数据等太多因素影响,难以精确,只能求安全的上界。

54.3 WCET 的精确定义

给出定义和相关概念:

★WCET (Worst-Case Execution Time):
  一段代码在【所有可能的输入和执行环境下】, 执行所需的【最长】时间
  (在一个特定硬件平台上, 不含被抢占/中断打断的时间——那是调度的事)

相关概念:
  BCET (Best-Case): 最好情况执行时间(最短)
  ACET (Average):  平均执行时间
  实际执行时间分布在 [BCET, WCET] 之间

  ★关键: 我们要的WCET是【安全上界】(safe upper bound):
    保证 ≥ 真实的最坏, 但又不能太宽松(否则浪费CPU、判定过于保守)
    理想: 紧的安全上界(tight & safe)

  注意区分:
   WCET = 单纯执行这段代码的最坏时间(不含被打断)
   WCRT (Worst-Case Response Time) = 从触发到完成(含被高优先级抢占、
         优先级反转阻塞等) —— WCRT用WCET+调度分析算出(第52/53章)

WCET 是"一段代码本身最慢跑多久"(不含被别的任务打断)。要区分它和 WCRT(最坏响应时间):WCET 是代码纯执行的最坏时间;WCRT 是任务从触发到完成的最坏时间,包含了被高优先级抢占、优先级反转阻塞(第53章)等——WCRT 是用 WCET 加上调度分析(第52章)算出来的。WCET 是"砖",WCRT 是"墙"。而我们求 WCET 要的是"紧的安全上界":必须 ≥ 真实最坏(安全,不能低估),又不能太宽松(否则太保守、浪费 CPU)。这个"既安全又紧"的要求,正是 WCET 分析的难点。

54.4 为什么 WCET 极难求

WCET 看似简单(不就是代码跑最慢的情况吗),实则是实时系统最难的问题之一。原因:

WCET 难求的原因:
  ① 路径爆炸: 代码有很多分支(if/循环), 不同输入走不同路径,
     路径组合数量爆炸, 找"最慢路径"极难
  ② 循环次数: 循环跑多少次? 依赖输入, 有时无法静态确定上界
     (无界循环 = WCET无界 = 不可用于硬实时!)
  ③ ★硬件不确定(最难): Cache命中/未命中、流水线、分支预测、
     TLB、内存访问延迟 —— 同一条指令, 执行时间可能差几十倍!
     且这些依赖【历史执行状态】, 极难建模
  ④ 输入依赖: 数据不同, 执行时间不同(如查找、排序)

  → 结果: 精确的WCET几乎无法求出, 只能【估计上界】
     (估太松 → 浪费; 估太紧 → 可能不安全)

WCET 难求的根本在于:执行时间不是代码本身决定的,而是"代码 + 输入 + 硬件状态 + 历史"共同决定的,变量太多、组合爆炸。①程序有大量分支和循环,不同输入走不同路径,找"最慢路径"是组合爆炸;②循环次数可能依赖输入(如果无法确定循环上界,WCET 就无界,该代码根本不能用于硬实时);③最难的是硬件——同一条指令,Cache 命中还是未命中,执行时间差几十倍(第八章),而 Cache 状态依赖之前执行了什么(历史相关),几乎无法精确建模;④数据依赖(排序不同数据耗时不同)。这些叠加,让"精确的 WCET"在现代 CPU 上几乎是不可能求出的,只能求"安全的上界估计"。这就是为什么 WCET 是学术界研究了几十年仍未完全解决的难题——也是实时系统"看似简单实则深奥"的典型。

54.5 硬件带来的不确定

硬件是 WCET 最大的敌人,把第八~十章的知识串起来看它们如何影响 WCET:

硬件特性如何影响 WCET章节
Cache命中快、未命中慢几十倍;状态依赖历史,难预测第八章
流水线流水线满/空、冒险 stall,时间不定第三章
分支预测预测对则快,预测错要清流水线(几十周期)第三章
TLB地址翻译缓存命中/未命中第九章
乱序执行/超标量指令重排、并行,时间极难分析第三章
DRAM 刷新/总线竞争内存访问延迟波动第八章

讽刺的是:现代 CPU 为提升"平均性能"引入的所有优化(Cache、流水线、分支预测、乱序执行),恰恰都是 WCET 的敌人——它们让执行时间高度依赖运行时状态,变得不可预测。Cache 让平均访问快,但命中与否差几十倍且依赖历史;分支预测让平均快,但预测错要付出几十周期代价;乱序执行让平均快,但时间几乎无法分析。这是第50章"吞吐 vs 实时"矛盾在硬件层的体现:这些优化都是为平均性能(吞吐),代价是最坏情况不可预测(伤实时)。所以严格的硬实时系统有时反而用简单、可预测的处理器(如某些无 Cache 或 Cache 可锁定、无乱序的 MCU),宁可慢也要可预测——飞控的 MCU 往往就是这类。这再次印证:实时系统常常"反性能优化"地追求确定性。

54.6 测量法 vs 静态分析法

实际求 WCET 上界,有两大方法:

测量法(measurement)静态分析法(static analysis)
做法大量运行,测实际执行时间,取最大值(再加余量)不运行,分析代码+硬件模型,计算上界
优点简单、贴近真实硬件能给出保证的安全上界
缺点不保证测到了真正最坏(可能漏掉最坏路径/状态)难、需精确硬件模型;结果常偏保守(松)
适合软实时、参考估计硬实时、安全认证(航空 DO-178C 等)

两种方法各有致命短板:测量法简单但不安全——你测了 100 万次的最大值,但不能保证第 100 万零一次不会更慢(可能没触发到最坏路径或最坏 Cache 状态),所以测量法给不出"保证的上界"(实践中常测量+加安全余量,如 max×1.5)。静态分析法不运行代码,通过分析程序结构和硬件模型计算上界,能给出保证安全的上界,但需要精确的硬件模型(现代 CPU 极难建模)、结果往往过于保守(高估很多)。硬实时/安全认证(如航空 DO-178C、汽车 ISO 26262)要求静态分析给出保证的上界;一般工程用测量+余量。还有混合方法(测量各代码段+静态组合)。实践中,没有完美方法,都要在"安全"和"紧"之间权衡。

54.7 为可预测性而编程

既然 WCET 难求,一个更重要的思路是:写代码时就让 WCET 容易确定、容易变小——"为可预测性而编程"。

让代码WCET可预测/可控的编程准则:
  ① ★所有循环有明确的、静态可知的次数上界
     for(i=0; i

核心思想:与其事后费力去求一段"混乱代码"的 WCET,不如一开始就写"WCET 容易确定"的代码。最关键的准则是"所有循环必须有静态可知的次数上界"——一个 while(链表非空) 遍历不定长链表的循环,WCET 就无界(链表可能任意长),不能用于硬实时;改成有明确上界的 for 循环才可分析。同理,避免递归(深度难界定)、避免动态内存(malloc 时间不定,用内存池——第24章)、避免数据依赖强的算法。这些准则听起来是"限制",但正是硬实时代码的编程规范(如航空的 MISRA C 就禁止很多不可预测的构造)。这体现了实时系统"从设计源头保证确定性"的哲学,而不是事后补救。你写飞控/控制代码时遵循这些,WCET 才可控,系统才可能通过可调度性证明。

54.8 让代码 WCET 可控的手段

除了编程准则,还有一些系统级手段减小 WCET 或让它可预测:

手段作用
Cache 锁定把关键代码/数据锁进 Cache,消除未命中不确定(第八章)
关闭/简化 Cache严格场景直接不用 Cache,时间可预测(慢但确定)
mlockall 锁内存防换页(第十章),消除缺页延迟
预热(warm-up)先跑一遍把 Cache/TLB 填好,再测/运行
关闭 DVFS/固定频率频率恒定,执行时间稳定
CPU 隔离(isolcpus)关键任务独占核,无其他任务干扰 Cache(第八卷)
内存池替代 malloc,分配时间确定(第24章)

这些手段的共同目标:消除或限定影响 WCET 的不确定源,让代码执行时间可预测。它们和第51章"降低延迟的手段"高度重合——因为 WCET(执行时间)和延迟(响应时间)都受同样的不确定源影响。典型组合:mlockall 锁内存 + 内存池 + CPU 隔离 + 固定频率 + Cache 预热/锁定,把一个实时任务的执行环境"净化"成可预测的。代价还是那句:牺牲平均性能(不用某些优化、独占核浪费),换执行时间的确定性。这是实时工程师为保证 WCET 常做的系统配置。

54.9 WCET 与可调度性的闭环

把 WCET 放回实时分析的整体流程,形成闭环:

实时系统设计的完整闭环:
  ① 确定每个任务的 WCET (本章): 测量/静态分析/为可预测编程
        │  C_i = WCET_i
        ▼
  ② 可调度性分析 (第52章): 用WCET作为C, 套RMS/EDF判据
        + 加入优先级反转阻塞时间 (第53章)
        + 加入中断/调度延迟 (第51章)
        │
        ▼
  ③ 判定: 每个任务的 WCRT ≤ deadline?
        是 → 系统可调度, 保证不误点 ✓
        否 → 优化WCET(减小C)/ 调整优先级/ 换硬件/ 减少任务
        │
        ▼
  ④ cyclictest/实测 验证 (第51章), 留安全余量

  ★这就是实时系统"可证明满足deadline"的完整工程方法论

WCET 是实时分析闭环的第一环,没有它整个分析无从谈起。完整流程:①求每个任务的 WCET(本章)→ ②以 WCET 为 C 做可调度性分析(第52章),并计入优先级反转阻塞(第53章)、中断/调度延迟(第51章)→ ③判断每个任务的最坏响应时间 WCRT 是否 ≤ deadline → ④用 cyclictest/实测验证并留余量。这就是实时系统"可证明满足 deadline"的完整工程方法论——把第七卷前面几章(延迟、调度、反转、WCET)全串起来了。如果判定不通过(某任务 WCRT>deadline),就回头优化:减小 WCET(优化代码/加 Cache 锁定)、调整优先级分配、升级硬件、或减少任务。这个闭环是实时工程师的核心工作流,也是"硬实时"区别于"能跑就行"的本质——一切都要有分析和证明支撑。

54.10 现代 CPU 的挑战

WCET 在现代高性能 SoC(如 RK3588)上尤其困难,这是前沿话题:

现代多核、乱序、深流水、多级 Cache 的高性能 SoC,让 WCET 分析几乎不可能做到"紧且安全"。多核共享 Cache/内存总线——一个核的任务执行时间受其他核在干什么影响(它们抢 Cache、抢内存带宽),这叫"核间干扰",让单核 WCET 分析失效。乱序执行、深流水、复杂分支预测让指令时序几乎无法建模。结果:在 RK3588 这类芯片上,你很难给出严格保证的 WCET。业界的应对:①硬实时关键任务放在专用的简单核/MCU 上(如飞控用独立 Cortex-M,不用大核 A76)——这就是第八卷 AMP(非对称多处理)的动机之一;②在 SoC 上用 CPU 隔离、Cache 分区(如 ARM 的 MPAM)、内存带宽限制来减少核间干扰;③对不那么严格的软实时,用测量+大余量。这解释了一个重要的系统架构选择:为什么复杂系统常"MCU 跑硬实时控制 + SoC 跑软实时的复杂应用"分工——因为高性能 SoC 给不了硬实时需要的 WCET 保证。理解这点,你就懂了车企"域控制器 + 安全 MCU"混合架构的深层原因。

54.12 面试常问问题

Q1:什么是 WCET?为什么实时分析要用它而非平均?
最坏执行时间:代码在所有输入和环境下的最长执行时间。实时要保证最坏不误点,用平均分析万一实际慢就失效;用 WCET,"最坏都不误点"才能推出"实际一定不误点"。

Q2:WCET 和 WCRT 的区别?
WCET 是代码纯执行的最坏时间(不含被打断);WCRT 是任务从触发到完成的最坏时间(含被抢占、优先级反转阻塞)。WCRT 由 WCET + 调度分析算出。

Q3:为什么 WCET 极难精确求?★
路径爆炸(分支组合)、循环次数依赖输入、最主要是硬件不确定(Cache 命中/未命中差几十倍且依赖历史、流水线、分支预测、乱序),加数据依赖。现代 CPU 上几乎无法精确,只能求安全上界。

Q4:为什么现代 CPU 优化反而伤 WCET?
Cache/流水线/分支预测/乱序都为平均性能,使执行时间高度依赖运行时状态、不可预测。这是吞吐 vs 实时矛盾在硬件层的体现。严格硬实时反而用简单可预测的处理器。

Q5:求 WCET 的两种方法及优劣?
测量法(大量运行取最大+余量):简单但不保证测到真正最坏(不安全)。静态分析法(分析代码+硬件模型算上界):能给保证的安全上界但难、偏保守。硬实时/安全认证用静态,一般工程用测量+余量。

Q6:怎么写代码让 WCET 可控?
循环有静态可知次数上界、避免递归、避免动态内存(用内存池)、避免数据依赖强的算法、控制分支复杂度、慎用不确定库函数。为可预测性而编程。

Q7:为什么硬实时任务常放 MCU 而非高性能 SoC?
多核乱序 SoC 有核间干扰(共享 Cache/带宽),WCET 无法保证;简单 MCU 时序可预测,能给 WCET 保证。故常"MCU 跑硬实时控制 + SoC 跑复杂应用"(AMP,第八卷)。

54.13 常见误区

  1. "WCET 就是多测几次取最大"——测量不保证测到真正最坏,不安全,要加余量或静态分析。
  2. "可调度性分析用平均执行时间"——必须用 WCET(最坏),否则证明失效。
  3. "WCET 和 WCRT 一样"——WCET 是纯执行,WCRT 含被抢占/阻塞。
  4. "现代 CPU 越快越适合实时"——快是平均快,其优化让 WCET 不可预测,可能不适合硬实时。
  5. "任何代码都能求出 WCET"——无界循环/递归的 WCET 无界,不能用于硬实时。
  6. "WCET 是编译器的事"——是系统级问题,涉及硬件、代码结构、编程规范。

54.14 本章总结

  • WCET(最坏执行时间):代码在所有输入/环境下最长执行时间,是可调度性分析(第52章)的输入 C;必须用最坏(非平均),且要安全上界(宁高估不低估)。
  • 区分:WCET(纯执行,不含打断)vs WCRT(响应,含抢占/反转阻塞,由 WCET+调度分析算出)。
  • WCET 极难求★:路径爆炸、循环次数、硬件不确定(Cache/流水线/分支预测/乱序,同指令差几十倍且依赖历史)、数据依赖 → 只能求上界。
  • 讽刺:现代 CPU 为平均性能的优化恰是 WCET 的敌人(吞吐 vs 实时);严格硬实时反用简单可预测处理器。
  • 求法:测量法(简单不安全,加余量)vs 静态分析法(保证上界但难/保守);硬实时用静态。
  • 为可预测性而编程:循环有静态上界、避免递归/动态内存/数据依赖算法 → 让 WCET 可控。
  • 系统手段:Cache 锁定/mlockall/预热/固定频率/CPU 隔离/内存池 → 消除不确定源。
  • 闭环:WCET → 可调度性分析(+反转+延迟)→ WCRT≤deadline? → 验证;现代 SoC 核间干扰使硬实时任务常放 MCU(AMP,第八卷)。
