本章目标：你已经把 FreeRTOS 内核骨架看透了。RT-Thread 是国产开源 RTOS,内核骨架和 FreeRTOS 几乎一样(位图调度、就绪表、上下文切换),但它有一个 FreeRTOS 没有的更高级的设计:用 C 语言实现了一套"面向对象"的内核对象系统,让线程、信号量、设备等都继承自统一的基类。这一章带你看这个对象模型和它的调度器。因为骨架你已熟悉,重点放在"RT-Thread 有什么不同、为什么这么设计"。

25.1 为什么在 FreeRTOS 之后学 RT-Thread

三个理由:

  • 它是国产主流 RTOS:大量国内产品用它,国内公司(包括做机器人、IoT 的)面试会问。你做过 RT-Thread(题目背景),正好深挖。
  • 它比 FreeRTOS 更"完整":有设备驱动框架、文件系统、网络协议栈、FinSH 命令行、包管理器——更接近一个"小型操作系统",是通往 Linux 的过渡。
  • 它展示了"C 语言如何做面向对象设计":这个技巧在 Linux 内核(第五卷)里无处不在(内核也是 C 写的"面向对象"),先在 RT-Thread 这个更清晰的地方学会它。

学习策略延续:FreeRTOS 让你看清"最精简的内核骨架",RT-Thread 让你看到"骨架上如何长出丰富组件、如何用 C 做优雅的抽象",Linux 则是这套思想的终极复杂版。三者递进。因为你已经懂了任务/调度/IPC 的原理,这一卷可以走得快些,聚焦 RT-Thread 的独特之处。

25.2 RT-Thread 的定位：RTOS 与 Linux 之间

RT-Thread 有两种形态,理解它的定位:

RT-Thread NanoRT-Thread 标准版
规模极小(几 KB),纯内核较大,带组件
类比相当于 FreeRTOS相当于"迷你 Linux"
组件只有调度/IPC/内存+ 设备框架、文件系统、网络、FinSH、GUI...
适用资源紧张的 MCU资源较宽裕、功能复杂的产品

它的野心是"用 RTOS 的体量,提供接近 Linux 的开发体验"——有类似 Linux 的设备模型(rt_device)、类似 shell 的 FinSH 命令行、类似 POSIX 的 API。所以它常用在"比单片机复杂、又用不起 Linux(没 MMU 或资源不够)"的中间地带。这个定位决定了它比 FreeRTOS 多出的那些设计,核心是为了"可扩展、可管理大量组件"——而这正需要一套统一的对象抽象。

25.3 内核对象：用 C 实现面向对象

RT-Thread 最有特色的设计:把线程、信号量、互斥量、邮箱、消息队列、定时器、设备……全都抽象成"内核对象(kernel object)",让它们继承自一个统一的基类 rt_object。这是用 C 语言模拟"面向对象"的经典手法。

为什么要这样?对比 FreeRTOS:FreeRTOS 里 TCB、Queue_t、Timer_t 是各自独立的结构,没有共同基类,管理它们的代码是分散的。而 RT-Thread 让它们共享一套"对象"的通用能力:每个对象都有名字、类型、能被统一地创建/删除/查找/遍历。这样:

  • 可以用统一的方式管理所有对象(比如 FinSH 命令 list_thread、list_sem 遍历所有线程/信号量);
  • 加新的对象类型时,复用基类的通用逻辑;
  • 调试工具能统一地枚举、显示所有内核对象。

这就是"用 C 语言做面向对象"——C 没有 class/继承,但可以用"结构体嵌套 + 把基类放在派生类结构体的第一个成员"来模拟继承。这个技巧(基类做第一个成员)你在第16章 FreeRTOS 的 pxTopOfStack 放第一个成员时见过类似思想,在 Linux 内核里(struct kobject、container_of 宏)会大规模见到。RT-Thread 是学这个技巧的最佳中间站——比 FreeRTOS 更体系化,又比 Linux 更清晰。

25.4 对象基类 rt_object

看基类定义(rtdef.h):

/* 所有内核对象的基类 */
struct rt_object
{
    char       name[RT_NAME_MAX];  /* 对象名字(调试/查找用) */
    rt_uint8_t type;               /* 对象类型(线程?信号量?设备?) */
    rt_uint8_t flag;               /* 标志 */
    rt_list_t  list;               /* 链表节点: 挂进对象容器 */
};
typedef struct rt_object *rt_object_t;

/* 对象类型枚举 */
enum rt_object_class_type
{
    RT_Object_Class_Thread = 0,   /* 线程 */
    RT_Object_Class_Semaphore,    /* 信号量 */
    RT_Object_Class_Mutex,        /* 互斥量 */
    RT_Object_Class_Event,        /* 事件 */
    RT_Object_Class_MailBox,      /* 邮箱 */
    RT_Object_Class_MessageQueue, /* 消息队列 */
    RT_Object_Class_Timer,        /* 定时器 */
    RT_Object_Class_Device,       /* 设备 */
    /* ... */
};

然后每个具体对象,都把 rt_object 作为自己结构体的第一个成员——这就是 C 语言的"继承":

/* 线程 = 继承 rt_object */
struct rt_thread
{
    struct rt_object parent;   /* ★第一个成员: 基类! 相当于"继承" */
    /* ↑ 因为放第一个, rt_thread指针可以直接当rt_object指针用(地址相同) */

    rt_list_t   tlist;         /* 线程链表节点 */
    void       *sp;            /* 栈指针(相当于FreeRTOS的pxTopOfStack) */
    void       *entry;         /* 线程入口函数 */
    rt_uint8_t  current_priority; /* 当前优先级 */
    rt_uint8_t  stat;          /* 线程状态 */
    /* ... */
};

/* 信号量 = 继承 rt_object(通过rt_ipc_object中间层) */
struct rt_semaphore
{
    struct rt_ipc_object parent;  /* parent里又含rt_object */
    rt_uint16_t value;            /* 信号量值 */
};

妙处:因为 parent(基类)是第一个成员,一个 rt_thread* 指针,强制转换成 rt_object* 就能当基类用(它们地址相同,偏移 0)——和第16章 pxTopOfStack 放第一个的道理一样。反过来,拿到一个 rt_object*,如果知道它其实是个线程,就能转回 rt_thread*。这就是 C 语言"向上/向下转型"的实现。

25.5 对象容器：统一管理所有内核对象

有了统一基类,RT-Thread 用一个对象容器(object container)管理所有对象——每种类型一个链表:

对象容器(rt_object_container): 每种对象类型一个信息结构 + 链表

  ┌──────────────────────────────────────┐
  │ [Thread]    → 线程1 → 线程2 → 线程3     │  所有线程挂这
  │ [Semaphore] → 信号量A → 信号量B         │  所有信号量挂这
  │ [Mutex]     → 互斥量X                   │
  │ [Timer]     → 定时器T                   │
  │ [Device]    → uart0 → spi1 → i2c2       │  所有设备挂这
  └──────────────────────────────────────┘

  创建对象 rt_object_init/allocate: 挂进对应类型的链表
  删除对象: 从链表摘除
  查找/遍历: 按类型遍历链表(rt_object_find 按名字查找)

这带来 FreeRTOS 没有的能力:

  • 按名字查找对象:rt_object_find("uart0", RT_Object_Class_Device)——像 Linux 里按路径找设备。
  • 统一枚举:FinSH 里敲 list_thread 就遍历线程链表打印所有线程状态,list_device 列所有设备。调试极方便。
  • 生命周期统一:静态对象(rt_object_init)和动态对象(rt_object_allocate,从堆分配)用统一接口管理。

这套"基类 + 对象容器"的设计,是 RT-Thread 相比 FreeRTOS 最大的架构差异。它的价值在"可管理性和可扩展性":当系统有几十个线程、信号量、设备时,统一的对象模型让管理、调试、扩展都变简单。代价是稍多的内存(每个对象带个 rt_object 头)和一点性能。这正是"RTOS 向操作系统演进"的体现——功能丰富了,就需要更强的抽象来组织。Linux 内核把这套思想发挥到极致(kobject、sysfs、设备模型),RT-Thread 是它的清晰预习版。

25.6 线程 rt_thread vs FreeRTOS TCB

RT-Thread 的"线程"就是 FreeRTOS 的"任务"(叫法不同,概念相同)。对照它们的控制块:

概念FreeRTOSRT-Thread
控制块TCB_tstruct rt_thread
栈指针pxTopOfStack(第一个成员)sp
优先级uxPriority(数值大=高)current_priority(数值小=高,和 FreeRTOS 相反!)
是否有对象基类无有(parent=rt_object)
创建xTaskCreatert_thread_create(动态)/rt_thread_init(静态)
入口函数返回不允许(死循环)允许(线程结束会被自动清理)

两个关键差异要记住(面试易混):

  • 优先级方向相反:FreeRTOS 数值越大优先级越高;RT-Thread 数值越小优先级越高(0 最高,和 Linux、和 Cortex-M 中断优先级一致)。搞反了调度全乱。
  • 线程可以正常结束:FreeRTOS 任务函数必须死循环(第16章);RT-Thread 线程入口函数可以 return,返回后线程进入 close 状态被系统回收(有个"空闲线程钩子"负责清理)。设计更接近普通函数/Linux 线程。

25.7 调度器：位图 + 就绪表（和 FreeRTOS 一个套路）

RT-Thread 的调度算法和 FreeRTOS 几乎一模一样——你已经会了。核心是优先级位图 + 就绪线程表:

/* 就绪线程优先级表: 每个优先级一个链表(和FreeRTOS pxReadyTasksLists一样) */
rt_list_t rt_thread_priority_table[RT_THREAD_PRIORITY_MAX];

/* 优先级位图: 哪些优先级有就绪线程(和FreeRTOS uxTopReadyPriority一样) */
rt_uint32_t rt_thread_ready_priority_group;
/* 若优先级>32, 用rt_uint8_t数组扩展 */

选最高优先级线程:
  用 __rt_ffs(rt_thread_ready_priority_group)  找最低的置1位
  (__rt_ffs = find first set, 底层用CLZ类指令, O(1))
  → 得到最高优先级(数值最小), 从对应链表取线程

看,和第17章 FreeRTOS 的 taskSELECT_HIGHEST_PRIORITY_TASK 完全是一个套路:位图记录哪些优先级有就绪线程,用硬件找位指令 O(1) 找出最高优先级,从对应链表取线程。唯一区别:FreeRTOS 用 CLZ 找最高位(因为数值大=高),RT-Thread 用 FFS(find first set)找最低位(因为数值小=高)——方向不同,原理相同。你在 FreeRTOS 学的位图调度,直接迁移过来。

这印证了第三卷收束说的:不同 RTOS 的内核骨架是同一套。位图 + 就绪表 + O(1) 找位,是所有优先级抢占式 RTOS 的通用实现(甚至 Linux 老的 O(1) 调度器也是)。你把它在 FreeRTOS 学透了,看 RT-Thread、看任何 RTOS 的调度器都是"哦,又是这套"。这就是深入学一个的价值——获得的是可迁移的洞察,不是某个 API 的记忆。

25.8 rt_schedule 与上下文切换

RT-Thread 的调度函数 rt_schedule() 对应 FreeRTOS 的 vTaskSwitchContext + 触发切换。逻辑同样熟悉:

void rt_schedule(void)
{
    /* 关中断 */
    rt_base_t level = rt_hw_interrupt_disable();

    /* ① 找到最高优先级的就绪线程 */
    to_thread = 用位图选出最高优先级就绪线程;

    /* ② 如果它不是当前线程, 才需要切换 */
    if (to_thread != rt_current_thread)
    {
        from_thread = rt_current_thread;
        rt_current_thread = to_thread;

        /* ③ 触发上下文切换(架构相关的汇编, 和FreeRTOS PendSV一样) */
        rt_hw_context_switch((rt_ubase_t)&from_thread->sp,
                             (rt_ubase_t)&to_thread->sp);
    }
    rt_hw_interrupt_enable(level);
}

底层 rt_hw_context_switch 在 Cortex-M 上的实现,和 FreeRTOS 的 PendSV 汇编几乎一样:也用 PendSV 做上下文切换(第18章讲的所有道理——为什么用 PendSV、双栈 MSP/PSP、硬件压 8 个+软件压 R4-R11——原样适用)。RT-Thread 的 libcpu/arm/cortex-m4/context_rvds.S 里那段 PendSV 汇编,你拿第18章的知识去读,一眼就懂。移植层的思想、上下文切换的机制,两个 RTOS 完全一致。

25.9 线程状态机

RT-Thread 线程状态和 FreeRTOS 对应(名字略不同):

RT-Thread 状态FreeRTOS 对应含义
RT_THREAD_INIT(创建中)刚创建,还没启动
RT_THREAD_READYReady就绪,等 CPU
RT_THREAD_RUNNINGRunning正在运行
RT_THREAD_SUSPENDBlocked/Suspended挂起(等 IPC/延时/被挂起)
RT_THREAD_CLOSE(Deleted)结束,待回收

状态转移逻辑和 FreeRTOS 一致:就绪的在就绪表,挂起的从就绪表摘除挂到 IPC 对象的等待队列/定时器链表。你在第16章学的"状态本质=线程挂在哪个链表",在 RT-Thread 一样成立。RT-Thread 把"阻塞"和"挂起"合并成一个 SUSPEND 状态(用不同的挂起原因区分),比 FreeRTOS 略简化。

25.10 与 FreeRTOS 的对照总结

把两个 RTOS 的核心对照,这张表帮你快速迁移知识:

维度FreeRTOSRT-Thread
任务/线程Task / TCB_tThread / rt_thread
优先级方向数值大=高数值小=高
对象模型各结构独立★统一 rt_object 基类 + 容器
调度算法位图+CLZ位图+FFS(同套路)
上下文切换PendSV 汇编PendSV 汇编(同)
IPC 基础一切基于队列各 IPC 独立(基于 rt_ipc_object)
组件丰富度内核为主★丰富(设备/文件系统/网络/FinSH)
线程可 return?否是

核心结论:内核骨架(线程、调度、切换)两者本质相同,你的 FreeRTOS 知识直接迁移;RT-Thread 的独特价值在"统一对象模型"和"丰富组件"——它是从 RTOS 走向操作系统的中间形态。下一章看它的 IPC 和设备框架(设备框架是它最像 Linux 的部分),第27章看内存、FinSH、BSP。因为骨架你已懂,这一卷我们主要看"多出来的、更高级的东西"。

25.11 面试常问问题

Q1:RT-Thread 和 FreeRTOS 的主要区别?
内核骨架(线程/调度/切换)本质相同;RT-Thread 有统一的内核对象模型(rt_object 基类+容器)、更丰富的组件(设备框架/文件系统/网络/FinSH),定位更接近小型操作系统。优先级方向相反(RT-Thread 小=高),线程可 return。

Q2:RT-Thread 的内核对象是什么?为什么这么设计?
把线程/信号量/设备等抽象成继承 rt_object 基类的对象,用对象容器统一管理。好处:统一创建/删除/查找/枚举(如 list_thread),便于管理大量组件、可扩展。用 C 模拟面向对象。

Q3:C 语言怎么实现"继承"?
把基类结构体作为派生类的第一个成员,这样派生类指针可直接当基类指针用(地址相同/偏移0)。RT-Thread 的 rt_thread.parent=rt_object 就是这样;Linux 内核 kobject/container_of 同理。

Q4:RT-Thread 的调度算法?和 FreeRTOS 一样吗?
基本一样:优先级位图 + 就绪线程表,用找位指令(FFS)O(1) 选最高优先级。区别只是 FreeRTOS 数值大=高用 CLZ、RT-Thread 数值小=高用 FFS。

Q5:RT-Thread 的上下文切换和 FreeRTOS 有区别吗?
没本质区别,Cortex-M 上都用 PendSV 做切换,第18章讲的道理(为何用 PendSV、双栈、寄存器分工)完全适用。

Q6:RT-Thread Nano 和标准版的区别?
Nano 是纯内核(≈FreeRTOS),标准版带设备框架/文件系统/网络/FinSH 等组件(≈迷你 Linux)。

25.12 常见误区

  1. "RT-Thread 优先级数值越大越高"——相反,数值越小越高(和 FreeRTOS 相反)。
  2. "RT-Thread 线程也不能 return"——可以 return,返回后被系统回收。
  3. "内核对象模型只是花架子"——它带来统一管理/查找/枚举能力,是可扩展性的基础。
  4. "RT-Thread 调度和 FreeRTOS 完全不同"——骨架几乎一样,都是位图+就绪表+O(1)选择。
  5. "RT-Thread 就是国产 FreeRTOS"——内核像,但组件丰富度和对象模型是它的独特价值。

25.13 本章总结

  • RT-Thread 定位在 RTOS 与 Linux 之间:Nano≈FreeRTOS,标准版≈迷你 Linux(带丰富组件)。
  • 核心特色:内核对象模型——线程/信号量/设备等都继承统一基类 rt_object,用对象容器统一管理。
  • C 实现"继承":基类作派生类第一个成员,指针可互转(同 Linux 内核套路)。
  • 对象容器带来:按名查找、统一枚举(list_thread)、生命周期统一——可管理性/可扩展性。
  • 线程 rt_thread ≈ FreeRTOS TCB;但优先级数值小=高(相反)、线程可 return。
  • 调度器:位图 + 就绪表 + FFS 找位(O(1)),和 FreeRTOS 同套路。
  • 上下文切换:Cortex-M 上同样用 PendSV,第18章知识完全适用。
  • 内核骨架两者本质相同,FreeRTOS 知识可迁移;RT-Thread 独特价值在对象模型和组件。
