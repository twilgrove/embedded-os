不传数据的队列，专门用来"数数"和"打招呼"

本章目标：第19章说"信号量是特殊的队列"。这一章讲清它特殊在哪、怎么用来做同步(中断通知任务)和资源计数(管理一组相同资源)。信号量是 RTOS 最常用的同步工具,但也最容易误用(尤其拿它当互斥锁)。学完你要能区分二值/计数信号量、讲清 give/take 底层就是队列的 send/receive、理解为什么信号量不该用来做互斥(留给第21章的互斥量)。

20.1 信号量要解决什么

队列解决"传数据",但很多场景不需要传数据,只需要传一个"信号":

  • 同步:任务 A 要等某件事发生才能继续。比如中断收到数据,想通知处理任务"可以干活了"——不用传数据(数据在别处),只要一个"叮"的信号。
  • 资源计数:有 3 个相同的资源(如 3 个 DMA 通道),多个任务抢用。需要记录"还剩几个可用",没了就等。

这两种场景,用队列传数据是"杀鸡用牛刀"(还得定义传什么数据)。信号量就是"退化的队列":不关心传什么,只关心"有没有信号 / 还剩几个"。它本质是一个受保护的计数器 + 阻塞唤醒机制。

20.2 生活类比：停车场的车位牌

信号量就像停车场入口那块"剩余车位"电子牌:

  • 牌子上的数字 = 信号量的计数值 = 当前可用资源数。
  • 一辆车进场(take/获取):数字减 1(占用一个资源)。
  • 一辆车离场(give/释放):数字加 1(归还一个资源)。
  • 数字为 0 时(车位满),想进的车在门口等(阻塞),直到有车离场(数字变正)。

二值信号量就是只有 1 个车位的停车场(数字只能是 0 或 1),常用于"同步"——像一个"事件发生了没"的旗子。计数信号量是有 N 个车位的,用于"管理 N 个相同资源"。两者底层是同一个东西,只是最大计数不同。

20.3 二值信号量：一把只有 0/1 的钥匙

二值信号量(Binary Semaphore)的计数只有 0 和 1,表示"信号无 / 有"。最经典用途是中断与任务的同步(第七章上半部/下半部的标准实现):

典型模式: 中断给信号, 任务等信号

创建: xSemaphore = xSemaphoreCreateBinary();  // 初始为0(无信号)

中断ISR(上半部, 快):              处理任务(下半部, 慢):
  收到数据存缓冲                    for(;;) {
  xSemaphoreGiveFromISR(              xSemaphoreTake(xSemaphore, 永久等);
    xSemaphore, &woken);             // 拿到信号 → 被唤醒
  portYIELD_FROM_ISR(woken);         处理刚才中断存的数据;
                                    }

时序:
  平时: 任务 Take 时信号量=0 → 任务阻塞睡着(不占CPU)
  中断来: Give → 信号量 0→1 → 唤醒任务
  任务醒: Take 成功(信号量 1→0) → 处理数据 → 再次 Take → 又睡

这个"中断 give、任务 take"的模式,是嵌入式最常见的同步范式,完美实现第七章的上半部/下半部:ISR 只做最少的事(存数据 + give 信号),把耗时处理推给一个专门的任务(take 后处理)。好处:ISR 极短(实时性好),重活在任务上下文做(可以从容、可以调用会阻塞的 API)。相比第七章那个"ISR 置标志 + 主循环轮询标志"的土办法,信号量让任务能阻塞睡眠(不占 CPU),有事才被精确唤醒——这才是 RTOS 的正确姿势。

20.4 give / take 就是 send / receive

信号量的两个操作,直接对应队列(第19章):

信号量操作底层队列操作含义
xSemaphoreGive(释放/给信号)xQueueSend(队列发送)计数+1;若有任务在等,唤醒它
xSemaphoreTake(获取/等信号)xQueueReceive(队列接收)计数-1;若计数为0,阻塞等待

看头文件里的宏定义,一目了然:

/* semphr.h —— 信号量 API 全是队列 API 的宏! */
#define xSemaphoreGive( xSemaphore )  \
    xQueueGenericSend( ( QueueHandle_t )( xSemaphore ), NULL, 0, queueSEND_TO_BACK )

#define xSemaphoreTake( xSemaphore, xBlockTime )  \
    xQueueSemaphoreTake( ( xSemaphore ), ( xBlockTime ) )

#define xSemaphoreCreateBinary()  \
    xQueueGenericCreate( 1, semSEMAPHORE_QUEUE_ITEM_LENGTH, queueQUEUE_TYPE_BINARY_SEMAPHORE )
    /* 长度=1, itemSize=0(semSEMAPHORE_QUEUE_ITEM_LENGTH=0) */

看 xSemaphoreGive 传给 xQueueGenericSend 的数据指针是 NULL、长度参数隐含 itemSize=0——它 send 一个"空数据",不拷贝任何东西,只是让队列的 uxMessagesWaiting 从 0 变 1(表示"有信号了")。take 则让它从 1 变 0。整个信号量就是"用队列的消息计数当信号计数,但不传实际数据"。第19章说的"一切 IPC 皆队列",这里是最直接的证据。

20.5 源码：信号量如何复用队列

信号量的 take 有个专门的 xQueueSemaphoreTake(和普通 receive 略有不同,因为不用拷贝数据),但骨架完全一样:

BaseType_t xQueueSemaphoreTake( QueueHandle_t xQueue, TickType_t xTicksToWait )
{
    for( ;; )
    {
        taskENTER_CRITICAL();
        {
            /* 信号量的"消息数"就是计数值 */
            if( pxQueue->uxMessagesWaiting > 0 )
            {
                /* 有信号: 计数-1 (不拷贝数据, 这是和普通队列receive的区别) */
                pxQueue->uxMessagesWaiting--;

                /* 若是互斥量, 记录持有者(第21章); 普通信号量跳过 */
                /* 唤醒等待"发送"的任务(如果有) */
                taskEXIT_CRITICAL();
                return pdPASS;
            }
            else
            {
                if( xTicksToWait == 0 ) { taskEXIT_CRITICAL(); return errQUEUE_EMPTY; }
                /* 计数为0: 准备阻塞 */
            }
        }
        taskEXIT_CRITICAL();

        /* 阻塞在"等接收"列表上, 睡到有人give或超时 */
        vTaskPlaceOnEventList( &( pxQueue->xTasksWaitingToReceive ), xTicksToWait );
        portYIELD_WITHIN_API();
    }
}

和第19章的 xQueueReceive 对比,几乎一样,只是不做数据拷贝(prvCopyDataFromQueue 那一步省了,因为 itemSize=0)。所以信号量的阻塞、唤醒、优先级等待列表、超时——全部继承自队列,一行新逻辑都不用写。这就是"信号量是退化的队列"的源码证据:去掉数据拷贝,队列就变成了纯粹的计数信号量。

20.6 同步模式：中断通知任务

把 20.3 的模式讲得更完整,包括"多次 give"的行为(面试常问的坑):

问题: 二值信号量, 如果中断连续 give 两次(任务还没来得及 take), 会怎样?

  信号量=0
  中断give第1次: 0→1 ✓
  中断give第2次: 已经是1了, 再give... 队列满(长度1), give失败!
  → 第2次的"信号"丢失了!

结论: 二值信号量【不累计】。多次give只等于一次。
      任务take一次就把信号清了, 期间来的多个事件只当一个。

这是二值信号量的重要特性和常见坑:它不计数,只表示"有没有信号"。如果中断在任务处理不过来时连续来了 5 次,任务只知道"有信号",不知道来了 5 次——会丢事件。如果你需要"来了几次都不丢、任务要处理相应次数",就得用计数信号量(20.7):每次 give 计数+1,任务 take 一次减 1,直到处理完所有累积的事件。"该用二值还是计数信号量"取决于:你在乎"有没有事件"还是"发生了几次事件"。面试爱问"中断频繁 give 二值信号量会不会丢事件"——答:会,该用计数信号量。

20.7 计数信号量：管理一组资源

计数信号量(Counting Semaphore)计数值可以是 0 到 N,两大用途:

用途一:管理 N 个相同资源

场景: 有 3 个 DMA 通道, 多个任务抢用

创建: xSem = xSemaphoreCreateCounting(3, 3);  // 最大3, 初始3(都可用)

任务要用DMA通道:
  xSemaphoreTake(xSem, 等待);   // 计数-1, 拿到一个通道(没了就等)
  ... 使用DMA通道 ...
  xSemaphoreGive(xSem);          // 计数+1, 归还通道

效果: 最多3个任务同时用, 第4个来的会阻塞, 直到有任务归还

用途二:事件计数(不丢事件)

场景: 中断可能快速连续发生, 任务要处理每一次, 不能丢

创建: xSem = xSemaphoreCreateCounting(255, 0);  // 最大255, 初始0

中断: xSemaphoreGiveFromISR(xSem, ...);  // 每次事件计数+1(累加!)
任务: for(;;){ xSemaphoreTake(xSem, 等待); 处理一次事件; }
      // 中断来了几次, 计数就是几, 任务 take 几次, 一个不落

底层实现:计数信号量就是 xQueueGenericCreate(N, 0, ...)——长度为 N、itemSize 为 0 的队列。uxMessagesWaiting(0~N)就是计数值,uxLength(N)就是最大计数。give=send(计数+1,最多到 N),take=receive(计数-1,到 0 就阻塞)。又是队列,只是长度从 1 变成 N。

20.8 信号量做互斥的问题

一个极其重要的知识点,面试高频、也是新手常犯的错。能不能用二值信号量做互斥锁(保护共享资源)?看起来可以:

用二值信号量当"锁"(初始给1):
  xSemaphoreTake(xSem);   // "加锁"
  访问共享资源;
  xSemaphoreGive(xSem);   // "解锁"

功能上能实现互斥,但有两个致命问题:

  1. 优先级反转(Priority Inversion):低优先级任务持有"锁",高优先级任务等这个锁,结果被一个中优先级任务插队,导致高优先级任务被迫等待很久。二值信号量不解决这个问题(第21章详讲)。
  2. 没有所有权概念:信号量可以被任何任务 give(A 加的锁,B 能解),容易出错。真正的互斥锁应该"谁加锁谁解锁"。

核心结论:信号量用于"同步"(一个任务/中断通知另一个),互斥量用于"互斥"(保护共享资源)。虽然二值信号量和互斥量底层都是长度 1 的队列,但互斥量多了"优先级继承"和"所有权",专门为互斥场景设计。用信号量做互斥是能跑,但会踩优先级反转的坑——这正是第21章互斥量存在的理由。记住这个区分:"同步用信号量,互斥用互斥量",是 RTOS 编程的基本功,也是面试区分度很高的一问。为什么它俩容易混?因为都能"锁",但设计意图和适用场景不同。

20.9 任务通知：更轻量的替代

FreeRTOS 后来引入了任务通知(Task Notification),在很多场景可以替代二值/计数信号量,而且更快、更省内存。原理:每个 TCB 里内置一个 32 位的通知值和一个通知状态,不用单独创建信号量对象。

TCB 里内置(不用额外创建对象):
  volatile uint32_t ulNotifiedValue;      // 通知值
  volatile uint8_t  ucNotifyState;        // 通知状态

用作二值信号量:
  发送方: xTaskNotifyGive(任务句柄);       // 相当于 give
  接收方: ulTaskNotifyTake(pdTRUE, 等待);  // 相当于 take

优势:

  • 快:直接操作 TCB 内的变量,不走队列那套(不用等待列表插入等),据官方约 45% 更快。
  • 省内存:不用为信号量分配单独的对象。

限制:只能"点对点"(一个特定任务接收),不能像信号量那样多个任务等同一个。所以:如果是"通知某个特定任务",优先用任务通知(快省);如果是"多个任务等同一事件"或"计数资源被多任务共享",还得用信号量。这是 FreeRTOS 性能优化的一个重要工具,面试会问"知道任务通知吗,和信号量比有什么优势"。

20.10 信号量 vs 队列 vs 通知怎么选

把 IPC 工具的选型总结成决策表(实战和面试都实用):

需求推荐理由
传递数据队列带缓冲、拷贝语义
纯同步(通知特定任务)任务通知最快最省
纯同步(多任务等同一事件)二值信号量通知做不到多接收方
事件计数(不丢)计数信号量累加计数
管理 N 个相同资源计数信号量计数=可用资源数
保护共享资源(互斥)互斥量(第21章)有优先级继承,防反转
等多个事件的组合事件组(第22章)按位标志,可等"与/或"

这张表是 RTOS 编程的"工具箱索引"。核心判断逻辑:①要不要传数据?要→队列。②只是同步?→通知(点对点)或信号量(多接收/计数)。③是保护资源(互斥)?→互斥量,别用信号量。④等多个条件组合?→事件组。选对工具,代码简洁又正确;选错(比如拿信号量做互斥),就埋下优先级反转的雷。记住:它们底层大多是队列的变体,但语义和适用场景不同,别混用。

20.11 面试常问问题

Q1:信号量和队列的关系?二值信号量底层是什么?
信号量是退化的队列(不传数据)。二值信号量=长度1、itemSize0 的队列;give=send、take=receive,用消息计数(0/1)表示信号有无。

Q2:二值信号量和计数信号量的区别?各用于什么?
二值只有0/1,表示"有无信号",用于同步(如中断通知任务)、不累计事件;计数0~N,用于管理N个资源或事件计数(累加,不丢事件)。

Q3:中断频繁 give 二值信号量会怎样?
会丢事件——二值不累计,多次give等于一次,任务处理不过来时后续信号被覆盖。要处理每次事件应用计数信号量。

Q4:中断和任务如何用信号量同步?(上半部/下半部)
ISR 里 xSemaphoreGiveFromISR + portYIELD_FROM_ISR(上半部,快);任务 xSemaphoreTake 阻塞等待,拿到后处理(下半部)。ISR 短、重活在任务做。

Q5:能用信号量做互斥锁吗?为什么不推荐?
功能上能,但二值信号量不解决优先级反转、且无所有权(谁都能give)。互斥应该用互斥量(有优先级继承和所有权)。同步用信号量,互斥用互斥量。

Q6:任务通知是什么?比信号量好在哪?
TCB 内置通知值/状态,不用单独对象,更快(约快45%)更省内存。但只能点对点(通知特定任务),多任务等同一事件仍需信号量。

20.12 常见误区

  1. "二值信号量会累计信号"——不会,多次 give 等于一次,可能丢事件;要累计用计数信号量。
  2. "信号量用来保护共享资源最合适"——那是互斥量的活;信号量做互斥有优先级反转和无所有权问题。
  3. "中断里用 xSemaphoreGive"——要用 xSemaphoreGiveFromISR,并配 portYIELD_FROM_ISR。
  4. "信号量和队列是两码事"——信号量是队列的特例(itemSize=0)。
  5. "任务通知能替代所有信号量"——只适合点对点;多接收方/多任务共享资源仍需信号量。
  6. "take 到信号量就一定不阻塞"——计数为0时会按 xTicksToWait 阻塞或超时返回。

20.13 本章总结

  • 信号量=不传数据的队列,用于同步(传信号)和资源计数。
  • 二值信号量(0/1):表示"有无信号",用于中断通知任务(上半部/下半部);不累计,可能丢事件。
  • 计数信号量(0~N):管理 N 个相同资源,或事件计数(累加不丢)。
  • give=xQueueSend、take=xQueueReceive;信号量 API 全是队列 API 的宏,take 不拷贝数据(itemSize=0)。
  • 阻塞/唤醒/优先级等待列表/超时全继承自队列,不写新逻辑。
  • 别用信号量做互斥:二值信号量不解决优先级反转、无所有权;互斥用互斥量(第21章)。
  • 任务通知:TCB 内置,点对点,更快更省,可替代很多信号量场景。
  • 选型:传数据→队列;点对点同步→通知;多任务同步/计数→信号量;互斥→互斥量;多事件组合→事件组。
