第廿二章 EventGroup 事件组

用一个整数的每一位，等待多个事件的"与/或"组合

本章目标：信号量一次只能等"一个"事件。但现实里常要等"多个事件的组合"——比如"等 WiFi 连上且传感器就绪且配置加载完,才启动主逻辑"。这就是事件组(Event Group)。它用一个整数的每一位表示一个事件,支持等待"与(全部满足)/或(任一满足)"。学完你要能用事件组做多条件同步,理解它和信号量的区别。这一章相对短小。

22.1 信号量不够用的场景

信号量/队列一次只处理一个同步点。但看这个需求:一个任务要等三件事都完成才能启动:

  • WiFi 连接成功;
  • 传感器初始化完成;
  • 配置文件加载完毕。

用信号量做?你得建 3 个信号量,任务里 take 三次——但这样代码笨重,且如果想"任一完成就先做点事"(或逻辑)就更难写。更本质的问题:信号量表达不了"等多个事件的组合关系"。事件组专门解决"多事件的与/或等待"——一个对象、一次调用,等一组事件的任意逻辑组合。

22.2 生活类比：出发前的检查清单

事件组就像旅行出发前贴在门上的检查清单:

  • 清单上有几项:☐带钥匙 ☐带钱包 ☐带手机 ☐锁门。
  • 每完成一项就打个勾(设置对应的位)。
  • "全部打勾才能出发"= 等待所有位都置 1(逻辑与)。
  • "任一项完成就提醒我"= 等待任一位置 1(逻辑或)。

这张"清单"就是一个整数,每一位是一个待办项(事件)。事件组 = 一个共享的位图 + 阻塞等待"某些位满足条件"的机制。

22.3 事件组的本质：一个位图

事件组的核心是一个整数(通常 24 个可用位,因为高 8 位内核保留):

事件组 = 一个整数, 每一位代表一个事件:

  bit:  23 ...  3   2   1   0
        ─  ... ─   1   0   1
                   │   │   │
              事件2已发生 │ 事件0已发生
                    事件1未发生

  设置事件 = 把某位置1(SetBits)
  等待事件 = 阻塞直到某些位满足"与/或"条件(WaitBits)
  清除事件 = 把某位清0(ClearBits)

每个位的含义由你自己定义(用宏起名,如 #define WIFI_READY (1<<0))。多个任务/中断可以设置不同的位,一个或多个任务可以等待位的组合。这让"多方协调"变得简洁——每一方负责点亮自己那一位,等待方检查组合。

22.4 数据结构

event_groups.c 里的定义,出乎意料地简单:

typedef struct EventGroupDef_t
{
    EventBits_t      uxEventBits;        /* ★ 就是那个位图整数(事件标志) */
    List_t           xTasksWaitingForBits; /* 等待事件的任务列表 */
} EventGroup_t;

/* EventBits_t 通常是 uint32_t(configUSE_16_BIT_TICKS=0时) */
/* 24个可用位(bit0-23), 高8位内核保留用于控制 */

就两个字段:一个位图整数 + 一个等待任务列表。比队列/信号量还简单——因为它不传数据、不缓冲,只是一个共享的标志位集合 + 阻塞机制。等待任务同样用 TCB 的 xEventListItem 挂进 xTasksWaitingForBits(和队列一样的机制,第19章)。

22.5 等待：xEventGroupWaitBits 的与/或

核心 API,支持"与/或"两种等待逻辑:

EventBits_t xEventGroupWaitBits(
    EventGroupHandle_t xEventGroup,
    const EventBits_t  uxBitsToWaitFor,   /* 关心哪些位(掩码) */
    const BaseType_t   xClearOnExit,      /* 满足后是否自动清这些位 */
    const BaseType_t   xWaitForAllBits,   /* pdTRUE=与(全满足), pdFALSE=或(任一满足) */
    TickType_t         xTicksToWait );    /* 超时 */

用法示例:
  #define WIFI_READY    (1 << 0)
  #define SENSOR_READY  (1 << 1)
  #define CONFIG_READY  (1 << 2)

  /* 等三个事件【都】发生(逻辑与) */
  xEventGroupWaitBits( xEG,
      WIFI_READY | SENSOR_READY | CONFIG_READY,  /* 关心这三位 */
      pdTRUE,   /* 满足后清除这些位 */
      pdTRUE,   /* ★与逻辑: 三位全为1才唤醒 */
      portMAX_DELAY );
  /* 到这里, 说明三件事都完成了 */

判断逻辑(简化源码):

/* 与逻辑: 关心的位【全部】置1 才满足 */
if( xWaitForAllBits == pdTRUE )
    满足 = ( ( uxCurrentEventBits & uxBitsToWaitFor ) == uxBitsToWaitFor );
/* 或逻辑: 关心的位【任一】置1 就满足 */
else
    满足 = ( ( uxCurrentEventBits & uxBitsToWaitFor ) != 0 );

if( 满足 )  return 立即返回(拿到当前位);
else        阻塞, 挂进 xTasksWaitingForBits, 记下等待条件, 让出CPU;

与逻辑:关心的位全部为 1(用 (bits & mask) == mask 判断);或逻辑:关心的位任一为 1(用 (bits & mask) != 0)。这就是位运算的经典用法——用掩码 & 检查特定位。不满足就阻塞,直到别人 SetBits 使条件满足。

22.6 设置事件：xEventGroupSetBits

某个任务/中断完成了自己的事,就设置对应的位,并检查是否有等待任务的条件被满足:

EventBits_t xEventGroupSetBits( EventGroupHandle_t xEventGroup, const EventBits_t uxBitsToSet )
{
    /* ① 置位 */
    pxEventBits->uxEventBits |= uxBitsToSet;

    /* ② 遍历等待列表, 检查每个等待任务的条件是否现在满足 */
    pxListItem = 等待列表头;
    while( pxListItem != 列表尾 )
    {
        uxBitsWaitedFor = 该任务等待的位;
        xWaitForAll     = 该任务是与还是或;

        if( 该任务的条件现在满足(用上面的与/或判断) )
        {
            /* ★ 满足! 把这个任务从等待列表移回就绪列表(唤醒) */
            vTaskRemoveFromUnorderedEventList( pxListItem, ... );
        }
        pxListItem = 下一个;
    }
    return uxEventBits;
}

关键:SetBits 不是简单置个位,而是置位后主动检查所有等待任务——谁的条件被满足了就唤醒谁。因为一次 SetBits(可能同时置多个位)可能让多个等待任务(不同的与/或条件)同时满足。这就是为什么事件组的 SetBits 比信号量的 give 逻辑复杂——它要评估多个不同的等待条件。

注意一个重要限制:xEventGroupSetBitsFromISR(中断版)不直接在 ISR 里做上面的遍历唤醒,因为遍历不确定长度、可能较久,不适合在中断里做。它把"设置位"这个动作推给定时器守护任务(timer daemon task)去执行(通过给定时器命令队列发消息)。这是第七章"中断里只做最少的事、把耗时推给任务"思想的又一次体现,也是为什么用事件组的 FromISR 版需要开启软件定时器(第23章)。

22.7 清除位与"取走"语义

事件满足后,那些位要不要清 0?由 xClearOnExit 参数控制,这决定了两种语义:

  • xClearOnExit = pdTRUE(取走):任务被唤醒后,自动把它等的那些位清 0。语义:"我把这个事件消费了"。类似信号量 take——事件是一次性的,处理完就没了。
  • xClearOnExit = pdFALSE(不清):位保持为 1。语义:"这个状态持续存在",别的任务也能看到。类似"标志位/状态"。

还有单独的 xEventGroupClearBits 手动清位。"取走还是保留"取决于事件是"一次性动作"还是"持续状态"——比如"收到一个包"是一次性的(取走),"WiFi 处于连接状态"是持续的(保留)。选对语义,逻辑才正确。

22.8 任务同步：xEventGroupSync

事件组有个特色 API xEventGroupSync,实现多任务集合点(rendezvous)——多个任务互相等待,都到齐了才一起继续:

场景: 三个任务要在某个点"会合", 都到了才一起往下走

每个任务:
  xEventGroupSync( xEG,
      TASK_N_ARRIVED,        /* 我到了, 设置我的位 */
      ALL_TASKS_ARRIVED,     /* 等待所有任务的位都置1 */
      portMAX_DELAY );
  /* 到这, 说明三个任务都到齐了, 一起继续 */

xEventGroupSync = 原子地"设置自己的位 + 等待所有位" + 满足后自动清位

这在"多阶段并行处理、每阶段结束要同步"的场景很有用(比如三个任务各自算一部分,都算完才进入下一阶段)。它是"设置+等待+清除"三步的原子组合,保证多任务安全会合。普通的 SetBits+WaitBits 分开做会有竞态(可能错过别人的信号),Sync 把它们原子化了。

22.9 和信号量/队列的取舍

什么时候用事件组?对比其他 IPC:

需求用什么
等单个事件信号量/任务通知(更轻)
等多个事件的"与/或"组合事件组(专长)
传数据队列
一个事件通知多个任务(广播)事件组(一次 SetBits 可唤醒多个等待者)
需要计数事件次数计数信号量(事件组的位只有0/1,不计数)

事件组的两个独门优势:①等多事件组合(与/或)②一对多广播(一次 SetBits 唤醒所有条件满足的任务)。信号量做不到这两点(信号量 give 只唤醒一个等待者)。但事件组的位不计数(只有 0/1),要计数用计数信号量。选型逻辑:等"多个条件的逻辑组合"或"广播"→事件组;等单个/要计数→信号量;传数据→队列。另外,任务通知也能模拟事件位(用其 32 位值当标志),在点对点场景更省——FreeRTOS 官方甚至建议优先考虑任务通知。

22.10 面试常问问题

Q1:事件组解决什么信号量解决不了的问题?
等待多个事件的"与/或"组合,以及一对多广播(一次设置位唤醒多个等待任务)。信号量只能等单个事件、只唤醒一个等待者。

Q2:事件组的本质是什么?
一个整数位图(通常24可用位),每位一个事件 + 一个等待任务列表。SetBits 置位并检查唤醒满足条件的任务,WaitBits 阻塞等待位的与/或组合。

Q3:与逻辑和或逻辑怎么判断?
与:(bits & mask)==mask(关心的位全为1);或:(bits & mask)!=0(任一为1)。位运算掩码检查。

Q4:xClearOnExit 的作用?
决定条件满足后是否自动清除等待的位。清=消费一次性事件(类信号量take);不清=保留持续状态(类标志位)。

Q5:为什么 xEventGroupSetBitsFromISR 特殊?
唤醒需遍历等待列表(不定长),不适合在中断做;它把设置位的动作推给定时器守护任务执行,需开启软件定时器。

Q6:事件组能计数吗?
不能,位只有0/1。要计数事件次数用计数信号量。

22.11 常见误区

  1. "事件组的位能计数"——只有0/1,不计数;计数用计数信号量。
  2. "SetBits 只唤醒一个任务"——会唤醒所有条件被满足的等待任务(广播)。
  3. "WaitBits 默认是与逻辑"——由 xWaitForAllBits 参数决定,与/或都可以。
  4. "满足后位会自动清"——取决于 xClearOnExit;为 pdFALSE 时不清。
  5. "中断里可以随便 SetBits"——FromISR 版把工作推给定时器任务,需开软件定时器。
  6. "事件组能传数据"——不能,只传"事件发生了"的标志;传数据用队列。

22.12 本章总结

  • 事件组解决信号量的短板:等待多个事件的与/或组合 + 一对多广播。
  • 本质=一个位图整数(24可用位)+ 等待任务列表;每位一个事件,含义自定义。
  • xEventGroupWaitBits:与(位全为1)/或(任一为1),不满足则阻塞。
  • xEventGroupSetBits:置位后遍历等待列表,唤醒所有条件被满足的任务(可多个)。
  • xClearOnExit 决定"取走(消费一次性事件)"还是"保留(持续状态)"。
  • xEventGroupSync:多任务会合点(原子的设置+等待+清除)。
  • FromISR 版把工作推给定时器守护任务,需开软件定时器。
  • 选型:多条件组合/广播→事件组;单事件/计数→信号量;传数据→队列。

EventGroup 事件组
├─ 一、需求:等多个事件的组合(信号量做不到)
├─ 二、类比:出发检查清单(每项打勾)
├─ 三、本质:位图整数(24可用位), 每位一事件
├─ 四、数据结构:uxEventBits + 等待任务列表(超简单)

├─ 六、SetBits:置位+遍历唤醒所有满足的任务(广播)
