本章目标:多核 CPU(第14章)上,两个核同时改一个变量会出错——这是竞态(race condition)。第21章你学过 FreeRTOS 的 Mutex 和优先级继承,第14章你学过 MESI 和内存屏障。这一章把 Linux 内核的同步工具讲全:原子操作、自旋锁(SpinLock)、互斥锁(Mutex),以及最关键的"什么场景该用哪个"。这是多核编程的命根子,面试重灾区。

37.1 竞态：并发的根本问题

看一个经典 bug。两个 CPU 核同时执行 count++:

count++ 看似一步, 实际是三步(第三章指令):
   ① load  把count从内存读进寄存器
   ② add   寄存器+1
   ③ store 写回内存

两个核同时做(count初始=5):
  CPU0: load(5)          CPU1: load(5)     ← 都读到5!
  CPU0: add →6           CPU1: add →6
  CPU0: store(6)         CPU1: store(6)    ← 都写6!
  结果 count=6, 但应该是7! 丢了一次加法 ✗

这就是竞态:多个执行流并发访问同一共享数据,结果依赖于它们交错的时序,可能出错。根源是 count++ 不是"一步到位"的——它中间可以被打断/交错。多核让这个问题变得普遍(两个核真的能同时跑);单核也有(中断打断、任务切换,第21章 FreeRTOS 讲过)。凡是"多个执行流共享可变数据"的地方,都要考虑竞态。

37.2 生活例子：两人改同一账本

夫妻俩共用一个存款账本(共享变量)。都想存 100 元。出错的时序:妻子看到余额 500(读),丈夫也看到 500(读);妻子算 500+100=600 写上去,丈夫也算 500+100=600 写上去。存了两次 100,账本却只加了 100——丢了一次(和 count++ 一模一样)。

怎么办?加把锁:谁改账本前先拿到那把唯一的锁,改完再放锁。妻子拿锁→读 500→写 600→放锁;丈夫这时才能拿锁→读 600→写 700。锁保证"读-改-写"这一串操作不被别人插进来——这就是互斥。同步的所有工具,本质都在解决"如何让一段操作不被并发打断"。

37.3 临界区与互斥

固定两个核心术语:

  • 临界区(critical section):访问共享数据的那段代码(如 count++ 那三步)。
  • 互斥(mutual exclusion):保证同一时刻最多一个执行流在临界区里。
       lock();        ← 进临界区前上锁
       count++;       ← 临界区: 同一时刻只有一个执行流在这
       unlock();      ← 出临界区放锁

同步机制的目标就一个:保护临界区,实现互斥。但"怎么实现互斥"有很多方案,代价各不同——有的用一条特殊硬件指令(原子操作),有的忙等(自旋锁),有的睡眠(互斥锁)。选哪个取决于临界区多长、在什么上下文(能不能睡眠,第36章)。下面逐个看。

37.4 原子操作(atomic)

最轻量的同步:原子操作——把"读-改-写"合并成一个不可分割的硬件动作,别人无法插进中间。

atomic_t count;              // 原子整数类型
atomic_inc(&count);          // 原子地 +1, 一步到位, 不会被打断
atomic_add(5, &count);       // 原子地 +5
int v = atomic_read(&count); // 原子读

对比: 普通 count++ 会出竞态; atomic_inc 保证正确
      (硬件保证 load-add-store 作为一个整体完成)

原子操作适合"对一个简单变量做简单修改"(计数器、标志位)。它不用锁,靠 CPU 提供的特殊指令直接保证不可分割,开销极小(没有加锁解锁、没有睡眠/唤醒)。引用计数(第32章 file 的 f_count、第31章 page 引用计数)、统计计数器,全用原子操作。但它只能保护"单个变量的单个操作";要保护"一段涉及多个变量的复杂逻辑",原子操作不够,得用锁。那原子操作本身怎么实现的?靠硬件——下一节。

37.5 硬件基础：LL/SC 与 CAS

原子操作和锁,最终都建立在 CPU 提供的原子指令上。ARM 用 LL/SC(Load-Linked/Store-Conditional,第14章提过):

ARM 的 LDXR / STXR (独占访问):
  loop:
    LDXR  r1, [addr]   ; 独占load: 读值 + 标记"我在盯着这个地址"
    ADD   r1, r1, #1   ; 改
    STXR  r2, r1, [addr]; 条件store: 若期间没别人动过addr → 成功(r2=0)
                       ;            若被别人动过 → 失败(r2=1)
    CBNZ  r2, loop     ; 失败就重试整个过程

关键: STXR会检查"从LDXR到现在, 这个地址有没有被别的核动过"
      没动过才写成功 → 保证了原子性

LL/SC 的巧思:不是"锁住地址不让别人碰",而是"乐观地做,提交时检查有没有被人动过,被动过就重来"。LDXR 读值并做个标记,STXR 提交时若发现期间有人改过这个地址(通过第14章 Cache 一致性 MESI 监测到),就失败重试。这叫"乐观并发"。x86 则用 CAS(Compare-And-Swap,cmpxchg 指令):"如果内存里的值还等于我预期的旧值,就换成新值,否则失败"——思想相同。所有更高级的同步(原子操作、自旋锁、Mutex、无锁数据结构、甚至第38章 RCU 的一部分)最终都靠这条 LL/SC 或 CAS 硬件指令搭起来。这是第14章 MESI/原子性硬件知识在软件层的直接兑现——没有硬件的原子指令,软件层根本无法实现可靠的互斥。

37.6 SpinLock 自旋锁

要保护一段(涉及多变量的)临界区,最基础的锁是 SpinLock(自旋锁)。特点:拿不到锁就"原地打转"忙等,不睡眠。

spinlock_t lock;
spin_lock(&lock);      // 拿锁: 拿不到就在这里死循环"自旋"等待
   ... 临界区 ...        // 拿到了, 干活
spin_unlock(&lock);    // 放锁

自旋锁的"自旋":
  while (锁被别人占着)
      ;   // 空转! 不断重试, CPU在这里"打转"不干别的也不睡
  拿到锁

自旋锁的关键性质:等待时"忙等"(空转 CPU),不睡眠、不切换。好处:拿到锁没有切换开销,响应极快(锁一放马上抢到)。坏处:等待时白白烧 CPU。所以自旋锁只适合"临界区非常短"的场景——短到"忙等一下就拿到"比"切换出去睡再切回来"更划算。致命前提:持有自旋锁时绝对不能睡眠(睡了别人一直空转等你,浪费;单核还会死锁)。这就决定了它是"中断上下文(第36章)唯一能用的锁"——因为中断上下文本来就不能睡。

37.7 Mutex 睡眠锁

另一种锁是 Mutex(互斥锁)。特点:拿不到锁就睡眠(让出 CPU),等锁释放时被唤醒。这就是第21章 FreeRTOS 那个 Mutex 的 Linux 版。

struct mutex lock;
mutex_lock(&lock);     // 拿锁: 拿不到就【睡眠】, 让出CPU给别人跑
   ... 临界区 ...        // (可以是长临界区, 甚至临界区里再睡眠)
mutex_unlock(&lock);   // 放锁: 唤醒等待的进程

对比自旋锁:
  自旋锁: 拿不到 → 原地空转(不让CPU)
  Mutex:  拿不到 → 睡眠(让出CPU, 别人能用这个核)

Mutex 等待时睡眠——不烧 CPU,让出核给别的进程跑。代价是有睡眠+唤醒+上下文切换的开销(第30章)。所以 Mutex 适合"临界区较长"的场景:临界区长,与其忙等浪费,不如睡一觉让别人干活。因为 Mutex 会睡眠,所以只能在进程上下文用,不能在中断上下文/持自旋锁时用(第36章:中断上下文不能睡)。Mutex 也支持优先级继承(第21章那个机制),防优先级反转。

37.8 SpinLock vs Mutex 怎么选(核心)

这是本章最重要、面试必考的决策。两个锁的对比:

SpinLock 自旋锁Mutex 睡眠锁
拿不到锁时忙等(自旋,占 CPU)睡眠(让出 CPU)
等待开销烧 CPU,但无切换不烧 CPU,但有切换开销
适合临界区短长
持锁时能睡眠吗❌ 绝对不能✅ 可以
能用在中断上下文吗✅ 能(唯一选择)❌ 不能(会睡)
决策树:
  临界区里/需要睡眠吗? 或 在中断上下文吗?
   ├─ 是(中断上下文, 或临界区要睡眠) → 只能 SpinLock
   └─ 否(进程上下文, 不睡)
        │
        临界区长还是短?
         ├─ 很短(几条指令) → SpinLock(避免切换开销)
         └─ 较长(可能耗时) → Mutex(睡眠, 不烧CPU)

一句话决策:能睡的进程上下文里,短临界区用 SpinLock、长临界区用 Mutex;中断上下文(或任何不能睡的地方)只能用 SpinLock。核心权衡是"忙等的 CPU 浪费" vs "睡眠切换的开销":临界区短到"忙等一下 < 切换两次",用自旋锁;临界区长到"忙等浪费 > 切换开销",用 Mutex。而"能不能睡眠"(第36章上下文)是一票否决项——中断上下文不能睡,直接排除 Mutex。面试极爱问"自旋锁和互斥锁的区别、什么时候用哪个",把这张表和决策树答出来就满分。

37.9 中断与锁：更深的坑

一个真实又致命的死锁场景,面试进阶必问。假设进程和中断处理都要访问同一个共享数据,用自旋锁保护:

危险场景(单核):
  进程上下文:
     spin_lock(&lock);     // 进程拿到锁
     ... 操作共享数据 ...
     ← 此时来了个中断!
        中断handler: spin_lock(&lock);  // 想拿同一把锁
                     → 锁被进程占着, 自旋等待...
                     → 但进程被中断打断了, 没机会放锁!
                     → 【死锁】互相等, 系统卡死 ✗

解决: 进程侧用 spin_lock_irqsave(&lock, flags)
     → 拿锁的同时【关本地中断】, 临界区内不会被中断打断
     → 中断进不来, 就不会来抢锁 → 无死锁

这个坑的本质:如果一把锁会被"中断上下文"和"进程上下文"同时争用,进程侧持锁时必须关中断(spin_lock_irqsave),否则中断在进程持锁期间打断进来抢同一把锁,而进程又因为被打断没法放锁,形成死锁。规则:被中断和进程共用的自旋锁,进程侧要用 spin_lock_irqsave(关中断版)。这把第36章(中断上下文)、第七章(关中断)、本章(锁)全串起来了——多核/中断环境下的并发,必须同时考虑"谁会碰这个数据""在什么上下文""要不要关中断"。这类分析能力,正是嵌入式/内核工程师的核心功力,也是这类公司重点考察的。

37.10 读写锁与 seqlock

一个常见优化:很多数据读多写少(如配置、路由表)。普通锁不管读写都互斥,但多个"读者"其实可以同时读(读不改数据,不冲突)。为此有专门的锁:

锁特点适合
读写锁 rwlock多个读者可同时持有;写者独占(排斥所有读者和其他写者)读多写少
seqlock写者优先、不被读者挡;读者读完检查"序号"变没变,变了(说明读期间有写)就重读读极多、写要低延迟(如系统时间)

读写锁放宽了"读者之间"的互斥(读者可并行),提升读多场景的并发。seqlock 更激进:写者不等读者(用一个序号,写前后各加一),读者读完比对序号,若期间发生过写就重读——让写者永不被读者阻塞(适合写要低延迟的场景,如内核维护的系统时间 gettimeofday)。这些是针对特定访问模式的优化,思想上都是"减少不必要的互斥"。把"减少互斥"推到极致,就是第38章的 RCU——读者完全无锁。

37.11 回扣内存屏障与 FreeRTOS

本章和前面几章的连接非常密:

本章概念连接到
原子操作靠 LL/SC/CAS第14章 MESI、原子指令硬件
锁内部要内存屏障(保证顺序)第14章 DMB/DSB、acquire/release
Mutex 睡眠/唤醒第30章 调度、第21章 FreeRTOS Mutex
Mutex 优先级继承第21章 优先级反转/继承
SpinLock 只能用在不能睡的上下文第36章 中断上下文
关中断保护临界区第七章 关中断、FreeRTOS 临界区

一个重要补充:锁不只是"互斥",还隐含"内存屏障"(第14章)。spin_lock 有 acquire 语义、spin_unlock 有 release 语义——保证临界区内的内存操作不会"跑到"锁外面去(编译器/CPU 乱序,第14章)。所以锁同时解决了两件事:互斥(同一时刻一个执行流)+ 内存序(临界区内的修改对下一个拿锁者可见)。回想第21章:FreeRTOS 的 Mutex + 优先级继承,就是本章 Mutex 的"小系统版";FreeRTOS 的临界区(关中断),就是本章 spin_lock_irqsave 的单核版。你在 FreeRTOS 里建立的"共享数据要保护、临界区要短、关中断防打断"的直觉,在 Linux 多核环境下依然成立,只是工具更丰富、还要额外考虑"多核真并行"和"内存序"。

37.12 面试常问问题

Q1:什么是竞态?举例。
多执行流并发访问共享数据,结果依赖交错时序而可能出错。如两核同时 count++(load-add-store 交错),丢失一次加法。

Q2:自旋锁和互斥锁的区别?什么时候用哪个?
自旋锁拿不到锁忙等(不睡,占 CPU),适合短临界区、可用于中断上下文;互斥锁拿不到锁睡眠(让 CPU,有切换开销),适合长临界区、只能进程上下文。中断上下文/持锁要睡→只能自旋锁;进程上下文短临界区→自旋锁,长→互斥锁。

Q3:持有自旋锁时能睡眠吗?为什么?
不能。持锁睡眠会让等锁者一直空转浪费 CPU,单核还会死锁(睡了没人放锁)。所以持自旋锁期间不能调用任何可能睡眠的函数。

Q4:原子操作怎么实现的?
靠 CPU 原子指令:ARM 的 LL/SC(LDXR/STXR,乐观地做、提交时检查有没有被别人动过,被动过重试),x86 的 CAS(cmpxchg)。

Q5:一个数据被进程和中断都访问,用自旋锁要注意什么?
进程侧要用 spin_lock_irqsave(拿锁同时关本地中断),否则进程持锁时中断打断进来抢同一锁,进程没法放锁→死锁。

Q6:读多写少的数据用什么?
读写锁(读者可并行,写者独占)或 seqlock(写者不被读者挡,读者靠序号检测重读);极致是 RCU(读者无锁,第38章)。

Q7:锁除了互斥还保证什么?
内存序:lock 有 acquire、unlock 有 release 语义(内存屏障),保证临界区内的修改不越界、对下一个拿锁者可见。

37.13 常见误区

  1. "count++ 是原子的"——不是,它是 load-add-store 三步,多核会竞态,要用 atomic_inc。
  2. "自旋锁比互斥锁好/差"——没有绝对好坏,看临界区长短和上下文;各有适用。
  3. "持自旋锁可以调用 kmalloc(GFP_KERNEL)/mutex"——不能,这些可能睡眠,持自旋锁不能睡。
  4. "单核不用考虑竞态"——单核也有(中断打断、任务切换),FreeRTOS 也要临界区。
  5. "锁只管互斥"——还隐含内存屏障,保证内存操作顺序和可见性。
  6. "加了锁就万无一失"——用错(忘关中断、加锁顺序不一致)会死锁;锁多了还伤性能。

37.14 本章总结

  • 竞态:多执行流并发访问共享数据,结果依赖交错时序而出错(count++ 经典例)。多核让它普遍,单核也有。
  • 目标:保护临界区、实现互斥。工具按代价从轻到重:原子操作 → 自旋锁 → 互斥锁。
  • 原子操作:单变量单操作(计数/标志),靠硬件 LL/SC(ARM)/CAS(x86) 保证不可分割,最轻量。
  • SpinLock:拿不到锁忙等不睡,适合短临界区、中断上下文唯一可用;持锁绝不能睡。
  • Mutex:拿不到锁睡眠,适合长临界区、只能进程上下文;支持优先级继承(第21章)。
  • 选择:不能睡/中断上下文→SpinLock;进程上下文短→SpinLock、长→Mutex。核心权衡:忙等浪费 vs 切换开销;能否睡眠一票否决。
  • 坑:进程与中断共用的自旋锁,进程侧要 spin_lock_irqsave(关中断) 防死锁。
  • 优化:读多写少用读写锁/seqlock;极致读无锁是 RCU(第38章)。锁还隐含内存屏障(第14章)。

37.15 思维导图 / 课后题

并发同步:原子/SpinLock/Mutex
├─ 一、★竞态:count++(load-add-store)多核交错→丢更新
├─ 二、目标:保护临界区, 互斥
├─ 三、原子操作:单变量单操作, 最轻量
│      └─ 硬件:★LL/SC(ARM LDXR/STXR) / CAS(x86 cmpxchg)
├─ 四、★SpinLock:拿不到→忙等不睡; 短临界区; 中断上下文唯一; 持锁不能睡
├─ 五、★Mutex:拿不到→睡眠; 长临界区; 只进程上下文; 优先级继承
├─ 六、★选择:能睡? 短→spin/长→mutex; 中断上下文→只能spin

  2. 用 C11 <stdatomic.h> 的 atomic_fetch_add 重做上题,对比性能和正确性。
  3. 用 perf 或计时对比:极短临界区下自旋锁 vs 互斥锁的性能差异,理解切换开销。
  4. 思考并画出:两把锁 A、B,线程1 先 A 后 B,线程2 先 B 后 A,如何死锁(加锁顺序不一致);怎么避免(统一加锁顺序)。
  5. 读内核 kernel/locking/spinlock.c 和 mutex.c,找到自旋等待和睡眠等待的实现,对照本章。
