本章目标:你写过 socket 程序、用过 TCP/UDP、跑过 ROS2(底层是 DDS 走 UDP)。这章讲数据从你的 send() 到网线上、再从网线到对方 recv(),在内核里怎么一层层封装/解封装。核心是理解"分层"这个网络的灵魂思想,以及 socket 如何借助第32章 VFS 变成"网络的文件"。

34.1 为什么需要分层

网络要解决的问题极其复杂:两台机器隔着千山万水、经过无数路由器、链路会丢包会乱序、硬件五花八门(以太网/WiFi/光纤)……如果让一个程序把这些全处理,没法写。解法还是那个老朋友——分层抽象(第32章 VFS 也是分层)。

把网络任务切成若干层,每层只干一件事,只和相邻层打交道:应用层管"数据内容",传输层管"可靠传给对方的某个程序",网络层管"跨网络找到对方主机",链路层管"在一根线上把比特发出去"。每层向上提供简单接口,向下隐藏复杂细节。换 WiFi 还是有线?只是最底层变,上面不用改。这就是分层的威力:把一个巨复杂问题,拆成几个各自可控的小问题。

34.2 生活例子：寄快递的层层封装

你寄一封信给朋友。你只写信纸内容(应用层),塞进信封写上收件人姓名(传输层:给对方的哪个人);信封放进快递箱写上详细地址(网络层:哪个城市哪条街);快递公司装上卡车,司机只管"把这箱货从 A 站开到 B 站"(链路层:一段路的运输)。

关键:每一层只关心自己那层的"地址",并把上层的东西当成"不透明的内容"包起来。司机不看信的内容,只看到货箱;分拣员不拆信封,只看快递单。收件方则反过来层层拆封:签收拆箱→看到信封→拆信封→读信。这个"发送方层层打包、接收方层层拆包"就是网络协议栈的工作方式。

34.3 网络分层模型

经典 OSI 七层太学术,实际用 TCP/IP 四层模型(Linux 就这么实现):

层干什么协议例"地址"数据单位
应用层具体应用的数据HTTP/DNS/DDS(ROS2)-message
传输层端到端,送到对方的哪个程序TCP / UDP端口号段 segment
网络层跨网络找到对方主机、路由IPIP 地址包 packet
链路层在一段物理链路上传比特以太网/WiFiMAC 地址帧 frame
发送时数据被逐层加"头"(封装):

  应用数据                          [ data ]
  传输层加TCP头(端口)         [TCP| data ]
  网络层加IP头(IP地址)     [IP|TCP| data ]
  链路层加以太网头(MAC) [ETH|IP|TCP| data ]  → 发到网线

接收时反过来, 每层拆掉自己的头, 交给上层。

每层加一个"头(header)",里面是这层需要的控制信息(端口、IP、MAC)。就像快递的层层包装,每层贴自己的单子。接收方每层读并撕掉对应的头。记住三个关键地址的分工:端口号(找主机上的哪个程序)、IP 地址(找哪台主机)、MAC 地址(找同一链路上的哪个网卡)。

34.4 socket：网络的文件接口

应用程序怎么用网络?通过 socket(套接字)。socket 是内核给应用提供的"网络端点"抽象:

// 服务端典型流程
int fd = socket(AF_INET, SOCK_STREAM, 0);  // 创建socket, 得到一个fd!
bind(fd, &addr, ...);      // 绑定IP:端口
listen(fd, backlog);       // 开始监听
int cfd = accept(fd, ...); // 接受一个连接, 得到新fd
read(cfd, buf, n);         // 像读文件一样收数据!
write(cfd, buf, n);        // 像写文件一样发数据!
close(cfd);

注意 socket() 返回一个 fd,之后能用 read/write/close 操作它——就像文件!这正是第32章"一切皆文件"的体现:socket 在内核里也有对应的 struct file 和 f_op,只不过它的 read/write 实现是"走网络协议栈收发",而不是"读写磁盘"。所以你在第32章学的 fd→file→(这里)→socket 链条,让网络编程和文件编程用同一套接口。这也是为什么 epoll(第35章)能同时监控文件和 socket。

34.5 TCP vs UDP

传输层两个主角,面试必问、选型必知:

TCPUDP
连接面向连接(先握手)无连接(直接发)
可靠性可靠(确认+重传,不丢不乱序)不可靠(可能丢/乱序)
顺序保证顺序不保证
速度/开销较慢,开销大快,开销小
流 vs 报文字节流(可能粘包)报文边界清晰
典型用途HTTP、文件传输(要求可靠)视频/游戏/DDS(要求实时)

选型核心:要可靠还是要实时。TCP 用确认重传保证不丢,代价是延迟(丢了要等重传)和开销。UDP 简单快速,但要应用自己处理丢包。关键给机器人场景:ROS2 底层的 DDS 默认走 UDP——因为机器人/无人机的传感器数据(激光雷达、图像)要求低延迟,丢一两帧无所谓(下一帧马上来),不能为等重传而卡顿。这就是为什么实时系统偏爱 UDP。另一个 TCP 的坑:字节流会"粘包"——TCP 把数据看成连续字节流,不保留你 send 的边界,收方要自己按协议切分消息(所以应用层协议要带长度字段)。UDP 保留报文边界,不粘包。

34.6 TCP 三次握手 / 四次挥手

TCP 的"面向连接"靠握手建立、挥手断开。面试高频,画出来:

三次握手(建连接):        四次挥手(断连接):
 client      server        client      server
   │─ SYN ───→│              │─ FIN ───→│  我要关了
   │←SYN+ACK ─│              │←── ACK ──│  知道了
   │─ ACK ───→│              │          │  (server还能发剩余数据)
   建立连接 ✓                │←── FIN ──│  我也关了
                             │─── ACK ─→│
                             连接关闭 ✓

为什么三次握手?要双方都确认"我能发、我能收,对方也能发能收"。三次是让双方都确认收发能力的最小次数(两次不够:server 无法确认 client 能收)。为什么挥手要四次?因为关闭是"半关闭"的——一方说"我发完了"(FIN),但另一方可能还有数据要发,所以对方的 ACK 和它自己的 FIN 分开发(不能像握手那样合并),就成了四次。这些细节面试常问,理解"为什么"比背步骤重要。

34.7 sk_buff：数据包的载体

内核里,一个网络数据包用 struct sk_buff(简称 skb)表示。它是网络子系统的核心数据结构,贯穿所有层:

sk_buff 的巧妙设计:
  ┌──────────────────────────────────┐
  │ [head]...[各层的头]...[data...]...[tail] │  一块数据缓冲区
  └──────────────────────────────────┘
       指针: head/data/tail/end

  发送: 数据在中间, 每下沉一层就把data指针前移,
        在前面"腾出"空间填这层的头 → 不用拷贝数据!
  接收: 每上浮一层就把data指针后移, "跳过"这层的头

skb 的精髓:用移动指针代替拷贝数据。加一层头(封装)不是把整个数据往后挪、再在前面插头(那要拷贝),而是预留了前置空间,只把 data 指针前移,在腾出的位置填头。拆头也只是 data 指针后移。网络要处理海量包,任何多余的内存拷贝都是性能杀手(回想第八章:内存拷贝要过 Cache、耗带宽),所以 skb 极力避免拷贝。理解这个"零拷贝"设计,是理解内核网络性能的关键。

34.8 发送：一路封装下沉

把 write(sockfd, buf, n) 到网线的完整旅程串起来:

write(sockfd, buf, n) 发送之旅:
  ① 系统调用陷入内核 → 找到socket对应的file/sock
  ② 数据拷入内核, 装进 sk_buff
  ③ 传输层(TCP): 加TCP头(端口、序号、校验), 处理可靠性
  ④ 网络层(IP): 加IP头(源/目的IP), 查路由表决定从哪个网卡出、下一跳
  ⑤ 链路层: 加以太网头(MAC), ARP查目的MAC
  ⑥ 交给网卡驱动 → 网卡DMA把skb数据搬到硬件 → 发到线上
  ⑦ 网卡发完触发中断, 内核回收skb

整个过程数据(skb)不断"下沉",每层加自己的头(移动指针,不拷贝)。第④步查路由表决定"这个包该往哪走",是 IP 层的核心工作。第⑥步网卡用 DMA(第六卷)直接搬数据,不劳 CPU 逐字节拷。发送基本是"进程主动调用 → 一路同步下沉到网卡"。接收则不同,因为数据是"突然到来"的,要靠中断——见下节。

34.9 接收：软中断与协议栈上浮

接收是网络里最能体现前面知识串联的部分。数据什么时候来不确定,所以靠中断(第七章)通知:

数据包到达的接收之旅:
  ① 网卡收到包, DMA存入内存, 触发【硬中断】(第七章)
  ② 硬中断处理极短: 只标记"有包要处理", 触发【软中断NET_RX】
        —— 硬中断里不做耗时处理! (第七章原则: ISR要短)
  ③ 软中断(SoftIRQ, 第36章)里慢慢处理:
        取出skb → 链路层拆以太头 → IP层拆IP头 → TCP层拆TCP头
  ④ 数据放进socket的接收缓冲区
  ⑤ 唤醒阻塞在recv的进程 (或通知epoll, 第35章)
  ⑥ 进程 read/recv 把数据从内核缓冲拷到用户buf

这里三章知识漂亮地汇合了:①网卡用中断通知(第七章);②但硬中断只做最少的事(标记+触发软中断),因为"ISR 要短"(第七章原则)——耗时的协议栈处理推迟到软中断(第36章将详解)去做;⑤处理完唤醒睡眠的进程(第30章调度),让它从内核返回用户态(第33章递送时机同一关口)拿数据。"硬中断快速响应 + 软中断延后处理"这个"上半部/下半部"拆分,是第七章"ISR 只置标志、主体后处理"思想在 Linux 网络里的正统实现——你在 STM32 里写"中断只置 flag,主循环处理"时,已经在用这个思想了,Linux 只是把它做成了系统化的软中断机制。

34.10 socket 与 VFS 的统一

回到第32章的主题,看 socket 如何融入"一切皆文件":

socket 在内核里的样子:
  fd ──→ struct file ──→ struct socket ──→ struct sock
  (整数)  (VFS的file,      (socket层)        (协议相关状态:
           f_op指向            │              TCP连接状态、
           socket的读写)       │              收发缓冲、序号...)
                          实际收发走协议栈

  read(sockfd) → file->f_op->read → socket的read实现 → 从sock接收缓冲取数据

socket 有 file、有 f_op,所以能 read/write/close/poll——完全是个"文件"。只是它的 f_op 实现不是读磁盘,而是操作网络协议栈。这就是第32章"一切皆文件"最有说服力的例子:连网络这么复杂的东西,也套进了统一的文件接口。好处是巨大的:同一套 epoll(第35章)能同时等文件、管道、socket;同一个 close 关闭一切;shell 能重定向 socket。统一抽象让整个系统的部件可以自由组合。

34.11 回扣中断 / VFS / 嵌入式

网络这章几乎串起了全书,梳理这些连接:

网络里的机制用到的前面知识
接收靠网卡中断第七章 中断、第13章 GIC
硬中断短、软中断处理协议栈第七章"ISR 要短"、第36章 SoftIRQ
socket 是文件第32章 VFS、fd→file
收到数据唤醒进程第30章 调度、进程唤醒
skb 避免拷贝第八章 Cache/内存带宽、DMA
数据缓冲、skb 分配第31章 Slab(skb 用专用缓存)
分层抽象第32章 VFS 同样是分层

网络子系统是 Linux 内核里综合性最强的部分,它把中断、调度、内存、文件抽象全用上了。对机器人工程师尤其重要:ROS2 的通信(DDS over UDP)、无人机图传、车载以太网都建立在这套协议栈上。理解"数据如何经中断进入、经软中断上浮、唤醒进程"这条链,你才能诊断实际的网络延迟、丢包、性能问题(比如"为什么高负载下网络延迟抖动"——可能是软中断处理不及时,第36章会讲)。这一章也验证了本书的整体设计:所有前面的砖,最终砌成了网络这栋楼。

34.12 面试常问问题

Q1:为什么网络要分层?TCP/IP 四层?
分层把复杂问题拆成各层独立小问题,每层只管一件事、只和相邻层交互,可替换。四层:应用/传输(TCP·UDP,端口)/网络(IP,IP 地址)/链路(以太网,MAC)。发送逐层加头,接收逐层拆头。

Q2:TCP 和 UDP 的区别?机器人/ROS2 为什么用 UDP?
TCP 面向连接、可靠(确认重传)、有序但慢;UDP 无连接、不可靠但快、保留报文边界。ROS2/DDS 用 UDP 因为传感器数据要低延迟,丢帧可容忍,不能为重传卡顿。

Q3:TCP 三次握手、四次挥手为什么是这个次数?
三次握手让双方都确认彼此收发能力(两次不够);四次挥手因为关闭是半关闭,一方 FIN 后对方可能还有数据要发,ACK 和 FIN 分开发,故四次。

Q4:一个包接收的完整流程?
网卡 DMA 收包→硬中断(只标记,触发软中断)→软中断 NET_RX 里逐层拆头→放 socket 接收缓冲→唤醒 recv 进程/通知 epoll→用户 read 拷走。

Q5:为什么接收用"硬中断+软中断"两段?
硬中断要短(第七章),但协议栈处理耗时,放硬中断会长时间关中断丢包。所以硬中断只标记,把耗时处理推给软中断(下半部)。

Q6:socket 和文件什么关系?
socket 在内核有 file/f_op,能 read/write/close/poll,是"文件"(一切皆文件)。故 epoll 能同时管文件和 socket。

Q7:sk_buff 为什么高效?TCP 粘包怎么回事?
skb 用移动指针加/拆头,避免拷贝数据(零拷贝思想)。TCP 是字节流不保留 send 边界,收方要按应用协议(带长度)切分,即"粘包";UDP 保留报文边界不粘包。

34.13 常见误区

  1. "UDP 一定丢包不能用"——UDP 不保证可靠,但低延迟场景(视频/DDS)正合适,可在应用层做必要的可靠性。
  2. "TCP 发一次 send 对方 recv 一次收全"——TCP 是字节流会粘包/拆包,要按协议切分。
  3. "三次握手是随便定的"——是双方确认收发能力的最小次数,有明确道理。
  4. "网卡收到包 CPU 立刻全程处理"——硬中断只标记,协议栈处理在软中断,避免长时间关中断。
  5. "socket 不是文件"——socket 有 file/f_op,能 read/write/epoll,是文件抽象的一员。
  6. "每层加头都要拷贝数据"——skb 用移动指针避免拷贝。

34.14 本章总结

  • 网络靠分层驯服复杂:应用 / 传输(TCP·UDP,端口) / 网络(IP,IP 地址) / 链路(以太网,MAC)。发送逐层加头封装,接收逐层拆头。
  • socket 返回 fd,能 read/write——是"网络的文件"(第32章一切皆文件),内核里 fd→file→socket→sock。
  • TCP:面向连接、可靠(确认重传)、有序、字节流(粘包)、慢;UDP:无连接、不可靠、快、保留报文边界。ROS2/机器人用 UDP 求低延迟。
  • TCP 三次握手(确认双方收发能力)、四次挥手(半关闭,FIN 与 ACK 分开)。
  • sk_buff:包的载体,用移动指针加/拆头避免拷贝(零拷贝)。
  • 发送:进程调用→装 skb→逐层加头→查路由→网卡 DMA 发出。
  • 接收:网卡硬中断(只标记)→软中断 NET_RX(逐层拆头)→放缓冲→唤醒进程——第七章"上半部/下半部"的正统实现。
  • 网络综合了中断/调度/内存/VFS,是全书知识的汇合点,也是 ROS2/图传/车载网的基础。

  3. 用 tcpdump 或 Wireshark 抓一次 TCP 连接,亲眼看三次握手(SYN/SYN-ACK/ACK)和四次挥手。
  4. cat /proc/softirqs 看 NET_RX/NET_TX 软中断计数;大流量时再看,理解网络走软中断。
  5. 读内核 net/ipv4/tcp.c 的 tcp_sendmsg 和 net/core/dev.c 的 netif_rx/napi,对照发送/接收路径。
