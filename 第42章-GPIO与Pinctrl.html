本章目标:GPIO(通用输入输出)是最简单的外设——控制一根引脚的高低电平(点灯、读按键)。但它引出两个重要概念:引脚复用(一根物理引脚能当 GPIO、也能当 UART/I2C)和管理它的 Pinctrl 子系统。这一章你会第一次把第41章的设备树、寄存器操作真正用起来,写出点灯驱动。这是所有外设驱动的"Hello World"。

42.1 GPIO：最基础的引脚控制

GPIO = General Purpose Input/Output(通用输入输出)。就是芯片上一根可以被软件控制的引脚,能干两件最基本的事:

  • 输出:软件让这根引脚输出高电平(3.3V)或低电平(0V)——比如点亮/熄灭一个 LED、控制一个继电器。
  • 输入:软件读这根引脚当前是高还是低——比如读一个按键按没按、读一个传感器的数字信号。

GPIO 是数字世界和物理世界的最简单接口:一根线,要么高要么低,软件读或写。你在 STM32 上点过灯(HAL_GPIO_WritePin),就是操作 GPIO。它是最基础的外设,理解它就理解了"软件如何通过寄存器控制物理引脚"这个驱动的本质动作。无人机上的状态灯、蜂鸣器、简单开关量传感器,都靠 GPIO。

42.2 生活例子：开关和门铃

GPIO 输出像墙上的电灯开关:你(软件)拨到"开"(写 1),灯亮(引脚高电平);拨到"关"(写 0),灯灭。你控制的是"给不给电"。

GPIO 输入像看门铃按钮的状态:你(软件)去看"现在有没有人按着门铃"(读引脚),按着=高电平(读到 1),没按=低电平(读到 0)。你观察的是"外面给没给电"。同一根引脚,配成输出你就是"发号施令"的开关,配成输入你就是"察言观色"的观察者——方向由软件配置决定。这个"方向"就是 GPIO 第一个要配的东西。

42.3 寄存器视角：GPIO 怎么工作

GPIO 在硬件上就是几个寄存器,软件读写这些寄存器来控制引脚(和 STM32 一样):

一个GPIO控制器的核心寄存器(简化):
  ┌────────────┬──────────────────────────────┐
  │ DIR  方向寄存器 │ 每个bit配一根引脚: 0=输入, 1=输出 │
  │ DATA 数据寄存器 │ 输出时写: 1=高 0=低; 输入时读: 引脚电平 │
  │ (还有: 上下拉、驱动强度、中断使能等)              │
  └────────────┴──────────────────────────────┘

  点灯(GPIO5 输出高):
    ① DIR  的第5位 置1  (设为输出)
    ② DATA 的第5位 置1  (输出高 → 灯亮)

  读按键(GPIO3 输入):
    ① DIR 的第3位 置0   (设为输入)
    ② 读 DATA 的第3位   (1=按下 或 松开, 看接法)

本质:GPIO 控制器是一块寄存器,每个引脚对应寄存器里的一个 bit。配方向(DIR)、读写电平(DATA),就是对这些 bit 做位操作(第四章寄存器、位运算)。这和你在 STM32 裸机里 GPIOA->ODR |= (1<<5) 点灯是完全一样的操作——只不过在 Linux 里,这些寄存器地址来自设备树(第41章),而且内核提供了封装好的 GPIO 子系统 API(42.7),你通常不用直接碰寄存器。但理解底层是这些 bit,才不会把 GPIO 当黑魔法。

42.4 一根引脚身兼数职：复用

关键概念来了。芯片的引脚数量有限(几百根),但功能需求很多(UART、I2C、SPI、PWM、GPIO……)。怎么办?让一根物理引脚能"身兼数职"——通过配置,同一根引脚可以当 GPIO,也可以当 UART 的 TX,还可以当 I2C 的 SCL。这叫引脚复用(pin multiplexing,pinmux)。

一根物理引脚 PIN_37, 通过"复用选择"决定它是什么:

   复用值 = 0 → 这根引脚是 GPIO
   复用值 = 1 → 这根引脚是 UART2_TX
   复用值 = 2 → 这根引脚是 I2C1_SCL
   复用值 = 3 → 这根引脚是 PWM3
   (芯片内部有个多路选择器MUX, 按复用值把引脚接到不同的内部功能模块)

   同一时刻只能选一个功能! 选了UART_TX就不能同时当GPIO

复用是 SoC 的必然设计:引脚是稀缺资源,靠复用让有限的引脚支持丰富的功能组合。芯片内部有个"多路选择器",复用配置值决定这根引脚接到哪个内部功能模块。关键约束:同一根引脚同一时刻只能是一种功能——你把某引脚配成 UART_TX,它就不能同时当 GPIO 或 I2C。这就带来一个管理问题:谁来分配和记录"每根引脚现在是什么功能",防止冲突(两个驱动都想用同一根引脚)?答案是 Pinctrl 子系统(下一节)。你在 STM32 里配 GPIO_AF(alternate function)就是选复用,Linux 把它系统化成了 Pinctrl。

42.5 Pinctrl：引脚复用管理器

Pinctrl(pin control)子系统是内核里专门管理"引脚复用和电气属性"的框架:

Pinctrl 管什么说明
引脚复用(mux)每根引脚当什么功能(GPIO/UART/I2C…)
电气配置上拉/下拉电阻、驱动强度、开漏/推挽
冲突管理记录引脚占用,防止两个设备抢同一引脚

Pinctrl 的核心价值:统一管理引脚这个稀缺共享资源,自动配置和防冲突。驱动不用自己去写复用寄存器,而是在设备树里声明"我这个 UART 要用这组引脚、配成 UART 功能",Pinctrl 在设备 probe 时(第40章)自动把这些引脚配置好(写复用寄存器、设上下拉),并记录占用。这样各设备驱动只管声明需求,Pinctrl 统一落实,避免了引脚冲突和重复代码。又是一个"子系统统一管理某类资源"的经典设计(和第31章 Slab 管内存、GPIO 子系统管引脚同理)。

42.6 设备树里配 Pinctrl 和 GPIO

实际工程中,引脚复用和 GPIO 都在设备树里配置。看典型写法:

// ① Pinctrl: 定义一组引脚的复用(某UART用的引脚)
&pinctrl {
    uart2_pins: uart2-pins {
        rockchip,pins =
            <1 RK_PB0 1 &pcfg_pull_none>,  // 引脚配成功能1(UART_TX)
            <1 RK_PB1 1 &pcfg_pull_up>;    // 引脚配成功能1(UART_RX), 上拉
    };
};

// ② 设备引用这组引脚配置
&uart2 {
    status = "okay";
    pinctrl-names = "default";
    pinctrl-0 = <&uart2_pins>;   // ★probe时Pinctrl自动应用这组配置
};

// ③ 一个用GPIO的设备(如LED)
myled {
    compatible = "myvendor,myled";
    gpios = <&gpio3 5 GPIO_ACTIVE_HIGH>;  // 用gpio3的第5号引脚, 高电平有效
};

模式很清晰:①在 pinctrl 节点里定义"引脚组"(哪些引脚、配成什么功能、上下拉);②设备节点用 pinctrl-0 引用它——设备 probe 时 Pinctrl 自动应用(把这些引脚配成 UART 功能);③用 GPIO 的设备用 gpios 属性声明"我要用哪个 GPIO 控制器的第几号引脚"。这就是第41章设备树 + 本章 Pinctrl/GPIO 的结合:硬件的引脚分配全在设备树描述,驱动只管用。

42.7 内核 GPIO 子系统 API

驱动里操作 GPIO,用内核 GPIO 子系统提供的 API(现代是 gpiod 描述符接口),不直接碰寄存器:

#include <linux/gpio/consumer.h>

// 在probe里, 从设备树的 gpios 属性获取GPIO描述符
struct gpio_desc *led;
led = devm_gpiod_get(&pdev->dev, "led", GPIOD_OUT_LOW);
//    从设备树"led-gpios"取, 配成输出、初始低电平

// 控制GPIO:
gpiod_set_value(led, 1);       // 输出高(点灯)
gpiod_set_value(led, 0);       // 输出低(灭灯)
int v = gpiod_get_value(btn);  // 读输入(读按键)
gpiod_direction_input(btn);    // 设为输入
gpiod_direction_output(led,0); // 设为输出

GPIO 子系统把"操作寄存器"封装成统一的 gpiod_* 函数:devm_gpiod_get 从设备树拿到 GPIO(自动解析 gpios 属性、找到对应 GPIO 控制器);gpiod_set_value/gpiod_get_value 读写电平;gpiod_direction_* 配方向。好处:驱动代码和具体 SoC 的 GPIO 寄存器无关——RK3588 和别的芯片 GPIO 寄存器布局不同,但 gpiod_set_value 接口一样,底层由各 SoC 的 GPIO 控制器驱动实现。这就是子系统的威力:上层驱动写一次,适配所有 SoC。(devm_ 前缀=设备移除时自动释放,第41章。)

42.8 GPIO 中断

GPIO 不只能被动读,还能配成中断源:引脚电平变化时触发中断(比如按键按下、传感器信号跳变)——这样不用轮询,事件来了自动通知(第七章中断思想)。

GPIO中断: 把一个输入GPIO配成"电平变化时触发中断"

  int irq = gpiod_to_irq(btn);   // 把GPIO转成中断号
  request_irq(irq, btn_isr,       // 申请中断, 注册处理函数(第43章)
              IRQF_TRIGGER_FALLING, // 下降沿触发(按下瞬间)
              "mybtn", dev);

  irqreturn_t btn_isr(int irq, void *dev){  // 按键按下时自动调用
      // 处理按键... (上半部要短! 第36章)
      return IRQ_HANDLED;
  }

GPIO 中断让"读按键/传感器"从"不停轮询"变成"事件驱动":配成下降沿触发,按键一按下就自动调用你的 ISR,CPU 平时不用管。gpiod_to_irq 把 GPIO 转成中断号,再用 request_irq 注册(第43章详讲中断处理)。这把第七章中断、第36章上半部/下半部的知识用到了 GPIO 上——ISR 里要短,复杂处理推给下半部。无人机很多按键、限位开关、外部传感器信号都用 GPIO 中断,避免轮询浪费 CPU。

42.9 用户空间用 GPIO（sysfs / libgpiod）

有时不想写内核驱动,想在用户空间(应用/脚本)直接控制 GPIO(快速原型、上层逻辑)。有两种方式:

方式说明状态
sysfs(旧)/sys/class/gpio/ 下 echo 导出、写方向和值已废弃(deprecated)
libgpiod(新)通过 /dev/gpiochipN 字符设备,用库/命令行(gpioset/gpioget)操作现在推荐
# 旧 sysfs 方式(了解, 已废弃):
echo 133 > /sys/class/gpio/export
echo out > /sys/class/gpio/gpio133/direction
echo 1   > /sys/class/gpio/gpio133/value    # 点灯

# 新 libgpiod 方式(推荐):
gpioset gpiochip0 5=1     # 把gpiochip0的5号引脚设为1
gpioget gpiochip0 3       # 读3号引脚
gpiodetect / gpioinfo     # 查看GPIO控制器和引脚

用户空间访问 GPIO 适合快速验证和上层控制逻辑(不用编译内核模块)。老的 /sys/class/gpio sysfs 接口已废弃(全局编号混乱、不安全),现在用 libgpiod(基于 /dev/gpiochip 字符设备——又是"一切皆文件"第32章)。ROS2/Python 快速控制无人机上的 GPIO 常走这条路。但要接中断、要精确时序或高性能,还是得写内核驱动。

42.10 一个 LED 驱动实例

把本章串起来,一个完整的 platform LED 驱动骨架(第40章 platform + 本章 GPIO):

struct led_data { struct gpio_desc *gpio; };

static int led_probe(struct platform_device *pdev){
    struct led_data *d;
    d = devm_kzalloc(&pdev->dev, sizeof(*d), GFP_KERNEL);
    d->gpio = devm_gpiod_get(&pdev->dev, "led", GPIOD_OUT_LOW); // 取GPIO
    platform_set_drvdata(pdev, d);
    gpiod_set_value(d->gpio, 1);   // 点灯
    dev_info(&pdev->dev, "LED on\n");
    return 0;
}
static int led_remove(struct platform_device *pdev){
    struct led_data *d = platform_get_drvdata(pdev);
    gpiod_set_value(d->gpio, 0);   // 灭灯
    return 0;
}
static const struct of_device_id led_of[] = {
    { .compatible = "myvendor,myled" }, { }
};
static struct platform_driver led_drv = {
    .probe = led_probe, .remove = led_remove,
    .driver = { .name="myled", .of_match_table=led_of },
};
module_platform_driver(led_drv);
MODULE_LICENSE("GPL");

这个驱动完整体现了第40-42章的主干:设备树里 myled 节点(compatible + led-gpios,第41章)→ platform 总线匹配 compatible="myvendor,myled"(第40章)→ 调 led_probe → probe 里 devm_gpiod_get 从设备树拿 GPIO、gpiod_set_value 点灯。短短几十行,但它是所有外设驱动的骨架模板——UART/I2C/SPI 驱动只是把"点灯"换成"操作对应硬件",结构完全一样。这就是你的驱动"Hello World"。

42.11 回扣寄存器 / STM32 / 子系统思想

本章的知识连接:

本章概念连接到
GPIO 就是操作寄存器 bit第四章寄存器、STM32 GPIO->ODR
引脚复用(pinmux)STM32 的 alternate function(AF)
GPIO 中断第七章中断、第36章上/下半部、STM32 EXTI
GPIO/Pinctrl 子系统"子系统统一管理资源"(第31章 Slab、VFS 第32章)
寄存器地址来自设备树第41章设备树

GPIO 是你从 STM32 迈向 Linux 驱动最好的桥梁:同样是点灯,STM32 里你直接 GPIOA->ODR |= (1<<5)(操作寄存器),Linux 里你 gpiod_set_value(led, 1)——底层完全一样(都是置那个 bit),区别是 Linux 多了几层抽象:设备树描述引脚(而非硬编码地址)、Pinctrl 管复用(而非手动配 AF)、GPIO 子系统封装寄存器操作(而非直接写)。这些抽象让你的驱动代码能跨 SoC 复用、能和别的驱动和平共处(不抢引脚)、能被上层统一管理。你会发现 Linux 驱动的"复杂"都不是无意义的——每一层抽象都在解决"多设备、多硬件、多用户"环境下的真实问题。掌握了 GPIO 这个最简单外设的完整 Linux 处理方式,后面复杂外设不过是"同样的框架 + 更复杂的硬件操作"。

42.12 面试常问问题

Q1:GPIO 是什么?能做什么?
通用输入输出引脚。输出:软件控制引脚高低电平(点灯/控制继电器);输入:软件读引脚电平(读按键/数字传感器)。是数字世界和物理世界最简单的接口。

Q2:什么是引脚复用?为什么需要?
一根物理引脚通过配置可当不同功能(GPIO/UART/I2C/PWM)。因为引脚数量有限、功能需求多,复用让有限引脚支持丰富功能。同一引脚同一时刻只能一种功能。

Q3:Pinctrl 子系统的作用?
统一管理引脚复用和电气属性(上下拉、驱动强度),记录占用防冲突。驱动在设备树声明引脚需求,Pinctrl 在 probe 时自动配置。

Q4:驱动里怎么操作 GPIO?
用 GPIO 子系统 gpiod_* API:devm_gpiod_get 从设备树取 GPIO,gpiod_set_value/get_value 读写,gpiod_direction_* 配方向。不直接碰寄存器,跨 SoC 通用。

Q5:怎么用 GPIO 做中断?
gpiod_to_irq 把 GPIO 转成中断号,request_irq 注册 ISR 并指定触发方式(上升/下降沿)。让读按键/传感器从轮询变事件驱动,ISR 要短(上半部)。

Q6:用户空间怎么控制 GPIO?
旧的 /sys/class/gpio(已废弃),新的 libgpiod(基于 /dev/gpiochip 字符设备,gpioset/gpioget)。适合快速原型和上层逻辑;高性能/中断还得写内核驱动。

Q7:Linux 点灯和 STM32 点灯的区别?
底层都是置寄存器 bit。Linux 多了抽象:寄存器地址来自设备树(非硬编码)、Pinctrl 管复用(非手动 AF)、gpiod API 封装(非直接写寄存器),换来跨 SoC 复用和防冲突。

42.13 常见误区

  1. "一根引脚能同时当 GPIO 和 UART"——复用下同一时刻只能一种功能。
  2. "驱动里要直接写 GPIO 寄存器"——用 gpiod_* 子系统 API,跨 SoC 通用。
  3. "GPIO 地址写死在驱动里"——来自设备树的 gpios 属性和 GPIO 控制器节点。
  4. "读按键只能轮询"——可配 GPIO 中断,事件驱动更高效。
  5. "用 /sys/class/gpio 是标准做法"——已废弃,现在用 libgpiod。
  6. "Pinctrl 可有可无"——不配好引脚复用,外设(UART/I2C)根本不通,是硬件调试第一步。

42.14 本章总结

  • GPIO:可软件控制的引脚,输出(写高低,点灯)/输入(读电平,读按键)。底层是操作 GPIO 控制器寄存器的 bit(同 STM32)。
  • 引脚复用(pinmux):一根物理引脚可配成不同功能(GPIO/UART/I2C…),同一时刻只能一种;因引脚稀缺而生。
  • Pinctrl 子系统:统一管引脚复用+电气属性+防冲突;驱动在设备树声明,probe 时自动应用。
  • 设备树里配:pinctrl 节点定义引脚组,设备用 pinctrl-0 引用;GPIO 设备用 gpios 属性。
  • 驱动 API:gpiod_*(devm_gpiod_get 取、gpiod_set/get_value 读写、gpiod_direction_* 配向),封装寄存器、跨 SoC 通用。
  • GPIO 中断:gpiod_to_irq+request_irq,把引脚变化变成中断(事件驱动,ISR 要短)。
  • 用户空间:旧 sysfs(废弃)/ 新 libgpiod(/dev/gpiochip,gpioset/gpioget)。
  • 回扣:GPIO 是 STM32→Linux 驱动的桥梁;LED 驱动是所有外设驱动的骨架模板(第40-42章主干的最小实例)。
