本章目标：第20章说"互斥用互斥量,别用信号量",这一章讲清为什么。核心是优先级反转(Priority Inversion)——一个曾让火星探测器险些失败的经典问题,以及互斥量的优先级继承解决方案。这是 RTOS 最重要、面试最爱考的知识点之一,也是第七卷"实时系统"的核心前哨。学完你要能画出优先级反转的时序、讲清优先级继承怎么破解、读懂互斥量 give/take 的源码。

21.1 互斥量 vs 信号量：一字之差的鸿沟

互斥量(Mutex,Mutual Exclusion 互斥)和二值信号量,底层都是长度 1 的队列,长得几乎一样。但设计意图截然不同:

二值信号量(Binary Semaphore)互斥量(Mutex)
用途同步(通知)互斥(保护共享资源)
所有权无:谁都能 give有:谁 take 谁 give(谁加锁谁解锁)
优先级继承无★有(防优先级反转)
典型场景中断通知任务多任务访问同一外设/全局数据
初始状态0(无信号)1(可用/未锁)

一句话记忆:信号量是"我给你一个信号"(跨主体的通知,方向性),互斥量是"我借用一下这个资源,用完还"(同一主体借还,有所有权)。互斥量多的那两样——所有权和优先级继承——正是为了解决信号量做互斥时的两大问题(第20.8),而优先级继承是重中之重。

21.2 互斥要保护的：临界区

先明确互斥量保护什么。当多个任务要访问同一个不可同时使用的资源时,就需要互斥:

  • 一个 I2C 总线/SPI 外设:两个任务同时发数据会互相干扰,乱套。
  • 一个全局链表/数据结构:A 改到一半被 B 抢占也来改,结构损坏(第七、十九章讲过)。
  • 一个 printf/串口:两个任务同时打印,输出交错成乱码。

这段"同一时刻只能一个任务进入"的代码,就是临界区(Critical Section)。互斥量的用法:

xMutex = xSemaphoreCreateMutex();   // 创建互斥量(初始"未锁")

任务访问共享资源:
  xSemaphoreTake(xMutex, 等待);   // 加锁(拿到才能进, 否则等)
  访问共享资源;                    // ← 临界区, 保证独占
  xSemaphoreGive(xMutex);          // 解锁(用完归还)

效果:任何时刻最多一个任务持有互斥量、进入临界区,别的任务想进就阻塞等待,直到持有者归还。这保证了共享资源的独占访问。看起来和第20.8 用信号量做锁一样——问题就藏在"如果持有锁的是低优先级任务"这个情况里。

21.3 火星探路者的故事

1997 年,NASA 的火星探路者(Mars Pathfinder)成功登陆火星,但很快出现诡异故障:系统频繁自动重启,险些让整个任务失败。工程师远程调试后发现,罪魁祸首正是优先级反转。

探测器上跑着 VxWorks 实时系统,有三个相关任务:

  • 高优先级:数据总线管理任务(要频繁访问共享总线)。
  • 中优先级:通信任务(耗时长,不用总线)。
  • 低优先级:气象数据采集任务(也要访问共享总线)。

高优先级任务和低优先级任务共享一个用互斥量保护的总线。故障场景:低优先级任务拿着总线锁时,被中优先级的通信任务抢占(合理,通信优先级更高),而通信任务一跑就是很久;此时高优先级任务想用总线,却拿不到锁(锁在低优先级手里),只能干等——等到一个"看门狗"超时,以为系统卡死了,就重启!

这个真实事故是优先级反转致命性的最佳教材:一个价值几亿美元的火星任务,差点毁于一个调度问题。NASA 的解决方案就是打开 VxWorks 互斥量的优先级继承——远程改了个配置,故障消失。它说明:实时系统里,优先级反转不是"性能问题",而是"正确性/安全问题",可能导致系统彻底失败。的飞控、汽车的控制器同样是硬实时系统,这就是为什么这个知识点在这类公司面试中分量极重。

21.4 优先级反转：问题的精确定义

把火星探路者的问题抽象成精确的时序。三个任务:高 H、中 M、低 L(优先级 H>M>L)。H 和 L 共享一个互斥量:

时间 →

L: ──[拿到锁]──▓▓▓──(被H抢占)      (M抢占,L挂起)              ──▓[释放锁]──
                    │                                          ▲
H:                  └[想拿锁,拿不到!]═══════阻塞等待═══════════┘[终于拿到]
                                        ▲
M:                                      └──────[M运行很久]──────┘

图解:
① L 拿到锁, 进入临界区
② H 就绪, 抢占 L(H优先级高), H 想拿锁 → 拿不到(在L手里) → H阻塞
③ 此时 M 就绪, 抢占 L(M > L) → M 开始跑
④ M 跑很久, 期间 L 无法运行(被M压着) → L 无法释放锁 → H 一直等
⑤ 直到 M 跑完, L 才能继续 → L 释放锁 → H 终于拿到锁运行

结果: 最高优先级的 H, 实际上被【中优先级的 M】拖了很久
      → "优先级被反转了": M 变相地优先于 H 运行了!

精确定义:优先级反转 = 一个高优先级任务,因为等待被低优先级任务持有的锁,而间接地被与该锁无关的中优先级任务阻塞,导致高优先级任务的执行被不合理地推迟。

关键在于:H 等 L 释放锁是合理的(短暂),但 M 插进来把 L 压住、让 L 迟迟不能释放锁,才是问题。M 和这个锁毫无关系,却间接地让 H 等了很久——高优先级的 H 竟然要等中优先级的 M,优先级"反转"了。最可怕的是:如果有多个中优先级任务轮番插队,H 可能被无限期推迟(unbounded priority inversion),这在硬实时系统里就是灾难(火星探路者的看门狗超时重启)。

21.5 优先级继承：解决之道

FreeRTOS 的互斥量用优先级继承(Priority Inheritance)解决优先级反转。核心思想极其精妙:

当高优先级任务 H 因为等锁而阻塞时,让持有锁的低优先级任务 L 临时继承 H 的高优先级。这样 L 就不会被中优先级 M 抢占了(L 现在"暂时是高优先级"),L 能尽快跑完临界区、释放锁,H 就能尽快拿到锁。L 释放锁后,优先级恢复原样。

用继承后重看时序:

时间 →

L: ──[拿到锁]──▓▓(H来了,L继承H的优先级!)▓▓[释放锁,优先级恢复]──
                    │                        │
H:                  └[想拿锁,阻塞]═══════════┘[立即拿到锁]──运行
                                             ▲
M:                          [M就绪但拿不到CPU!因为L现在是高优先级]

图解:
① L 拿到锁
② H 就绪抢占, H 想拿锁 → 拿不到 → H 阻塞
   ★ 此时 L 继承 H 的优先级(临时变成和H一样高)
③ M 就绪, 但 M < L(继承后的) → M 抢不了L! → M 等着
④ L(高优先级)快速跑完临界区 → 释放锁 → L 优先级恢复原样
⑤ H 立即拿到锁, 运行

结果: H 只等了 L 临界区那一小段(可控!), M 没能插队捣乱。
      优先级反转被限制在"L临界区长度"内(bounded), 不再无限期。

优先级继承把优先级反转从"可能无限期"(灾难)变成"最多等一个临界区长度"(可控)。虽然没有完全消除反转(H 还是等了 L 一小会),但让它有界、可预测——这对实时系统足够了(第七卷会讲 WCET 最坏执行时间分析要用到这个"有界"性质)。

21.6 互斥量的数据结构：多出的两个字段

互斥量复用队列结构(第19章),但要实现继承和所有权,得记两样东西。回看第19章 Queue_t 里那几个"为互斥量准备"的字段:

Queue_t 里, 当它被用作互斥量时:

  /* union 里的这个成员被启用(替代 pcReadFrom) */
  UBaseType_t uxRecursiveCallCount;  /* 递归计数(21.9) */

  /* 队列结构里, 互斥量专用: */
  void *pxMutexHolder;   /* ★所有者: 指向当前持有此互斥量的任务TCB */

  /* TCB 里(第16章)配合的字段: */
  UBaseType_t uxBasePriority;  /* ★任务的"原始优先级"(继承前的) */
  UBaseType_t uxMutexesHeld;   /* 该任务当前持有几个互斥量 */

两个关键:

  • pxMutexHolder(在队列结构里):记录"谁现在持有这把锁"——这就是互斥量的所有权。信号量没有这个字段,所以谁都能 give;互斥量有它,才能实现"谁 take 谁 give"和优先级继承(知道要给谁提升优先级)。
  • uxBasePriority(在 TCB 里):记录任务被继承提升之前的原始优先级。因为继承会临时改高 uxPriority,释放锁后要恢复到 uxBasePriority——所以必须把原值存起来。

你现在理解第16章 TCB 里 uxBasePriority 和 uxMutexesHeld 两个字段的用途了——它们就是为优先级继承准备的。uxPriority 是"当前有效优先级"(可能被继承抬高),uxBasePriority 是"本来的优先级"。调度器用 uxPriority 调度,释放锁时用 uxBasePriority 恢复。uxMutexesHeld 计数是因为一个任务可能同时持有多个互斥量,要都释放且优先级正确恢复,逻辑会更复杂(取所有等待者中最高的)。前面章节埋的字段,在这里全部激活。

21.7 take 源码：记录所有者 + 继承

互斥量的 take(加锁)比信号量多做两件事:记录所有者、必要时触发优先级继承。看 xQueueSemaphoreTake 里互斥量相关的逻辑:

BaseType_t xQueueSemaphoreTake( QueueHandle_t xQueue, TickType_t xTicksToWait )
{
    for( ;; )
    {
        taskENTER_CRITICAL();
        {
            if( pxQueue->uxMessagesWaiting > 0 )   /* 锁可用 */
            {
                pxQueue->uxMessagesWaiting--;

                /* ★ 如果是互斥量: 记录"我"成为持有者 */
                if( pxQueue->uxQueueType == queueQUEUE_IS_MUTEX )
                {
                    pxQueue->u.xSemaphore.xMutexHolder = pvTaskIncrementMutexHeldCount();
                    /* 记下当前任务为持有者, 并给该任务的uxMutexesHeld+1 */
                }
                taskEXIT_CRITICAL();
                return pdPASS;
            }
            else   /* 锁被别人持有 */
            {
                if( xTicksToWait == 0 ) { taskEXIT_CRITICAL(); return errQUEUE_EMPTY; }
            }
        }
        taskEXIT_CRITICAL();

        vTaskSuspendAll();
        prvLockQueue( pxQueue );
        {
            /* ★★ 关键: 触发优先级继承 —— 让持有者继承我(申请者)的优先级 */
            if( pxQueue->uxQueueType == queueQUEUE_IS_MUTEX )
            {
                taskENTER_CRITICAL();
                vTaskPriorityInherit( pxQueue->u.xSemaphore.xMutexHolder );
                taskEXIT_CRITICAL();
            }
            /* 然后把自己挂进等待列表, 阻塞 */
            vTaskPlaceOnEventList( &( pxQueue->xTasksWaitingToReceive ), xTicksToWait );
        }
        ...
    }
}

核心是 vTaskPriorityInherit,看它的逻辑:

void vTaskPriorityInherit( TaskHandle_t const pxMutexHolder )
{
    TCB_t * const pxTCB = pxMutexHolder;   /* 持有锁的低优先级任务 */

    /* 如果持有者优先级 < 当前(申请锁的)任务优先级, 才需要提升 */
    if( pxTCB->uxPriority < pxCurrentTCB->uxPriority )
    {
        /* ★ 把持有者的优先级提升到和申请者一样高 */
        /* (uxBasePriority 保持不变, 记着原始值) */
        pxTCB->uxPriority = pxCurrentTCB->uxPriority;

        /* 如果持有者在就绪列表, 要把它移到新优先级的就绪链表 */
        if( 持有者在就绪列表 )
        {
            移动到 pxReadyTasksLists[新优先级];
        }
    }
}

清晰:当高优先级任务申请一个被低优先级任务持有的锁,就把持有者的 uxPriority 提升到申请者的水平(但 uxBasePriority 不变,记着原值)。这样持有者暂时"变高",能优先跑完临界区。这就是继承的源码。

21.8 give 源码：归还 + 恢复优先级

give(解锁)时,对称地要把继承来的优先级还回去。看 xQueueGenericSend 里互斥量的处理(通过 prvCopyDataToQueue 里的钩子):

/* 释放互斥量时调用, 恢复优先级 */
BaseType_t xTaskPriorityDisinherit( TaskHandle_t const pxMutexHolder )
{
    TCB_t * const pxTCB = pxMutexHolder;

    pxTCB->uxMutexesHeld--;   /* 持有的互斥量数-1 */

    /* 如果当前优先级 != 原始优先级(说明被继承提升过) */
    if( pxTCB->uxPriority != pxTCB->uxBasePriority )
    {
        /* 且没有再持有别的互斥量了 */
        if( pxTCB->uxMutexesHeld == 0 )
        {
            /* ★ 恢复到原始优先级 */
            pxTCB->uxPriority = pxTCB->uxBasePriority;
            /* 移回原优先级的就绪链表 */
            移动到 pxReadyTasksLists[uxBasePriority];
            return pdTRUE;   /* 需要重新调度 */
        }
    }
}

逻辑对称:释放锁时,若之前被继承提升过(uxPriority != uxBasePriority)且不再持有其他互斥量,就把 uxPriority 恢复成 uxBasePriority,并移回原优先级就绪链表。然后唤醒等锁的高优先级任务,它立即拿到锁运行。一次完整的"继承-恢复"闭环。

注意 uxMutexesHeld == 0 这个判断:如果任务同时持有多把锁,释放一把时不能立刻恢复(可能因另一把锁还在被更高优先级等待)。这是嵌套持锁的正确处理——细节比"提升一次恢复一次"复杂,但思想不变:只有当任务不再因任何互斥量而背负"继承的高优先级"时,才恢复原始优先级。FreeRTOS 这里的实现是简化版(完整的多锁优先级恢复要取所有持有锁的最高等待者优先级),但对绝大多数场景够用。这也是为什么"一个任务尽量别同时持有多把锁"是好实践。

21.9 递归互斥量

普通互斥量有个坑:同一个任务不能重复 take 自己已持有的互斥量,否则死锁(自己等自己释放)。但有时函数 A 持锁后调用了函数 B,B 里又要拿同一把锁——这就需要递归互斥量(Recursive Mutex)。

xMutex = xSemaphoreCreateRecursiveMutex();

任务:
  xSemaphoreTakeRecursive(xMutex, 等待);   // 第1次拿, 计数=1
    funcA() {
      xSemaphoreTakeRecursive(xMutex, 等待); // 同一任务再拿, 计数=2(不死锁!)
      ...
      xSemaphoreGiveRecursive(xMutex);       // 计数=1
    }
  xSemaphoreGiveRecursive(xMutex);           // 计数=0, 真正释放

实现:用第19章那个 union 里的 uxRecursiveCallCount。同一个持有者可以重复 take,每次计数+1;必须 give 相同次数(计数归 0)才真正释放锁。只有持有者本人能递归 take(靠 pxMutexHolder 判断)。这就是第19章那个 union 里 uxRecursiveCallCount 的用途——普通队列用读指针,递归互斥量用它当递归计数,共用内存。前面埋的每个字段,到这里全部有了着落。

21.10 互斥量的使用禁忌

互斥量威力大,但有严格使用规则,违反会出严重 bug:

  • 绝不能在中断里用!互斥量有所有权(持有者是某个任务),而中断不属于任何任务(第七章),没有"持有者"概念,谈不上优先级继承。所以互斥量没有 FromISR 版本。中断里要同步用信号量。
  • 持锁时间要短:临界区越长,继承持续越久、别的任务被压得越久。锁里别做耗时操作、别阻塞在别的东西上。
  • 谁 take 谁 give:别让 A 加锁 B 解锁(那是信号量的用法),破坏所有权语义。
  • 避免嵌套多把锁:容易死锁(21.12)。

21.11 优先级天花板（对比方案）

解决优先级反转,除了优先级继承,还有另一个经典方案优先级天花板(Priority Ceiling),面试可能对比问:

优先级继承(FreeRTOS 用)优先级天花板
机制持锁者被高优先级等待时,临时继承其优先级任务一拿到锁,就立即提升到"该锁的天花板优先级"(=所有可能用此锁的任务中最高的)
时机被动:有人来等才提升主动:一拿锁就提升
优点只在必要时提升,开销小能防死锁、更可预测
缺点不防死锁要预先知道每把锁的天花板,不灵活;可能过度提升

FreeRTOS 用继承(简单、开销小);一些更严格的实时系统(如某些汽车 OSEK/AUTOSAR)用天花板(可预测性更好、能防死锁)。两者都把优先级反转限制在有界范围内,选择是"简单灵活 vs 严格可预测"的权衡。第七卷会深入实时调度理论时再对比。

21.12 死锁：互斥的另一个陷阱

互斥量还有个经典问题——死锁(Deadlock):两个任务各持一把锁,又互相等对方的锁,永远僵住。

任务A:                    任务B:
  Take(锁1) ✓              Take(锁2) ✓
  Take(锁2) ← 等B释放      Take(锁1) ← 等A释放
     ↓                        ↓
  永远等待 ←──── 死锁 ────→ 永远等待

死锁的四个必要条件(缺一不可,面试常考):①互斥(资源不可共享)②持有并等待(拿着一个还要另一个)③不可剥夺(不能强抢)④循环等待(A等B、B等A形成环)。破坏任一条件即可避免。最常用的破坏方法:规定所有任务按固定顺序获取多把锁(破坏"循环等待")——比如永远先拿锁1再拿锁2,就不会有 A、B 反向获取形成环。

死锁 vs 优先级反转:优先级反转是"高优先级等太久但最终能跑",死锁是"永远僵住谁也跑不了"。都是互斥引入的问题。工程实践中避免死锁的黄金法则:①尽量少用多把锁 ②必须用多把时,全局统一加锁顺序 ③持锁时间尽量短 ④别在持锁时阻塞在其他事件上。第五卷讲 Linux 时,会看到内核有专门的死锁检测工具(lockdep),就是帮你抓这类循环等待。面试常把优先级反转和死锁一起考,能区分清楚是加分项。

21.13 面试常问问题

Q1:互斥量和二值信号量的区别?
都是长度1的队列,但互斥量有所有权(谁take谁give)和优先级继承(防优先级反转);信号量无,且用于同步而非互斥。互斥量初始"可用",信号量初始"无信号"。

Q2:什么是优先级反转?画出时序。
高优先级任务等低优先级持有的锁时,被无关的中优先级任务插队拖延,导致高优先级被间接推迟(甚至无限期)。见 21.4 时序。火星探路者的真实事故。

Q3:优先级继承如何解决优先级反转?
当高优先级任务因等锁阻塞,让持锁的低优先级任务临时继承其高优先级,使其不被中优先级抢占、尽快释放锁;释放后恢复原优先级。把反转限制在临界区长度内(有界)。

Q4:优先级继承在源码里怎么实现?涉及哪些字段?
队列的 pxMutexHolder 记持有者;TCB 的 uxBasePriority 存原始优先级、uxPriority 是当前(可被提升)优先级。take 时 vTaskPriorityInherit 提升持有者;give 时 xTaskPriorityDisinherit 恢复。

Q5:互斥量能在中断里用吗?为什么?
不能。互斥量有所有权(持有者是任务),中断不属于任何任务,无法优先级继承。没有 FromISR 版。中断同步用信号量。

Q6:什么是死锁?怎么避免?
多任务互相持锁又等对方的锁,僵死。四条件:互斥/持有并等待/不可剥夺/循环等待。破坏任一即可,最常用是全局统一加锁顺序破坏循环等待。

Q7:优先级继承和优先级天花板的区别?
继承是被动的(有人等才提升),天花板是主动的(一拿锁就提升到预设最高)。天花板能防死锁、更可预测,但要预知天花板值。FreeRTOS 用继承。

21.14 常见误区

  1. "互斥量和二值信号量一样"——底层像,但互斥量有所有权和优先级继承,信号量没有。
  2. "优先级继承能完全消除优先级反转"——不能,只是把它限制在有界(一个临界区长度)内。
  3. "互斥量可以在中断里用"——不能,无所有权概念,没有 FromISR 版。
  4. "A 加锁可以 B 解锁"——互斥量必须谁 take 谁 give;那样用是信号量。
  5. "同一任务可以重复 take 普通互斥量"——会死锁,要用递归互斥量。
  6. "死锁和优先级反转是一回事"——反转是等太久最终能跑,死锁是永远僵住。
  7. "用了互斥量就不会死锁"——多把锁乱序获取照样死锁,要统一加锁顺序。

21.15 本章总结

  • 互斥量 vs 二值信号量:底层都是长度1队列,但互斥量有所有权(谁take谁give)和优先级继承,用于互斥(保护共享资源)。
  • 优先级反转:高优先级任务等低优先级持有的锁时,被无关中优先级任务插队拖延(可能无限期)——火星探路者真实事故。
  • 优先级继承:持锁的低优先级任务临时继承等待者的高优先级,尽快释放锁,把反转限制在有界范围;释放后恢复。
  • 数据结构:队列 pxMutexHolder(所有者)+ TCB uxBasePriority(原始优先级)/uxPriority(当前可被提升)/uxMutexesHeld。
  • take → vTaskPriorityInherit 提升持有者;give → xTaskPriorityDisinherit 恢复原优先级。
  • 递归互斥量:同一任务可重复 take(用 union 里 uxRecursiveCallCount 计数),give 同样次数才真释放。
  • 禁忌:不能在中断用(无所有权)、持锁要短、谁take谁give、避免嵌套多锁。
  • 对比方案:优先级天花板(主动提升,防死锁,更可预测)vs 继承(被动,简单)。
  • 死锁:多任务互相持锁等对方,四条件(互斥/持有等待/不可剥夺/循环等待),破坏循环等待(统一加锁顺序)避免。
