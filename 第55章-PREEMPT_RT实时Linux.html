本章目标:第50章说原生 Linux 不硬实时,但 Linux 功能太强大了(网络、文件、ROS2、丰富驱动),我们又想在它上面跑实时任务。能不能把 Linux 改造成实时的?能——PREEMPT_RT 补丁(现已大部分合入主线)就是干这个的。本章讲它如何把 Linux 那些"不确定源"(第51章)逐一改造掉,让延迟从毫秒级降到微秒级。作为第七卷收官,这是理论落地到 RK3588 实际工程的一章,车企做 Linux 实时必考。

55.1 既要 Linux 的功能，又要实时

一个现实的两难:

  • RTOS(FreeRTOS):实时性好,但功能少——没有丰富的文件系统、网络栈、进程隔离、ROS2、复杂应用生态。
  • Linux:功能极强(网络、文件、驱动、ROS2、AI 框架),但默认不硬实时(第50章)。

现代机器人/无人机/汽车既需要 Linux 的强大功能(跑视觉、导航、通信),又需要实时(控制回路)。怎么办?两条路:①分开——MCU 跑 RTOS 做硬实时控制,SoC 跑 Linux 做复杂应用(AMP,第八卷),这是最稳妥的;②合并——把 Linux 本身改造成实时的,让实时任务和复杂应用都跑在一个 Linux 上。本章讲第二条路:PREEMPT_RT 如何改造 Linux。它让你在一颗跑 Linux 的 SoC(如 RK3588)上,既能跑 ROS2/视觉,又能跑有一定实时保证的控制任务——省去额外 MCU,简化系统。这对成本/体积敏感的产品很有吸引力。

55.2 生活例子：给货车装急救通道

原生 Linux 像一辆满载的货车(吞吐优先,第50章):装得多、效率高,但一旦开动就不好停,急事(紧急任务)插不进来——货装到一半不能随便卸(不可抢占段)、司机在打电话时不接别的呼叫(关中断)。

PREEMPt_RT 的改造,就像给这辆货车动手术:装可以随时暂停装卸的机构(可抢占)、给司机配个"急事优先"的对讲机(中断线程化、优先级继承),让紧急任务能随时打断当前工作插进来处理。改造后,货车还是那辆功能强大的货车(Linux 的功能全保留),但获得了"急事能及时响应"的能力(实时性)。代价是:为了能随时被打断,平时的运输效率(吞吐)略降。这就是 PREEMPT_RT 的本质——用一点吞吐,换 Linux 的实时能力。

55.3 PREEMPT_RT 是什么

明确 PREEMPT_RT 的定位:

问题答案
是什么一套让 Linux 内核近乎完全可抢占的补丁/配置(CONFIG_PREEMPT_RT)
目标大幅降低最坏调度延迟,让 Linux 能满足软实时到部分硬实时
手段消除第51章那些不确定源:改可抢占、中断线程化、优先级继承
现状历经 20 年,核心已合入 Linux 主线(6.12 起大部分正式支持)
不改变Linux 的 API、功能、生态全保留(应用不用改)

PREEMPT_RT 的核心目标:把 Linux 内核改造成"几乎处处可抢占",从而大幅降低最坏调度延迟(第51章),让高优先级实时任务能几乎立即抢占运行。它不是另一个操作系统,而是对标准 Linux 内核的一系列修改(通过 CONFIG_PREEMPT_RT 配置启用),完全保留 Linux 的 API、功能和生态——你的 ROS2、网络程序、驱动照常跑,只是内核的实时性大大增强。这是一个了不起的工程成就:历经约 20 年开发,如今核心已合入 Linux 主线(6.12 版本起大部分特性正式支持,不再需要外部打补丁)。它能把 Linux 的最坏延迟从毫秒级(甚至几十毫秒)压到几十微秒,满足软实时和相当一部分硬实时需求。下面看它具体改了什么。

55.4 Linux 抢占模型的演进

理解 PREEMPT_RT 前,先看 Linux 抢占能力的几个档位(内核配置选项):

抢占模型可抢占程度适合
PREEMPT_NONE内核态不可抢占(只在返回用户态时切换)服务器(要吞吐)
PREEMPT_VOLUNTARY内核里加了一些"自愿抢占点"桌面(改善响应)
PREEMPT(低延迟)内核态大部分可抢占(除持锁/中断)桌面/低延迟需求
PREEMPT_RT内核态几乎完全可抢占(连自旋锁临界区都可抢占)实时

这是一个"可抢占程度"递增的谱系,PREEMPT_RT 是最激进的一档。普通 Linux(服务器 NONE、桌面 PREEMPT)在内核态有大量不可抢占的地方(第51章),高优先级任务要等。PREEMPT_RT 把"几乎所有"内核代码都变成可抢占——包括原本绝对不可抢占的自旋锁临界区(55.5)。可抢占程度越高,调度延迟越小(高优先级任务越能及时插入),但抢占/切换开销越大、吞吐略降——又是第50章吞吐 vs 实时的权衡。你在编译内核时通过这个配置选择档位;做实时就选 PREEMPT_RT。下面看它用哪些具体手段实现"几乎完全可抢占"。

55.5 改造一：自旋锁变可睡眠锁

PREEMPT_RT 最核心、最巧妙的改造——把大部分自旋锁变成可睡眠、可抢占的锁:

原生Linux的自旋锁(第37章)问题:
  持自旋锁时【不可抢占】(第51章不可抢占段的主要来源)
  → 高优先级实时任务就绪, 但当前任务持自旋锁 → 只能等它解锁
  → 自旋锁临界区 = 一段不可抢占的延迟

PREEMPT_RT的改造:
  ★把大部分 spinlock 转换成 rt_mutex(可睡眠的互斥锁)
   → 持这种"自旋锁"时【可以被抢占】了!
   → 高优先级任务就绪 → 立即抢占持锁任务
   → 持锁任务被抢占后睡眠(而非自旋), 高优先级先跑
   → 而且rt_mutex带【优先级继承】(第53章)! 防反转
   (少数真正需要的原子上下文, 保留raw_spinlock不变)

这是 PREEMPT_RT 最核心的改造:把 Linux 内核里大量的自旋锁(第37章)从"持锁不可抢占"改成"持锁可抢占的可睡眠锁(rt_mutex)"。原生 Linux 里,持自旋锁是不可抢占段(第51章)的主要来源——一段自旋锁临界区就是一段高优先级任务无法插入的延迟。PREEMPT_RT 让这些"自旋锁"实际变成可睡眠的 rt_mutex:持锁的任务可以被高优先级任务抢占(它会睡眠让路,而非霸占 CPU 自旋),大大降低了调度延迟。更妙的是,rt_mutex 自带优先级继承(第53章),顺便解决了优先级反转问题。代价是:锁操作从"简单自旋"变成"可能睡眠+切换",开销增大、吞吐略降(实时的代价)。少数真正需要原子性、绝不能睡的地方(如调度器核心、真正的中断处理)保留为 raw_spinlock 不变。这个"自旋锁可睡眠化"是 PREEMPT_RT 削减不可抢占段的主力。

55.6 改造二：中断线程化

第二大改造——把中断处理线程化(第43章线程化中断的全面推广):

原生Linux的中断问题:
  硬中断ISR在中断上下文运行, 【不可被调度/抢占】,
  而且ISR期间可能关中断 → 高优先级实时任务被ISR挡住
  一个耗时的ISR = 一段不可控的延迟

PREEMPT_RT的改造:
  ★把大部分中断处理【放进内核线程】运行(threaded IRQ, 第43章)
   → 中断处理变成一个【普通的内核线程】, 有优先级、可被调度
   → 如果实时任务优先级比某中断线程【更高】,
     实时任务可以抢占中断线程! (中断不再无条件优先)
   → 中断处理也纳入统一的优先级调度体系

  只保留极短的硬中断上半部(标记+唤醒中断线程)
  → 硬中断关中断时间极短 → 中断延迟大降

第二大改造:把中断处理从"不可抢占的中断上下文"搬进"可调度的内核线程"。原生 Linux 的硬中断 ISR 优先于一切任务运行、不可抢占,一个耗时 ISR 就是一段挡住实时任务的延迟。PREEMPT_RT 把大部分中断处理做成内核线程(第43章线程化中断的全面化):中断处理变成有优先级、可被调度的线程,于是一个高优先级的实时任务可以抢占低优先级的中断线程——中断不再无条件压倒实时任务,而是纳入统一的优先级体系。只保留极短的硬中断上半部(标记+唤醒中断线程),硬中断关中断时间极短。这让"中断延迟"和"中断对调度的干扰"都大幅降低,并让你能给关键实时任务设比中断线程更高的优先级,保证它优先。

55.7 改造三：优先级继承等

PREEMPT_RT 还有一系列配套改造,共同消除不确定源:

改造作用对应
rt_mutex 优先级继承内核锁普遍带 PI,消除优先级反转第53章
软中断线程化softirq 也放线程,可被抢占第36章
高精度定时器hrtimer,定时精度到纳秒,精确唤醒-
缩短/消除长临界区把内核里长的不可抢占段拆短第51章
可抢占的 RCURCU 读侧也可抢占(preemptible RCU)第38章

PREEMPT_RT 是一整套系统性改造,把第51章列举的每一个"不确定源"逐个攻克:自旋锁可抢占(改造一)、中断线程化(改造二)、软中断也线程化、内核锁普遍带优先级继承消除反转(第53章)、用高精度定时器 hrtimer 实现纳秒级精确唤醒、把内核里残留的长不可抢占段拆短、连 RCU(第38章)读侧都改成可抢占。你会发现,这套改造就是第51章"降低延迟手段"和第53章"优先级继承"在内核层面的系统性落地。每一项都在削减最坏调度延迟。这也再次印证第七卷的知识是环环相扣的:PREEMPT_RT 不是凭空的黑科技,而是把前面所有实时原理(延迟分解、优先级继承、可抢占、高精度定时)工程化地实现在 Linux 内核里。理解了前面几章,你就能看懂 PREEMPT_RT 每项改造的动机。

55.8 效果：延迟从毫秒到微秒

改造的实际效果,用第51章的 cyclictest 衡量:

cyclictest 测最坏调度延迟(Max), 同样负载下:

  原生Linux(PREEMPT_NONE/桌面):
    Max 常在 【几百微秒 ~ 几十毫秒】, 且有不可预测长尾
    → 只能软实时(视频/音频), 做不了控制

  PREEMPT_RT:
    Max 通常 【< 100微秒】(好的平台/配置下几十us), 且【有界】
    → 能满足软实时和相当一部分硬实时(如几百Hz~kHz控制回路)

  RTOS(FreeRTOS)/裸机:
    Max 更小(us级甚至更低)且更稳定
    → 严格硬实时(高频电机控制等)

  → PREEMPT_RT让Linux的实时性【接近但一般不及】专用RTOS,
    但换来了Linux的全部功能生态

效果显著:PREEMPT_RT 把 Linux 的最坏调度延迟从"毫秒级且有不可预测长尾"降到"通常 <100 微秒且有界"。这个量级的提升,让 Linux 从"只能软实时"变成"能满足软实时和相当一部分硬实时"——足以跑几百 Hz 到 kHz 的控制回路(很多机器人/无人机的姿态环在这个范围)。但要客观:PREEMPT_RT 的实时性仍一般不及专用 RTOS/裸机(后者 Max 更小更稳定),因为 Linux 内核毕竟庞大复杂,还有些难以完全消除的延迟源、以及现代 SoC 的硬件不确定(第54章)。所以选择是:极严格的硬实时(高频电机 FOC)用 RTOS/MCU;中等实时 + 需要 Linux 功能的用 PREEMPT_RT。车企会根据具体控制回路的严格程度来选。

55.9 另一条路：双内核(Xenomai)

把 Linux 变实时,除了 PREEMPT_RT(改造单内核),还有另一种思路——双内核:

双内核方案(Xenomai / RTAI 等):
  在Linux【底下】塞一个微型实时内核(co-kernel),
  它优先级高于整个Linux:
    ┌──────────────────────────┐
    │  实时任务 ← 跑在实时内核上(硬实时保证)│
    ├──────────────────────────┤
    │  Linux(及其所有应用)← 作为实时内核的  │
    │       "空闲任务"运行, 随时被实时内核抢占 │
    └──────────────────────────┘
  → 实时内核直接管硬件中断, 保证实时任务的确定性
  → Linux只在实时内核空闲时才运行

对比:
  PREEMPT_RT(单内核): 改造Linux本身, 简单统一, 用标准Linux API,
                     实时性好但不及双内核
  双内核(Xenomai): 实时性更强(接近RTOS), 但架构复杂、
                   实时任务要用专门API、维护成本高

双内核是另一条实时化路线:不改造 Linux,而是在它"下面"放一个微型实时内核(co-kernel),这个实时内核优先级高于整个 Linux。实时任务跑在实时内核上(获得接近 RTOS 的硬实时保证),Linux 整个作为实时内核的"空闲任务"运行——只在没有实时任务时才跑。对比 PREEMPT_RT:双内核实时性更强(实时内核小而可控),但架构复杂、实时任务要用专门 API(不是标准 Linux API)、维护成本高。PREEMPT_RT 则是改造 Linux 单内核本身——统一、简单、用标准 Linux API,实时性虽略逊但足够多数场景,且随主线维护。趋势上 PREEMPT_RT 合入主线后成为主流选择(维护和生态优势),双内核(Xenomai)用于对实时性要求极致的特定场合。了解两条路的存在和取舍即可。

55.10 实时 Linux 调优实战清单

把 Linux 调成实时,不只是打开 PREEMPT_RT,还要一系列配置(实战 checklist,车企做 RK3588 实时必用):

实时 Linux 调优清单:
  □ 用 PREEMPT_RT 内核(或至少 PREEMPT)
  □ 关键任务用 SCHED_FIFO/RR 高优先级 (chrt, 第52章)
  □ mlockall() 锁定内存, 防换页 (第10/51章)
  □ 用内存池/预分配, 避免运行时 malloc (第24/54章)
  □ CPU 隔离: isolcpus + 把实时任务绑到隔离核 (亲和, 第八卷)
  □ 关闭该核的中断/RCU回调/负载均衡, 减少干扰
  □ 关闭 DVFS/CPU 调频, 固定最高频率 (第54章)
  □ 禁用不必要的电源管理(C-states 深睡眠会增大唤醒延迟)
  □ 用带优先级继承的锁 (PTHREAD_PRIO_INHERIT, 第53章)
  □ 调整 RT throttling (或确保实时任务不死循环)
  □ 用 cyclictest 验证 Max 延迟达标 (第51章)
  □ 循环有界、避免递归 (为可预测编程, 第54章)

这份清单是实时 Linux 工程的"作战手册",几乎每一项都对应第七卷前面某章的原理。把 Linux 调成实时,PREEMPT_RT 内核只是起点,还要:用实时调度策略(第52章)、锁内存防换页(第51章)、内存池避免 malloc(第54章)、CPU 隔离把实时任务绑到专用核并关掉该核的各种干扰(第八卷 affinity/isolcpus)、固定频率(第54章)、用优先级继承锁(第53章),最后用 cyclictest 验证(第51章)。你会发现,这份清单就是把第七卷所有知识落地成可执行的工程步骤——这正是"理论到实践"的桥梁。对目标车企的你,这份 checklist 极其实用:在 RK3588 上跑实时控制任务,就是照着它一项项配置和验证。面试问"怎么在 Linux 上做实时",答出这份清单(而不只是"用 PREEMPT_RT")才显功力。

第七卷完整地讲了实时系统:从"什么是实时"(50)的观念,到延迟分析(51)、调度理论(52)、优先级反转(53)、WCET(54)的方法论,最后到 PREEMPT_RT(55)把这一切落地到真实的 Linux 系统。你现在具备了完整的实时系统能力:理解实时的本质、能分析和降低延迟、能设计可证明满足 deadline 的调度、能防治优先级反转、能处理 WCET、能把 Linux 调成实时。这是从"会用 RTOS"到"懂实时系统工程"的完整跃迁。

但本章也揭示了一个更大的工程现实:单靠 PREEMPT_RT,Linux 的实时性仍不及专用 RTOS;而现代产品(无人机、自动驾驶域控)既要 Linux 的强大功能(视觉、AI、通信),又要 RTOS 级的硬实时控制。怎么两全?答案是第八卷的核心主题——AMP(非对称多处理):在一颗 SoC(如 RK3588)里,让大核跑 Linux(复杂应用)、小核跑 RTOS(硬实时控制),两者协同。这样飞控的硬实时用 RTOS 保证、视觉导航的功能用 Linux 满足,各取所长。第八卷会讲 RK3588 上的这种混合架构:Linux + RT-Thread/RTOS、多核协同(AMP/SMP)、核间通信(RPMsg)、共享内存、CPU 亲和隔离——把第七卷的实时理论和第六卷的驱动能力,综合应用到工业级别的真实工程平台上。你已经打通了从计算机原理、ARM、RTOS、Linux 内核、驱动到实时系统的完整链路,第八卷将是这一切的工程集大成。恭喜你征服了实时系统这座硬核高峰!

55.12 面试常问问题

Q1:为什么原生 Linux 不硬实时?PREEMPT_RT 解决什么?
原生 Linux 有大量不可抢占段、长关中断、中断优先于任务等(第51章),最坏延迟毫秒级且有长尾。PREEMPT_RT 把内核改造成几乎完全可抢占,把最坏延迟降到微秒级且有界。

Q2:PREEMPT_RT 的核心改造有哪些?★
①自旋锁改成可睡眠可抢占的 rt_mutex(消除不可抢占段,还带优先级继承);②中断线程化(中断处理放内核线程,可被高优先级任务抢占);③软中断线程化、高精度定时器、可抢占 RCU、缩短长临界区。

Q3:自旋锁改可睡眠锁为什么能提升实时性?
原生持自旋锁不可抢占,是延迟主要来源。改成 rt_mutex 后持锁可被高优先级抢占(持锁者睡眠让路),大降调度延迟,还顺带优先级继承防反转。少数原子场景保留 raw_spinlock。

Q4:PREEMPT_RT 能达到什么延迟?和 RTOS 比?
cyclictest Max 通常 <100us 且有界(原生毫秒级有长尾)。能满足软实时和部分硬实时(百 Hz~kHz 控制)。但一般不及专用 RTOS/裸机(us 级更稳),因 Linux 庞大+硬件不确定。

Q5:PREEMPT_RT 和双内核(Xenomai)的区别?
PREEMPT_RT 改造 Linux 单内核,统一、标准 API、已入主线;双内核在 Linux 下塞微实时内核,实时性更强但架构复杂、要专门 API。趋势是 PREEMPT_RT 主流。

Q6:在 Linux 上做实时,除了 PREEMPT_RT 还要做什么?
SCHED_FIFO 高优先级、mlockall 锁内存、内存池避 malloc、CPU 隔离绑核关干扰、固定频率、优先级继承锁、cyclictest 验证、循环有界。一整套调优。

Q7:既要 Linux 功能又要硬实时,最佳方案?
AMP:大核跑 Linux(复杂应用)、小核/MCU 跑 RTOS(硬实时控制),核间通信协同(第八卷)。PREEMPT_RT 适合中等实时+要 Linux 功能的场景。

55.13 常见误区

  1. "装了 PREEMPT_RT 就自动硬实时"——还要一整套调优(绑核/锁内存/优先级等),且仍不及 RTOS。
  2. "PREEMPT_RT 会让 Linux 变快"——不是变快,是最坏延迟变小可预测;平均吞吐略降。
  3. "PREEMPT_RT 改变了 Linux API"——API/功能不变,应用不用改。
  4. "PREEMPT_RT 能完全替代 RTOS"——实时性一般不及专用 RTOS,极严格场景仍需 RTOS/AMP。
  5. "自旋锁全改成了睡眠锁"——大部分改,少数原子上下文保留 raw_spinlock。
  6. "实时 Linux 只需内核配置"——还要应用侧(实时优先级、锁内存、内存池)和系统调优。

55.14 本章总结

  • 两难:RTOS 实时但功能少,Linux 功能强但默认不实时。PREEMPT_RT 把 Linux 改造成实时,保留全部功能生态。
  • 抢占谱系:NONE → VOLUNTARY → PREEMPT → PREEMPT_RT(几乎完全可抢占);可抢占越多延迟越小、吞吐略降。
  • 核心改造★:①自旋锁→可睡眠可抢占的 rt_mutex(消除不可抢占段+优先级继承);②中断线程化(中断处理放线程,可被实时任务抢占);③软中断线程化、hrtimer、可抢占 RCU、拆短长临界区。
  • 效果:最坏调度延迟从毫秒级有长尾 → <100us 有界(cyclictest Max);满足软实时+部分硬实时,但一般不及 RTOS。
  • 另一路:双内核(Xenomai)——Linux 下塞微实时内核,实时性更强但复杂;PREEMPT_RT 单内核更统一、已入主线,成主流。
  • 实战调优清单:PREEMPT_RT + SCHED_FIFO + mlockall + 内存池 + CPU 隔离绑核 + 固定频率 + 优先级继承锁 + cyclictest 验证。
  • 终极方案:极严格硬实时用 AMP(大核 Linux + 小核 RTOS,第八卷);PREEMPT_RT 适合中等实时+要 Linux 功能。
