本章目标：RTOS 里创建任务是 xTaskCreate——直接给个函数和栈。Linux 完全不同:它用 fork(分裂出一个自己的副本)+ exec(把副本变成别的程序) 这套独特机制。这一章讲清 fork/exec/clone 三个系统调用、写时复制、以及"线程和进程底层都是 clone"的统一。这是 Linux 进程管理最核心、面试最高频的部分。学完你要能讲清"shell 敲一个命令,系统怎么把程序跑起来"。

29.1 进程从哪来：一切进程的祖先

RTOS 里任务凭空 xTaskCreate 出来。Linux 不一样:Linux 里新进程只能由已有进程"分裂"产生(除了开机时内核手工造的第一个)。所以所有进程构成一棵树:

进程树(所有进程的家谱):

  内核启动 → 手工创建 PID 1 (init/systemd)  ← 树根, 一切进程的祖先
                │ fork
      ┌─────────┼─────────┐
    getty     sshd      各种服务
      │fork      │fork
    login     bash(你的shell)
                │fork
              你敲的命令(ls, gcc...)

PID 1(init/systemd)是内核唯一"凭空造"的进程,之后所有进程都是通过 fork 从已有进程分裂出来的。"进程只能由进程创建"是 Linux 的基本模型——这和 RTOS "任务随便建"不同,也带来了进程树、父子关系、回收等一整套语义(第28章的 parent/children)。

29.2 fork：进程的"分裂繁殖"

fork() 是 Linux 创建进程的核心,行为很反直觉:它把当前进程完整复制一份,产生一个几乎一模一样的子进程。就像细胞分裂——一个变两个,两个几乎相同。

pid_t pid = fork();

fork 做的事: 复制父进程, 产生子进程
  子进程复制了父进程的:
    - 地址空间(代码/数据/堆/栈的内容) —— 但用写时复制(29.4)
    - 打开的文件描述符
    - 当前工作目录、信号处理设置、环境变量...
  子进程有自己新的:
    - PID(不同的进程号)
    - parent 指向父进程
    - 独立的地址空间(逻辑上, 物理上先共享)

复制出来的子进程,从 fork 返回的下一行开始执行(不是从头!),拥有和父进程当时几乎相同的状态。这就是"分裂繁殖":不是造一个空白进程,而是克隆一个现有的。为什么这么设计?因为这样简单而强大——子进程继承了父进程的一切环境(打开的文件、变量),配合 exec 能实现极灵活的进程创建(29.6)。

29.3 fork 一次返回两次

fork 最让新手困惑、面试最爱考的一点:fork 调用一次,却返回两次——在父进程里返回一次,在子进程里返回一次。返回值用来区分你是谁:

pid_t pid = fork();
if (pid > 0) {
    // ★ 父进程执行这里: fork 返回【子进程的PID】(正数)
    printf("我是父进程, 我的孩子是 %d\n", pid);
} else if (pid == 0) {
    // ★ 子进程执行这里: fork 返回 【0】
    printf("我是子进程\n");
} else {
    // fork 失败(返回-1)
}
// 父子进程都会执行到这里(各自独立)

为什么一次调用返回两次?因为 fork 之后,内存里有了两个进程,它们都从 fork 返回处继续跑。fork 在父进程里返回子进程的 PID(让父进程知道孩子是谁),在子进程里返回 0(子进程想知道父进程用 getppid())。同一行 pid = fork(),在两个进程里给 pid 赋了不同的值,于是 if/else 让父子走了不同的分支——这就是"一次调用,两个进程,两种返回"。

理解这点的关键:fork 后就是两个独立的进程在各自的地址空间里跑同一份代码,只是 fork 的返回值不同。它们之后互不影响(改各自的变量,对方看不到,因为地址空间独立——第九、十章的隔离)。这和 RTOS 完全不同(RTOS 任务共享地址空间)。"fork 一次返回两次"是理解 Unix 进程模型的第一道坎,跨过去就豁然开朗。

29.4 写时复制：fork 快的秘密

fork 要"复制整个地址空间",如果父进程用了 2GB 内存,复制 2GB 岂不很慢?这里正是第十章讲的写时复制(COW)大显身手的地方!回顾第十章 10.7:

fork 时【不真正复制内存】:
  父子进程的页表都指向【同一批物理页】, 标记为只读

  父进程页表 ─┐
              ├→ 同一批物理页(只读)   ← 不复制! 共享!
  子进程页表 ─┘

任何一方要【写】某页时:
  → 写只读页 → 触发缺页异常(第九章权限位)
  → 内核这时才复制那一页, 给写的人一份私有可写副本
  → 只复制"被修改的页", 没改的一直共享

所以 fork 几乎瞬间完成(只复制页表,不复制数据),内存也极省(大量只读页父子共享,尤其代码段永远共享)。这就是第十章埋的伏笔在进程创建里的落地。COW 让"复制整个进程"这个昂贵操作变得廉价——这是 Unix fork 模型能高效工作的技术基石。如果没有 COW,fork 这种"先完整复制再可能立刻 exec 替换掉"的模型会浪费巨大,根本不实用。

29.5 exec：进程的"灵魂替换"

fork 造出的子进程是父进程的副本,跑的还是父进程的代码。但我们通常想让子进程跑别的程序(比如 shell fork 后想跑 ls)。这就要 exec——它把当前进程的"内容"整个换成另一个程序,但进程本身(PID 等)不变。

execl("/bin/ls", "ls", "-l", NULL);   // 把当前进程变成 ls 程序

exec 做的事:
  - 丢弃当前进程的地址空间(旧代码、数据、堆、栈全扔掉)
  - 加载新程序(/bin/ls)的代码和数据到地址空间
  - 从新程序的入口开始执行
  - 但保留: PID、父子关系、打开的文件描述符(默认)、...

关键: exec 成功后【不返回】! 因为原来的代码已经被替换掉了,
      "返回到哪"的那个"哪"已经不存在了。

形象比喻:fork 是"生了个和你一样的孩子",exec 是"给这个孩子做了灵魂替换,让他变成完全不同的一个人,但还是你的孩子(PID/父子关系不变)"。exec 有一族函数(execl/execv/execlp/execvp...),区别只是参数传递方式,核心一样。exec 成功不返回是个重要特性——因为进程的代码都被换了,没有"原路"可回。

29.6 fork + exec：shell 启动程序的真相

现在揭晓:你在 shell 里敲 ls -l,系统到底怎么把它跑起来?答案是 fork + exec 的经典组合。

你在 bash 敲: ls -l

bash 做的事:
① fork()  ─────→ 分裂出一个子进程(bash的副本)
      │
   ┌──┴──────────────┐
父进程(bash)        子进程(bash副本)
   │                  │
② wait(子进程)     ③ exec("/bin/ls", "ls", "-l")
   等孩子干完           把自己变成 ls 程序, 开始执行 ls
   │                  │
   │              ④ ls 执行完 exit
   │                  │
⑤ wait 返回 ←─────────┘ 回收子进程
   bash 打印提示符, 等你下一条命令

拆解这个"fork 生孩子 → 孩子 exec 变身目标程序 → 父亲 wait 收尸"的三步舞:

  • fork:bash 先复制一个自己(子进程),这样不影响 bash 本身。
  • exec:子进程把自己替换成 ls——现在这个子进程就是 ls 了。
  • wait:bash(父进程)等 ls 跑完,回收它(避免僵尸,第28章),然后继续等你的下条命令。

"fork + exec" 是 Unix/Linux 启动新程序的标准范式,理解它就理解了整个系统怎么运转。为什么要分成 fork 和 exec 两步,而不像 Windows 的 CreateProcess 一步到位?因为在 fork 之后、exec 之前的那个空隙,子进程可以做很多定制——比如重定向输入输出(把 stdout 接到文件,实现 ls > out.txt)、设置环境变量、改变工作目录。这个"两步之间可插入定制"的灵活性,是 Unix 进程模型的精妙之处,也是 shell 能实现管道、重定向的基础(|、>、< 都在这个空隙里设置)。面试常问"fork 和 exec 为什么分开",这就是标准答案。

29.7 clone：fork 和线程的统一底层

第28章说"线程是共享资源的进程,fork 和 pthread_create 底层都是 clone"。现在揭晓 clone。它是比 fork 更底层、更灵活的系统调用:fork 是"复制一切",而 clone 让你精确指定子进程和父进程共享哪些资源。

clone(flags, ...) 用 flags 控制共享什么:

  CLONE_VM      共享地址空间(内存)  ← 关键!
  CLONE_FILES   共享打开的文件表
  CLONE_FS      共享文件系统信息(工作目录)
  CLONE_SIGHAND 共享信号处理
  CLONE_THREAD  同一个线程组(共享tgid)
  ...

不同的 flags 组合 = 不同的"创建":
  ┌────────────────────────────────────────┐
  │ 什么都不共享(各自独立)   = fork(创建进程) │
  │ 共享VM+FILES+FS+SIGHAND  = 创建线程        │
  └────────────────────────────────────────┘

这就是第28章那个深刻设计的实现:进程和线程的区别,不是"两种东西",而是"clone 时共享了多少资源"。

  • fork = clone(几乎不共享)→ 子进程有独立的一切 → 这是"进程"。
  • 创建线程 = clone(CLONE_VM | CLONE_FILES | CLONE_THREAD | ...)→ 新任务和创建者共享地址空间、文件、信号 → 这是"线程"。

clone 用一组开关,把"进程"和"线程"统一成了一个连续谱系。你甚至可以创建"介于两者之间"的东西(共享一部分资源)。这种设计的优雅在于:内核不需要为进程和线程写两套代码,只需一个 clone,用 flags 区分共享程度。关键洞察:共享 CLONE_VM(地址空间)与否,是进程和线程最根本的分界——共享地址空间(同一个 mm,第28章)就是线程(能直接访问彼此的变量),不共享(各自的 mm)就是进程(隔离)。这完美呼应第一章"进程=独立操作台,线程=共享操作台"的比喻,现在你看到了它在 clone flags 里的精确实现。

29.8 线程创建：pthread_create → clone

你写多线程用 pthread_create,它底层就是 clone。链条:

pthread_create(&tid, NULL, thread_func, arg)
    │ (glibc 的 pthread 库)
    ▼
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND |
      CLONE_THREAD | CLONE_SETTLS | ..., ...)
    │ (系统调用)
    ▼
内核 copy_process(): 创建一个新 task_struct,
    但它的 mm 指向【和创建者同一个 mm】(因为CLONE_VM)
    → 新线程和主线程共享地址空间
    → 它们能访问同一批全局变量(所以多线程要加锁, 第37章!)

关键结论:Linux 的线程,在内核里就是一个 task_struct(和进程一样),只是它的 mm 等字段指向和同组线程共享的对象。所以:

  • 内核调度的基本单位是 task_struct——线程和进程被一视同仁地调度(第30章)。这就是"Linux 线程是内核级线程"的含义:每个线程内核都知道、都单独调度(不像某些用户级线程库,内核只看到一个进程)。
  • 同进程的线程共享地址空间,能直接读写彼此的全局变量——方便但危险(数据竞争),所以需要第37章的锁。这就是第14章"多核内存序"、第37章"并发同步"要解决的问题的来源。

29.9 copy_process：创建的核心

fork/clone 最终都进入内核的 copy_process(kernel/fork.c)——真正创建 task_struct 的地方。骨架:

static struct task_struct *copy_process(...unsigned long clone_flags...)
{
    /* ① 分配新的 task_struct 和内核栈 */
    struct task_struct *p = dup_task_struct(current);

    /* ② 根据 clone_flags, 决定"复制"还是"共享"各种资源 */
    copy_files(clone_flags, p);    /* 文件表: 共享 or 复制 */
    copy_fs(clone_flags, p);       /* 文件系统信息 */
    copy_sighand(clone_flags, p);  /* 信号处理 */
    copy_mm(clone_flags, p);       /* ★地址空间: CLONE_VM则共享, 否则复制(COW) */
    copy_thread(clone_flags, p);   /* 架构相关: 设置新任务的内核栈和寄存器 */

    /* ③ 分配 PID */
    p->pid = alloc_pid(...);

    /* ④ 设置父子关系, 加入进程链表 */
    /* ⑤ 返回新任务(还没运行, 由调用者唤醒加入运行队列) */
    return p;
}

看那些 copy_* 函数:每一个都根据 clone_flags 决定"共享还是复制"某类资源。copy_mm 是关键——如果 CLONE_VM 就让新任务的 mm 指向同一个(线程),否则复制一份(进程,用 COW)。copy_thread 是架构相关的,它设置新任务的内核栈和初始寄存器现场——这不就是第16章 FreeRTOS 的 pxPortInitialiseStack 吗?它同样要"伪造"一个初始现场,让新任务第一次被调度时能正确开始运行。Linux 和 FreeRTOS 在这里做的是同一件事(伪造初始栈现场),只是 Linux 更复杂。你在第16章学的东西,直接迁移到了 Linux。

29.10 进程退出与回收：exit 与 wait

进程结束和回收(呼应第28章僵尸进程):

  • exit(do_exit):进程主动结束(调 exit() 或 main 返回)或被信号杀死。内核释放它的大部分资源(地址空间、文件等),但保留 task_struct 和退出状态,进程进入僵尸(ZOMBIE)状态。
  • wait(wait/waitpid):父进程调用它来"收尸"——读取子进程的退出状态,然后内核才彻底释放子进程的 task_struct。
进程生命周期:
  fork/clone → 运行 → exit(变僵尸, 留退出码) → 父进程wait(读退出码) → 彻底释放

  若父进程不wait: 子进程一直是僵尸(占PID)
  若父进程先死:  子进程变孤儿, 被PID 1(init)收养, init会wait它们

为什么要留个僵尸阶段,不直接彻底删除?因为父进程可能想知道子进程"怎么死的"(正常退出?退出码几?被信号杀?)。这个信息必须保留到父进程来读。所以进程死后不能立即完全消失,要留个"尸体"(僵尸)带着退出信息,等父进程 wait 来读取和确认。这是 Unix 进程模型的一部分,也是为什么写多进程程序必须处理 wait(否则堆积僵尸)。

29.11 回扣 RTOS：创建任务的对比

把 Linux 和 RTOS 创建执行流的方式对照,理解各自的取舍:

RTOS(xTaskCreate)Linux(fork+exec / clone)
怎么创建直接给函数+栈,凭空建从已有进程分裂(fork)或指定共享(clone)
地址空间共享(所有任务一个)进程独立(fork)/线程共享(clone)
创建开销小(分配栈+TCB)较大(但 COW 让 fork 不算太慢)
初始现场pxPortInitialiseStack 伪造copy_thread 伪造(同思想)
回收vTaskDelete(简单)exit→僵尸→父进程 wait(有语义)

核心差异还是那句话:RTOS 任务共享地址空间、创建简单直接;Linux 进程要隔离(独立地址空间),所以有 fork(复制)这套更重但更强大的机制,配合 COW 优化开销。但底层的"伪造初始栈现场让新执行流能第一次跑起来"是完全一样的(pxPortInitialiseStack vs copy_thread)。你在 FreeRTOS 学的内核骨架,在 Linux 里全都能找到对应,只是穿了更复杂的外衣。下一章看 Linux 怎么调度这些进程/线程(CFS)——你会看到,它和 FreeRTOS 的位图调度是同一个问题的两种不同答案。

29.12 面试常问问题

Q1:fork 做了什么?为什么返回两次?
复制当前进程产生子进程(用 COW)。一次调用后内存里有两个进程,都从 fork 返回处继续;父进程里返回子 PID,子进程里返回 0,用返回值区分身份。

Q2:fork 为什么快?写时复制原理?
不真复制内存,父子页表共享同一批物理页并标只读;谁写触发缺页异常才复制那一页。只复制被改的页,故快而省(第十章)。

Q3:exec 做了什么?为什么不返回?
把当前进程的地址空间换成新程序(代码/数据全替换),PID 等不变,从新程序入口执行。原代码已被替换,没有"返回处",故成功不返回。

Q4:为什么 shell 用 fork+exec 而不是一步创建?
fork 和 exec 之间的空隙可做定制(重定向 stdin/stdout、设环境变量、改目录),这是管道/重定向的实现基础,灵活性高。

Q5:进程和线程的区别?底层如何统一?
底层都是 clone,用 flags 区分共享哪些资源。进程 clone 几乎不共享(独立地址空间);线程 clone 共享地址空间(CLONE_VM)、文件、信号等。共享 mm 与否是根本分界。

Q6:僵尸进程和孤儿进程?
僵尸:进程 exit 后 task_struct 留着(存退出码)等父进程 wait 回收;父不 wait 就成僵尸占 PID。孤儿:父进程先死,子进程被 PID 1 收养,init 会 wait 它们。

Q7:pthread_create 底层是什么?线程共享和独立哪些资源?
底层是 clone(CLONE_VM|CLONE_FILES|CLONE_THREAD...)。共享地址空间/文件/信号,独立栈和寄存器现场。内核把线程当 task_struct 独立调度。

29.13 常见误区

  1. "fork 会立即复制全部内存"——用 COW,只复制页表,写时才复制被改的页。
  2. "fork 返回值是新进程还是老进程决定的"——同一 fork,父进程得子 PID,子进程得 0,是两个进程的两次返回。
  3. "exec 会创建新进程"——不,exec 替换当前进程的内容,PID 不变,不创建新进程。
  4. "进程和线程是内核里两种不同的东西"——都是 task_struct,clone flags 区分。
  5. "exec 成功后会返回继续执行原代码"——不返回,原代码已被替换。
  6. "僵尸进程占用大量内存"——主要占 PID 和一个 task_struct,危害是耗尽 PID。
  7. "多线程共享栈"——线程共享地址空间但各有独立栈;共享的是全局/堆。

29.14 本章总结

  • Linux 进程只能由已有进程分裂产生,构成以 PID 1 为根的进程树。
  • fork:复制当前进程产生子进程,一次调用返回两次(父得子 PID,子得 0)。
  • 写时复制让 fork 快而省:父子共享只读页,写时才复制(第十章)。
  • exec:把当前进程内容换成新程序(PID 不变),成功不返回。
  • fork+exec 是启动新程序的标准范式;两步之间的空隙可做重定向/环境定制(shell 管道/重定向的基础)。
  • clone 是统一底层:用 flags 指定共享哪些资源。fork=不共享(进程),线程=共享地址空间等(CLONE_VM)。
  • pthread_create 底层是 clone;线程是内核级的、被独立调度的 task_struct。
  • copy_process 按 flags 复制/共享资源;copy_thread 伪造初始栈现场(同 FreeRTOS pxPortInitialiseStack)。
  • 退出回收:exit(变僵尸留退出码)→ 父进程 wait(读退出码并释放);不 wait 成僵尸,父先死成孤儿被 init 收养。

29.15 思维导图 / 课后题
