本章目标:上一章说"实时=消灭不确定的延迟"。这一章把"延迟"拆开——从一个事件(如传感器中断)发生,到对应任务真正开始处理,这段时间到底花在哪几个环节?每个环节都是一个可能失控的"不确定源"。看清延迟的完整分解(中断延迟、关中断、调度延迟、不可抢占、优先级反转、硬件抖动),你才知道该攻击哪里来降低最坏延迟。这是实时系统调优的地图。

51.1 延迟：实时的头号敌人

实时系统的核心指标是最坏响应延迟——从事件发生到系统完成响应的最长时间。它必须 ≤ deadline(第50章)。要保证这一点,先得知道这段延迟由哪些部分组成、每部分的最坏值是多少。

这就像做预算:要保证"总花费不超过 10ms",你得列出每一笔开销(中断响应花多少、调度花多少、执行花多少),把每笔的最坏值加起来,看总和是否 ≤ 10ms。任何一笔开销失控(比如某处关中断 5ms),整个预算就爆了。所以本章的任务是把"响应延迟"这笔总账,拆成一个个可分析、可优化的子项——只有拆开,才能逐项攻击、逐项保证上界。

51.2 生活例子：120 急救的时间线

你突发疾病打 120(事件发生),到医生开始救治(响应完成),中间经历:

打120 ─────────────────────────────→ 医生开始救治
  │ ① 接线员接电话的时间(可能占线等待)   ← 中断延迟
  │ ② 派车、救护车赶来的路上(可能堵车)    ← 调度延迟
  │ ③ 救护车被别的紧急任务耽搁            ← 抢占/优先级问题
  │ ④ 到医院还要等有空的手术室            ← 资源等待(锁)
  ▼
  总延迟 = 各环节之和; 要保证"黄金抢救时间内",
          每个环节都不能失控(占线太久?堵车?手术室被占?)

关键:总延迟是各环节延迟的累加,而"能不能在黄金时间内救治"取决于每个环节的最坏情况。接线占线久(中断延迟)、堵车(调度延迟)、救护车被别的事耽搁(抢占问题)、手术室被占(锁/资源等待)——任何一环卡住,总时间就爆。实时系统的响应延迟也是这样层层累加的,本章就是分析计算机里对应的每一环。

51.3 响应延迟的完整分解

把"事件发生→任务处理完"的延迟完整拆解,这是本章的核心地图:

事件发生(如传感器中断触发)
  │
  ├─① 中断延迟: 到CPU开始执行ISR
  │    包含: 当前指令执行完 + 更高优先级中断 + ★【关中断】时间
  │
  ├─② ISR执行: 上半部处理(第36章)
  │
  ├─③ 调度延迟: ISR唤醒了实时任务, 到该任务真正上CPU运行
  │    包含: 调度器决策 + ★【不可抢占段】+ ★【优先级反转】等待
  │
  └─④ 任务执行: 实时任务跑完(受Cache/TLB/换页等硬件抖动影响)

  最坏响应延迟 = ①+②+③+④ 各自的最坏值之和
  实时系统的工作: 让每一项都有确定的、尽量小的上界

这张分解图是整个实时调优的地图。响应延迟分四段:中断延迟(事件到 ISR 开始)、ISR 执行、调度延迟(ISR 唤醒任务到任务上 CPU)、任务执行。其中最危险的三个"不确定源"我用★标出:关中断时间(藏在①里)、不可抢占段和优先级反转(藏在③里)、硬件抖动(藏在④里)。这些正是让最坏延迟失控的元凶。实时系统工程,就是逐项限定这四段的上界,尤其是消灭那三个★不确定源。下面逐个深入。记住这张图,你就有了分析任何实时延迟问题的框架:出问题时,问"是哪一段爆了?"

51.4 中断延迟

第一段:中断延迟(interrupt latency)——从中断信号产生,到 CPU 真正开始执行 ISR 的时间。它由几部分组成:

中断延迟 = ① 当前指令执行完(CPU在指令边界才响应中断, 第七章)
         + ② 更高优先级中断先被处理(如果有)
         + ③ ★【中断被屏蔽的时间】(关中断期间, 中断被挂起)
         + ④ 进入ISR的固定开销(保存现场、查向量表, 第七/12章)

  ①④ 通常小且固定(几十~几百ns), 可预测
  ② 取决于中断优先级设计
  ③ ★是最大变数! 如果有代码关中断很久, 中断就被压着

中断延迟里,固定开销(保存现场、查中断向量表,第七/12章)是可预测的小量;真正的变数是"中断被屏蔽的时间"(③)。如果系统中有某段代码关闭了中断很长时间,那么在这期间来的中断只能干等着,直到中断被重新打开——这段关中断时间直接加到了中断延迟上,而且往往是最大、最难控制的部分。所以降低中断延迟的关键,是缩短关中断的时间(下一节)。硬件层面,GIC(第13章)的中断优先级和分发也影响②。

51.5 关中断：最隐蔽的杀手

把关中断单独拎出来讲,因为它是实时延迟最隐蔽、最致命的来源:

为什么要关中断? 保护临界区(第37章 spin_lock_irqsave):
  当代码访问"中断和主流程共享的数据"时, 要关中断防打断

问题: 关中断期间, 【所有】中断都被挡住!
  spin_lock_irqsave(&lock, flags);   // 关中断
     ... 临界区 ...                   // 如果这里执行很久
                                      // (比如遍历长链表、复杂计算)
  spin_unlock_irqrestore(&lock, flags); // 才开中断
   → 这整段时间, 紧急中断(如飞控传感器)都进不来!
   → 关中断500us = 中断延迟至少+500us

实时系统的铁律: 关中断的临界区必须【极短】
  某个驱动/内核路径一个长的关中断段, 就能毁掉整个系统的实时性

关中断是实时性的头号隐形杀手:一段长时间关中断的代码,会让这期间所有中断(包括最紧急的)都被延迟。而且它隐蔽——可能藏在某个不起眼的驱动、某个内核锁的临界区里,平时看不出问题,直到某个最坏时刻紧急中断恰好撞上它,延迟就爆了。这就是为什么第37章反复强调"自旋锁临界区要极短"、第36章强调"ISR 要短":它们本质都是在限制关中断/不可抢占的时间。实时系统里,评估一个驱动能不能用,一个关键指标就是它的最长关中断时间。原生 Linux 有些路径关中断较久(为了简单/性能),这正是它不硬实时的原因之一;PREEMPT_RT(第55章)的一大工作就是把这些长关中断段拆短或改成可抢占。降低关中断时间,是提升实时性最直接有效的手段。

51.6 调度延迟

第三段:调度延迟(scheduling latency)——从实时任务"就绪"(被 ISR 唤醒)到它"真正在 CPU 上运行"的时间。这是实时性最核心的指标之一。

调度延迟 = ① 调度器做决策的时间(选出下一个任务, 第17/30章)
         + ② ★等待当前任务让出CPU:
              - 当前任务在【不可抢占的内核段】→ 要等它出来
              - 当前是更高优先级任务 → 合理, 但要有界
         + ③ ★优先级反转: 实时任务等一个被低优先级持有的锁

  理想(RTOS): 最高优先级任务就绪 → 立即抢占, 调度延迟极小且确定
  现实(通用OS): 可能因不可抢占段、反转 等, 延迟大且不定

调度延迟回答一个关键问题:"最紧急的任务就绪后,多久能真正开始跑?"理想的实时系统(RTOS,第17章严格优先级抢占)应该是"立即"——最高优先级任务一就绪就抢占当前任务上 CPU,调度延迟极小且确定。但现实中有两个拖后腿的:不可抢占段(当前任务正在内核里执行不能被打断的代码,第51.7)和优先级反转(第51.8)。这两个是调度延迟里的不确定源。降低调度延迟,就是减少不可抢占的时间、消除优先级反转。cyclictest(51.10)测的主要就是调度延迟。

51.7 不可抢占：内核临界区

调度延迟的一大来源——不可抢占段:内核执行某些代码时,不允许被抢占,高优先级任务只能等。

为什么内核有不可抢占段?
  ① 持有自旋锁时(第37章): 不能被抢占(否则死锁/长等)
  ② 中断上下文/软中断处理中(第36章): 不是进程, 不能被调度切换
  ③ 显式禁止抢占(preempt_disable)保护per-CPU数据等

  后果: 高优先级实时任务就绪了, 但当前任务正在这些段里,
        调度器不能立即切换 → 实时任务干等 → 调度延迟增大

  这段"不能抢占"的最长时间, 直接决定了调度延迟的下界
  原生Linux这类段较多且较长 → 实时性差
  PREEMPT_RT(第55章): 把大部分不可抢占段改成可抢占(如自旋锁改可睡眠锁)

不可抢占段是"当前任务霸占 CPU、高优先级任务只能等"的时间。持自旋锁(第37章)、中断/软中断上下文(第36章)、显式 preempt_disable 都会造成不可抢占。这段时间越长,调度延迟越大。原生 Linux 为了性能和简单,有不少较长的不可抢占段(如某些遍历、批处理)——这是它不硬实时的另一主因。PREEMPT_RT(第55章)的核心改造之一,就是把内核里大量不可抢占的地方变成可抢占(比如把自旋锁改成可以睡眠、可被抢占的锁),从而大幅降低调度延迟。理解不可抢占段,你就理解了 PREEMPT_RT 到底在改什么。

51.8 优先级反转的延迟

调度延迟里最"阴险"的一种——优先级反转(priority inversion),第21章见过,这里从延迟角度看(第53章详解):

优先级反转: 高优先级任务被低优先级任务"堵住"

  低优先级任务L 持有锁M(在临界区里)
  高优先级任务H 就绪, 要这个锁M → 被迫等L释放
    → H(高)竟然在等L(低)! 优先级"反转"了
  更糟(无界反转): 中优先级任务Mid抢占了L,
    → L迟迟跑不完、不释放锁 → H等的时间【不可控】!
    → 高优先级任务H的延迟被一个中优先级任务无限拉长

  这是实时系统的经典灾难(火星探路者就栽在这, 第53章)
  解法: 优先级继承 / 优先级天花板(第53章)

优先级反转让高优先级任务被迫等待低优先级任务,破坏了"高优先级优先"的实时保证。更可怕的是"无界优先级反转":如果中优先级任务插进来抢占持锁的低优先级任务,高优先级任务的等待时间就不可控了——这直接摧毁实时性(延迟无上界)。这是实时系统必须解决的经典问题(NASA 火星探路者号就因此死机重启,第53章会详讲这个著名案例)。解法是优先级继承(第21章、第53章)或优先级天花板。从延迟角度看,优先级反转是调度延迟里可能失控的一项,必须用继承等机制把它限定成有界。

51.9 硬件层的不确定：Cache/TLB/换页

第四段(任务执行)里也藏着不确定——来自硬件的时间抖动:

硬件不确定源为什么造成延迟抖动章节
Cache 未命中命中读 Cache 快,未命中读内存慢几十倍,时间波动大第八章
TLB 未命中地址翻译缓存未命中要查页表,慢第九章
页面换出(换页)访问被换出到磁盘的内存,缺页要从磁盘调入,极慢且不定第十章
动态内存 malloc可能触发缺页、碎片整理,时间不定第31章
DVFS/频率调节CPU 动态调频,执行时间随频率变-

即使任务开始运行了,硬件层面的不确定性仍会让执行时间波动。Cache/TLB 命中与否差几十倍(第八/九章)、访问被换出的内存要从磁盘调回(第十章,可能几毫秒)、malloc 可能触发缺页或碎片整理(第31章)、CPU 动态调频改变执行速度。这些让"任务执行时间"本身就不确定,给 WCET 分析(第54章)带来困难。实时系统的应对手段:用 mlockall 锁定内存防换页、预热 Cache、用内存池代替 malloc(第24章)、关闭 DVFS 固定频率、甚至用 Cache 锁定。核心思想还是第50章那句:宁可平均慢(比如不用动态优化),也要时间可预测。这些硬件不确定源虽然单个不大,但累加起来影响 WCET,实时严格的系统必须逐个处理。

51.10 如何测量延迟：cyclictest

理论要落地——怎么实际测一个系统的延迟?标准工具是 cyclictest:

cyclictest 的原理:
  ① 起一个高优先级实时线程, 让它周期性睡眠(如每1ms)
  ② 睡眠到点被唤醒时, 测"实际唤醒时刻" - "本应唤醒时刻"
     = 这次的【调度延迟】
  ③ 大量重复, 统计: 最小/平均/【最大】延迟

  cyclictest -p 99 -i 1000 -l 100000   # 优先级99, 每1ms, 跑10万次
  输出: Min / Avg / Max 延迟(微秒)
  ★关键看 Max! 它就是最坏调度延迟

  典型结果:
   原生Linux:  Max可能 几ms ~ 几十ms(不定, 有长尾)
   PREEMPT_RT: Max通常 < 100us(有界, 可用于软/部分硬实时)
   RTOS:       更小更稳定

cyclictest 是评估 Linux 实时性的黄金标准:它模拟一个实时任务周期性唤醒,测量"实际唤醒 vs 应该唤醒"的偏差(即调度延迟),重复大量次数看最大值。关键看 Max(最坏延迟)——平均值再好,只要 Max 有长尾(偶尔几十 ms),就不能用于硬实时。原生 Linux 的 Max 常有不可预测的长尾,PREEMPT_RT(第55章)能把 Max 压到 100us 以下且有界——这是判断"这个系统能不能满足某 deadline"的直接依据。做实时 Linux 调优,第一件事就是跑 cyclictest 看 Max,再针对性优化(找长关中断/不可抢占段)。车企做实时系统一定会用它。

51.11 回扣与降低延迟的手段

把延迟来源和对应的降低手段汇总(实用速查表):

延迟来源降低手段章节
关中断时间长缩短临界区、少用 irqsave、拆分第37章
ISR 太长上半部极短、重活推下半部/线程化中断第36/43章
不可抢占段PREEMPT_RT(自旋锁改可抢占)、缩短临界区第55章
优先级反转优先级继承/天花板第53章
换页/动态内存mlockall 锁内存、内存池、预分配第24/31章
Cache/TLB 抖动预热、Cache 锁定、CPU 隔离(isolcpus)第8/9章
调度不确定用实时调度类 SCHED_FIFO/RR、CPU 亲和第30/52章

这张表是实时系统调优的"作战手册"。降低最坏延迟,就是针对上面每个来源逐一处理:缩短关中断和不可抢占段(治本)、消除优先级反转(用继承)、锁定内存防换页、隔离 CPU 减少干扰、用实时调度策略保证抢占。回顾你会发现,这些手段几乎每一个都在前面章节出现过——第七卷不是引入全新技术,而是从"实时性"这个目标出发,把前面的知识重新组织成"如何保证时序"的方法论。本章给你的最重要能力是"延迟分解"的思维:遇到实时问题(如"偶尔响应超时"),不要瞎猜,而是按"中断延迟→ISR→调度延迟→执行"逐段测量分析,找到爆掉的那一段,再用对应手段治它。这种系统化的延迟分析和优化能力,正是车企实时工程师的核心价值。下一章讲如何用实时调度算法从理论上保证 deadline。

51.12 面试常问问题

Q1:从事件发生到任务处理完,延迟由哪几部分组成?
中断延迟(事件到 ISR 开始)+ ISR 执行 + 调度延迟(ISR 唤醒任务到任务上 CPU)+ 任务执行。最坏响应=各段最坏值之和,实时系统要限定每段上界。

Q2:中断延迟由什么决定?最大变数是什么?
当前指令执行完+更高优先级中断+进入 ISR 固定开销+关中断被屏蔽的时间。最大变数是关中断时间——某段长关中断会直接加到中断延迟上。

Q3:为什么关中断是实时性头号杀手?★
关中断期间所有中断被挡,一段长关中断代码让紧急中断也被延迟;且它隐蔽(藏在驱动/锁临界区),平时看不出直到最坏时刻。所以临界区必须极短。

Q4:调度延迟是什么?来源?
从实时任务就绪到真正上 CPU 运行的时间。来源:调度决策+等待当前任务让出 CPU(不可抢占段)+优先级反转。理想 RTOS 立即抢占,通用 OS 因不可抢占段/反转而大且不定。

Q5:什么是不可抢占段?怎么减少?
内核执行不能被抢占的代码(持自旋锁、中断/软中断上下文、preempt_disable)。这段时间高优先级任务只能等。PREEMPT_RT 把大部分改成可抢占(自旋锁改可睡眠锁)来减少。

Q6:硬件层有哪些延迟不确定源?怎么应对?
Cache/TLB 未命中、换页、动态内存、DVFS。应对:mlockall 锁内存防换页、内存池代替 malloc、预热/锁定 Cache、关 DVFS、CPU 隔离。

Q7:怎么测量 Linux 的实时延迟?
cyclictest:高优先级线程周期唤醒,测实际 vs 应该唤醒的偏差,看 Max(最坏调度延迟)。原生 Linux Max 有长尾,PREEMPT_RT 能压到 100us 以下有界。

51.13 常见误区

  1. "延迟就看平均值"——实时看最坏值(Max),平均好但有长尾也不行。
  2. "关中断没什么影响"——是实时头号隐形杀手,长关中断直接拉高中断延迟。
  3. "高优先级任务一定立即运行"——可能被不可抢占段、优先级反转挡住。
  4. "任务开始跑了延迟就确定了"——Cache/换页等硬件抖动仍影响执行时间。
  5. "cyclictest 平均低就实时好"——要看 Max,长尾才是实时杀手。
  6. "降延迟只靠调度器"——要综合治理关中断、不可抢占、反转、内存、Cache。

51.14 本章总结

  • 最坏响应延迟 = 中断延迟 + ISR 执行 + 调度延迟 + 任务执行,各段最坏值累加;实时=限定每段上界。
  • 中断延迟:固定开销(可预测)+ 关中断被屏蔽时间(最大变数)。
  • 关中断是头号隐形杀手★:关中断期间所有中断被挡,长关中断直接拉高延迟,且隐蔽——临界区必须极短(第37章)。
  • 调度延迟(实时任务就绪→上 CPU):来源=调度决策+不可抢占段+优先级反转。
  • 不可抢占段(持自旋锁/中断上下文/preempt_disable)让高优先级任务干等;PREEMPT_RT 改可抢占来减少。
  • 优先级反转:高优先级被低优先级持锁堵住,无界反转致延迟不可控(第53章),用继承/天花板解。
  • 硬件不确定源:Cache/TLB 未命中、换页、malloc、DVFS;应对=锁内存/内存池/预热/关调频/隔离 CPU。
  • 测量:cyclictest 看 Max(最坏调度延迟);原生 Linux 有长尾,PREEMPT_RT 有界(<100us)。
  • 核心能力:延迟分解思维——按段测量定位爆掉的环节,用对应手段治理。

51.15 思维导图 / 课后题

延迟 Latency 的来源
├─ 一、实时=限定最坏响应延迟, 先拆解各环节
├─ 二、★★完整分解:中断延迟+ISR执行+调度延迟+任务执行
├─ 三、中断延迟:固定开销 + ★关中断被屏蔽时间(最大变数)
├─ 四、★★关中断=头号隐形杀手:期间所有中断被挡, 临界区必须极短
├─ 五、调度延迟(就绪→上CPU):决策+★不可抢占段+★优先级反转

│      └─ 应对:mlockall/内存池/预热/关DVFS/隔离CPU
├─ 九、★测量:cyclictest 看Max(最坏调度延迟)
└─ 十、★延迟分解思维:按段定位爆点 + 对应手段(作战手册)
