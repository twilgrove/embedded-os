本章目标:第七章你学了中断的硬件原理,第34章你看到网卡接收用"硬中断+软中断"两段处理。这一章把 Linux 的中断处理机制讲透:为什么要拆成"上半部/下半部",SoftIRQ / Tasklet / Workqueue 三种下半部机制各是什么、怎么选。这是你在 STM32 里"中断只置 flag,主循环处理"那个朴素做法的系统化、工业级版本。

36.1 中断处理的两难

回顾第七章的核心原则:中断处理函数(ISR)要短要快。为什么?因为处理中断时通常要关闭(屏蔽)中断——如果 ISR 里干很久,这段时间别的中断都被挡着(丢事件、延迟飙升)。想象网卡收包 ISR 里把整个 TCP/IP 协议栈跑完(几十微秒),这期间别的中断全被压着,系统就"卡"了。

但矛盾来了:有些中断确实有很多活要干(网卡收包要拆协议栈、磁盘完成要处理数据)。又要响应快(ISR 短),又有很多活要干——怎么办?Linux 的解法是"分身":把中断处理拆成两半——紧急的一小部分立刻在中断里做(上半部),不紧急的大部分推迟到中断外做(下半部)。

36.2 生活例子：接电话记便签

你在专心工作,电话响了(中断)。笨做法:接起电话当场把对方要的事全办完(可能要查资料、打印、跑腿,半小时),这期间电话线一直占着,别人打不进来(中断被屏蔽,别的事件进不来)。

聪明做法:接起电话,快速记下"谁、要什么"到便签上,说"知道了我稍后处理",挂断(上半部:极短,几秒,立刻释放电话线)。然后等手头忙完,再照便签一件件办(下半部:慢慢干,不占电话线)。关键:接电话(响应中断)必须快,好让电话线尽快空出来;真正费时的活挪到"挂断后"做。这就是上半部/下半部的精髓。

36.3 上半部与下半部

把这套术语和职责固定下来:

上半部(Top Half)下半部(Bottom Half)
别名硬中断处理、ISR软中断处理、延后处理
何时执行中断发生时立刻,关中断环境稍后,开中断环境
要求极短极快可以耗时
能否睡眠/阻塞❌ 绝对不能看机制(softirq/tasklet 不能,workqueue 能)
典型工作应答硬件、存数据、标记"有活"、触发下半部真正的数据处理(拆协议栈等)

分工原则:上半部只做"必须立刻做且做不了多久会出事"的最小工作——比如读走网卡里的数据(不读硬件缓冲会溢出)、应答硬件、然后触发下半部并立刻返回。下半部做"能等一等的重活"。判断依据:这件事推迟几十微秒会不会出问题?不会→放下半部;会(如硬件缓冲要溢出)→放上半部。这个"紧急最小化 + 重活延后"的拆分,你在 STM32 写"中断里只把数据丢进队列/置 flag,主循环慢慢处理"时早就用过——Linux 只是给了它三种工业级的"下半部"实现。

36.4 Linux 中断处理全流程

串一遍一次中断在 Linux 里的完整旅程(结合第13章 GIC):

一次设备中断的完整流程:
  ① 设备拉中断线 → GIC(第13章)仲裁 → 送给某个CPU
  ② CPU跳到内核的中断入口(异常向量, 第12章)
  ③ 内核保存现场, 查中断号 → 找到注册的handler
  ④ 【上半部】执行设备的ISR(request_irq注册的):
        - 极短: 应答设备、读走紧急数据、
        - 触发下半部(raise softirq / schedule tasklet/work)
        - 返回
  ⑤ 中断返回前, 检查有没有待处理的软中断 → 有就执行【下半部】
  ⑥ 【下半部】(softirq/tasklet)在此处理, 或推给ksoftirqd线程
  ⑦ 恢复现场, 返回被打断的代码

注意第⑤步:下半部(softirq/tasklet)通常在"中断返回时"顺便执行——此时已经开中断了,可以慢慢干,还能被更高优先级中断打断。如果下半部太多处理不完,推给专门的内核线程 ksoftirqd(36.9)。整个流程把第12章异常入口、第13章 GIC、第七章中断原则都串起来了。

36.5 下半部三种机制

Linux 提供三种下半部机制,这是本章重点。先总览再逐个讲:

机制运行上下文能否睡眠并发特点典型用途
SoftIRQ中断上下文❌ 不能同类型可多核并行网络收发、块设备(高性能核心)
Tasklet中断上下文❌ 不能同一 tasklet 不会并行(串行化)一般驱动的下半部
Workqueue进程上下文(内核线程)✅ 能可调度、可阻塞需要睡眠/耗时长的下半部

核心分野是"运行在什么上下文":SoftIRQ/Tasklet 运行在"中断上下文"(不能睡眠),Workqueue 运行在"进程上下文"(内核线程里,能睡眠)。这个区别决定了一切:能不能睡眠、能不能调用可能阻塞的函数(如某些内存分配、等锁)。下面逐个看。

36.6 SoftIRQ 软中断

SoftIRQ 是最底层、性能最高的下半部机制,但也最"硬核"(限制多):

SoftIRQ 特点:
  ├─ 类型是【静态编译时定的】, 数量固定(如NET_RX/NET_TX/TIMER/BLOCK等)
  ├─ 运行在中断上下文, 【不能睡眠】
  ├─ 同一种softirq可以【在多个CPU上并行】运行 ★
  │    → 高性能, 但要自己处理并发(用per-CPU数据或锁, 第37章)
  └─ 一般驱动【不直接用】, 主要给网络/块设备等核心子系统

  cat /proc/softirqs  ← 看各类softirq在各CPU的计数

SoftIRQ 的杀手锏是"同类型可多核并行":两个 CPU 可以同时跑 NET_RX 软中断处理两个网卡的包——这对高吞吐网络至关重要(第34章接收就走 NET_RX)。但代价是你得自己处理并发安全(多核同时跑,共享数据要保护,第37章)。因为限制多、易出错,SoftIRQ 类型是内核编译时固定的,普通驱动不直接用,只有网络、块设备、定时器这些核心子系统用。普通驱动用它的"友好封装"——Tasklet。

36.7 Tasklet

Tasklet 是建立在 SoftIRQ 之上的"更好用的下半部",给普通驱动用:

Tasklet 特点(底层就是一种softirq):
  ├─ 运行在中断上下文, 【不能睡眠】(和softirq一样)
  ├─ 动态创建, 用起来简单(驱动常用)
  ├─ ★同一个tasklet【不会在多核并行】(自动串行化)
  │    → 你不用担心自己这个tasklet的并发, 省心
  └─ 不同tasklet可以在不同CPU并行

  用法: tasklet_init(&t, func, data);  在ISR里 tasklet_schedule(&t);

Tasklet 相比 SoftIRQ 的关键便利:同一个 tasklet 保证不会在两个 CPU 上同时运行(自动串行化)。这意味着你写 tasklet 处理函数时,不用担心它自己和自己并发——省去很多加锁的麻烦。代价是同一 tasklet 不能并行(损失一点并行度),但对普通驱动够用了。所以选择很清晰:要极致并行性能(网络)→SoftIRQ 但自己管并发;普通驱动图省心→Tasklet。两者都不能睡眠——要睡眠得用 Workqueue。

36.8 Workqueue 工作队列

前两者都不能睡眠,但有些下半部工作必须能睡眠(比如要申请可能阻塞的资源、要等一个锁、要做慢速 IO)。Workqueue 就是为此而生:

Workqueue 特点:
  ├─ ★运行在【进程上下文】(内核线程里, 如kworker)
  ├─ ★【可以睡眠、可以阻塞、可以被调度】 ← 关键区别!
  ├─ 因为在内核线程, 受调度器管理(第30章)
  └─ 用法: 定义work → schedule_work(&work) → kworker线程稍后执行

  代价: 有线程切换开销, 延迟比softirq/tasklet大一点

Workqueue 和前两者的本质区别:它把下半部工作交给一个内核线程(kworker)去执行,所以运行在"进程上下文",能睡眠、能阻塞、受调度器调度(第30章)。SoftIRQ/Tasklet 是"借中断返回的时机顺便执行"(中断上下文,不能睡);Workqueue 是"派个工人线程慢慢干"(进程上下文,能睡)。判断用哪个的黄金标准:下半部里需不需要睡眠?需要(等锁、慢 IO、可能阻塞的分配)→ 只能 Workqueue;不需要且要快 → SoftIRQ/Tasklet。代价是 Workqueue 有线程调度开销,延迟略大。ps aux | grep kworker 能看到这些工人线程。

36.9 ksoftirqd 内核线程

一个补充但重要的机制。SoftIRQ 通常在中断返回时执行,但如果软中断来得太猛(比如网络流量暴涨,NET_RX 软中断一个接一个),在中断返回时一直处理软中断,会导致用户进程饿死(CPU 全在处理软中断,没空跑进程)。

ksoftirqd 解决"软中断太多饿死进程":
  正常: 软中断在中断返回时就地处理(快)
  但若软中断处理不完(反复来):
    → 内核不再无限就地处理, 而是唤醒【ksoftirqd内核线程】
    → 剩下的软中断交给这个线程, 让它和普通进程【一起被调度】
    → 这样软中断不会完全抢占进程, 系统保持响应

  每个CPU一个: ksoftirqd/0, ksoftirqd/1, ...

ksoftirqd 是"软中断"和"调度公平"之间的平衡阀:平时软中断就地快速处理;一旦量大到威胁进程运行,就把它降级成一个受调度器管理的内核线程,和普通进程公平竞争 CPU。这解释了实际现象:高网络负载时你会看到 ksoftirqd 占很高 CPU——那是内核在用线程化的方式消化海量软中断,同时不让进程完全饿死。诊断"网络高负载下系统卡顿"时,ksoftirqd 的 CPU 占用是关键线索。

36.10 三者怎么选(决策)

把下半部选择做成决策流程,面试和实战都用得上:

下半部选哪个? 决策树:

  下半部里需要睡眠/阻塞吗?
   ├─ 需要(等锁/慢IO/可能阻塞的操作)
   │     → 用 Workqueue (进程上下文, 能睡)
   │
   └─ 不需要(纯计算/搬数据, 要快)
        │
        需要同类型多核并行的极致性能吗?
         ├─ 是(网络/块设备核心) → SoftIRQ(自己管并发)
         └─ 否(普通驱动) → Tasklet(串行化, 省心)

一句话记忆:要睡眠用 Workqueue;不睡眠图省心用 Tasklet;不睡眠要极致并行(核心子系统)用 SoftIRQ。90% 的普通驱动开发用 Tasklet 或 Workqueue 就够了,SoftIRQ 是内核核心子系统的专属。

36.11 回扣 STM32 / FreeRTOS / 网络

把这套机制和你已有的经验对齐:

概念你熟悉的对应
上半部要短STM32 里"ISR 只置 flag/丢队列"(第七章)
下半部延后处理STM32 里"主循环 while(1) 处理 flag"
Workqueue(线程化)FreeRTOS 里"中断给任务发通知,任务处理"(第22章 EventGroup / 任务通知)
SoftIRQ 处理网络第34章网卡 NET_RX 软中断拆协议栈
不能睡眠的上下文第七章"ISR 里不能阻塞"

你其实早就在用"上半部/下半部"思想了。在 STM32 裸机里,你写"串口中断只把字节存进环形缓冲区,主循环再解析协议"——这就是上半部(存字节)/下半部(解析)!在 FreeRTOS 里,你写"中断里 xTaskNotifyFromISR 通知任务,任务被唤醒后处理"——这就是 Workqueue 式的"中断触发、线程处理"(第22章)!Linux 的 SoftIRQ/Tasklet/Workqueue,只是把这个你早已在用的朴素思想,做成了三种正规、可复用、考虑了多核的机制。理解了这层对应,你不是在学一个陌生的新东西,而是在给已有直觉配上专业工具和术语。这正是本书从 RTOS 通向 Linux 的意义。

36.12 面试常问问题

Q1:为什么中断要分上半部和下半部?
ISR 要短(处理时常关中断,久了丢事件、延迟大),但有些中断有重活。所以拆:上半部在中断里做紧急最小工作并触发下半部;下半部延后做重活(开中断环境,可耗时)。

Q2:上半部有什么限制?
极短、不能睡眠/阻塞、通常关中断。只做应答硬件、读走紧急数据、触发下半部。

Q3:下半部三种机制及区别?
SoftIRQ(中断上下文,不能睡,同类型可多核并行,自管并发,核心子系统用);Tasklet(基于 softirq,不能睡,同一 tasklet 串行化省心,普通驱动用);Workqueue(进程上下文/内核线程,能睡能阻塞,受调度)。

Q4:怎么选下半部机制?
需要睡眠→Workqueue;不睡且要极致并行(网络)→SoftIRQ;不睡且图省心(普通驱动)→Tasklet。

Q5:SoftIRQ 和 Tasklet 最大区别?
同一 SoftIRQ 类型可在多核并行(高性能但要自己处理并发),同一 Tasklet 不会并行(自动串行化,省心)。Tasklet 底层就是一种 softirq。

Q6:ksoftirqd 是什么?什么时候出现?
处理软中断的内核线程(每核一个)。软中断太多(如网络暴涨)时,内核不再无限就地处理,而唤醒 ksoftirqd 让软中断和进程一起被调度,防止进程饿死。高网络负载时它 CPU 占用高。

Q7:Workqueue 为什么能睡眠而 tasklet 不能?
Workqueue 运行在内核线程(进程上下文),受调度器管理,睡眠就是让出 CPU;tasklet 运行在中断上下文,没有可让出的进程实体,睡眠会导致系统错乱。

36.13 常见误区

  1. "下半部都不能睡眠"——SoftIRQ/Tasklet 不能,但 Workqueue(进程上下文)能睡。
  2. "Tasklet 和 SoftIRQ 完全不同"——Tasklet 底层就是一种 softirq,是它的易用封装。
  3. "同一个 tasklet 会多核并行"——不会,自动串行化;不同 tasklet 才可并行。
  4. "上半部可以做很多工作"——上半部要极短,重活必须推给下半部。
  5. "ksoftirqd 高 CPU 是 bug"——通常是网络/软中断负载高的正常表现(也可能提示需优化)。
  6. "中断上下文能调用可能睡眠的函数"——不能(如某些 GFP_KERNEL 分配、mutex),会导致内核错误。

36.14 本章总结

  • 矛盾:ISR 要短(处理时关中断,久了误事)vs 有些中断有重活 → 拆成上半部/下半部。
  • 上半部(硬中断/ISR):极短、不能睡、应答硬件+读紧急数据+触发下半部。下半部:延后做重活。
  • 下半部三机制:SoftIRQ(中断上下文,不睡,同类型多核并行,自管并发,核心子系统);Tasklet(基于 softirq,不睡,同一个串行化省心,普通驱动);Workqueue(进程上下文/内核线程,能睡,受调度)。
  • 选择:要睡→Workqueue;不睡要极致并行→SoftIRQ;不睡图省心→Tasklet。核心分野是"运行在中断上下文还是进程上下文"。
  • ksoftirqd:软中断太多时线程化处理,让软中断和进程公平调度,防饿死;高网络负载时 CPU 占用高。
  • 回扣:这就是 STM32"ISR 置 flag / 主循环处理"、FreeRTOS"中断通知任务"的系统化版本;网络 NET_RX 走 SoftIRQ(第34章)。
