第廿三章 SoftwareTimer 软件定时器

不占硬件定时器，也能安排"多久之后干件事"

本章目标：你经常要"5 秒后关灯""每 100ms 采样一次"这类定时任务。硬件定时器数量有限、还要写中断,用起来重。软件定时器让你创建任意多个"软"定时器,到期自动调用你的回调,不占硬件、不写中断。学完你要理解它靠一个"守护任务+命令队列"实现、以及回调函数的严格禁忌(不能阻塞)。这一章较短。

23.1 硬件定时器为什么不够用

MCU 有硬件定时器(TIM),但有局限:

  • 数量有限:一个芯片就几个 TIM,被 PWM、编码器、输入捕获等占用后,没剩几个给你做"延时办事"。
  • 用起来重:每个都要配寄存器、写中断服务程序、处理中断上下文限制(第七章)。想要 20 个不同的定时逻辑?写 20 个中断?不现实。

而很多需求其实不需要硬件级的精度,只要"大概几秒后/每隔多久做件事"——比如超时检测、周期上报、LED 慢闪、看门狗喂狗。软件定时器就是用软件模拟出任意多个这种"低精度但够用"的定时器,全部复用系统那一个 SysTick 心跳(第18章),不额外占硬件。

23.2 软件定时器的核心思想

软件定时器的原理其实你已经见过——和第17章 vTaskDelay 的延时列表一模一样:维护一个按到期时间排序的定时器列表,每个 tick 检查有没有到期的,到期就调用它的回调函数。

软件定时器 = 到期时间排序的列表 + tick驱动 + 回调

  定时器列表(按到期时间排序):
    ┌──────────────────────────────┐
    │ Timer A → 到期tick=1050        │ ← 最早到期, 在头部
    │ Timer B → 到期tick=1200        │
    │ Timer C → 到期tick=2000        │
    └──────────────────────────────┘
  每个tick检查: 当前tick >= 头部到期时间? → 调用其回调, 从列表移除(或周期性重新插入)

创建/启动一个定时器,就是算出它的到期时间、插入这个有序列表。到期时,调用你注册的回调函数(pxCallbackFunction)。一个 SysTick 心跳,驱动所有软件定时器——这就是"用一个硬件资源模拟出无数个软定时器"的本质。

23.3 守护任务：Timer Daemon

关键问题:这个"检查到期、调用回调"的活谁来干?不能在 SysTick 中断里干(回调可能耗时、还可能调用会阻塞的 API)。FreeRTOS 的做法:专门起一个任务——定时器守护任务(Timer Daemon Task,也叫 Timer Service Task)来管所有软件定时器。

vTaskStartScheduler() 时若开启软件定时器, 会自动创建:
  → 守护任务 prvTimerTask (优先级 configTIMER_TASK_PRIORITY)

守护任务的工作(简化):
  for(;;) {
    ① 算出"下一个定时器到期"还有多久
    ② 阻塞等待: 要么等到那个到期时间, 要么等命令队列来消息(取较早者)
    ③ 醒来后:
       - 若是定时器到期 → 调用它的回调函数(在守护任务上下文!)
       - 若是命令队列有消息 → 处理命令(启动/停止/复位定时器)
  }

回调函数在守护任务的上下文里执行,不是在中断里!这是理解软件定时器的关键。所以回调的优先级 = 守护任务的优先级(configTIMER_TASK_PRIORITY)。这意味着:①如果守护任务优先级低,回调可能被高优先级任务延迟(定时不准);②回调里绝对不能做太久的事(会拖累所有其他定时器)。软件定时器的精度和及时性,取决于守护任务能否及时被调度——这是它"软"的代价,也是它和硬件定时器的本质区别。

23.4 命令队列：一切操作先入队

一个精妙设计:你调 xTimerStart/Stop/Reset 等 API,并不直接操作定时器列表,而是往一个"定时器命令队列"发一条消息,由守护任务去实际执行。

你的任务调 xTimerStart(xTimer, ...):
    │
    ▼ 不直接启动! 而是:
  往命令队列发一条消息 { 命令=START, 定时器=xTimer }
    │
    ▼
  守护任务从命令队列收到消息 → 真正把定时器插入到期列表

为什么要绕这一圈(命令队列)?
  → 保证所有对定时器列表的操作【都在守护任务里串行进行】
  → 不用加锁! 天然线程安全(只有守护任务碰这个列表)

这是用"单线程串行化"代替"加锁"的经典并发设计:所有对定时器列表的修改都汇集到守护任务一个地方顺序执行,就没有多任务同时改列表的竞态,连锁都省了。"把共享数据的操作集中到一个专门的任务/线程,用消息传递代替共享+加锁"是一种优雅的并发范式(actor 模型的雏形),你在很多高质量系统里会见到它。命令队列也解释了为什么这些 API 有 xTicksToWait 参数——命令队列可能满,发命令本身可能要等。

23.5 数据结构与到期列表

定时器结构 Timer_t(timers.c,精简):

typedef struct tmrTimerControl
{
    const char       *pcTimerName;
    ListItem_t        xTimerListItem;    /* 挂进到期列表的钩子 */
    TickType_t        xTimerPeriodInTicks; /* 周期(多少tick到期一次) */
    void             *pvTimerID;         /* 用户ID(区分定时器/传数据) */
    TimerCallbackFunction_t pxCallbackFunction; /* ★到期回调函数 */
    UBaseType_t       uxAutoReload;      /* 单次 还是 周期(auto-reload) */
} Timer_t;

和 vTaskDelay 一样,FreeRTOS 用两个定时器列表 + 溢出处理(pxCurrentTimerList / pxOverflowTimerList),因为 tick 计数会回绕(第17章讲过完全相同的技巧)。到期列表按到期时间排序,守护任务只需看头部。你会发现整个第三卷反复出现同一套机制:有序列表 + tick 驱动 + 双列表处理溢出——延时任务、软件定时器都是它。看懂一处,处处贯通。

23.6 单次 vs 周期

由 uxAutoReload 决定两种模式:

模式行为用途
单次(One-shot)到期调一次回调,然后停止超时检测、延迟执行一次(如"3秒后关背光")
周期(Auto-reload)到期调回调后,自动重新计时,循环往复周期采样、周期上报、心跳(如"每100ms采一次")

周期模式的实现:回调执行后,守护任务把它以(当前到期时间 + 周期)重新插回到期列表——注意是基于"上次到期时间"而非"当前时间"重新计算,这样能避免周期漂移(如果基于当前时间,回调本身的延迟会累积成误差)。这是个容易忽略但重要的精度细节。

23.7 回调函数的禁忌

软件定时器回调有严格规则,违反会出严重问题:

  • 回调里绝不能阻塞!回调在守护任务上下文跑,如果它阻塞(如 take 一个拿不到的信号量、vTaskDelay),整个守护任务被卡住,所有其他软件定时器都停摆!这是最常见的坑。
  • 回调要短:回调期间守护任务不能处理其他到期定时器和命令,拖长会让其他定时器不准。
  • 回调里可以调用非阻塞的 API(如 xQueueSend 超时设 0)。

这条禁忌和第七章"中断里不能阻塞"、第20章"信号量同步"是一脉相承的:某些执行上下文(中断、软件定时器回调)是"公共资源",一旦阻塞就会拖垮依赖它的所有东西。软件定时器回调虽然在任务上下文(不像中断那么严格),但因为是共享的守护任务,一个回调阻塞会连累所有定时器,所以同样要求"绝不阻塞、尽量短"。如果定时到期要做耗时/可能阻塞的事,正确做法是:回调里只发个信号量/消息,唤醒另一个专门的任务去做——又是"回调/中断只触发,重活交给任务"的一贯模式。

23.8 为什么不在中断里直接调回调

回顾:硬件定时器中断可以直接在 ISR 里做事,为什么软件定时器要绕守护任务?两个原因:

  1. 回调可能耗时/想调用 FreeRTOS API:中断上下文限制多(第七章),不能阻塞、调 API 受限。放到任务上下文,回调能做的事更多、更从容。
  2. 操作定时器列表需要串行化:如果在 SysTick 中断里改列表,又在任务里通过 API 改,会有竞态。集中到守护任务就没这问题(23.4)。

代价就是 23.3 说的:精度和及时性依赖守护任务的调度。如果你需要高精度、硬实时的定时(如电机换相、精确 PWM),必须用硬件定时器,软件定时器达不到。软件定时器适合"低精度、要很多个、逻辑复杂"的场景;硬件定时器适合"高精度、硬实时、数量少"。

23.9 和硬件定时器怎么配合

实际工程里两者配合:

软件定时器硬件定时器
精度低(tick 级,依赖守护任务调度)高(时钟周期级,确定)
数量任意多(只受内存限)有限(几个)
回调上下文守护任务(可调 API,不能阻塞)中断(严格受限)
典型用途超时、周期上报、慢闪、看门狗喂狗PWM、电机换相、精确采样、编码器

经验法则:能用软件定时器就用(省硬件、好写);要精度/硬实时才用硬件定时器。甚至可以组合:硬件定时器中断里 give 信号量/发命令,由软件层处理——各取所长。飞控里,高频控制环用硬件定时器保证精度,而各种超时监控、状态上报用软件定时器,分工明确。

23.10 面试常问问题

Q1:软件定时器和硬件定时器的区别?
软件定时器用一个守护任务+SysTick 心跳模拟,数量任意、精度低(依赖任务调度)、回调在任务上下文;硬件定时器精度高、数量有限、回调在中断。见 23.9。

Q2:软件定时器的回调在什么上下文执行?有什么限制?
在定时器守护任务上下文,不是中断。绝不能阻塞(会卡住所有定时器),要短;要做耗时/阻塞的事应发信号唤醒别的任务。

Q3:为什么 xTimerStart 等 API 不直接操作定时器,而走命令队列?
让所有对定时器列表的操作都在守护任务里串行执行,天然线程安全、免加锁。用消息传递代替共享+锁。

Q4:软件定时器精度为什么不高?
回调由守护任务执行,若守护任务被更高优先级任务抢占或有回调拖长,到期处理会延迟;精度是 tick 级且不确定。要高精度用硬件定时器。

Q5:周期定时器如何避免漂移?
基于"上次到期时间+周期"重新计算下次到期,而非基于当前时间,避免回调延迟累积成误差。

Q6:回调里能调用 vTaskDelay 吗?
不能,会阻塞守护任务导致所有定时器停摆。回调只能用非阻塞操作。

23.11 常见误区

  1. "软件定时器回调在中断里执行"——在守护任务上下文,不是中断。
  2. "回调里可以阻塞/延时"——绝对不行,会卡住所有软件定时器。
  3. "软件定时器精度和硬件一样"——低得多,依赖守护任务调度,不适合硬实时。
  4. "xTimerStart 立即启动定时器"——是发命令给守护任务,略有延迟;命令队列满还可能失败。
  5. "周期定时器基于当前时间重算"——基于上次到期时间,避免漂移。
  6. "软件定时器不占任何资源"——占守护任务的栈和 CPU、命令队列内存,只是不占硬件 TIM。

23.12 本章总结

  • 软件定时器用一个守护任务 + SysTick 心跳模拟任意多个"软"定时器,不占硬件 TIM。
  • 原理=到期时间排序列表 + tick 驱动 + 回调(同 vTaskDelay 的延时列表,含双列表处理溢出)。
  • 守护任务(Timer Daemon)执行回调,回调在任务上下文(非中断),优先级=configTIMER_TASK_PRIORITY。
  • 命令队列:所有 xTimerStart/Stop 等操作先入队,由守护任务串行执行——免加锁的并发设计。
  • 单次(one-shot)vs 周期(auto-reload);周期基于上次到期时间重算,避免漂移。
  • 回调绝不能阻塞(会卡住所有定时器),要短;耗时/阻塞的事发信号交给别的任务。
  • 精度低、依赖守护任务调度;高精度/硬实时用硬件定时器。
  • 配合:软件定时器管超时/周期上报/慢闪,硬件定时器管 PWM/换相/精确采样。

课后练习

  1. 在 timers.c 找 Timer_t、prvTimerTask(守护任务)、命令队列 xTimerQueue、prvProcessExpiredTimer,对照本章读。
  2. 创建单次和周期两个软件定时器,回调里翻转不同 LED,观察行为差异。
  3. 故意在回调里加 vTaskDelay/长循环,观察其他软件定时器被拖延甚至停摆,亲历"回调不能阻塞"。
