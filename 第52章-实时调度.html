本章目标:实时系统里有多个任务(飞控:姿态环、位置环、遥控解析、日志……),各有周期和 deadline。怎么安排它们在 CPU 上的执行顺序,才能保证每个任务都不错过 deadline?这就是实时调度理论。本章讲两大经典算法——RMS(单调速率)和 EDF(最早截止期),以及如何用可调度性分析提前证明"这组任务一定不会误点"。这是实时系统从"能跑"到"可证明"的关键,面试常考。

52.1 实时调度要回答的问题

第30章讲过通用调度(CFS 追求公平)。实时调度目标完全不同:不是公平,而是保证每个任务在 deadline 前完成。它要回答两个问题:

  • 调度策略:多个任务就绪时,先运行哪个?(按什么规则排优先级)
  • 可调度性分析:给定一组任务(各自的周期、执行时间、deadline),用某调度策略,能不能保证都不误点?——这能在设计阶段数学证明,不用等运行出事。

实时调度最珍贵的能力是"可调度性分析"——在系统上线前,用数学证明"这组任务一定都能满足 deadline"。这对硬实时至关重要:你不能靠"跑跑看没出事"来保证飞控不炸机,而要提前证明最坏情况下也不误点。RMS 和 EDF 之所以经典,不只因为它们是调度规则,更因为它们有配套的可调度性判据(52.6、52.7),让你能算出"行还是不行"。这是实时系统工程的科学基础。

52.2 生活例子：护士排班

一个护士(CPU)要照顾几个病人(任务),每个病人有不同的照顾频率和紧迫性:病人 A 每 2 小时要量一次血压(周期 2h)、病人 B 每 5 小时喂一次药、病人 C 每 10 小时换一次液……每次照顾要花一定时间,而且必须在下一个周期到来前完成(错过就出问题)。

护士怎么排班,才能保证每个病人都按时被照顾到?一种思路:频率高的优先(A 每 2 小时就要,最频繁,给最高优先级)——这就是 RMS。另一种思路:看谁的"下一次截止时间"最近就先做谁(哪个病人马上就要错过了,先救他)——这就是 EDF。而且护士上岗前,护士长要能算出"以现在的病人数和照顾时间,一个护士忙得过来吗(会不会有人被耽误)"——这就是可调度性分析。本章就是把这套排班的智慧形式化。

52.3 周期任务模型

实时调度分析建立在一个标准模型上——周期任务:

每个实时任务用几个参数描述:
  ★T (Period, 周期):    每隔T时间, 任务要执行一次
  ★C (Computation, 执行时间): 每次执行需要的CPU时间(常用WCET, 第54章)
  ★D (Deadline, 截止期): 每次必须在触发后D时间内完成(常D=T)
  U (Utilization, 利用率): U = C/T (这个任务占用CPU的比例)

  例: 姿态控制任务 T=2ms, C=0.5ms
     → 每2ms要跑一次, 每次0.5ms, 利用率 U=0.5/2=25%
     → 必须在每个2ms周期内跑完那0.5ms

  一组任务的总利用率: U_total = Σ(Ci/Ti)
   → 显然 U_total > 1 (>100%) 绝对不可调度(CPU不够用!)
   → U_total ≤ 1 是必要条件, 但不一定充分(要看调度算法)

周期任务模型用三个参数刻画每个任务:周期 T、执行时间 C、截止期 D。飞控的姿态环、位置环等都是典型周期任务(固定频率跑)。利用率 U=C/T 是核心量——一个任务占多少 CPU;一组任务的总利用率 ΣCi/Ti 决定了它们总共要多少 CPU。总利用率 > 100% 铁定做不完(CPU 不够);≤ 100% 是必要条件,但能不能真的都不误点,还要看调度算法和具体判据(下面 RMS/EDF 各有不同的可调度门槛)。这个模型是所有可调度性分析的基础。

52.4 固定优先级 vs 动态优先级

实时调度算法分两大类,RMS 和 EDF 分属其一:

固定优先级(RMS)动态优先级(EDF)
优先级任务的优先级固定不变(设计时定死)优先级动态变化(按当前情况算)
依据按周期定(周期短=优先级高)按截止期定(deadline 近=优先级高)
实现简单(和第17章 FreeRTOS 优先级调度一样)复杂(每次要比较 deadline)
CPU 利用率上限较低(~69%~100%)可达 100%

固定优先级:每个任务的优先级在设计时定死、运行中不变(就是你熟悉的第17章 FreeRTOS 优先级抢占调度)。动态优先级:优先级随运行时情况变化(如按"谁的 deadline 最近"动态调整)。RMS 是固定优先级的代表,EDF 是动态优先级的代表。固定优先级实现简单、可预测、被 RTOS 广泛采用,但 CPU 利用率上限低;动态优先级(EDF)能榨干 CPU(利用率达 100%),但实现复杂、行为较难预测。这个分类决定了两种算法的根本特性差异。

52.5 RMS：单调速率调度

RMS = Rate Monotonic Scheduling(单调速率调度)。规则极简:

RMS 规则: 周期越短(速率越高)的任务, 优先级越高
          (rate = 1/T, 速率单调 → 优先级单调)

  例: 任务A T=2ms, 任务B T=5ms, 任务C T=10ms
     → 优先级: A > B > C  (A周期最短, 最高优先级)
     → 就绪时总是先运行周期最短的

  为什么"周期短的优先"合理?
   周期短的任务, 两次执行间隔小, "更急"(更快又要来一次),
   给它高优先级, 让它能及时跑完不误点。

  ★RMS是"最优的固定优先级算法":
   如果RMS都调度不了这组任务, 那没有任何固定优先级方案能调度
   (RMS optimality定理)

RMS 的规则简单到优雅:周期越短、优先级越高。直觉上,周期短的任务"催得紧"(很快又要跑一次),给它高优先级理所应当。RMS 的重要地位在于它是"最优的固定优先级调度算法"——数学证明:如果按 RMS 分配优先级都无法让某组任务全部满足 deadline,那么任何其他固定优先级分配方案也不可能做到。换句话说,用固定优先级,RMS 就是你能做到的最好。这个最优性让 RMS 成为固定优先级实时系统的标准做法。你在 FreeRTOS 里给"周期短的控制任务设高优先级、周期长的日志任务设低优先级"——不知不觉就在用 RMS 思想!本章让你知其所以然:这不是拍脑袋,而是有最优性理论支撑的。

52.6 RMS 的可调度性判据

RMS 配套一个著名的可调度性判据(Liu & Layland 定理),能算"这组任务用 RMS 能不能都不误点":

★Liu-Layland 判据(充分条件):
  n个周期任务, 总利用率 U = Σ(Ci/Ti)
  若  U ≤ n·(2^(1/n) − 1)   则一定可调度(RMS下都不误点)

  这个上界随n变化:
   n=1: 100%      n=2: 82.8%     n=3: 78.0%
   n=4: 75.7%     ...            n→∞: ln2 ≈ 69.3%

  含义: 用RMS, 只要总利用率不超过约69%(保守), 一定安全!
   ★这是【充分条件】: 满足则一定可调度;
     不满足也【未必】不可调度(可用更精确的响应时间分析RTA)

  例: 3个任务 U=0.7 > 0.78? 不, 0.7<0.78 → 一定可调度 ✓
      若 U=0.8 > 0.78 → 判据不保证, 要用精确分析再判

Liu-Layland 判据给出一个"安全线":用 RMS,只要总利用率不超过 n·(2^(1/n)−1)(n 多时约 69%),就保证所有任务都不误点。这是一个充分条件——满足了一定安全,是设计时的"保守但可靠"的门槛。关键洞察:RMS 保证安全的利用率上限只有约 69%(而非 100%)——为了固定优先级的简单和可预测,牺牲了约 30% 的 CPU 利用率。超过 69% 不代表一定不行,只是这个简单判据不再保证,要用更精确的"响应时间分析(RTA)"逐个算每个任务的最坏响应时间。面试可能让你算:给几个任务的 T 和 C,判断 RMS 下是否可调度——套这个公式即可。

52.7 EDF：最早截止期优先

EDF = Earliest Deadline First(最早截止期优先)。动态优先级的代表:

EDF 规则: 谁的【当前 deadline 最近】, 谁优先级最高
          (优先级动态变: 随时间推移, 各任务的deadline远近在变)

  例: 此刻 任务A还有3ms到deadline, 任务B还有1ms到deadline
     → 先运行B(它更急, deadline更近)
     过一会儿情况变了, 再重新比

  ★EDF 可调度性判据(超简单):
     总利用率 U = Σ(Ci/Ti) ≤ 1 (100%)  ⟺ 可调度
   → EDF能把CPU用到100%还保证不误点! (单核, D=T时)
   → 这是它相对RMS的最大优势: 利用率上限100% vs RMS的69%

  ★EDF 是"最优的动态优先级算法"(单核):
     只要U≤1, EDF一定能调度; 任何算法调度不了的(U>1), EDF也不行
     → 换句话说EDF能调度一切"理论上可调度"的任务集

EDF 的规则也很直觉:谁最快要误点(deadline 最近),就先做谁——像救火,哪里最急先救哪里。因为 deadline 随时间推移在变,任务的相对优先级动态变化(动态优先级)。EDF 的杀手锏是它的可调度性判据极其简单且强大:总利用率 ≤ 100% 就一定可调度(单核、D=T)。这意味着 EDF 能把 CPU 榨到 100% 还保证不误点,而 RMS 只能保证到约 69%——EDF 的 CPU 利用率上限碾压 RMS。而且 EDF 是"最优的动态优先级算法":凡是理论上可调度的任务集(U≤1),EDF 都能调度。听起来 EDF 完胜,为什么 RTOS 大多还用 RMS?因为 EDF 有实现复杂、过载时行为糟糕(一旦 U>1,EDF 可能"雪崩式"多个任务连环误点,而 RMS 过载时只有低优先级任务受影响,更可预测)等缺点——下一节对比。

52.8 RMS vs EDF 对比

两大算法全面对比(面试高频):

RMS(固定优先级)EDF(动态优先级)
优先级固定(周期短=高)动态(deadline 近=高)
CPU 利用率上限约 69%~(n 相关)100%
实现复杂度简单(比固定优先级)复杂(动态比 deadline)
可预测性高(优先级固定,行为好懂)较低
过载(U>1)时只低优先级任务受损(可控)可能雪崩(多任务连环误点)
采用大多数 RTOS、飞控(FreeRTOS 优先级)部分系统、Linux SCHED_DEADLINE

RMS 和 EDF 是一对经典权衡:EDF 利用率高(100%)但复杂、过载时会雪崩;RMS 利用率低(69%)但简单、可预测、过载时优雅降级。过载行为的差异尤其关键:RMS 过载时,高优先级(短周期、通常最关键的控制)任务不受影响,只有低优先级任务误点——可控、可预测;而 EDF 过载时,可能一个任务超时引发连锁反应,多个任务集体误点——灾难性。对硬实时安全系统,"过载时行为可预测"往往比"利用率高"更重要——所以飞控、汽车等安全关键系统大多用固定优先级(RMS 思想),宁可 CPU 只用到 69% 也要行为可控。EDF 更适合"利用率要求高、过载可控"的场景(Linux 的 SCHED_DEADLINE 用 EDF)。没有绝对好坏,选择取决于"你更看重利用率还是可预测性/安全裕度"——这又回到实时系统的核心权衡。

52.9 Linux 的实时调度类

回到 Linux(第30章讲过调度类),看它提供的实时调度策略:

调度策略类型说明
SCHED_FIFO实时,固定优先级先进先出:同优先级不轮转,一直跑到主动让出/被更高优先级抢占
SCHED_RR实时,固定优先级轮转:同优先级任务时间片轮流(Round Robin)
SCHED_DEADLINE实时,EDF基于 EDF,指定周期/运行时/deadline,内核保证
SCHED_OTHER(CFS)非实时普通任务,公平调度(第30章)
用法(给任务设实时调度):
  struct sched_param p = { .sched_priority = 80 };
  sched_setscheduler(pid, SCHED_FIFO, &p);   // 设为实时FIFO优先级80
  // 或 chrt -f 80 ./myprogram

  ★实时优先级(1-99)高于所有普通任务(CFS):
    只要有实时任务就绪, 普通任务就得不到CPU
    → 实时任务里死循环 = 系统卡死! (要小心)

Linux 提供三种实时调度策略:SCHED_FIFO(固定优先级、同级不轮转,对应 RMS 式固定优先级)、SCHED_RR(固定优先级但同级时间片轮转)、SCHED_DEADLINE(基于 EDF,直接指定 deadline)。实时优先级(1-99)高于所有普通 CFS 任务——只要有实时任务就绪,普通任务就靠边站。这带来一个危险:实时任务里若有死循环,会饿死所有普通任务、卡死系统(所以有 RT throttling 机制默认限制实时任务最多用 95% CPU 留口子)。做 Linux 实时应用,就是把关键任务设成 SCHED_FIFO 高优先级(第55章配合 PREEMPT_RT)。在 RK3588 上跑实时任务就用这些。

52.10 多核实时调度的难题

前面的 RMS/EDF 理论都是单核的。多核(SMP,第30章)让实时调度变复杂,了解即可:

多核实时调度两种思路:
  ① 分区调度(Partitioned): 每个任务固定绑到某个核
     → 每个核独立跑单核RMS/EDF, 简单、可预测
     → 但是"装箱问题": 任务往核上分配可能装不下(某核满了)
  ② 全局调度(Global): 任务可在任意核跑, 全局一个队列
     → 灵活、利用率高, 但分析极难(有Dhall效应等反常现象)

  ★多核实时的反直觉: 单核的漂亮结论(EDF U≤1)在多核不成立!
     多核上U远小于核数也可能不可调度(经典反例)
  实践: 安全关键系统常用【分区+绑核】(CPU affinity, 第八卷),
        牺牲灵活性换可预测性和可分析性

多核实时调度远比单核难,单核的漂亮定理(如 EDF U≤1 可调度)在多核不再成立。两种思路:分区调度(每个任务绑定固定核,各核独立跑单核算法——简单可分析,但分配是装箱问题)、全局调度(任务可跨核迁移——灵活但分析极难,有反常现象)。实践中,安全关键的实时系统倾向"分区+绑核"(CPU affinity,第八卷):把关键实时任务固定绑到专用核(甚至隔离该核 isolcpus 不让别的任务用),牺牲灵活性换来可预测性和可分析性。这再次体现实时的信条:宁可保守(不充分利用多核),也要确定性。车企在多核 SoC 上做实时,绑核隔离是常规操作。

52.11 回扣 FreeRTOS 与工程

本章理论和你的实践连接:

本章概念连接到
固定优先级抢占(RMS)第17章 FreeRTOS 调度器
周期短=优先级高你给控制任务设高优先级的直觉
SCHED_FIFO/DEADLINE第30章 Linux 调度类
可调度性分析WCET(第54章,C 的来源)

本章把你"凭经验给任务排优先级"的做法,升级成"有理论依据、可数学证明"的实时调度设计。你在 FreeRTOS 里给姿态控制设最高优先级、给日志设最低——现在你知道这是 RMS(周期短的优先),而且它是最优的固定优先级方案,还能用 Liu-Layland 判据证明"这组任务在 XX 利用率下一定不误点"。这种"可证明"的能力是硬实时系统的科学基础:飞控不能靠"测试没出事"来保证,而要靠可调度性分析证明最坏情况也安全。工程实践的完整链条是:①用 WCET(第54章)确定每个任务的 C;②按 RMS 分配优先级(或用 EDF/SCHED_DEADLINE);③用可调度性判据证明可调度;④处理优先级反转(第53章)、多核绑核;⑤用 cyclictest(第51章)验证实际延迟。掌握这套方法论,你就能设计并论证一个满足 deadline 的实时系统,而不只是"调调优先级看看行不行"——这正是车企实时工程师的核心竞争力,面试也常围绕"如何保证一组任务都满足 deadline"展开。

52.12 面试常问问题

Q1:实时调度和普通调度(CFS)的目标区别?
CFS 追求公平/吞吐;实时调度追求保证每个任务在 deadline 前完成(不是公平)。实时调度还能做可调度性分析,设计时证明不误点。

Q2:RMS 是什么?规则?
Rate Monotonic,固定优先级:周期越短优先级越高。是最优的固定优先级算法——RMS 调度不了的,没有固定优先级方案能调度。

Q3:RMS 的可调度性判据?利用率上限?
Liu-Layland:U ≤ n(2^(1/n)−1) 则一定可调度(充分条件),n→∞ 时约 69.3%。即用 RMS 保守保证到约 69% 利用率。超过要用响应时间分析(RTA)精确判。

Q4:EDF 是什么?判据?和 RMS 比?
Earliest Deadline First,动态优先级:deadline 最近的优先。判据 U≤1 即可调度(单核 D=T),利用率上限 100%(vs RMS 69%),是最优动态算法。但实现复杂、过载时可能雪崩。

Q5:为什么飞控等安全系统多用固定优先级(RMS)而非 EDF?
EDF 利用率高但过载时多任务连环误点(雪崩),不可预测;RMS 过载时只有低优先级任务受损,高优先级关键任务不受影响,行为可控。安全系统重可预测性胜过利用率。

Q6:Linux 的实时调度策略有哪些?
SCHED_FIFO(固定优先级不轮转)、SCHED_RR(固定优先级时间片轮转)、SCHED_DEADLINE(EDF)。实时优先级高于所有 CFS 任务,实时任务死循环会卡死系统(有 RT throttling)。

Q7:多核实时调度为什么难?
单核定理(EDF U≤1)在多核不成立,有反常现象。分区调度(绑核,可分析)vs 全局调度(灵活难分析)。安全系统多用分区+绑核隔离,牺牲灵活换可预测。

52.13 常见误区

  1. "实时调度追求公平"——追求满足 deadline,不是公平(那是 CFS)。
  2. "RMS 能用满 100% CPU"——保证上限约 69%(n 多时),超过要精确分析。
  3. "EDF 全面优于 RMS"——EDF 利用率高但复杂、过载雪崩、不可预测;各有适用。
  4. "总利用率≤100% 就一定可调度"——EDF 单核是,RMS 不是(约 69% 才保证);多核都不成立。
  5. "给任务设高优先级就实时了"——还要可调度性分析证明、处理反转、控制 WCET。
  6. "单核实时理论直接用于多核"——多核有反常,不能照搬,常用绑核。

52.14 本章总结

  • 实时调度目标:保证每个任务满足 deadline(非公平);珍贵能力=可调度性分析(设计时数学证明不误点)。
  • 周期任务模型:周期 T、执行时间 C、截止期 D;利用率 U=C/T,总利用率 ΣCi/Ti。
  • RMS(固定优先级):周期短=优先级高;是最优固定优先级算法。判据 U ≤ n(2^(1/n)−1)(约 69%)则可调度(充分条件)。
  • EDF(动态优先级):deadline 近=优先级高;判据 U≤1 可调度(利用率上限 100%),最优动态算法。
  • 对比:EDF 利用率高(100%)但复杂、过载雪崩;RMS 利用率低(69%)但简单、可预测、过载优雅降级——安全系统多选 RMS 式固定优先级。
  • Linux 实时策略:SCHED_FIFO / SCHED_RR / SCHED_DEADLINE(EDF);实时优先级高于 CFS,死循环会卡死(RT throttling)。
  • 多核实时难(单核定理失效);安全系统用分区+绑核隔离,牺牲灵活换可预测。
  • 工程链条:WCET(C)→ RMS/EDF 分优先级 → 可调度性证明 → 处理反转/绑核 → cyclictest 验证。
