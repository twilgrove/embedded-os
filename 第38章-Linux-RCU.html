本章目标:第37章的锁,读者也要付出代价(至少一次原子操作/内存屏障)。但内核里大量数据是读极多、写极少(路由表、进程列表、配置)——能不能让读者完全不加锁、零开销?这就是 RCU(Read-Copy-Update),Linux 最精妙也最难懂的同步机制。它是第37章"减少互斥"思路的终极形态。学完这章,第五卷 Linux 内核就圆满收官了。

38.1 锁的尽头：能不能读不加锁

第37章末尾我们看到一条演进线:普通锁(读写都互斥)→ 读写锁(读者可并行)→ seqlock(写者不被读者挡)。每一步都在"减少读者的代价"。推到极致的问题是:能不能让读者完全不加锁、不加屏障、零开销地读?

这听起来不可能——不加任何保护,读到一半数据被改了怎么办?但 RCU 做到了,靠一个巧妙的转念:不阻止读者读旧数据,而是保证"旧数据在还有读者用时不被销毁"。读者尽管读(哪怕读到的是"旧版本"),写者去准备"新版本",等所有还在读旧版本的读者都读完了,才回收旧版本。读写不再互斥,而是"新旧版本共存一段时间"。

38.2 生活例子：不停电换灯泡

房间里一盏灯亮着(旧数据),大家正靠它看书(读者在读)。你要换个更亮的灯泡(更新)。加锁做法:让所有人闭眼(阻塞读者),关灯、换泡、开灯,再让大家睁眼——期间没人能看书(读者被挡)。

RCU 做法:你先在旁边装好新灯并点亮(准备新副本),然后"啪"一下把大家的目光引到新灯(切换指针,一瞬间),旧灯先不拆——因为可能还有人正盯着旧灯看(还有读者在用旧数据)。你等到确认没人再看旧灯了,才把旧灯拆掉(回收旧数据)。全程没有人需要闭眼(读者从不被阻塞)!新旧灯共存了一小会儿,代价是"多留了会儿旧灯",换来"读者全程无干扰"。这就是 RCU 的精髓:更新时新旧共存,等旧的没人用了再清理。

38.3 RCU 的名字：读-拷贝-更新

RCU = Read - Copy - Update,名字就是它的三步工作法:

Read(读)读者直接读,不加锁(只标记"我在读临界区")
Copy(拷贝)写者要改时,不原地改,而是拷贝一份,在副本上改
Update(更新)改完把指针原子地切换到新副本;旧副本等没人用了再回收

关键在"Copy"和"Update":写者不在原数据上改(那会让正在读的读者看到"改到一半"的中间状态),而是复制一份、在副本上从容改好、再一次性把指针切过去。指针切换是原子的(第37章原子操作),所以读者要么看到完整的旧版本,要么看到完整的新版本,绝不会看到"改了一半"的破碎状态。这是 RCU 让读者能安全无锁读的基础。

38.4 核心思想：更新用新副本(而非原地改)

用一个链表删除节点的例子,看清"新旧共存":

链表: A → B → C, 要删除 B

传统(加锁): 锁住链表, 改A的next指向C, 释放B, 解锁
             → 期间读者被挡

RCU:
  ① A → B → C   (初始, 读者可能正停在B上读)
  ② 把A的next【原子地】改成指向C:
        A ──────→ C
             ╲   ↗
              → B      (B还在! 只是没人能"新"到达它了)
     此刻: 新来的读者走 A→C (看不到B了)
           但已经在B上的老读者, 还能继续读B (B没被释放!)
  ③ 等【所有可能还在读B的读者】都离开后 → 才真正free(B)

看第②步:改指针是原子的一瞬间。这一瞬之后,新读者顺着 A→C 走,再也到不了 B;但之前已经在读 B 的老读者,手里还握着 B 的指针,B 还没被释放,他们能安全读完。写者不能立刻 free(B)——万一还有老读者在读就崩了。所以 RCU 最难、最核心的问题是:写者怎么知道"所有老读者都读完了,可以安全 free 旧数据了"?这就引出 RCU 的灵魂概念——宽限期。

38.5 最大难题：何时释放旧数据

再强调这个核心难题,因为它是 RCU 全部复杂性的来源:

读者不加锁、不留痕迹地读,写者怎么知道"没有读者还在用旧数据"?如果写者太早 free 旧数据,还在读的读者就会访问已释放内存(崩溃);如果永不 free,就内存泄漏。必须精确地在"最后一个老读者离开后"回收。但读者又不加锁(没法用"引用计数归零"这种要读者付出代价的方式,那就违背了"读者零开销"的初衷)。这个"既要读者零成本,又要知道读者何时全部离开"的矛盾,RCU 用一个绝妙的观察解决——宽限期。

38.6 宽限期 grace period（RCU 的灵魂)

RCU 的关键洞察,基于一条规则:RCU 读者在"读侧临界区"内(rcu_read_lock 到 rcu_read_unlock 之间)不允许睡眠/被抢占。由这条规则,推出一个绝妙结论:

★核心推理:
  规则: RCU读者临界区内不能睡眠/切换出去
  推论: 如果某个CPU发生了一次【上下文切换】(第30章),
        说明它上面【原来的那个读者已经退出临界区了】
        (因为读者临界区内不会切换)

  所以: 只要【每个CPU都至少经历过一次上下文切换】,
        就能保证【所有旧读者都已离开临界区】!

  这段"等每个CPU都切换过一次"的时间 = 【宽限期 grace period】
  宽限期结束 → 旧数据保证没人用了 → 可以安全free

这是 RCU 最天才的地方,一定要理解:因为读者临界区内不能切换,那么"一个 CPU 发生了上下文切换"就是"它上面的读者已经读完了"的免费信号——不需要读者主动报告(读者零开销!),只要观察调度器(第30章)的上下文切换就行。写者切换指针后,只要等到"每个 CPU 都至少切换过一次"(这段时间叫宽限期),就 100% 确定所有可能持有旧数据的老读者都退出了,此时 free 旧数据绝对安全。RCU 把"检测读者是否离开"这个难题,巧妙转化成"观察每个 CPU 是否都调度过一次"——完全不打扰读者。这就是 RCU 能让读者零开销的秘密。synchronize_rcu() 就是"阻塞等待一个宽限期过去",call_rcu() 是"登记一个回调,宽限期后自动 free"。

38.7 RCU 读侧几乎零成本

看 RCU 读者的代码,体会它有多"轻":

RCU 读者:
  rcu_read_lock();               // ★几乎是空操作(只是标记"我在读临界区")
      p = rcu_dereference(ptr);  // 读指针(带个轻量屏障保证顺序)
      ... 使用 p ...              // 安全: 就算此刻被删, p指的旧数据也不会被free
  rcu_read_unlock();             // ★几乎是空操作

  对比 spinlock/mutex: 读者要原子操作+可能等待
  RCU读者: rcu_read_lock/unlock 在多数配置下几乎不产生指令!
           → 读性能 ≈ 无锁, 且【多个读者、读者与写者都不互相阻塞】

RCU 读者的 rcu_read_lock/unlock 在多数内核配置下几乎是空操作(就是关/开抢占,或什么都不做)——没有原子操作、没有总线锁、没有 Cache 行争抢(第14章)、绝不阻塞。这是任何基于锁的方案都做不到的读性能。代价全转移到了写者身上:写者要拷贝、要等宽限期(可能几毫秒到几十毫秒),开销大、延迟高。所以 RCU 是把"读者的成本"转嫁给"写者",只在读远多于写时才划算。

38.8 RCU 完整流程

把写者的完整更新流程串起来(以删除/替换为例):

RCU 写者更新一个数据:
  ① 拷贝一份旧数据 (Copy), 在副本上修改
  ② rcu_assign_pointer(ptr, 新副本);  // 原子切指针(带release屏障)
       —— 从此新读者看到新数据, 老读者还在看旧数据
  ③ synchronize_rcu();   // 阻塞, 等一个【宽限期】过去
       —— 或 call_rcu(&old, free_cb); 登记回调, 不阻塞
  ④ 宽限期结束(每个CPU都切换过一次)→ 老读者保证全退出
  ⑤ free(旧数据)         // 现在安全了!

写者之间仍需互斥(用spinlock/mutex), 因为RCU只解决读写并发,
不解决写写并发!

注意最后一句:RCU 只让"读-写"无锁并发,不解决"写-写"冲突。多个写者之间仍要用第37章的锁(spinlock/mutex)互斥。RCU 的优化专门针对"读多写少"——读者(多)零开销,写者(少)用传统锁互斥且承担宽限期开销。这个分工正是它高效的原因。

38.9 RCU 的适用与限制

RCU 不是万能药,明确它的边界(面试爱问):

适合 RCU不适合 RCU
读远多于写(路由表、进程列表、配置)读写相当或写多(宽限期开销划不来)
能接受读者短暂读到"旧版本"要求读到绝对最新值
数据能用"整体替换指针"的方式更新要就地改、无法拷贝替换的复杂结构
写延迟不敏感(可等宽限期)写要求低延迟

两个关键限制。第一,读者可能读到"旧版本"——在写者切指针后、老读者退出前,老读者读的是旧数据。RCU 保证的是"读到一个一致的、完整的版本"(不是破碎中间态),但不保证是"最新"版本。要求强一致最新值的场景不能用 RCU。第二,写侧开销大(拷贝+宽限期),写多了不划算。所以 RCU 的定位非常清晰:读多写少 + 能容忍读到稍旧值 + 可指针替换更新。符合这三条,RCU 给你近乎免费的读;不符合,老老实实用第37章的锁。内核选择同步机制时,正是按这些条件判断的。

38.10 内核里 RCU 的实际应用

RCU 在 Linux 内核用得极广,举几个你已学过的:

  • 进程列表遍历:遍历所有 task_struct(第28章)——读极多(各种统计、ps),写少(创建/退出进程),完美 RCU 场景。
  • 路由表/网络配置(第34章):每个包都要查路由(读疯狂),路由变化少(写少)。
  • dentry 缓存(第32章):路径查找读极频繁,目录变化少。
  • 文件描述符表(第32章):读 fd 频繁,增删 fd 少。

可以说,凡是内核里"读热路径"上的共享数据,大多用了 RCU。正是 RCU 让这些高频读操作几乎零成本,Linux 才能在多核上扩展到几十上百核而读性能不塌。这也是为什么理解 RCU 对高性能系统工程师(如做多核实时系统的)很重要——它是多核可扩展性的关键武器。你在 /proc 里看到的很多信息、ROS2 底层查表,背后可能都有 RCU 在默默让读操作飞快。

38.11 回扣全卷：同步机制全景

第五卷收官,把全部同步机制排成一条"读者代价递减"的谱系:

机制读者代价适用章
普通锁(spin/mutex)要拿锁(原子操作/可能阻塞)通用,读写都互斥37
读写锁 rwlock拿读锁(读者间可并行)读多写少37
seqlock读完检查序号(可能重读)读极多、写低延迟37
RCU几乎零(不加锁不屏障)读远多于写、可容忍旧值38

这条谱系是一个漂亮的"用限制换性能"的故事:每往下一层,读者代价更低,但对使用场景的限制更多。普通锁最通用但读者也有代价;RCU 读者近乎免费,但要求"读多写少+可容忍旧值+可指针替换"。没有免费的午餐——RCU 的读者零成本,是用"写者承担宽限期开销""读者可能读到旧值""只适用特定更新模式"换来的。作为工程师,你的价值就在于看清场景的特征(读写比例、一致性要求、更新方式),选对那个"限制正好能接受、性能正好最优"的机制。从 FreeRTOS 的关中断临界区(第21章),到 Linux 的原子/自旋锁/互斥锁(第37章),再到 RCU(本章)——你现在拥有了一整套并发工具箱,和选择它们的判断力。这正是第五卷、乃至整本教材想带你抵达的境界:不是记住每个机制,而是理解它们的取舍,在真实系统里做出正确工程决策。

38.12 面试常问问题

Q1:RCU 是什么?解决什么问题?
Read-Copy-Update,一种让读者几乎零开销(不加锁不加屏障)的同步机制,针对读远多于写的场景。核心:更新时拷贝新副本、原子切指针、新旧共存,等所有老读者离开后回收旧数据。

Q2:RCU 读者为什么能不加锁?写者怎么知道能释放旧数据了?
因为读者临界区内不能睡眠/切换,所以"每个 CPU 都发生过一次上下文切换"就意味着所有老读者都退出了——这段时间叫宽限期。写者等一个宽限期(synchronize_rcu)后即可安全 free,全程不打扰读者。

Q3:什么是宽限期(grace period)?
从写者切指针后开始,到"每个 CPU 都至少经历一次上下文切换"为止的时间。宽限期结束保证所有可能持有旧数据的读者都已离开,可安全回收旧数据。

Q4:RCU 保证读到最新数据吗?
不保证。切指针后老读者仍读旧版本,RCU 只保证读到一个完整一致的版本(非破碎中间态),不保证是最新。要求强一致最新值不能用 RCU。

Q5:RCU 解决写-写并发吗?
不解决。RCU 只优化读-写并发(读者无锁);多个写者之间仍要用 spinlock/mutex 互斥。

Q6:RCU 的适用条件?
读远多于写、能容忍读到稍旧值、数据可用整体替换指针更新、写延迟不敏感(可等宽限期)。典型:路由表、进程列表、dentry 缓存。

Q7:RCU 相比读写锁的优势?
读写锁的读者仍要拿读锁(原子操作、Cache 行争抢),多核下有开销;RCU 读者几乎零开销、读者与写者完全不互相阻塞,多核扩展性远好于读写锁。

38.13 常见误区

  1. "RCU 读者也要加锁"——rcu_read_lock 几乎是空操作,读者不加真正的锁。
  2. "RCU 读到的一定是最新值"——可能读到旧版本(一致但不一定最新)。
  3. "RCU 能替代所有锁"——只优化读多写少的读写并发,写写仍需锁,写多不划算。
  4. "宽限期靠读者报告"——靠观察每个 CPU 的上下文切换,读者零参与。
  5. "RCU 读者临界区能睡眠"——经典 RCU 不能睡(睡了破坏宽限期推理);有专门的可睡眠变体 SRCU。
  6. "写者切指针后立刻能 free 旧数据"——必须等一个宽限期,否则老读者访问已释放内存。

38.14 本章总结

  • RCU(Read-Copy-Update):让读者几乎零开销(不加锁不加屏障)的同步机制,针对读远多于写。
  • 思想:写者不原地改,拷贝副本改好、原子切指针→ 新旧版本共存;读者尽管读(旧读者读旧版本,新读者读新版本),都读到完整一致的版本。
  • 核心难题:写者何时能 free 旧数据(读者不加锁没痕迹)?→ 宽限期。
  • 宽限期:因读者临界区内不能切换,"每个 CPU 都切换过一次"⇒ 所有老读者已离开。等一个宽限期即可安全 free。把"检测读者离开"转化为"观察 CPU 调度",不打扰读者。
  • API:rcu_read_lock/unlock(读者,近乎空操作)、rcu_assign_pointer(切指针)、synchronize_rcu/call_rcu(等宽限期后回收)。
  • 限制:读者可能读到旧值(非最新);写侧开销大(拷贝+宽限期);写-写仍需锁;适用"读多写少+容忍旧值+指针替换"。
  • 应用:进程列表、路由表、dentry/fd 表等内核读热路径——多核可扩展性的关键。
  • 全卷同步谱系:普通锁 → 读写锁 → seqlock → RCU,读者代价递减、场景限制递增;工程的艺术在于按场景选对取舍。
