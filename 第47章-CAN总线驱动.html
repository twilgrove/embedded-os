本章目标:CAN 总线是汽车和机器人的"神经网络"——发动机、刹车、车身、雷达、电机之间的通信全靠它。对目标去理想/小鹏/Momenta车载的你,CAN 是必考核心。它的设计哲学和前面三种总线截然不同:面向消息、ID 即优先级、非破坏性仲裁、为可靠性而生。Linux 用 SocketCAN 把 CAN 做成"网络设备",让你用 socket(第34章)收发 CAN 报文。

47.1 为什么车里用 CAN

一辆车里有几十上百个电子单元(ECU):发动机控制、ABS 刹车、车身、仪表、雷达、空调……它们要频繁交换数据(车速、转速、刹车状态)。用点对点连线?几十个单元两两相连是线缆灾难(重量、成本、故障率爆炸)。需要一种总线,让所有单元挂在一条(两根线)总线上互相通信,而且极其可靠(汽车关乎人命,不能因一个节点故障或电磁干扰就通信崩溃)。

CAN(Controller Area Network,控制器局域网)就是博世为汽车设计的这种总线。它的核心诉求是可靠、实时、多节点、抗干扰——这些诉求塑造了它独特的设计(和追求速度的 SPI、省线的 I2C 完全不同)。在汽车、工业、机器人领域,CAN 是事实标准:的机器人、行业无人机的电机/云台/电池管理,车企的整车网络,大量用 CAN。理解 CAN 的设计,就理解了"为可靠性和实时性而生的通信"长什么样。

47.2 生活例子：会议室抢话筒

想象一个大会议室,几十人(节点)共用一套广播系统(一条总线),谁说话所有人都听得到(广播)。问题:同时几个人想说话怎么办(总线冲突)?

CAN 的规则很妙:每个人发言前先报一个"话题优先级编号"(CAN ID),编号越小优先级越高。大家同时开口报编号时,编号小(优先级高)的人继续说,编号大的人一听"有比我更紧急的",立刻主动闭嘴让行,等下一轮再说。关键是——让行的人不会破坏正在进行的发言(不像两个人同时喊话变成噪音),而是"无损地"退出。所以最紧急的消息(如刹车)总能第一时间发出去,不被打断。这就是 CAN 的"非破坏性仲裁"——用优先级解决冲突,且保证高优先级消息零延迟胜出。这对实时性至关重要(刹车信号不能等)。

47.3 CAN 的核心特性

先总览 CAN 区别于 UART/I2C/SPI 的关键特性:

特性说明
多主广播任何节点都能主动发,消息广播给所有节点(无主从之分)
面向消息报文按"消息 ID"标识内容,不是发给某个地址的节点(见 47.6)
ID 即优先级消息 ID 小=优先级高,仲裁时胜出
非破坏性仲裁多节点同发,高优先级无损胜出,低优先级自动退让重试
差分信号两根线 CAN_H/CAN_L 差分传输,强抗电磁干扰
强错误检测CRC、位监测、错误帧、故障节点自动隔离
速度经典 CAN 最高 1Mbps(CAN FD 更高),不追求高速,追求可靠实时

一句话概括 CAN 的设计哲学:牺牲速度,换取可靠性、实时性和多节点广播。它不快(1Mbps),但极其可靠、能让紧急消息优先、能容忍节点故障和强干扰——这正是汽车/工业需要的。下面逐个展开最核心的几点。

47.4 CAN 帧：ID 即优先级

CAN 传输的单位是"帧",核心字段是 ID 和数据:

CAN 数据帧(标准帧, 简化):
  ┌─────┬──────────┬─────┬──────────┬─────┬─────┐
  │ SOF │ 仲裁段(ID) │ 控制 │ 数据(0-8字节)│ CRC │ ACK │
  └─────┴──────────┴─────┴──────────┴─────┴─────┘
          ↑
     ★11位ID(标准)或29位ID(扩展)
     ID有双重作用:
      ① 标识这条消息是"什么内容"(如0x100=车速, 0x200=转速)
      ② 决定优先级(ID数值越小 → 优先级越高)

  数据段最多【8字节】(经典CAN) —— 很小! 只传关键信号, 不传大数据

CAN ID 的双重身份是理解 CAN 的钥匙:它既标识消息内容(0x100 代表"车速"这类消息),又决定优先级(ID 小的优先)。数据段最多 8 字节(经典 CAN)——非常小,因为 CAN 是传"控制信号"的(车速、开关状态、指令),不是传大数据流(那是以太网的活)。这个"小数据、ID 标识内容、ID 定优先级"的设计,完全服务于实时控制场景:每条消息小而快,紧急的先发。电机控制、车企的整车信号,都是这种小报文。

47.5 仲裁：非破坏性竞争

CAN 最精妙的机制——非破坏性逐位仲裁,面试必考,一定要懂原理:

物理基础: CAN线有"显性(0)"和"隐性(1)", 显性会"盖住"隐性
          (类似I2C开漏: 任何节点发0都能把线拉成0)

多节点同时发, 逐位比较ID:
  节点A的ID: 0 1 1 0 1 0 0...
  节点B的ID: 0 1 0 1 ...
             ↑ ↑ ↑
  第1,2位相同(都发, 线上一致, 都继续)
  第3位: A发1(隐性), B发0(显性) → 线上是0(显性盖过隐性)
    → A发1却看到线上是0 → "有人优先级比我高" → A立刻退出仲裁!
    → B继续发送(它赢了), 消息不受影响
  → B(ID更小)无损获得总线, A退出等下轮重发

非破坏性仲裁的精髓:节点边发送 ID 边监听总线,一旦发现自己发的位(隐性 1)和总线实际(被别人拉成显性 0)不符,就知道有更高优先级节点在发,立即主动退出——而这个退出对赢家毫无影响(赢家的报文完整发出)。ID 数值小的(高位是 0/显性多的)总是胜出。对比以太网(第34章)的 CSMA/CD:以太网冲突了要"都停下、随机退避重发"(破坏性,浪费带宽);CAN 是"高优先级无损胜出、低优先级悄悄退让"(非破坏性,零浪费,且确定性)。这个机制保证了最高优先级消息永远能在固定时间内发出(实时性!刹车信号不会被堵),同时总线利用率高(冲突不浪费)。这是 CAN 为实时控制而生的核心设计,车企面试极爱问"CAN 仲裁原理"。

47.6 面向消息而非地址

一个思维转变:CAN 不像 I2C/以太网那样"发给某个地址的节点",而是面向消息:

I2C/以太网(面向地址):
  "我要把数据发给 地址0x68 那个设备"  ← 点对点, 指定接收方

CAN(面向消息):
  "我广播一条 ID=0x100(车速) 的消息"   ← 不指定发给谁!
  → 所有节点都收到, 每个节点自己决定"这条消息我关不关心"
  → 关心车速的节点(仪表、雷达)用【过滤器】只接收ID=0x100
  → 不关心的节点硬件自动忽略

  好处: ① 一条消息多个节点同时用(广播天然适合传感器数据分发)
        ② 加新节点不影响别人(它自己订阅关心的ID即可)
        ③ 发送方不用知道谁在接收(解耦)

CAN 是"发布-订阅"式:发送方广播带 ID 的消息(不管谁收),接收方按 ID 过滤只收自己关心的。这和 I2C"点名某地址"、以太网"发给某 MAC"的"面向地址"完全不同。好处是天然解耦和广播:车速消息发一次,仪表、雷达、巡航都能收;加个新 ECU 只需让它订阅关心的 ID,不影响现有节点。这个"面向消息/发布订阅"思想,你在 ROS2(话题 topic 发布订阅)里已经熟悉了——CAN 在硬件总线层面就是这个模型。理解这点,CAN 的很多设计(过滤器、无地址)就通了。

47.7 差分信号与可靠性

CAN 为什么在充满电磁干扰的汽车里还能可靠通信?靠差分信号和多重错误机制:

差分信号: 用两根线 CAN_H 和 CAN_L 的【电压差】表示数据
  显性(0): CAN_H高、CAN_L低 (压差大)
  隐性(1): CAN_H=CAN_L      (压差~0)
  读数据看的是【两根线的差】, 不是绝对电压

  ★抗干扰原理: 电磁干扰会【同时】影响两根线(它们挨在一起,
     常绞在一起=双绞线) → 两根线一起被抬高/拉低
     → 但它们的【差】几乎不变 → 数据不受干扰! (共模抑制)

多重错误检测(可靠性):
  CRC校验 + 位监测 + 帧格式检查 + ACK确认
  + 错误帧(出错立即通知全网重发)
  + 故障节点自动"离线"(错误计数超限自我隔离, 不拖垮总线)

差分信号 + 双绞线是 CAN 抗干扰的关键:干扰同时打在两根线上(共模),但数据看的是两线之差,共模干扰被抵消——这就是"共模抑制",让 CAN 在汽车强电磁环境里依然可靠(以太网、USB 也用差分,同理)。再加上 CRC、位监测、错误帧、故障节点自隔离等多重机制,CAN 的可靠性达到了汽车级要求。对比前面几种总线,CAN 是唯一把"可靠性"放在第一位设计的——因为它的应用(刹车、转向)真的关乎安全。终端还要接 120Ω 终端电阻消除反射(常见硬件要点)。

47.8 CAN FD 与更高带宽

经典 CAN 有两个瓶颈:速度低(1Mbps)、数据段小(8 字节)。随着汽车电子数据量暴增(自动驾驶传感器),推出了 CAN FD:

经典 CANCAN FD(Flexible Data-rate)
数据段最多 8 字节最多 64 字节
速率最高 1Mbps数据段可提速到 5Mbps+
兼容-兼容经典 CAN(可混用)

CAN FD 在保持 CAN 仲裁/可靠性优点的基础上,提升了数据段大小(8→64 字节)和数据阶段速率。仲裁段仍用低速(保证仲裁的确定性),数据段切到高速(提升吞吐)。现代汽车和高端机器人越来越多用 CAN FD(域控制器间通信、传感器数据)。更高端的还有车载以太网(100M/1000M,传摄像头/雷达大数据)。但 CAN/CAN FD 因其确定性和可靠性,在控制类通信里仍不可替代。了解 CAN FD 的存在和定位即可,底层机制和经典 CAN 一致。

47.9 SocketCAN：把 CAN 当网络设备

转到 Linux 驱动。Linux 处理 CAN 的方式很巧妙,叫 SocketCAN——把 CAN 接口做成"网络设备",用 socket(第34章)收发 CAN 报文。

SocketCAN 的思路:
  不把CAN当字符设备(read/write原始数据), 而是当【网络设备】!

  CAN接口 → 像网卡一样是个 net_device: can0, can1...
  收发CAN报文 → 用 socket API(第34章)!

  配置(像配网卡):
    ip link set can0 type can bitrate 500000  # 设波特率
    ip link set can0 up                        # 启用
    ifconfig can0 / ip -d link show can0       # 查看状态

  好处:
   ① 复用成熟的网络子系统(第34章): 多个程序可同时收发、过滤
   ② 标准socket编程, 和写网络程序一样
   ③ 工具丰富: candump/cansend(can-utils)

SocketCAN 是一个漂亮的设计决策:CAN 本质是"广播消息、多节点、带过滤"的通信,这和网络(第34章)高度相似——所以 Linux 干脆把 CAN 纳入网络子系统,做成 net_device,用 socket 收发。这样就复用了整套成熟的网络基础设施(第34章):多个进程能同时打开同一个 CAN 接口收发、内核帮你做 ID 过滤、排队、和网络工具链集成。回想第46章 regmap 让 I2C/SPI 复用,这里 SocketCAN 让 CAN 复用网络栈——都是"把新东西套进已有的成熟抽象"的智慧。对比:早期非 SocketCAN 方案把 CAN 当字符设备,每个程序独占、要自己管过滤,远不如 SocketCAN 灵活。现在 Linux 上用 CAN 几乎都走 SocketCAN,这也是"面向消息的 CAN"和"面向消息的网络"在设计上的天然契合。

47.10 用 SocketCAN 收发

用 SocketCAN 收发 CAN 报文,就是写 socket 程序(第34章),对目标车企岗要会:

#include <linux/can.h>
#include <linux/can/raw.h>

int s = socket(PF_CAN, SOCK_RAW, CAN_RAW);   // ★创建CAN socket
struct sockaddr_can addr = { .can_family=AF_CAN };
addr.can_ifindex = if_nametoindex("can0");   // 绑定到can0接口
bind(s, (struct sockaddr*)&addr, sizeof(addr));

// 发送一帧:
struct can_frame frame = {
    .can_id = 0x123,           // 消息ID
    .can_dlc = 2,              // 数据长度(0-8)
    .data = {0xAB, 0xCD},
};
write(s, &frame, sizeof(frame));   // 发出去(广播)

// 接收(可设过滤器只收关心的ID):
struct can_frame rx;
read(s, &rx, sizeof(rx));          // 收到一帧
printf("ID=%x len=%d\n", rx.can_id, rx.can_dlc);

// 命令行工具(can-utils):
//   candump can0            # 监听所有报文
//   cansend can0 123#ABCD   # 发一帧

就是标准 socket 编程:socket(PF_CAN, SOCK_RAW, CAN_RAW) 创建 CAN socket,bind 到 can0 接口,然后 write 发 can_frame(带 ID、长度、数据)、read 收。能设置过滤器(只收关心的 ID,对应 47.6 的订阅)。命令行 candump(抓所有报文)和 cansend(发报文)是调试 CAN 的利器(属于 can-utils 工具集)——插上 CAN 分析仪 candump 一看,总线上跑什么报文一目了然。这就是车企里用 CAN 的日常方式:配好 can0、用 socket 或 candump 收发调试,和写网络程序体验一致。

CAN 是本卷面向"车载/机器人"最重要的一章,直接对口理想/小鹏/Momenta车载岗。它展示了一种和 UART/I2C/SPI 完全不同的设计哲学:不为速度、不为省线,而为可靠性、实时性、多节点广播——因为它的应用关乎安全和实时控制。其中"ID 即优先级 + 非破坏性仲裁"保证紧急消息(刹车)确定性优先,是实时系统(第七卷)思想在总线层的体现;"面向消息/发布订阅"和 ROS2、和网络广播一脉相承;而 SocketCAN 把 CAN 纳入网络栈(第34章),让你用熟悉的 socket 收发。实战上:配 can0、candump 抓包、cansend 发包、socket 编程解析报文,是车载/机器人开发的日常。更进一步,车里还有 CAN 之上的应用层协议(如汽车诊断的 UDS、机器人的 CANopen),以及 DBC 文件描述报文含义——这些是在 CAN 基础上的上层,理解了 CAN 本身,再学它们就水到渠成。掌握 CAN,你就拿到了进入智能汽车/机器人通信领域的门票。

47.12 面试常问问题

Q1:CAN 总线的特点?为什么汽车用它?
多主广播、面向消息、ID 即优先级、非破坏性仲裁、差分信号抗干扰、强错误检测。汽车需要多节点、极可靠、实时(刹车不能延迟)、抗强电磁干扰的通信,CAN 正为此设计。

Q2:CAN 的仲裁机制?为什么是"非破坏性"?★
逐位仲裁:节点边发 ID 边监听,显性(0)盖过隐性(1);发现自己发 1 但线上是 0,说明有更高优先级(ID 更小)在发,立即退出。赢家报文不受影响(非破坏性)。保证最高优先级消息确定性胜出,兼顾实时和总线利用率。

Q3:CAN ID 有什么作用?
双重:标识消息内容(如 0x100=车速)+ 决定优先级(ID 越小优先级越高,仲裁胜出)。

Q4:CAN 和以太网冲突处理的区别?
以太网 CSMA/CD 冲突后都停下随机退避重发(破坏性、不确定);CAN 非破坏性仲裁,高优先级无损胜出、低优先级退让,确定性强、不浪费带宽。

Q5:CAN 为什么面向消息而非地址?好处?
广播带 ID 的消息,接收方按 ID 过滤订阅(发布-订阅,类似 ROS2 话题)。好处:一条消息多节点共享、加节点不影响别人、收发解耦。

Q6:CAN 怎么抗干扰?
差分信号(CAN_H/CAN_L 双绞线,看电压差),干扰共模作用于两线被差分抵消(共模抑制);加 CRC、位监测、错误帧、故障节点自隔离、120Ω 终端电阻。

Q7:Linux 怎么用 CAN?SocketCAN 是什么?
SocketCAN 把 CAN 做成网络设备(can0),用 socket(PF_CAN)收发 can_frame,复用网络子系统。ip link 配波特率启用,candump/cansend 调试。像写网络程序一样用 CAN。

47.13 常见误区

  1. "CAN 追求高速"——不,追求可靠/实时/多节点,速度不高(1Mbps,CAN FD 高些)。
  2. "CAN 是发给某个地址的节点"——面向消息广播,接收方按 ID 过滤订阅。
  3. "CAN 冲突像以太网要退避重发"——非破坏性仲裁,高优先级无损胜出,不是破坏性冲突。
  4. "CAN ID 只是标识"——还兼任优先级(越小越高)。
  5. "Linux 用 CAN 要写字符设备驱动"——用 SocketCAN,当网络设备,socket 收发。
  6. "CAN 不用终端电阻"——总线两端要接 120Ω 消除反射,否则通信不稳。

47.14 本章总结

  • CAN:汽车/机器人的"神经网络",为可靠、实时、多节点广播、抗干扰而生(牺牲速度)。
  • 核心特性:多主广播、面向消息、ID 即优先级、非破坏性仲裁、差分信号、强错误检测;数据段小(经典 8 字节)。
  • CAN 帧:ID 双重身份=标识消息内容 + 定优先级(越小越高)。
  • 非破坏性仲裁★:逐位比较,显性(0)盖隐性(1),发 1 见 0 即退出;高优先级无损胜出、低优先级退让重试 → 确定性实时(vs 以太网 CSMA/CD 破坏性)。
  • 面向消息/发布订阅:广播带 ID 消息,接收按 ID 过滤(类 ROS2 话题);解耦、易扩展。
  • 差分信号(CAN_H/CAN_L 双绞线,看压差)共模抑制抗干扰;CRC/错误帧/故障自隔离/120Ω 终端电阻。
  • CAN FD:数据段 8→64 字节、数据段提速,兼容经典 CAN。
  • SocketCAN:把 CAN 做成网络设备(can0),用 socket(PF_CAN) 收发 can_frame,复用网络栈;ip link 配置,candump/cansend 调试。
