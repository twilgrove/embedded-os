本章目标：第九章讲了 MMU 这个硬件"翻译器"。虚拟内存是软件(操作系统)在 MMU 硬件上盖起的一整套内存管理体系。MMU 是砖,虚拟内存是楼。这一章把按需分页、换页、写时复制、mmap 这些让人"哦原来如此"的机制讲透——它们都是"利用缺页异常做文章"。学完你要能解释"malloc 了 1GB 为什么不占内存""fork 为什么瞬间完成""内存不够时系统怎么办"。这也是第一卷的收官,会把十章串成一条完整的线。

10.1 MMU 与虚拟内存的关系

先厘清一个常被混淆的点:MMU 是硬件,虚拟内存是机制。

  • MMU(第九章):一块硬件,只干两件事——把虚拟地址翻译成物理地址、检查权限。它不懂"文件""进程""换页",它只是个忠实的翻译机。
  • 虚拟内存(本章):操作系统利用 MMU 提供的能力(尤其是"翻译失败会触发缺页异常"这个能力),实现的一整套聪明的内存管理策略。

关键桥梁是缺页异常:每当程序访问一个"页表里没准备好"的地址,MMU 就触发缺页异常,把控制权交给操作系统。操作系统就在这个"异常处理程序"里做各种文章——这是虚拟内存所有魔法的统一入口。记住这句话,本章都是它的展开。

10.2 虚拟内存要兑现的四个承诺

虚拟内存对每个程序做出四个"美好承诺",让程序员写代码时可以假装:

  1. 独占:你独占一整块巨大的、连续的地址空间(32 位是 4GB,64 位大得用不完),别的程序不存在。
  2. 够大:你能用的内存可以比机器实际装的物理内存还多(靠硬盘顶)。
  3. 隔离与保护:别人碰不到你,你碰不到别人(第九章已实现)。
  4. 省与快:启动快、fork 快、多个程序能共享同一份物理内存(如共享库)。

这四个承诺,物理内存只有一份、还可能不够,凭什么能兑现?靠的就是"虚拟地址和物理内存不必一一对应、不必时刻对应"这个自由度。下面逐个看。

10.3 承诺一：每个进程独占的大地址空间

每个进程都拥有从 0 到最大的一整套虚拟地址(它自己的坐标系),仿佛独占整台机器。这在第九章已经实现:每进程一张页表,同样的虚拟地址在不同进程被翻译到不同物理页。

但有个反常识的关键:拥有"4GB 地址空间"不等于占用了 4GB 物理内存!地址空间只是"编号范围",大部分编号是空的、没有对应物理页的。只有你真正用到的那些页,才会被分配真实物理内存(见 10.5 按需分页)。

这解释了新手困惑:"每个进程都有 4GB,我开 100 个进程不就要 400GB 内存?"——不会。地址空间是虚拟的"门牌号范围",不占物理内存;只有真正住了人的门牌(映射了物理页的虚拟页)才占。100 个进程各自的 4GB 绝大部分是空号。

10.4 承诺二：内存可以比物理内存大(换页)

怎么让程序用的内存比物理内存还多?把不常用的页暂时挪到硬盘上,腾出物理内存给急用的。这叫换页(Swapping/Paging)。

类比:你桌子(物理内存)小,放不下所有文件。你把最近不看的文件收进抽屉(硬盘的 swap 区),桌上只留在用的。等哪天又要那份收起来的,再从抽屉拿出来、同时把另一份不用的收回去。桌子虽小,但通过和抽屉换来换去,你"感觉"能处理的文件远超桌子容量。

物理内存满了, 程序又要新页:
  ① 内核挑一个"最近不用"的物理页(如LRU)
  ② 把它的内容写到硬盘的 swap 区
  ③ 把这个腾出的物理页给急用的
  ④ 更新页表: 被换出的那个虚拟页标记为"在硬盘上, 不在内存"

以后程序又访问被换出的页:
  → MMU发现该页不在内存 → 缺页异常
  → 内核从swap把它读回物理内存, 更新页表 → 重执行指令
  → 程序毫无察觉(只是慢了一下)

又见缺页异常!换页就是"利用缺页异常,在程序访问被换出的页时,悄悄把它从硬盘读回来"。物理内存 + 硬盘 swap,共同撑起了"比物理内存更大"的假象。

10.5 按需分页：用到才给

虚拟内存最核心的策略:不到真正访问,绝不分配物理内存。叫按需分页(Demand Paging)。

看 malloc 的真相,这是超高频面试题:

char *p = malloc(1 * 1024 * 1024 * 1024);  // 申请 1GB
// 此刻: 物理内存几乎没被占用!
// 内核只是在页表里"划出"了1GB的虚拟地址范围, 标记为"有效但暂无物理页"
// 一个物理页都还没给你

p[0] = 'A';   // 你第一次访问第0页
// → MMU发现这页没物理页 → 缺页异常
// → 内核这时才分配一个真实的4KB物理页, 填好页表 → 重执行
// → 现在才占用了 4KB 物理内存

// 你只写了1个字节, 就只占1页(4KB), 剩下的1GB仍是"空头支票"

所以:malloc 1GB 几乎瞬间返回且不占内存,是因为它只给了你"地址空间的承诺",物理内存要等你真正访问每一页时才按需分配。这也解释了第九章练习里 top 看到的 VIRT(虚拟,承诺的)远大于 RES(常驻,真实占用的)。

按需分页的好处:①启动快(不用一次性分配);②省内存(没用到的不占);③支持"稀疏"的大地址空间。第三次见缺页异常了吧?按需分页 = 利用缺页异常,在你首次访问某页时才分配物理内存。虚拟内存的三大魔法(按需分页、换页、写时复制)有一个共同的实现套路:先在页表里"欠着",等你访问时缺页,内核在缺页处理里"补上"。

10.6 页面置换与抖动

换页要挑"换出哪个页",这就是页面置换算法,和第八章 Cache 替换同理,常用 LRU(最近最少使用)近似——赌最久没用的以后也少用。

抖动(Thrashing):性能悬崖

如果物理内存严重不足,程序又频繁访问很多页,会出现:刚把 A 页换出去,马上又要用 A,只好把 B 换出来读回 A;转头又要 B……系统绝大部分时间都在硬盘和内存之间搬页,几乎不干正事——这叫抖动(Thrashing)。现象是:内存一满,系统卡死、硬盘狂响、鼠标都动不了。

因为硬盘比内存慢约万倍(第四章金字塔),一旦频繁换页,性能断崖式下跌。这就是为什么"内存不够时加内存比换更快的 CPU 有用得多"。现代系统还有 OOM Killer(内存耗尽时杀掉占用大的进程)来避免彻底卡死,第五卷会讲。

10.7 写时复制(COW)：fork 为什么这么快

虚拟内存最优雅的技巧之一,面试爱考。fork() 创建一个子进程,子进程是父进程的完整副本。按理说要把父进程的全部内存复制一份给子进程——如果父进程用了 2GB,复制 2GB 会很慢很占内存。但 fork 几乎瞬间完成。为什么?

秘密是写时复制(Copy-On-Write, COW):fork 时根本不复制内存,而是让父子进程的页表都指向同一批物理页,并把这些页标记为只读。

fork 后(还没人写):
  父进程页表 ─┐
              ├─▶ 同一批物理页(标记只读)   ← 不复制! 共享!
  子进程页表 ─┘

某一方要写某页时:
  → 往只读页写 → MMU触发异常(第九章权限位!)
  → 内核在异常处理里: 这时才真正复制这一页, 给写的人一份私有的可写副本, 改它的页表
  → 双方从此各用各的这一页

结果: 只有"被修改的页"才复制, 没改的页一直共享 → 极省、极快

又见异常机制!COW 用的是第九章的"只读权限位 + 写触发异常"。好处:

  • fork 瞬间完成(只复制页表,不复制数据);
  • 省内存(大量只读页父子共享,如代码段永远不改,永远共享);
  • 典型场景 fork 后马上 exec(第五卷),那复制内存就完全白费了——COW 正好避免了这浪费。

写时复制是"能拖就拖、能省就省"这个虚拟内存哲学的极致体现:复制这件事,拖到"真的有人要写"的最后一刻,且只复制那一页。它和按需分页、换页一样,都是"先欠着,缺页/异常时再补"。你把这三个机制放一起看,会发现它们是同一个思想的三种应用。

10.8 mmap：把文件变成内存

还有一个把虚拟内存威力发挥到极致的机制:mmap(内存映射)——把一个文件"映射"到进程的虚拟地址空间,之后像访问内存一样访问文件,不用 read/write。

mmap 把文件映射到一段虚拟地址:
  char *data = mmap(..., fd, ...);   // 把文件fd映射进来
  char c = data[1000];               // 直接像数组一样读文件第1000字节!

原理: 映射时不真读文件, 只在页表建立"这段虚拟地址↔文件某部分"的关系
  第一次访问 data[1000] → 该页没在内存 → 缺页异常
  → 内核在缺页处理里从文件读入对应页 → 重执行 → 拿到数据

又是缺页异常!mmap 的妙处:

  • 省一次拷贝:传统 read 要"内核缓冲区→用户缓冲区"拷贝,mmap 直接映射,少一次拷贝(零拷贝思想);
  • 按需加载:大文件不用全读进内存,访问哪页读哪页;
  • 可执行文件/共享库的加载就是 mmap:程序启动时,内核把 .text/.so 用 mmap 映射进地址空间,按需分页加载,启动飞快;
  • 共享内存(进程间通信):两个进程 mmap 同一文件/同一物理页,就能共享数据——第八卷 RK3588 的 AMP 共享内存就用这个。

10.9 进程地址空间的完整布局

把前面所有概念落到一张图上——这就是每个 Linux 进程看到的虚拟地址空间(呼应第五章的内存布局,现在你能理解每一块了):

一个进程的虚拟地址空间(64位Linux, 示意):

  高地址 ┌────────────────────┐
         │   内核空间          │  ← 所有进程共享, 用户态不可访问(权限位)
         ├────────────────────┤
         │   栈 Stack ↓        │  ← 局部变量/调用栈(第五章), 向下长
         ├────────────────────┤
         │        ↕(空闲)      │
         ├────────────────────┤
         │  mmap 区            │  ← 共享库(.so)、mmap文件、大块malloc
         ├────────────────────┤
         │        ↕            │
         │   堆 Heap ↑         │  ← malloc/brk, 向上长
         ├────────────────────┤
         │  BSS(未初始化全局)   │
         │  Data(已初始化全局)  │
         ├────────────────────┤
  低地址 │  Text 代码段(只读)   │  ← 程序指令, 常被多进程共享
         └────────────────────┘

关键: 这整张图是"虚拟"的! 每块按需映射到物理页;
      多进程的代码段/共享库可映射到同一份物理页(省内存)

cat /proc/自己PID/maps 看到的就是这张图的真实内容。每一段都是一组虚拟页,通过页表按需映射到物理内存,有各自的权限位。你写的每个变量、每次 malloc、每个函数,都落在这张图的某一段里。

10.10 虚拟内存的代价与嵌入式取舍

虚拟内存这么好,为什么 FreeRTOS/飞控内环不用?因为它有代价,而实时系统最怕代价的不确定性:

  • 翻译开销:每次访存要经 MMU(有 TLB 缓解,但 TLB miss 时要多次访存)。
  • 缺页的不确定延迟:一次访问可能命中(纳秒级),也可能缺页要从硬盘读回(毫秒级)——延迟差百万倍,且不可预测!这对"必须在 100µs 内响应"的飞控是致命的。
  • 换页可能在任何时刻发生,打乱实时任务的时序。
  • 硬件成本:MMU 占芯片面积和功耗。

这就是第一章那条分界线的完整答案:Linux 要虚拟内存(为了隔离、安全、跑众多程序,牺牲确定性);实时系统(飞控内环)不要或谨慎用(为了确定性,牺牲隔离和灵活)。硬实时系统常把关键内存"锁定"在物理内存里(mlock)防止被换出,就是为了消除缺页的不确定延迟。这类公司两种系统都用、都考,就是因为无人机既需要 Linux 端的丰富功能,又需要飞控端的硬实时确定性——理解这个取舍,是这类岗位的核心素养。

10.11 回扣与全卷收束

第十章也是第一卷的终点。回头看,这十章其实讲了一个完整的故事:

第一卷主线:操作系统是怎么"长"在硬件上的

CPU会算(2章:ALU/寄存器) ── 靠指令一步步干活(3章)
        │
   但只有几十个寄存器(4章), 状态要存在栈里(5章)
        │
   函数调用把寄存器和栈拧成一股绳(6章)
        │
   ┌────┴─────────────────────────────┐
   │ 上下文 = 寄存器快照, 存在栈里, TCB记栈顶  │  ← 操作系统"多任务"的地基
   └────┬─────────────────────────────┘
        │
   中断(7章)强行打断CPU → 抢占式调度的引擎 → 操作系统"心脏"完整
        │
   而访存太慢, 用Cache(8章)缓解 → 影响切换成本、写代码方式
        │
   要让多个不可信程序安全共存 → MMU(9章)加地址翻译层 → 隔离与保护
        │
   在MMU上盖起虚拟内存(10章): 按需分页/换页/COW/mmap → 独占大内存的假象
        │
        ▼
   至此, 第一章那句"操作系统=复用硬件(并发)+隔离保护(安全)"
   的每一个字, 都有了硬件层面的完整解释。

第一卷回答了"操作系统赖以生存的硬件是什么样"。接下来第二卷(ARM 架构)会把这些硬件机制在真实的 ARM 处理器上精确化(Cortex-M/A、异常等级 EL、GIC 中断控制器、Cache 一致性),然后第三卷起,我们就带着这套硬件认知,去读 FreeRTOS、RT-Thread、Linux 的真实源码——你会发现,源码里的每一行,都是第一卷这些概念的实现。

10.12 面试常问问题

Q1:MMU 和虚拟内存什么关系?
MMU 是硬件(翻译地址+查权限);虚拟内存是操作系统利用 MMU(尤其缺页异常)实现的内存管理机制。

Q2:malloc 1GB 会立刻占用 1GB 物理内存吗?为什么?
不会。按需分页:malloc 只在页表划出虚拟地址范围,物理页等你首次访问某页时缺页才分配。故 VIRT 大、RES 小。

Q3:fork 为什么很快?写时复制原理?
COW:fork 不复制内存,父子页表共享同一批物理页并标只读;谁写触发异常,内核才复制那一页给它。只复制被改的页,极快极省。

Q4:内存比物理内存大是怎么做到的?
换页(swap):把不常用页换到硬盘,腾出物理内存;再访问时缺页从硬盘读回。物理内存+swap 撑起更大假象。

Q5:什么是抖动(thrashing)?
物理内存严重不足时频繁换进换出,系统大部分时间在搬页几乎不干活,性能断崖。因硬盘比内存慢万倍。

Q6:mmap 的原理和好处?
把文件映射到虚拟地址,像访问内存一样访问文件;按需(缺页)加载、少一次拷贝、可用于加载可执行文件/共享库、进程间共享内存。

Q7:实时系统为什么慎用虚拟内存?
缺页可能从硬盘读回,延迟不确定(纳秒 vs 毫秒),破坏实时确定性。硬实时用 mlock 锁定内存防换出。

10.13 常见误区

  1. "虚拟内存就是硬盘上的 swap 文件"——狭义误解。虚拟内存是整套地址虚拟化机制,swap 只是其中"内存比物理大"这一部分。
  2. "有了 4GB 地址空间就占了 4GB 内存"——地址空间是虚拟编号范围,按需才映射物理页。
  3. "malloc 成功就分配了物理内存"——只给了虚拟地址承诺,访问时才真分配。
  4. "fork 会复制父进程全部内存"——COW,只在写时复制被改的页。
  5. "虚拟内存一定让系统更快"——它是为隔离/灵活/大容量,带来翻译和缺页开销;实时场景反而是负担。
  6. "缺页异常都是坏事"——按需分页、COW、mmap、换页回读都靠正常缺页运作。

10.14 本章总结

  • MMU 是硬件,虚拟内存是操作系统在其上实现的机制;缺页异常是所有虚拟内存魔法的统一入口。
  • 四个承诺:独占的大地址空间、可比物理内存大、隔离保护、省与快共享。
  • 地址空间大 ≠ 占物理内存;只有真正访问的页才映射物理页。
  • 按需分页:malloc 只给虚拟承诺,首次访问缺页时才分配物理页(解释 VIRT≫RES)。
  • 换页(swap):不常用页挪硬盘,撑起"比物理内存大";严重不足会抖动。
  • 写时复制 COW:fork 不复制、共享只读页,写时才复制那一页 → fork 快而省。
  • mmap:文件映射进地址空间,按需加载、少拷贝,用于加载程序/共享库/共享内存。
  • 三大魔法(按需分页/换页/COW)本质同一套路:页表里先欠着,访问时缺页/异常再补上。
  • 代价是翻译开销 + 缺页的不确定延迟,故实时系统慎用(mlock 锁定)。
  • 全卷收束:第一章"复用硬件+隔离保护"的每个字,十章下来都有了硬件解释。
