本章目标：FreeRTOS 调度器你已吃透——位图选最高优先级(第17章)。Linux 的 CFS(完全公平调度器)是完全不同的哲学:不追求"谁优先级高谁跑",而追求"每个进程分到公平的 CPU 时间"。这一章讲清 CFS 的 vruntime、红黑树、调度类框架,以及 Linux 怎么同时兼顾实时和公平。把它和 FreeRTOS 对照,你能深刻理解"实时 vs 分时"两种调度哲学的根本差异——这是第一卷埋的最大伏笔之一。这是面试的核心考点。

30.1 Linux 调度要解决的新问题

FreeRTOS 调度简单粗暴:严格按优先级,高的先跑。这在 RTOS(少量、可控的实时任务)很完美。但 Linux 面对的是完全不同的场景:

  • 成百上千个进程,来源不明、行为各异。
  • 混合负载:有需要快速响应的交互程序(你的鼠标、键盘、编辑器),有闷头算的批处理(编译、渲染),还有需要实时的(音视频)。
  • 不能饿死任何人:如果严格按优先级,低优先级进程可能永远得不到 CPU(饿死)——通用系统不能接受。
  • 多核:要在多个 CPU 间均衡负载。

核心矛盾:RTOS 的"严格优先级"会饿死低优先级进程,不适合通用系统;但纯粹平均分配又照顾不了"交互程序要快速响应"的需求。Linux 需要一个既公平(不饿死谁)、又能照顾交互性(响应快)、还能支持实时(音视频/控制)的调度器。这比 RTOS 难得多。CFS(Completely Fair Scheduler,完全公平调度器,2.6.23 引入)是 Linux 对"公平+交互"这部分的答案,而实时需求交给专门的实时调度类。

30.2 调度类：一个可插拔的框架

Linux 不是只有一个调度算法,而是一个可插拔的调度类(sched_class)框架——不同类型的进程用不同的调度器,按优先级排列。这是第28章 task_struct 里 sched_class 字段的用途:

Linux 调度类(按优先级从高到低, 高的类先被选):

  ┌─────────────────────────────────────┐
  │ stop_sched_class     最高, 停机/迁移用 │
  ├─────────────────────────────────────┤
  │ dl_sched_class       Deadline(EDF), 硬实时 │ ← 第七卷EDF!
  ├─────────────────────────────────────┤
  │ rt_sched_class       实时(FIFO/RR), 固定优先级 │ ← 像FreeRTOS
  ├─────────────────────────────────────┤
  │ fair_sched_class     ★CFS, 普通进程(绝大多数) │ ← 完全公平
  ├─────────────────────────────────────┤
  │ idle_sched_class     空闲(没别的可跑时)        │
  └─────────────────────────────────────┘

调度时: 从最高的类往下问"你有就绪进程要跑吗?"
        有实时进程 → 实时类先跑(实时优先于普通)
        没实时进程 → CFS 选一个普通进程

这个框架很妙:实时进程(rt/dl 类)永远优先于普通进程(CFS 类)——保证了实时性;而普通进程之间用 CFS 公平调度——保证了公平。Linux 用"分层调度类"同时满足了实时和公平两个看似矛盾的需求。你的音视频、控制程序可以设成 rt 类(像 FreeRTOS 那样按优先级),普通程序走 CFS。这个可插拔设计,又是"函数指针/多态"(每个 sched_class 有一组函数指针:enqueue/dequeue/pick_next 等),和第26章 RT-Thread 设备框架同一个手法。

30.3 CFS 的核心思想：完全公平

CFS 管普通进程,它的理想:如果有 N 个进程,每个进程应该得到 1/N 的 CPU 时间。就像分蛋糕,N 个人平分。

但 CPU 一次只能跑一个进程,没法真的"同时各给 1/N"。CFS 的办法:记录每个进程"已经用了多少 CPU 时间",每次都挑那个"用得最少"的来跑。这样长期看,大家用的时间会趋于相等——就公平了。

CFS 理念: 让每个进程的"已运行时间"尽量相等

  进程A用了 10ms, 进程B用了 10ms, 进程C用了 5ms
  → C 用得最少 → 下次调度选 C 跑
  → C 跑一会到 10ms, 大家又相等了...

  就像排队打水: 谁打的水最少, 谁先来打, 保证长期公平

这比"优先级"高明在:没有进程会被饿死——哪怕一个进程"优先级低",它用的时间少,迟早会被选中。CFS 用"追踪已用时间、补偿用得少的",实现了动态的公平。但直接用"真实运行时间"还不够(要支持 nice 优先级),于是有了 vruntime。

30.4 虚拟运行时间 vruntime

CFS 的核心变量是 vruntime(虚拟运行时间),在 task_struct 的 se(sched_entity)里。它不是真实运行时间,而是"加权后的"运行时间:

vruntime 大致 = 真实运行时间 × (基准权重 / 进程自己的权重)

  → 权重高的进程(高优先级), vruntime 增长【慢】
    (真实跑了10ms, vruntime只加了5ms的样子)
  → 权重低的进程, vruntime 增长【快】

CFS 调度规则: 总是选 vruntime 【最小】的进程跑
  (谁的虚拟时间最少, 谁最"亏", 让它跑)

妙在:CFS 追求的"公平"是 vruntime 相等,不是真实时间相等。因为高优先级进程 vruntime 涨得慢,要让它的 vruntime 追上别人,它就得跑更多真实时间——于是高优先级进程自然分到更多 CPU 时间,但又不会完全饿死低优先级的。这就是 CFS 的精髓:用 vruntime 这一个变量,把"公平"和"优先级"统一了——优先级不再是"谁绝对先跑",而是"谁分到的时间份额更大"。

对比 FreeRTOS:FreeRTOS 优先级是"绝对的"(高优先级不阻塞就永远压着低优先级)。CFS 优先级是"相对的份额"(高优先级分更多时间,但低优先级也有份)。这就是"实时优先级"和"分时优先级(nice)"的本质区别——前者为确定性(必须让关键任务先跑),后者为公平和体验(大家都要照顾)。vruntime 是理解 CFS 的钥匙:一切都是"选 vruntime 最小的、让新唤醒的进程 vruntime 设得合适以获得响应性"。睡眠后被唤醒的进程 vruntime 较小(睡觉时没涨),所以会被优先调度——这就是交互程序响应快的原因(你按键,处理进程刚被唤醒,vruntime 小,立刻被调度)。

30.5 nice 值与权重

用户怎么调进程优先级?用 nice 值(-20 到 +19,越 nice 越谦让=优先级越低):

nice -20 (最高优先级) ←──────────→ nice +19 (最低, 最"谦让")

nice 值映射到"权重"(内核 sched_prio_to_weight 表):
  nice 0  → 权重 1024 (基准)
  nice -1 → 权重 1277 (每降1级nice, 权重约×1.25)
  nice +1 → 权重 820

CFS 时间分配 = 该进程权重 / 所有就绪进程权重之和
  nice每差1级, CPU时间份额差约10%(1.25倍关系)

关键:nice 值影响的是"CPU 时间的份额"(通过权重影响 vruntime 增长速度),不是"绝对优先"。nice -20 的进程分到的 CPU 时间多得多,但 nice +19 的进程也不会完全饿死。这和 RTOS 优先级(高优先级完全压制低的)是两码事。renice/nice 命令就是调这个。为什么每级差 1.25 倍?为了让"nice 差 1 级=CPU 份额差约 10%"这个体验相对一致(几何级数),这是精心调过的经验值。

30.6 红黑树：为什么不用位图

FreeRTOS 用位图 O(1) 选最高优先级(第17章)。CFS 为什么改用红黑树?因为需求不同:

  • FreeRTOS 要"选最高优先级"——优先级是离散的少数几十个值,位图完美。
  • CFS 要"选 vruntime 最小的"——vruntime 是连续变化的、几乎无限多的值,位图用不了。需要一个能"按 vruntime 排序、快速取最小、且插入删除都快"的结构——红黑树(平衡二叉搜索树)。
CFS 运行队列(每个CPU一个)用红黑树按vruntime排序:

              [15]
             /    \
          [9]      [20]
         /  \      /
      [6]   [12] [17]
      ↑
   最左节点 = vruntime最小 = 下一个要跑的进程!

  取下一个: 取最左节点 → O(log n)(实际缓存了最左节点, 近O(1))
  进程跑完/新入队: 插入/删除并保持平衡 → O(log n)

红黑树的好处:最左边的节点永远是 vruntime 最小的(下一个该跑的),取它很快;插入删除保持平衡,都是 O(log n)。CFS 还缓存了最左节点指针,取下一个进程接近 O(1)。数据结构的选择取决于操作需求:选离散最高优先级用位图,选连续最小值用红黑树。这是"没有最好的数据结构,只有最适合场景的"的经典案例。

历史插曲:Linux 2.6 早期用的是 O(1) 调度器(和 FreeRTOS 类似的优先级位图+队列),但它对"交互性"处理得很复杂、很多启发式 hack。2.6.23 换成 CFS(红黑树+vruntime),用一个优雅的模型统一处理了公平和交互,代码更简洁、行为更可预测。从 O(1) 调度器到 CFS 的演进,是"用简单统一的模型代替一堆特殊情况 hack"的经典重构——O(1) 调度器虽然选进程快(O(1)),但为了交互性堆砌了大量特判;CFS 虽然选进程是 O(log n)(略慢),但模型干净、无需特判。这个取舍告诉你:算法复杂度不是唯一标准,可维护性和行为可预测性同样重要。

30.7 调度周期与最小粒度

CFS 怎么决定每个进程一次跑多久(时间片)?它是动态计算的(不像 FreeRTOS 固定 1 tick):

  • 调度周期(sched_latency):CFS 保证在这段时间内,所有就绪进程都能跑一次(默认约 6ms×log(cpu 数) 之类)。目标:让每个进程都及时得到运行,保证交互响应。
  • 每个进程的时间片 = 调度周期 × (自己权重 / 总权重)。权重高的分到的时间片长。
  • 最小粒度(min_granularity):如果进程太多,均分下来每个时间片会非常小,频繁切换开销大。所以设一个下限(默认约 0.75ms),时间片不小于它——牺牲一点"周期内都跑到"的理想,换取不过度切换。

这里体现了 CFS 的另一个权衡:进程少时,追求"低延迟"(每个进程都快速轮到,交互好);进程多时,追求"低切换开销"(用最小粒度限制切换频率,否则光切换就把 CPU 耗光了,呼应第八章切换的 Cache 成本)。这个"延迟 vs 吞吐"的平衡,是通用调度器必须做的、而 RTOS 不用操心的(RTOS 任务少、固定时间片就行)。你能看到,Linux 调度器的复杂性,几乎全部来自"要照顾各种场景"这个通用性要求。

30.8 schedule() 与 __switch_to

调度的核心函数 schedule()(kernel/sched/core.c),对应 FreeRTOS 的 vTaskSwitchContext:

asmlinkage void schedule(void)
{
    struct task_struct *prev, *next;

    prev = current;                    /* 当前进程 */

    /* ① 从最高的调度类往下, 找下一个要跑的进程 */
    next = pick_next_task(rq, prev);   /* CFS则取红黑树最左节点 */

    if (prev != next) {
        /* ② 上下文切换! */
        context_switch(rq, prev, next);
        /* 内部调用 switch_mm(换地址空间/页表!) + switch_to(换寄存器/栈) */
    }
}

关键:Linux 的上下文切换比 RTOS 多一步——switch_mm(切换地址空间):

context_switch 做两件事:
  ① switch_mm(prev->mm, next->mm)   ★切换地址空间(换页表TTBR!)
     - 若切换到不同进程 → 换页表 → 导致TLB失效(第九章!)
     - 若切换到同进程的线程 → 共享mm, 不用换(便宜!)
  ② switch_to(prev, next)           切换寄存器和内核栈
     - 架构相关(arch/arm64/.../switch_to)
     - 本质和FreeRTOS PendSV一样: 存旧寄存器, 恢复新寄存器

第九章埋的"进程切换比线程切换贵"在这里落地了:switch_mm 换页表(TTBR)会导致 TLB 失效、Cache 冷启动(第八、九章),这是进程切换的隐性大成本;而同进程的线程共享 mm,switch_mm 发现 mm 相同就跳过,所以线程切换便宜。switch_to 则和 FreeRTOS 的 PendSV 汇编做同一件事(存恢复寄存器换栈,第18章),只是 ARM64 上寄存器更多、还要处理 FPU/SIMD 等。你在第18章逐行读懂的 PendSV,就是 Linux __switch_to 的简化版——原理完全一样。Linux 用 ASID(第九章)让切换进程时不必全清 TLB,缓解成本。

30.9 实时调度类：RT 与 Deadline

Linux 也支持"像 FreeRTOS 那样"的实时调度,给 rt/dl 调度类(优先于 CFS):

策略调度类行为像什么
SCHED_FIFOrt固定优先级,高的先跑,同级不轮转(跑到阻塞/让出)FreeRTOS 关时间片
SCHED_RRrt固定优先级 + 同级时间片轮转FreeRTOS 开时间片
SCHED_DEADLINEdl按"截止时间"调度(EDF),最紧急的先跑第七卷 EDF

SCHED_FIFO/RR 就是 FreeRTOS 那套"严格优先级抢占"!你用 chrt 命令或 sched_setscheduler 把一个进程设成 SCHED_FIFO + 高优先级,它就会像 FreeRTOS 高优先级任务一样,抢占所有普通(CFS)进程、不被饿死顾虑约束。这就是在 Linux 上跑"软实时"任务的方法——在 RK3588 上跑对时延敏感的控制/感知任务,常把它们设成 SCHED_FIFO 高优先级,把 CFS 的普通进程压下去。SCHED_DEADLINE 是更先进的 EDF 算法(第七卷详讲),按截止期限调度,理论上能达到更高的 CPU 利用率。

重要认知:Linux 不是"只有 CFS",它通过调度类同时提供了"实时(rt/dl,像 RTOS)"和"公平(CFS)"两种模式。普通程序走 CFS(公平、防饿死),需要实时的程序设成 SCHED_FIFO/DEADLINE(优先级抢占/截止期,像 RTOS)。所以"Linux 能不能做实时"的答案是:能做软实时(SCHED_FIFO),配合 PREEMPT_RT 补丁能做到接近硬实时。但因为有虚拟内存的缺页不确定性(第十章)、更长的中断关闭区间等,原生 Linux 的实时性仍不如专门的 RTOS——这就是为什么飞控内环用 RTOS、而 RK3588 主系统用 Linux+SCHED_FIFO(第八卷 AMP 架构)。

30.10 多核负载均衡

RTOS 多是单核(FreeRTOS 单核为主)。Linux 是多核 SMP,调度器要处理负载均衡——让各个 CPU 核的负载大致均衡,别一个核忙死、别的核闲着。

  • 每个 CPU 核有自己的运行队列(runqueue)——per-CPU(第28章讲的多核手法),各自的红黑树。这样各核选进程不用抢一个全局锁。
  • 周期性负载均衡:内核定期检查各核负载,把进程从忙的核"迁移"到闲的核。
  • 考虑 CPU 拓扑和亲和性:迁移优先在"近"的核之间(同 L2/L3 Cache,迁移代价小);还要尊重进程的 CPU 亲和性设置(taskset/sched_setaffinity)。
  • NUMA/big.LITTLE 感知:在 RK3588 这种大小核上,EAS(能耗感知调度)会把重任务放大核、轻任务放小核(第十一章 big.LITTLE)。

负载均衡是"多核"带来的、RTOS 基本不用操心的复杂性。核心权衡:迁移进程能均衡负载,但迁移有代价(目标核的 Cache 里没有这进程的数据,冷启动,第八章)。所以内核不会频繁乱迁,而是权衡"负载不均的损失 vs 迁移的 Cache 代价"。CPU 亲和性(把进程绑在固定核)就是让你手动干预这个——把关键进程绑在专用核,避免被迁移、独占该核的 Cache,提升确定性。第八卷讲 RK3588 实践时,CPU 隔离(isolcpus)+ 亲和性绑定 + SCHED_FIFO 是做 Linux 端实时优化的组合拳。

30.11 抢占与内核抢占

抢占(第一卷)在 Linux 里分两个层次,面试常问:

  • 用户态抢占:进程在用户态运行时,时钟中断或更高优先级进程唤醒会触发重新调度、抢占它。这个所有 Linux 都有。
  • 内核态抢占(CONFIG_PREEMPT):进程陷入内核态执行系统调用时,能不能被抢占?
    • 不可抢占内核(老/服务器配置):内核代码执行时不被抢占,直到它主动让出或返回用户态。简单,但一个长系统调用会拖延调度,延迟高。
    • 可抢占内核(PREEMPT):内核代码执行时也能被抢占(在安全的抢占点),延迟低,适合桌面/实时。
    • PREEMPT_RT(实时补丁):把内核几乎所有地方都变成可抢占、把中断线程化、把自旋锁换成可睡眠的——最大限度降低延迟,让 Linux 接近硬实时。

为什么内核抢占是个问题?因为进程在内核态时,可能正持有锁、正操作内核数据结构——随便抢占会导致并发问题(第37章)。所以内核抢占只能在"安全点"(没持锁等)发生,这需要精细的设计。抢占越彻底,延迟越低(实时性越好),但实现越复杂、开销略增。PREEMPT_RT 补丁(现在大部分已合入主线)就是把 Linux 往"硬实时"方向推的努力——、机器人、工业控制领域用 Linux 做实时,PREEMPT_RT 是关键。这又是"实时性 vs 复杂度/吞吐"的权衡,贯穿全书。

30.12 对比 FreeRTOS 调度器

把两种调度哲学做终极对照(这是第一卷最大伏笔的收束):

维度FreeRTOS(RTOS)Linux CFS(普通进程)
目标确定性、实时公平、吞吐、交互体验
优先级含义绝对(高的压制低的)相对份额(nice→CPU时间比例)
会饿死低优先级吗会(这是"特性")不会(公平)
选进程位图 O(1) 选最高优先级红黑树 O(log n) 选最小 vruntime
时间片固定 1 tick动态(按权重和进程数)
多核基本单核per-CPU 队列 + 负载均衡
实时支持本身就是实时另有 rt/dl 调度类(SCHED_FIFO/DEADLINE)

第一卷 1.6 那张表——"FreeRTOS 严格优先级为实时,Linux CFS 公平为吞吐"——现在你看到了两边完整的源码级实现。核心洞察:调度没有"最好",只有"最适合场景"。RTOS 场景(少量关键实时任务)适合严格优先级(简单、确定);通用 OS 场景(海量各异进程)适合公平调度(不饿死、照顾交互),同时用调度类给实时需求留后门。你现在能同时深入理解这两种哲学,并知道它们各自为什么这样设计——这就是从"会用"到"真懂"的跨越,也是这类既用 RTOS 又用 Linux 的公司最看重的素养。

30.13 面试常问问题

Q1:CFS 的核心思想?怎么实现公平?
让每个进程分到公平的 CPU 时间。用 vruntime(虚拟运行时间)追踪,每次选 vruntime 最小的进程跑,长期使各进程 vruntime 趋于相等,不饿死任何进程。

Q2:vruntime 是什么?nice 值如何影响它?
加权后的运行时间(真实时间×基准权重/进程权重)。高优先级(权重大)vruntime 涨得慢,要涨到和别人一样得跑更多真实时间,故分到更多 CPU 时间。nice 值映射权重,每级差约 1.25 倍(份额差约 10%)。

Q3:CFS 为什么用红黑树而不是位图?
vruntime 是连续值(几乎无限多),位图只能处理离散的少数优先级。红黑树能按 vruntime 排序、取最小(最左节点)、插删都 O(log n)。

Q4:CFS 和 FreeRTOS 调度器的本质区别?
FreeRTOS 严格优先级(高压低、会饿死、位图 O(1)、为确定性);CFS 完全公平(nice 是相对份额、不饿死、红黑树、为吞吐和交互)。

Q5:Linux 能做实时吗?怎么做?
能做软实时:用调度类框架,rt 类(SCHED_FIFO/RR,严格优先级像 RTOS)和 dl 类(SCHED_DEADLINE,EDF)优先于 CFS。配 PREEMPT_RT 补丁接近硬实时。但原生 Linux 因虚拟内存等不确定性,实时性仍不如专用 RTOS。

Q6:Linux 上下文切换和 RTOS 有什么不同?
多了 switch_mm(切地址空间/页表):切进程换页表致 TLB 失效、Cache 冷启动(贵);切同进程线程共享 mm 不换(便宜)。switch_to 和 FreeRTOS PendSV 做同样的存恢复寄存器。

Q7:多核负载均衡怎么做?为什么有 CPU 亲和性?
每 CPU 一个运行队列,周期性把进程从忙核迁到闲核(考虑 Cache 拓扑)。迁移有 Cache 冷启动代价,故用 CPU 亲和性把关键进程绑定固定核,避免迁移、独占 Cache、提升确定性。

30.14 常见误区

  1. "CFS 完全平均分配 CPU"——是按权重(nice)加权公平,不是绝对平均;高优先级分更多。
  2. "nice 值高优先级就绝对高"——nice 是相对份额,低 nice 也不会饿死。
  3. "Linux 不能做实时"——能做软实时(SCHED_FIFO/DEADLINE),PREEMPT_RT 接近硬实时。
  4. "CFS 用优先级队列"——用红黑树按 vruntime 排序,不是优先级位图。
  5. "进程和线程切换开销一样"——进程切换多了 switch_mm(换页表/TLB 失效),更贵。
  6. "vruntime 就是真实运行时间"——是加权后的,权重高涨得慢。
  7. "绑核一定更快"——绑核提升确定性和 Cache 命中,但可能牺牲负载均衡;要看场景。

30.15 本章总结

  • Linux 调度要兼顾公平(不饿死)、交互(响应快)、实时、多核,比 RTOS 难。
  • 调度类框架(可插拔):dl > rt > CFS > idle;实时类优先于普通类,兼顾实时和公平。
  • CFS 核心:让每个进程分到公平 CPU 时间,选 vruntime 最小的跑,不饿死。
  • vruntime=加权运行时间;高优先级(权重大)涨得慢→分更多时间。nice 值→权重(每级≈1.25×)。
  • 用红黑树按 vruntime 排序(取最左=最小),O(log n);因 vruntime 连续,不能用位图。
  • 时间片动态算(调度周期×权重占比,受最小粒度下限约束):平衡延迟与切换开销。
  • schedule()→context_switch=switch_mm(换地址空间,进程切换致 TLB 失效)+ switch_to(换寄存器,同 PendSV)。
  • 实时:SCHED_FIFO/RR(像 FreeRTOS 严格优先级)、SCHED_DEADLINE(EDF);PREEMPT_RT 接近硬实时。
  • 多核:per-CPU 运行队列 + 负载均衡(权衡迁移的 Cache 代价);CPU 亲和性提升确定性。
  • 第一卷"实时 vs 分时"两种调度哲学,在此完整对照收束。
