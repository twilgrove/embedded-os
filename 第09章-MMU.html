给每个进程一套"假地址"，让它们以为各自独占整个内存

本章目标：第一章说 Linux 靠 MMU 做内存隔离,那只是结论。这一章把 MMU 这个硬件单元彻底讲透:它凭什么能让每个进程"独占整个内存"、凭什么能阻止 A 进程访问 B 的内存。这是 RTOS 工程师转 Linux 最大的认知台阶,也是理解虚拟内存(第十章)、进程(第五卷)的钥匙。学完你要能画出地址翻译流程、解释 TLB 和缺页。

9.1 从一个崩溃说起：没有 MMU 的世界

回到你熟悉的 STM32(无 MMU)。你写:

int *p = (int*)0x20001000;   // 直接写一个内存地址
*p = 123;                    // 直接就写进去了

在裸机/无 MMU 系统里,你写的地址就是真实物理内存的地址,直接命中 RAM 的那个位置。方便,但灾难性地危险:

  • 你算错一个指针,写到了别的任务的数据区——直接踩坏,系统诡异崩溃(第五章栈溢出那种)。
  • 两个程序都想用地址 0x20001000?冲突,没法共存。
  • 任何程序都能读写任何地址,毫无隔离和安全可言。

当系统只跑几个你自己写的、测试过的任务(RTOS 场景),这还能忍。但当系统要跑几百个来源不明、随时可能有 bug 或恶意的程序(Linux/手机/服务器场景),这绝对不能忍。MMU 就是为了解决"多个不可信程序如何安全共享一块物理内存"而生。

9.2 三个必须解决的问题

要让众多程序安全共享物理内存,必须同时解决三件事:

  1. 隔离:A 进程绝对不能访问 B 进程的内存(第一章"独立操作台")。
  2. 地址冲突:每个程序都想用简单固定的地址(比如都从 0 开始),不能因为物理内存就一份而打架。
  3. 灵活分配:程序需要的内存有大有小、随时变化,物理内存还可能不连续、甚至不够用(要能用硬盘顶上,第十章)。

一个精妙的机制能同时解决这三个——就是"地址翻译"。

9.3 核心思想：加一层地址翻译

计算机领域有句名言:"任何问题都可以通过增加一个中间层来解决。"MMU 就是这句话的完美体现。

核心思想:让程序用的地址,不再是真实的物理地址,而是一套"假地址"(虚拟地址)。CPU 和内存之间插一个硬件(MMU),负责把程序用的"假地址"实时翻译成真实的"物理地址"。

没有MMU:
  CPU ──(程序用的地址 = 物理地址)──▶ 内存   ← 直接、危险

有MMU:
  CPU ──(虚拟地址 VA)──▶ ┌─────┐ ──(物理地址 PA)──▶ 内存
                         │ MMU │  实时翻译
                         └─────┘  查"地图"(页表)
  • 程序里所有地址都是虚拟地址(Virtual Address, VA)——它是"假"的,是程序自己的一套坐标系。
  • 真实内存条上的地址是物理地址(Physical Address, PA)。
  • MMU 按一张"地图"(页表)把 VA 翻译成 PA。不同进程用不同的地图,于是同一个 VA(比如都用 0x1000),在 A 进程被翻译到物理位置 X,在 B 进程被翻译到物理位置 Y——互不干扰,还都能用简单地址,三个问题一次解决!

这就是 MMU 的全部精髓:用"每个进程一套虚拟地址 + 各自的翻译地图"制造出"我独占整个内存"的假象,同时用地图实现隔离。A 的地图里根本没有指向 B 内存的条目,所以 A 连"表达"访问 B 的能力都没有——隔离是从根上做到的。

9.4 生活类比：前台转接电话

把 MMU 想成公司前台。每个员工(进程)对外只知道一个"分机号"(虚拟地址),比如都用"1 号线"。外面打进来找不同部门的"1 号线",前台(MMU)根据"是哪个部门打来的"(哪个进程),查转接表(页表),把同样的"1 号线"转到不同的真实电话(物理地址)。

  • 员工不需要知道真实的外线号码,只管用简单的分机号——程序不用管物理地址。
  • 不同部门可以用相同的分机号而不冲突——不同进程可用相同虚拟地址。
  • 转接表里没有的号码,前台拒绝转接——没权限的地址访问不了。

9.5 分页：把内存切成小块

翻译不能按每个字节一条一条记(那地图比内存还大)。办法:把内存切成固定大小的小块,以"块"为单位翻译。这个块叫页(Page),通常 4KB。

  • 虚拟地址空间被切成一个个 虚拟页(Virtual Page);
  • 物理内存被切成一个个同样大小的 物理页框(Page Frame);
  • 翻译就是"这个虚拟页 → 对应哪个物理页框"。页内的偏移(4KB 内是第几个字节)不用翻译,直接照搬。
虚拟地址(以32位、4KB页为例) 拆成两部分:

  31            12 11         0
 ┌───────────────┬────────────┐
 │  虚拟页号 VPN  │  页内偏移    │
 │  (20位)       │  (12位=4KB) │
 └───────────────┴────────────┘
         │              │
     查页表翻译       直接照搬(页内位置不变)
         ▼              ▼
 ┌───────────────┬────────────┐
 │  物理页框号 PFN │  页内偏移    │  = 物理地址
 └───────────────┴────────────┘

为什么选 4KB?太小则页表项太多(地图太大),太大则浪费(一个程序只用几字节也得给它一整页)。4KB 是几十年权衡下的经典折中(也有 2MB、1GB 的"大页"用于特殊场景,减少页表开销)。

9.6 页表：虚拟页到物理页的地图

那张"翻译地图"就是页表(Page Table)。它本质是个数组:下标是虚拟页号,内容是对应的物理页框号(以及权限位)。每个条目叫页表项(PTE, Page Table Entry)。

页表(每个进程一张):
  虚拟页号   →   页表项(物理页框号 + 权限位)
  ┌──────┬──────────────────────────┐
  │ VPN0 │ PFN=5,  可读可写, 有效     │
  │ VPN1 │ PFN=99, 只读,   有效       │
  │ VPN2 │ ------  无效(没映射)       │ ← 访问它→缺页异常
  │ ...  │ ...                       │
  └──────┴──────────────────────────┘

关键点:

  • 每个进程有自己的一张页表——这就是"各自的地图"、隔离的根源。
  • 页表存在物理内存里。有个专门的寄存器指向当前进程的页表基地址(ARM64 叫 TTBR,Translation Table Base Register)。
  • 切换进程 = 换 TTBR 指向另一张页表 = 换一整套地图 = 瞬间切换整个地址空间的视图。第五卷 task_struct 里的 mm(第一章见过!)就管着这张页表。

9.7 多级页表：为什么不用一张大表

问题:32 位地址、4KB 页,有 2²⁰≈100 万个虚拟页,一张表就要 100 万项。64 位更是天文数字。而且大部分虚拟地址空间是空的(没映射)——为一堆空条目预留巨大的表,极其浪费。

解决:多级页表(像书的多级目录)。不用一本厚厚的、每页都列出来的索引,而是"卷→章→节→页"逐级查找,只为真正用到的部分建立下级表。

多级页表(以两级示意):

虚拟地址: [一级索引][二级索引][页内偏移]
             │         │
             ▼         │
        一级页表 ───▶ 二级页表 ───▶ 物理页框
        (目录)        (只为用到的区域建立)

好处: 没用到的地址范围, 对应的二级页表根本不创建 → 省内存
ARM64 常用 3~4 级页表(L0/L1/L2/L3)

代价:一次翻译要查好几级表,每级都是一次内存访问。翻译一个地址要访存 3~4 次!这慢得无法接受——于是需要下面的 TLB 来救场。

9.8 一次地址翻译全过程

把上面串起来,CPU 访问一个虚拟地址时发生什么(先不考虑 TLB):

CPU 要访问虚拟地址 VA
     │
     ▼
① 从 TTBR 找到当前进程页表基址
     │
     ▼
② 用 VA 的各级索引, 逐级查页表(每级一次访存)
     │
     ▼
③ 得到最终页表项 PTE
     │
   ┌─┴─────────────┐
 有效且有权限      无效/越权
   │               │
 ④取出物理页框号   触发缺页/权限异常(9.11)
   拼上页内偏移
   = 物理地址PA
   │
   ▼
⑤ 用 PA 去(经过Cache)访问真实内存

注意:每次访存都要先翻译,翻译本身又要多次访存——如果每次都这么走,性能崩溃。这就是 TLB 存在的理由。

9.9 TLB：翻译也要缓存

翻译太慢(多级查表),而地址翻译也有局部性(程序反复访问同一批页)。既然有局部性,就能缓存——把"最近用过的 VA→PA 翻译结果"存进一个小而快的缓存,叫 TLB(Translation Lookaside Buffer,translation 旁路缓存)。它就是"页表的 Cache"。

CPU 要翻译 VA:
     │
     ▼
  TLB 里有 VA 的翻译吗?
   ┌──┴──┐
 有(TLB hit)      没有(TLB miss)
   │               │
 直接拿到PA        去查多级页表(慢) → 把结果存入TLB → 再用
 ~1周期            几次访存
  • TLB 命中率也很高(局部性),所以绝大多数翻译是 1 周期搞定的,多级页表的慢只在 miss 时才付出。
  • TLB 和 Cache 是两个不同的缓存:TLB 缓存"地址翻译",Cache 缓存"数据/指令"。别混。一次访存:先 TLB 翻译地址,再拿物理地址去 Cache 找数据。

第一章那个面试加分点"上下文切换导致 TLB 失效",现在能讲透了:切换进程时换了页表,旧进程缓存在 TLB 里的翻译全部作废(它们是按旧地图翻的),必须清空 TLB。新进程开跑时要重新一条条翻译、重填 TLB,这批 TLB miss 就是进程切换的隐性大成本之一(和第八章 Cache 冷启动并列)。为缓解,现代 ARM 给 TLB 项打上 ASID(地址空间标识),标明"这条翻译属于哪个进程",切换时就不必全清,不同进程的翻译能共存于 TLB——这是重要优化,第二卷会提。

9.10 页表项里的权限位：保护的实现

MMU 不只翻译地址,还顺便做权限检查——这是"内存保护"的硬件实现。每个页表项(PTE)除了物理页框号,还有一堆权限位:

权限位作用
有效位(Valid)这个虚拟页有没有映射;无效→缺页异常
读/写权限(R/W)能不能写;往只读页写→异常(如写代码段被拦)
用户/内核(U/K)用户态能不能访问;用户态碰内核页→异常(第一章的 EL0/EL1 保护落地在这!)
可执行(X / NX)这页的内容能不能当指令执行;数据页标 NX 防止执行注入的代码(防栈溢出攻击)

翻译地址的同时,MMU 硬件检查这些位,违规立刻触发异常。于是:

  • 用户程序访问内核内存 → 被拦(内核/用户位);
  • 程序试图修改自己的代码段 → 被拦(只读);
  • 往栈上注入代码并跳去执行 → 被拦(NX,第五章的栈溢出攻击防御);
  • 访问一个没映射的地址(野指针、NULL)→ 缺页异常 → 段错误。

这才是"内存保护"的真相:不是软件在检查,而是 MMU 在每次访存翻译时用硬件顺手检查权限位,零额外开销。第一章讲的用户态/内核态隔离、进程间隔离,物理上就落实在这些页表权限位上。你的野指针之所以能被 Linux 抓成 Segmentation Fault 而不是默默踩坏内存(对比无 MMU 的裸机),全靠这里。

9.11 缺页异常(Page Fault)

当 MMU 翻译时发现"这个虚拟页无效/没映射/越权",它触发一个异常(第七章的异常机制!)叫缺页异常(Page Fault),把控制权交给内核处理。缺页不一定是错误,分两种:

  • 正常的缺页(内核借机干活):比如你 malloc 了内存但还没用,内核暂时没给真实物理页(懒分配);你第一次访问时缺页,内核这时才分配一个物理页、填好页表,然后让你的指令重新执行,你毫无感觉。这是虚拟内存的核心技巧,第十章详讲(按需分页、写时复制、换页)。
  • 真正的错误:访问野指针、NULL、越权——内核发现这是非法访问,给进程发 SIGSEGV 信号,就是你熟悉的 段错误(Segmentation Fault)。

所以"段错误"的本质你现在完全清楚了:你的程序访问了一个页表里没有合法映射的虚拟地址,MMU 翻译失败触发缺页异常,内核判定为非法,杀掉你的程序。

9.12 MMU 与上下文切换 / TLB 刷新

收束到操作系统主线。进程切换时,和 MMU 相关的动作:

切换进程 A → B:
  ① 保存 A 的寄存器现场(前几章)
  ② 把 TTBR 指向 B 的页表        ← 换地图!整个地址空间视图切换
  ③ 处理 TLB(有ASID则无需全清, 否则清空)
  ④ 恢复 B 的寄存器现场 → B 运行, 它的虚拟地址现在按B的地图翻译

关键对比,把第一章的"进程 vs 线程"讲到硬件层:

  • 切换进程:要换 TTBR(换页表/地址空间),伴随 TLB 和 Cache 冲刷 → 贵。
  • 切换同进程的线程:同一个地址空间、同一张页表,不用换 TTBR、不用刷 TLB → 便宜得多。

这就从硬件层解释了第一章的结论:"进程间切换比线程间切换贵",贵就贵在换地址空间(换页表)导致的 TLB/Cache 失效。线程共享地址空间,切换时省了这一大笔。而 RTOS 通常压根没有 MMU、所有任务共享一个地址空间,切换时既不换页表也没 TLB 问题,所以又快又确定——第一章说的"RTOS 切换快"的根本原因,到这里给全了。第十章会在 MMU 这个地基上,盖起"虚拟内存"的完整大厦(按需分页、swap、mmap、写时复制)。

9.13 面试常问问题

Q1:MMU 是干什么的?解决什么问题?
把虚拟地址翻译成物理地址,并做权限检查。解决多程序安全共享物理内存:隔离、地址不冲突、灵活分配。

Q2:虚拟地址怎么翻译成物理地址?
分页:VA=虚拟页号+页内偏移,用虚拟页号查(多级)页表得物理页框号,拼上偏移得 PA。TTBR 指向当前进程页表。

Q3:为什么用多级页表?
单级表太大且大量空条目浪费。多级只为用到的地址范围创建下级表,省内存;代价是多次访存,靠 TLB 弥补。

Q4:TLB 是什么?和 Cache 区别?
页表的缓存,缓存 VA→PA 翻译结果,靠翻译的局部性提速。Cache 缓存数据/指令,TLB 缓存地址翻译,两个不同缓存。

Q5:内存保护/进程隔离怎么实现的?
每进程独立页表(A 的地图里没 B 的映射);翻译时 MMU 硬件检查 PTE 权限位(读写/用户内核/NX/有效),违规触发异常。

Q6:段错误(Segmentation Fault)的本质?
访问页表里无合法映射或越权的虚拟地址,MMU 触发缺页异常,内核判定非法,发 SIGSEGV。

Q7:为什么进程切换比线程切换贵?和 MMU 什么关系?
进程切换要换页表(TTBR),导致 TLB 失效、Cache 冷启动;线程共享地址空间无需换,省了这些。

9.14 常见误区

  1. "程序里的地址就是物理地址"——有 MMU 时全是虚拟地址,经翻译才到物理。
  2. "页表在 CPU 里"——页表在物理内存里,TTBR 寄存器指向它;TLB 才在 CPU 里缓存翻译。
  3. "TLB 就是 Cache"——不同东西:TLB 缓存地址翻译,Cache 缓存数据。
  4. "缺页一定是错误"——很多是正常的按需分页,内核借机分配后重执行指令;只有非法访问才是段错误。
  5. "内存保护是操作系统软件逐次检查"——是 MMU 硬件在翻译时用权限位检查,几乎零开销。
  6. "进程和线程切换成本一样"——进程切换多了换页表+TLB失效,更贵。

9.15 本章总结

  • 无 MMU 时地址即物理地址,方便但无隔离无保护,不适合跑众多不可信程序。
  • MMU 核心思想:加一层地址翻译,程序用虚拟地址,MMU 按页表翻译成物理地址。
  • 分页:内存切成 4KB 页,以页为单位翻译;VA=虚拟页号+页内偏移。
  • 页表是翻译地图,每进程一张,存内存里,TTBR 指向它;换进程=换 TTBR=换地图=隔离的根源。
  • 多级页表省内存(只建用到的部分),代价是多次访存,靠 TLB(翻译的缓存)弥补。
  • PTE 里的权限位(读写/用户内核/NX/有效)由 MMU 翻译时硬件检查,实现内存保护;违规触发异常。
  • 缺页异常:正常(按需分页,内核分配后重执行)或错误(非法访问→SIGSEGV 段错误)。
  • 回扣:进程切换换页表致 TLB/Cache 失效,故比线程切换贵;RTOS 无 MMU 故切换又快又确定。ASID 可减少 TLB 全清。

9.16 思维导图

MMU
├─ 一、无MMU:地址即物理,无隔离无保护
├─ 二、要解决:隔离/地址不冲突/灵活分配
├─ 三、核心思想:加一层地址翻译(VA→PA)
│  └─ 每进程一套虚拟地址+各自地图=独占假象+隔离
├─ 四、类比:前台按部门转接同一分机号
├─ 五、分页:内存切4KB页,VA=页号+页内偏移
├─ 六、页表:翻译地图,每进程一张,TTBR指向
│  └─ 换进程=换TTBR=换地图(task_struct.mm)
├─ 七、多级页表:省内存(只建用到的),代价多次访存
├─ 八、翻译全过程:TTBR→逐级查表→PTE→拼PA
├─ 九、TLB:翻译的缓存(≠数据Cache)
│  └─ 切换换页表→TLB失效(隐性成本);ASID减少全清
├─ 十、PTE权限位:读写/用户内核/NX/有效
│  └─ MMU硬件翻译时检查=内存保护;落实EL0/EL1隔离
├─ 十一、缺页异常:正常(按需分页)/错误(SIGSEGV段错误)
└─ 十二、回扣:进程切换贵在换页表+TLB失效;RTOS无MMU故快

9.17 课后练习与面试题

课后练习

  1. Linux 上写程序打印某变量的地址,多次运行(或多进程同时跑)观察虚拟地址可能相同,理解"不同进程可用相同虚拟地址"。
  2. 写一个访问 NULL 或野指针的程序,触发段错误,用 dmesg 或 core dump 看信息,对照 9.11 理解其本质。
  3. 用 cat /proc/self/maps 看当前进程的虚拟地址空间布局(代码/数据/堆/栈/库),对照第五章内存布局图。
  4. 写一个 malloc 大块内存但不写入 vs 写入的程序,用 top 看 VIRT 和 RES 的差异,初步感受按需分页(第十章深入)。
  5. 查 RK3588(ARMv8)的页大小和页表级数,了解 TTBR0/TTBR1 分别管用户/内核空间。
