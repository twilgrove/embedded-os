本章目标：终于到 Linux 内核。别怕它的几千万行代码——它的骨架和你已经吃透的 FreeRTOS/RT-Thread 是同一套:进程就是"任务/线程"、task_struct 就是"TCB/rt_thread"、调度就是"选最高优先级"、切换就是"存恢复寄存器换栈"。这一章先建立全局地图,然后深入 task_struct——Linux 里"一个进程/线程"的化身,相当于放大版、字段多几百倍的 TCB。学完你要能对照着 RTOS 理解 Linux 进程的本质。源码基于 Linux 6.x。

28.1 带着 RTOS 的认知看 Linux

很多人学 Linux 内核直接懵，因为它太大。但你有别人没有的优势:你已经在 FreeRTOS/RT-Thread 里完整理解了一个操作系统内核的每个核心机制。Linux 只是把这些机制做得更复杂、更强大,以应对"通用、多核、多用户、跑海量不可信程序"的需求。核心概念的映射:

RTOS 概念(你已懂)Linux 对应复杂在哪
任务/线程 + TCB进程/线程 + task_struct+ 地址空间、文件、信号、命名空间...
位图选最高优先级CFS 红黑树 + 多调度类为公平、多核负载均衡
PendSV 上下文切换switch_to / __switch_to+ 切地址空间、TLB、FPU
队列/信号量/互斥量同名 + 管道/共享内存/socket+ 跨进程、支持阻塞在 VFS
关中断/挂起调度自旋锁/RCU/内存屏障多核并发
heap/内存池Buddy + SLAB+ 虚拟内存、按需分页

学习心法:每遇到一个 Linux 机制,先问"它对应 RTOS 的哪个?多解决了什么问题?"这样你就不是在学一堆陌生东西,而是在给已有的骨架"升级"。Linux 复杂性的来源就三个:①多核(需要精细的并发控制)②虚拟内存/隔离(MMU,第九、十章)③通用性(要照顾各种场景,不能像 RTOS 那样简单粗暴)。抓住这三点,Linux 的复杂就有了逻辑,不再是无序的细节。

28.2 宏内核：Linux 的架构选择

操作系统内核架构有两大流派,面试常问:

宏内核(Monolithic)Linux微内核(Microkernel)
理念所有核心服务(调度/内存/文件/驱动/网络)都在内核态,一个大程序内核只留最核心(调度/IPC/内存),文件/驱动等放用户态服务进程
性能高(服务间直接函数调用)较低(服务间要 IPC 通信,频繁陷入)
可靠性一个驱动崩溃可能拖垮内核一个服务崩溃不影响内核(隔离)
代表LinuxQNX、seL4、Mach、部分 RT-Thread 配置

Linux 选了宏内核(为性能),但用可加载内核模块(LKM)缓解了它的僵化——驱动可以动态加载/卸载(insmod/rmmod),不用重编内核。所以 Linux 是"带模块化的宏内核"。这个选择解释了很多:为什么 Linux 驱动跑在内核态(第六卷)、为什么一个坏驱动能搞崩系统(kernel panic)、为什么 Linux 性能好。这类做实时机器人的,有时会关注微内核(如 QNX)的可靠性优势,面试可能对比问。

28.4 task_struct：进程/线程的化身

Linux 里"一个进程/线程"就是一个 struct task_struct(在 include/linux/sched.h)——相当于 FreeRTOS 的 TCB、RT-Thread 的 rt_thread,但大得多(有几百个字段,结构体本身几 KB)。第一章 1.12 惊鸿一瞥过它,现在正式深入。

/* include/linux/sched.h —— 极度精简, 只列灵魂字段 */
struct task_struct {
    /* ── 状态与调度 ── */
    volatile long   __state;        /* 进程状态(运行/睡眠/僵尸...) */
    void            *stack;         /* 指向内核栈 */
    int             prio, static_prio, normal_prio;  /* 优先级(多个) */
    const struct sched_class *sched_class;  /* ★调度类(CFS/RT/...) */
    struct sched_entity   se;       /* CFS 调度实体(红黑树节点在里面) */

    /* ── 内存 ── */
    struct mm_struct  *mm;          /* ★地址空间(进程的虚拟内存, 第九/十章!) */
    struct mm_struct  *active_mm;

    /* ── 身份 ── */
    pid_t             pid;          /* 进程ID(其实是线程ID) */
    pid_t             tgid;         /* 线程组ID(用户看到的"进程ID") */

    /* ── 进程关系 ── */
    struct task_struct  *parent;        /* 父进程 */
    struct list_head    children, sibling; /* 子进程/兄弟进程链表 */

    /* ── 打开的文件 ── */
    struct files_struct *files;     /* 打开的文件表(第32章) */
    struct fs_struct    *fs;        /* 工作目录等 */

    /* ── 信号 ── */
    struct signal_struct  *signal;  /* 信号相关(第33章) */
    struct sighand_struct *sighand;

    /* ── 命名空间(容器的基础) ── */
    struct nsproxy    *nsproxy;

    char              comm[TASK_COMM_LEN];  /* 进程名(如"bash") */
    /* ... 还有几百个字段 ... */
};

别被字段数量吓到。它就是个"大号 TCB":TCB 有的(栈、优先级、状态、链表节点),它都有;它多出来的,全是 Linux 特有的复杂性——地址空间(mm)、打开的文件(files)、信号、命名空间等。这些"多出来的",正是第一章说的"Linux 比 RTOS 多做的隔离与通用性"的体现。

28.5 和 TCB/rt_thread 的对照

直接对照三者的核心字段,你会豁然开朗——Linux 进程本质就是放大的 RTOS 任务:

作用FreeRTOS TCBRT-Thread rt_threadLinux task_struct
栈pxTopOfStackspstack(内核栈)
优先级uxPrioritycurrent_priorityprio + sched_class
状态(挂在哪个链表)stat__state
就绪/等待链表节点xStateListItemtlistse(CFS)/链表节点
名字pcTaskNameparent.namecomm
地址空间❌ 无❌ 无✅ mm(关键区别!)
打开的文件❌❌✅ files

第一章 1.12 那个洞察在这里完整验证了:Linux 的 task_struct 和 RTOS 的 TCB,最本质的区别就是那个 mm 字段(地址空间)。RTOS 任务共享一个地址空间(所以 TCB 没有 mm),Linux 进程各有独立地址空间(所以 task_struct 有 mm,指向独立的页表,第九章)。这一个字段,就是"进程 vs 任务""有 MMU vs 无 MMU""隔离 vs 不隔离"的全部分野。而 files、signal、nsproxy 等,是 Linux 作为通用多用户系统额外需要的。你带着 RTOS 的认知看,task_struct 就不再是几百个陌生字段,而是"TCB + 一堆 Linux 特有的东西"。

28.6 task_struct 的关键字段群

把几百个字段按"功能群"归类,理解就有条理了。每个群对应内核的一个子系统:

task_struct 的字段群(每群对应一个子系统):

  ┌─ 调度群: __state, prio, sched_class, se ──→ 第30章调度
  ├─ 内存群: mm, active_mm ──────────────────→ 第31章内存
  ├─ 文件群: files, fs ──────────────────────→ 第32章VFS
  ├─ 信号群: signal, sighand, pending ────────→ 第33章信号
  ├─ 身份群: pid, tgid, cred(权限) ──────────→ 本章+安全
  ├─ 关系群: parent, children, sibling ───────→ 本章进程树
  ├─ 命名空间群: nsproxy ────────────────────→ 容器(Docker基础)
  └─ 时间统计群: utime, stime(用户/内核态耗时)→ top命令数据来源

看 task_struct 就像看一个人的完整档案:它的调度信息、拥有的内存、打开的文件、待处理的信号、身份、家庭关系(父子进程)……都记在这一个结构里。内核对进程的所有操作,归根结底都是在读写这个结构的某些字段。你后面每一章(调度、内存、文件、信号),其实都是在深入 task_struct 的某个字段群。task_struct 是整个 Linux 进程管理的中心枢纽,就像 TCB 是 FreeRTOS 的中心一样。

28.7 进程状态

Linux 进程状态(__state),比 RTOS 多几个,但核心一样:

状态含义RTOS 对应
TASK_RUNNING就绪或正在运行(在运行队列里)Ready/Running
TASK_INTERRUPTIBLE可中断睡眠(等事件,能被信号唤醒)Blocked
TASK_UNINTERRUPTIBLE不可中断睡眠(等关键 IO,信号叫不醒)Blocked(更强)
__TASK_STOPPED被停止(收到 SIGSTOP,如 Ctrl+Z)Suspended
EXIT_ZOMBIE僵尸:进程死了但父进程还没回收它(RTOS 无)
EXIT_DEAD最终死亡,即将被清理(Deleted)

两个 Linux 特有的重要状态:

①可中断 vs 不可中断睡眠:可中断睡眠(等一般事件)能被信号打断唤醒;不可中断睡眠(如等磁盘 IO 的关键阶段)连信号都叫不醒——这就是为什么有时进程卡死、kill -9 都杀不掉(它在 TASK_UNINTERRUPTIBLE,俗称 "D 状态",ps 里显示 D)。

②僵尸进程(Zombie):进程 exit 后,它的大部分资源被释放,但 task_struct 还留着(保留退出状态),等父进程调 wait() 来"收尸"读取退出码。如果父进程不 wait,就成了僵尸——占着 PID 不释放。这是 Linux 特有的、面试高频的概念(RTOS 没有,因为 RTOS 任务没有"父子关系"和"退出码回收"的语义)。

28.8 PID 与进程关系

Linux 进程有身份和家族关系,这是 RTOS 没有的:

  • PID vs TGID:一个易混点。pid 其实是线程ID,tgid(线程组 ID)才是用户 ps 看到的"进程 ID"。一个多线程进程里,所有线程共享同一个 tgid(=主线程的 pid),但各有不同的 pid。在 Linux 内核眼里,线程就是"共享了地址空间等资源的进程"(第29章 clone)——所以内核用 task_struct 同时表示进程和线程,用 tgid 把同一进程的线程归组。
  • 进程树:每个进程有父进程(parent)、子进程链表(children)。所有进程构成一棵树,根是 init/systemd(PID 1)。fork 创建子进程(第29章)。
  • 孤儿与僵尸:父进程先死,子进程成"孤儿",被 PID 1 收养;子进程先死父进程没收尸,成"僵尸"。

"线程就是共享资源的进程"是 Linux 一个深刻而独特的设计。Windows 等系统把进程和线程做成两种不同的东西;Linux 则统一用 task_struct 表示,通过"创建时共享哪些资源"来区分:完全独立=进程,共享地址空间/文件等=线程。这就是为什么 Linux 的线程叫"轻量级进程(LWP)"。第29章讲 clone 时会看到,fork(建进程)和 pthread_create(建线程)底层都是同一个 clone 系统调用,只是参数(共享哪些资源)不同。这个"统一表示、参数区分"的设计极其优雅,是理解 Linux 进程/线程的钥匙。

28.9 内核栈与 thread_info

每个进程有两个栈(RTOS 通常一个):

  • 用户栈:进程在用户态(EL0)运行时用的栈,在进程的虚拟地址空间里(第五、十章的栈)。
  • 内核栈:进程陷入内核态(系统调用、中断,第十二章)后用的栈,由内核分配(通常 8KB 或 16KB),task_struct->stack 指向它。

为什么要两个栈?因为用户态和内核态是隔离的(第十二章 EL0/EL1)。进程通过系统调用进内核后,不能用用户栈(不安全、可能不够),要切到专属的内核栈。这对应第十二章 ARM64 的 SP_EL0(用户栈)/SP_EL1(内核栈),你已经懂了。

历史上,内核栈底部还放了个 thread_info 结构(存一些底层标志、指回 task_struct),用"栈指针对齐后取低位"就能快速找到它——这是个经典技巧。新版内核多把 thread_info 移进了 task_struct,但概念仍在。

28.10 current：如何找到当前进程

内核代码里到处用 current 宏表示"当前正在运行的进程的 task_struct"——相当于 FreeRTOS 的 pxCurrentTCB、RT-Thread 的 rt_current_thread。但 Linux 的 current 实现更精巧:

  • 老办法(x86 早期):内核栈是 8KB 对齐的,栈底放 thread_info,thread_info 里有指向 task_struct 的指针。用当前 SP掩掉低位就得到栈底、进而拿到 current。不用全局变量,靠"栈指针算出来"。
  • 现代(ARM64/x86-64):用一个专用寄存器存 current(ARM64 用 sp_el0 在内核态被复用来存 current;x86-64 用 per-CPU 变量 + %gs 寄存器)。每个 CPU 核有自己的 current(因为多核每核跑不同进程)。

对比 RTOS:FreeRTOS 单核,一个全局 pxCurrentTCB 就够。Linux 多核,每个 CPU 核都在跑不同的进程,所以 "current" 必须是 per-CPU 的——不能用一个全局变量。这是"多核"带来复杂性的一个小例子:RTOS 里理所当然的全局变量,到多核 Linux 就得变成 per-CPU。per-CPU 变量(每个核一份)是 Linux 内核处理多核的核心手法之一,你会在调度队列(每核一个 runqueue)、软中断、内存分配缓存里反复见到。记住这个从"单核全局"到"多核 per-CPU"的转变,是理解 Linux 多核设计的一把钥匙。

28.11 进程的组织：链表与红黑树

内核怎么管理成百上千个 task_struct?用多种数据结构,每种服务一个目的(和 RTOS 用链表管 TCB 同理,但更丰富):

数据结构组织什么目的
双向链表(task_list)所有进程串起来遍历所有进程(如 ps)
红黑树(每个 CFS 运行队列一棵)就绪进程按虚拟运行时间排序O(log n) 选下一个跑谁(第30章)
哈希表(pid hash)PID → task_struct按 PID 快速查找进程
父子/兄弟链表进程树fork/wait/信号传递

关键:同一个 task_struct 同时挂在多个数据结构上(通过内嵌的多个链表节点/树节点)——这正是第16章 FreeRTOS "侵入式链表、一个 TCB 挂多个列表"的放大版!task_struct 里有多个 list_head/rb_node 字段,分别把它挂进不同的组织结构。Linux 内核的 list_head 侵入式链表和 container_of 宏,就是你在 FreeRTOS/RT-Thread 见过的"侵入式链表 + 从节点反查宿主"技巧的正统实现。第25章 RT-Thread 的对象容器预习了这个,现在在 Linux 里见到全貌。

28.12 面试常问问题

Q1:Linux 的 task_struct 和 RTOS 的 TCB 有什么关系和区别?
本质都是"进程/任务的控制块"。task_struct 是放大版,多了地址空间(mm)、打开文件(files)、信号、命名空间、进程关系等。最本质区别是 mm 字段——进程有独立地址空间,RTOS 任务共享。

Q2:Linux 里进程和线程的区别?内核怎么表示?
内核用同一个 task_struct 表示两者;线程是"共享地址空间等资源的进程"(轻量级进程)。同进程的线程共享 tgid,各有独立 pid。fork 和 pthread_create 底层都是 clone,区别在共享哪些资源。

Q3:pid 和 tgid 的区别?
pid 是线程 ID,tgid(线程组 ID)是用户看到的进程 ID。多线程进程里所有线程共享 tgid=主线程 pid,各有不同 pid。

Q4:什么是僵尸进程?怎么产生和避免?
进程退出后 task_struct 还留着(保存退出状态)等父进程 wait 回收;父进程不 wait 就成僵尸,占 PID。避免:父进程 wait/waitpid,或用 SIGCHLD 处理,或双 fork。

Q5:可中断睡眠和不可中断睡眠的区别?
可中断(TASK_INTERRUPTIBLE)等一般事件,能被信号唤醒;不可中断(D 状态)等关键 IO,信号叫不醒,kill -9 都杀不掉。

Q6:current 是什么?多核下怎么实现?
当前运行进程的 task_struct(类比 pxCurrentTCB)。多核下是 per-CPU 的(每核一份),用专用寄存器或 per-CPU 变量实现,不能用全局变量。

Q7:宏内核和微内核的区别?Linux 是哪种?
宏内核所有服务在内核态(性能高,一处崩溃可能拖垮全部);微内核只留核心,其余放用户态服务(隔离好,IPC 开销大)。Linux 是宏内核 + 可加载模块。

28.13 常见误区

  1. "Linux 内核和 RTOS 完全不同要重新学"——骨架相同,是放大升级,概念可迁移。
  2. "进程和线程是两种完全不同的东西"——Linux 内核用同一个 task_struct,线程是共享资源的进程。
  3. "pid 就是进程 ID"——pid 是线程 ID,tgid 才是用户看的进程 ID。
  4. "僵尸进程占大量内存"——主要占的是 PID 和一个 task_struct,危害是耗尽 PID,不是内存。
  5. "kill -9 能杀死任何进程"——不可中断睡眠(D 状态)的进程杀不掉,要等 IO 完成。
  6. "current 是全局变量"——是 per-CPU 的,多核每核一份。
  7. "task_struct 太复杂没法理解"——按字段群(对应各子系统)理解,就是"大号 TCB+Linux 特有字段"。

28.14 本章总结

  • 带 RTOS 认知看 Linux:概念一一映射,复杂性来自多核、虚拟内存/隔离、通用性三点。
  • Linux 是宏内核 + 可加载模块(性能优先,驱动可动态加载)。
  • task_struct = 放大版 TCB/rt_thread,几百字段,是进程/线程管理的中心枢纽。
  • 与 TCB 最本质区别是 mm(独立地址空间)字段——进程 vs 任务、隔离 vs 共享的分野。
  • 字段按群理解:调度/内存/文件/信号/身份/关系/命名空间——每群对应一个子系统。
  • 状态:RUNNING/可中断睡眠/不可中断睡眠(D)/停止/僵尸(Linux 特有)。
  • 线程=共享资源的进程;pid(线程)vs tgid(进程);fork 和 pthread_create 底层都是 clone。
  • 两个栈:用户栈 + 内核栈(对应 SP_EL0/SP_EL1);current 是 per-CPU 的(多核关键)。
  • 进程用链表(遍历)+ 红黑树(调度)+ 哈希(按PID查)+ 进程树组织,靠侵入式 list_head(FreeRTOS 侵入式链表的正统版)。
