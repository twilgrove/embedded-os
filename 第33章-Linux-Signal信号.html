内核给进程发的"软件中断"

本章目标:你按 Ctrl+C 停掉程序,用过 kill -9,程序崩溃时看到 "Segmentation fault"——这些背后都是信号(signal)。信号是内核向进程发送的异步通知,本质是"软件层面的中断"。学完第七章硬件中断,你会发现信号就是它在软件/进程层的翻版:同样是"打断正常流程去处理突发事件"。

33.1 为什么需要信号

进程正在运行,突然发生了一件它没预料到、但必须知道的事:用户按了 Ctrl+C 想终止它、它访问了非法内存、它的子进程死了、定时器到点了……这些事件异步发生(不知道什么时候来),进程不可能一直"轮询"检查。需要一种机制,让内核能"打断"进程,通知它"出事了"。

这个需求和第七章硬件中断一模一样:CPU 正在跑,外设突然有事(按键、数据到了),需要打断 CPU 去处理。硬件中断解决"硬件→CPU"的异步通知;信号解决"内核→进程"的异步通知。所以信号又叫"软件中断"——它是中断思想在进程层的复刻。

33.2 生活例子：拍肩膀通知

你在专心写代码(进程在运行)。同事有急事要通知你,他不会等你自然停下(那太久),而是走过来拍你肩膀——你被打断,抬头处理这件事,处理完再回去写代码。这一"拍肩膀"就是信号:异步、打断当前工作、迫使你处理。

不同的拍法对应不同信号:轻拍(SIGTERM,"请你停一下")你可以说"等我保存一下再停";使劲拽走(SIGKILL,"立刻走")你没得商量必须走;告诉你"你孩子放学了"(SIGCHLD,子进程结束)。有的通知你能选择怎么回应(捕获处理),有的(SIGKILL)你无权拒绝。

33.3 信号就是软件中断

把信号和硬件中断(第七章)并列,你会秒懂信号的整个模型:

硬件中断(第七章)信号(本章)
触发方硬件设备内核 / 其他进程
打断谁CPU 当前执行流进程当前执行流
处理程序中断处理函数 ISR信号处理函数 handler
处理表中断向量表进程的信号处理表
能否屏蔽可屏蔽/不可屏蔽(NMI)可阻塞 / 不可阻塞(SIGKILL/STOP)
处理后返回被打断处继续返回被打断处继续

对应关系严丝合缝:信号 = 进程层面的中断。硬件中断有向量表映射"中断号→ISR",信号有处理表映射"信号号→handler";硬件中断可屏蔽(第七章 PRIMASK/GIC),信号可阻塞(信号掩码);都是"打断→处理→返回"。你在第七章建立的整个中断心智模型,直接搬到信号上就对了。这也是本书反复强调的:核心机制会在不同层次反复出现,学通一个,触类旁通。

33.4 常见信号一览

记住这些高频信号(面试和调试都要):

信号号含义默认动作
SIGINT2Ctrl+C 中断终止
SIGKILL9强制杀死(kill -9)终止(不可捕获/阻塞)
SIGTERM15请求终止(kill 默认)终止(可捕获,优雅退出)
SIGSEGV11段错误(非法内存访问)终止 + core
SIGCHLD17子进程状态变化(结束等)忽略
SIGSTOP/SIGCONT19/18暂停/继续进程停止/继续(STOP 不可捕获)
SIGALRM14定时器到点终止
SIGPIPE13写已关闭的管道/socket终止
SIGUSR1/210/12用户自定义终止

两个"不可抗拒"的信号:SIGKILL(9)和 SIGSTOP(19)——不能被捕获、阻塞或忽略。这是内核给自己留的"最终手段":无论进程怎么设置,内核总能用这俩杀死/暂停它(否则失控进程就没法收拾了)。所以 kill -9 是"必杀"。

33.5 信号的一生：产生 → 挂起 → 递送

一个信号从产生到被处理,走三步。这三步之间可能有延迟,是理解信号的关键:

① 产生(generate): 事件发生, 内核决定给进程发信号
      来源: 键盘(Ctrl+C)、kill命令、硬件异常(段错误)、内核(SIGCHLD)
              │
              ▼
② 挂起(pending): 信号记录在进程的"未决信号集"里
      —— 但进程可能正在运行/睡眠, 还没处理它
      —— 如果该信号被"阻塞", 就一直挂起, 等解除阻塞
              │
              ▼
③ 递送(deliver): 进程真正处理这个信号
      —— 在合适的时机(见33.8), 内核让进程执行handler或默认动作

核心洞察:信号的"产生"和"递送"之间有时间差。产生只是"记一笔"(设置未决位),真正处理(递送)要等一个合适时机。中间可以用"信号掩码"阻塞某些信号,让它们停在"挂起"状态暂不递送。这个"产生≠立即处理"的异步性,是信号所有复杂行为的根源。

33.6 三种处理方式

进程收到信号,有三种处置(用 signal()/sigaction() 设置):

方式说明
默认动作(SIG_DFL)按信号的默认行为(终止/忽略/停止),见 33.4 表
忽略(SIG_IGN)丢弃该信号(SIGKILL/STOP 除外)
捕获(自定义 handler)注册一个函数,信号来时执行它(SIGKILL/STOP 除外)
void handler(int sig){ printf("收到信号 %d\n", sig); }
int main(){
    signal(SIGINT, handler);   // 捕获Ctrl+C
    while(1) pause();          // 按Ctrl+C不再退出, 而是打印
}

捕获是最有用的:程序可以拦截 SIGTERM,在退出前做清理(存盘、关文件、通知对端)——这叫"优雅退出"。这就是为什么服务程序用 kill(SIGTERM,可捕获)而不是 kill -9(SIGKILL,不可捕获)——给程序一个善后的机会。的无人机/机器人程序退出前要安全停桨、保存状态,就靠捕获 SIGTERM。

33.7 内核数据结构

信号相关的状态存在 task_struct(第28章)里,核心是三样:

task_struct 里的信号相关字段:
  ├─ pending       未决信号集(哪些信号在挂起, 位图)
  ├─ blocked       信号掩码(哪些信号被阻塞, 位图)★
  └─ sighand       信号处理表(每个信号→handler/默认/忽略)
        struct sigaction action[64];  // 类比中断向量表!

判断"有信号要处理":  pending & ~blocked  != 0
  (有未决的, 且没被阻塞的)

两个位图 pending 和 blocked:pending 记"哪些信号来了还没处理",blocked 记"哪些信号被暂时挡住"。真正要递送的信号 = pending & ~blocked(来了且没被挡)。sighand 就是信号版的中断向量表——数组下标是信号号,内容是处理方式。fork 时子进程复制这些,exec 时捕获的 handler 被重置为默认(因为代码换了,handler 函数地址失效)。

33.8 信号在何时递送(关键)

最容易被忽略、面试最爱问的点:信号不是产生就立刻处理,而是在特定时机递送。什么时机?

信号的递送时机: 进程"从内核态返回用户态"的那一刻检查

  典型场景:
   ① 系统调用返回时(如read返回前, 内核检查有没有信号)
   ② 中断/异常处理返回用户态时(第七章的返回路径)
   ③ 从睡眠中被唤醒时

  内核在返回用户态前问: pending & ~blocked 有信号吗?
    有 → 先去执行信号handler(或默认动作), 再回用户代码

为什么是"返回用户态时"?因为这是内核能安全介入进程执行流的天然检查点。进程要么在跑用户代码(内核插不进去),要么在内核里(执行系统调用/被中断)。内核在"即将回到用户代码"的关口做检查,顺势把控制权导向 handler,是最自然的切入点。这直接解释了一个经典现象:一个卡在 read() 阻塞的进程,信号能"叫醒"它——信号唤醒睡眠的进程,使其从内核返回用户态,在返回点递送信号(read 则返回 EINTR)。这也和第30章"系统调用返回是检查 need_resched 抢占的时机"呼应——返回用户态是内核统一的"处理待办事项"关口:检查抢占、检查信号,都在这。

33.9 可靠信号与不可靠信号

历史遗留但面试会问:信号分两类。

不可靠信号(1-31)可靠信号(34-64,实时信号)
会不会丢同种信号连发多次,可能只留一个(不排队)排队,每个都送达
顺序无保证按顺序
携带数据不能能(sigqueue 带值)

关键坑:传统信号(如 SIGINT)不排队。如果一个信号还挂起未处理时,同种信号又来了,它们会"合并"成一个——因为 pending 是位图,一个信号只占一位,来 10 次也是那一位置 1。所以"收到 N 个 SIGCHLD"可能只触发一次 handler——这是处理子进程回收的经典 bug(必须在 SIGCHLD handler 里用 while(waitpid(...)) 循环回收所有子进程,不能假设一个信号对应一个子进程)。实时信号(34+)才排队、不丢。

33.10 异步信号安全(async-signal-safe)

一个进阶但重要的坑。信号 handler 是异步打断主流程执行的——就像中断打断主程序(第七章)。想象主程序正在 malloc(第31章)中途(改到一半内部链表),信号来了,handler 里也调 malloc……会破坏 malloc 的内部状态(重入问题)。

所以信号 handler 里只能调用"异步信号安全"的函数(可重入的,如 write、_exit),不能调 malloc、printf 等(它们不可重入,handler 打断主程序时它们可能正被执行到一半)。这和第七章"中断 ISR 里不能做耗时/阻塞操作、要注意重入"是同一个问题的两个层次!ISR 打断主程序、handler 打断主流程,都面临"打断了正在操作共享状态的代码"的重入风险。解决思路也一样:handler 尽量短,只设个标志位(volatile sig_atomic_t flag=1),真正的活留给主循环做——正如中断里"ISR 只置标志,主循环处理"。又一次,中断的经验直接迁移到信号。

33.11 回扣中断与 RTOS

信号这一章,处处是第七章中断的影子,把它们连起来:

概念硬件中断(第七章)信号(本章)
本质硬件异步打断 CPU软件异步打断进程
屏蔽机制PRIMASK / GIC 使能位信号掩码 blocked
不可屏蔽NMISIGKILL / SIGSTOP
处理表中断向量表sighand 处理表
递送/处理时机指令边界返回用户态时
重入陷阱ISR 不可重入 → 只置标志handler 只能调 async-safe → 只置标志

RTOS 里没有"信号"这个完整机制(FreeRTOS 用任务通知/事件组做类似的异步通知,第22章),但"异步打断 + 处理表 + 屏蔽 + 重入注意"这套思想是通用的。Linux 的信号,是把硬件中断的整套模型"抬"到进程层,让进程之间、内核对进程也能做"中断式"的异步通知。学到这里你应该形成一个统一认识:无论硬件中断、软件信号、还是 RTOS 的事件通知,都是"异步事件通知"这一需求的不同实现,共享同一套设计考量(屏蔽、优先级、重入、处理时机)。抓住这个统一模型,以后遇到任何"异步通知"机制都能快速理解。

33.12 面试常问问题

Q1:信号是什么?和中断什么关系?
信号是内核向进程发送的异步通知,本质是"软件中断"——进程层面的中断。有处理表、可屏蔽(掩码)、打断→处理→返回,和硬件中断模型完全对应。

Q2:SIGKILL 和 SIGTERM 的区别?
SIGTERM(15)可被捕获,程序能拦截做优雅退出(存盘、清理);SIGKILL(9)不可捕获/阻塞/忽略,内核直接杀,是必杀手段。服务程序退出应先发 SIGTERM。

Q3:信号从产生到处理经历什么?何时递送?
产生(设未决位)→挂起(pending,可能被 blocked 挡住)→递送(在进程"从内核返回用户态"时检查 pending&~blocked 并处理)。产生≠立即处理。

Q4:为什么信号能打断阻塞的 read?
信号唤醒睡眠进程,使其从内核返回用户态,在返回点递送信号,read 被中断返回 EINTR(需重试)。

Q5:传统信号会丢吗?SIGCHLD 处理要注意什么?
传统信号(1-31)不排队,同种多次挂起会合并成一个。所以 SIGCHLD handler 里必须 while(waitpid(-1,..,WNOHANG)>0) 循环回收所有子进程,不能假设一信号对应一子进程。

Q6:信号 handler 里能调 printf/malloc 吗?
不能。handler 异步打断主流程,若主流程正在 malloc/printf 中途,handler 再调会重入破坏状态。只能调 async-signal-safe 函数(write/_exit),最佳实践是 handler 只设标志位,主循环处理。

Q7:哪两个信号不可捕获?为什么?
SIGKILL 和 SIGSTOP。内核给自己保留的最终控制手段,保证任何进程都能被杀死/暂停,防止失控进程无法收拾。

33.13 常见误区

  1. "信号一发就立刻处理"——产生后先挂起,在返回用户态时才递送,有延迟。
  2. "SIGKILL 能被捕获做清理"——不能,SIGKILL/SIGSTOP 不可捕获;要清理用 SIGTERM。
  3. "发 N 个 SIGCHLD 就触发 N 次 handler"——传统信号不排队会合并,要循环 waitpid。
  4. "handler 里能随便调库函数"——只能调 async-signal-safe 的,别调 malloc/printf。
  5. "信号能携带大量数据"——传统信号只是个号;实时信号(sigqueue)才能带一个值。
  6. "信号是进程间通信的主力"——信号是通知(轻量、易丢),传数据用管道/共享内存/socket。

33.14 本章总结

  • 信号 = 进程层面的软件中断:内核/进程向进程发的异步通知,模型与硬件中断一一对应(处理表、屏蔽、打断-处理-返回)。
  • 常见:SIGINT(Ctrl+C)、SIGKILL(9,必杀不可捕获)、SIGTERM(15,可捕获优雅退出)、SIGSEGV(段错误)、SIGCHLD(子进程变化)。SIGKILL/SIGSTOP 不可捕获/阻塞。
  • 一生:产生(设未决位)→挂起(pending,被 blocked 掩码可挡)→递送(返回用户态时处理)。产生≠立即处理。
  • 三种处理:默认 / 忽略 / 捕获(自定义 handler)。捕获用于优雅退出。
  • 结构在 task_struct:pending(未决位图)、blocked(掩码)、sighand(处理表=信号版中断向量表)。
  • 递送时机:进程从内核态返回用户态时(和抢占检查同一关口);故能打断阻塞的 read(→EINTR)。
  • 坑:传统信号不排队会合并(SIGCHLD 要循环 waitpid);handler 只能调 async-signal-safe 函数(和 ISR 重入问题同源,最佳实践"只置标志")。

课后练习

  1. 写程序捕获 SIGINT,按 Ctrl+C 不退出而是打印计数,连按几次观察;再试捕获 SIGTERM 做"存盘后退出"。
