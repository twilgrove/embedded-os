本章目标：第16章起,任务、队列的创建都在调 pvPortMalloc——FreeRTOS 自己的内存分配器。为什么不直接用 C 标准库的 malloc?因为标准 malloc 在嵌入式/实时场景有致命问题。这一章讲清这些问题、以及 FreeRTOS 提供的 5 种不同取舍的 heap 实现(heap_1~5)。这既是第三卷的收官,也是理解"实时系统为什么排斥动态内存"(贯穿全书的主题)的最后一块拼图。学完你要能为不同项目选对 heap。

24.1 为什么不用标准 malloc

C 标准库有 malloc/free,为什么 FreeRTOS 要自己造轮子?标准 malloc 在嵌入式实时环境有几个硬伤:

  • 不确定的执行时间:标准 malloc 找合适空闲块可能要遍历、可能慢,执行时间不可预测——这对实时系统是灾难(第七卷:实时要求时间有上界)。
  • 不是线程安全的(默认):多任务同时 malloc 会破坏内部数据结构;要线程安全得加锁,标准库的实现未必配合 RTOS 的调度。
  • 碎片化:长期运行后内存被切得七零八落(下节),可能"总空闲很多但没有一块够大",分配失败。
  • 占用大、依赖 C 库:标准 malloc 实现复杂、代码大,小 MCU 吃不消。
  • 没有统一的失败处理/统计:嵌入式需要知道"堆用了多少、还剩多少、失败了怎么办"。

所以 FreeRTOS 提供自己的 pvPortMalloc/vPortFree,并给出 5 种实现(heap_1.c ~ heap_5.c),让你按项目需求选择不同的取舍。这本身就是一堂"没有万能方案,只有权衡"的工程课。

24.2 碎片化：动态内存的头号敌人

碎片化(Fragmentation)是动态内存最阴险的问题,必须讲透。分两种:

外部碎片(External Fragmentation):
  反复 malloc/free 不同大小的块后, 空闲内存被切成很多小段:

  [已用][空16][已用][空8][已用][空24][已用]
        └─ 空闲总共48字节, 但分散成 16+8+24 三小段

  现在要 malloc(40): 失败! 虽然总空闲48>40, 但没有【连续】的40字节!
  → 这就是外部碎片: 有空间, 但用不上(不连续)

内部碎片(Internal Fragmentation):
  分配器按固定块/对齐分配, 你要33字节, 给你48字节(下一个对齐单位)
  → 多给的15字节浪费了, 藏在已分配块内部

外部碎片是嵌入式长期运行的隐形杀手:一个设备跑了几天后突然 malloc 失败、崩溃,总内存明明够——就是外部碎片。它难复现(要长时间特定的分配/释放序列)、难调试。这就是为什么很多 heap 实现在 free 时要"合并相邻空闲块"(heap_4)来对抗碎片,也是为什么安全关键系统干脆禁止运行时动态分配(用静态分配,第16章),从根上杜绝碎片。理解碎片化,你就理解了后面 5 种 heap 的核心区别——它们的差异主要就在"怎么处理 free 和碎片"。

24.3 pvPortMalloc：统一的接口

5 种 heap 实现,对外都提供同一套接口,你的代码不用改:

void *pvPortMalloc( size_t xWantedSize );   /* 分配 */
void  vPortFree( void *pv );                /* 释放 */
size_t xPortGetFreeHeapSize( void );        /* 当前剩余堆 */
size_t xPortGetMinimumEverFreeHeapSize( void ); /* 历史最低剩余(水位线!) */

任务创建(xTaskCreate,第16章)、队列创建等内部都调 pvPortMalloc。换 heap 实现只需在工程里换一个 heap_N.c 文件、不用改任何业务代码——这就是统一接口的好处。堆的总大小由 configTOTAL_HEAP_SIZE 配置,FreeRTOS 从一个静态大数组 ucHeap[configTOTAL_HEAP_SIZE] 里分配(而不是用 C 库的堆),这样堆大小编译期确定、位置可控。

注意那个 xPortGetMinimumEverFreeHeapSize(历史最低剩余堆)——它是嵌入式调堆大小的利器,类似第16章任务栈的 high water mark。它告诉你"运行以来堆最紧张时还剩多少",帮你判断 configTOTAL_HEAP_SIZE 设得够不够、有没有内存泄漏(这个值持续下降就是泄漏征兆)。实战调试必用。

24.4 heap_1：只分配不释放

最简单的实现。只能 malloc,不能 free(vPortFree 是空函数)。

heap_1 原理:
  一个大数组 ucHeap[configTOTAL_HEAP_SIZE]
  一个指针 pucAlignedHeap 记录"下一个可分配位置"

  pvPortMalloc(size):
    从当前位置切出 size 字节, 指针后移, 返回
    (就像切香肠, 只往前切, 切完的不回收)

  vPortFree(): 什么都不做!
  • 优点:极简、确定(分配就是移指针,时间恒定 O(1))、绝无碎片(因为从不释放)、代码最小。
  • 缺点:不能释放内存。
  • 适用:任务/队列一次性创建后永不删除的系统(很多嵌入式产品就是开机建好所有任务,一直跑到关机)。这种场景下 heap_1 是最安全的选择——没有 free 就没有碎片、没有 free 相关的 bug。

heap_1 体现了一个反直觉的智慧:在很多嵌入式系统里,"不支持释放"反而是优点。因为绝大多数嵌入式产品的对象(任务、队列、信号量)都是启动时创建、永不销毁的。既然不 free,那支持 free 只会带来碎片和复杂性。heap_1 用"砍掉功能"换来了绝对的确定性和安全性——这正是实时/安全系统的价值观。飞控这类系统很可能就用 heap_1 或静态分配。

24.5 heap_2：能释放但不合并

heap_2 支持 free,用空闲块链表管理释放的内存,分配时用最佳适配(best fit)找最接近大小的空闲块。但它不合并相邻的空闲块。

heap_2:
  维护一个按大小排序的空闲块链表
  malloc: 遍历找【最接近且够大】的空闲块(best fit), 切一块给你
  free: 把释放的块加回空闲链表
  ★ 但不合并相邻空闲块 → 相邻的两个空闲块永远是分开的两小块

问题:因为不合并,如果你反复分配释放不同大小的块,会产生严重外部碎片。所以 heap_2 已被官方不推荐(deprecated),说"用 heap_4 代替"。它只在一种情况安全:反复分配释放的块都是相同大小(这样释放的块能被同样大小的请求完美复用,不产生碎片)。heap_2 是"有 free 但没做好"的中间产物,了解它主要是为了理解"为什么合并相邻块很重要"(下节 heap_4)。

24.6 heap_4：能释放且合并相邻块

heap_4 是最常用、最推荐的通用实现。它在 heap_2 基础上,增加了关键的一步:free 时合并相邻的空闲块(coalescence)。

heap_4 核心: free 时把相邻的空闲块合并成一个大块

  free 前: [已用A][空闲X][刚释放的B][空闲Y][已用C]
  free B 后, 检查左右邻居都空闲 → 合并:
  free 后: [已用A][──── 空闲 X+B+Y 合成一大块 ────][已用C]

  → 对抗外部碎片! 小碎片被合并回大块, 又能满足大分配了
  • 分配策略:首次适配(first fit)——找到第一个够大的空闲块就用(比 best fit 快)。
  • 关键优势:合并相邻空闲块,大幅减少外部碎片,适合长期运行、分配释放各种大小的通用场景。
  • 适用:绝大多数需要 free 的项目的默认选择。

合并相邻块是 heap_4 对抗碎片的核心武器,也是它和 heap_2 的本质区别。它的实现:每个内存块有个头部(BlockLink_t)记录块大小和指向下一个空闲块的指针;free 时检查"被释放块的物理相邻块是否也空闲",是就把它们的头部信息合并成一个更大的空闲块。虽然合并让 free 稍慢(要检查邻居、调整链表),但换来了对碎片的长期抵抗力——这个取舍对长期运行的系统非常值。Linux 内核的伙伴系统(Buddy System,第五卷)也用"合并相邻空闲块"对抗碎片,思想完全一致——heap_4 是它的迷你版。

24.7 heap_3：包装标准 malloc

heap_3 最简单粗暴:直接调用 C 标准库的 malloc/free,只是在外面加了一层"挂起调度器"保证线程安全。

heap_3:
  pvPortMalloc(size):
    vTaskSuspendAll();   // 挂起调度器(第17章), 保证线程安全
    void *p = malloc(size);   // 调标准库malloc
    xTaskResumeAll();
    return p;

它把内存管理甩给了 C 库,自己只负责加锁。适用:你已经有一个可靠的、配置好的标准 malloc(比如某些 SDK 提供),想直接用它。注意:此时 configTOTAL_HEAP_SIZE 无效(堆由 C 库和链接脚本决定),且继承了标准 malloc 的所有问题(不确定时间、碎片)。heap_3 是"图省事、复用现成 malloc"的选择,不适合追求确定性的场景。

24.8 heap_5：多块非连续内存

heap_5 = heap_4 + 支持从多块不连续的内存区域分配。前面几种都假设堆是一整块连续内存(ucHeap 数组);但有些芯片(如 RK 系列、带外部 SDRAM 的 MCU)有多块地址不连续的 RAM(片内 SRAM + 外部 SDRAM + TCM 等)。

heap_5: 通过 vPortDefineHeapRegions 登记多块内存

  HeapRegion_t xHeapRegions[] = {
    { (uint8_t*)0x20000000, 0x8000 },   // 片内SRAM, 32KB
    { (uint8_t*)0x60000000, 0x80000 },  // 外部SDRAM, 512KB
    { NULL, 0 }                          // 结束标记
  };
  vPortDefineHeapRegions( xHeapRegions );  // 使用前必须先登记!

  之后 pvPortMalloc 可以从这几块内存里分配(逻辑上当成一个堆)

heap_5 的分配/合并逻辑和 heap_4 一样,只是管理多块内存。适用:内存分散在多个不连续地址区的芯片(嵌入式常见,尤其带外部内存的)。关键:用之前必须先调 vPortDefineHeapRegions 登记,否则 malloc 会失败——这是 heap_5 的常见坑。

24.9 五种 heap 对比与选型

实现能 free?合并碎片?多块内存?确定性适用场景
heap_1❌ 不能—❌最强(无碎片)对象只创建不删除(飞控/安全系统首选)
heap_2✅❌ 不合并❌差(碎片)已弃用;仅同大小块反复分配安全
heap_3✅看 C 库看 C 库差想直接用标准 malloc
heap_4✅✅ 合并❌较好★通用首选(需 free 的项目)
heap_5✅✅ 合并✅ 支持较好内存分多块不连续区域

选型决策树:①对象只建不删(启动时创建所有任务)→ heap_1(最安全无碎片)。②需要动态创建删除、单块连续内存 → heap_4(合并抗碎片,通用首选)。③内存分多块不连续 → heap_5。④已有可靠的标准 malloc 想复用 → heap_3。heap_2 别用(用 heap_4 代替)。核心判断:你到底需不需要释放内存?不需要→heap_1(把复杂性砍掉);需要→heap_4/5(用合并对抗碎片)。这张表和选型逻辑是面试的高频考点,能答出"heap_4 因为合并相邻块所以抗碎片、heap_1 因为不释放所以最安全"就很到位。

24.10 实时系统为什么慎用动态内存

这是贯穿全书的主题,在这里做总结(第5、10、16章都提过)。实时/安全关键系统(飞控、汽车、医疗、航空)对动态内存极其谨慎,甚至完全禁止,原因:

  • 时间不确定:malloc 找块/合并的时间不可预测,破坏实时性(WCET 无法界定,第七卷)。
  • 碎片导致运行时失败:长期运行后可能突然分配失败,不可接受(飞行中飞控 malloc 失败=坠机)。
  • 难以验证:安全认证(DO-178C 航空、ISO 26262 汽车、MISRA)要求行为可预测、可穷尽验证,动态内存的行为依赖运行时历史,难验证。

应对策略:①用静态分配(第16章 xTaskCreateStatic,所有内存编译期确定);②只在启动阶段分配、之后不再动(用 heap_1);③用内存池(memory pool)——预分配固定大小块的池子,分配/释放就是从池子拿/还,时间恒定、无碎片(比通用 malloc 可控得多)。

这就是为什么第16章说安全系统偏爱静态创建、第5章说嵌入式慎用栈上大数组和堆、第10章说实时系统用 mlock 锁内存——一以贯之的价值观:实时/安全系统宁可牺牲灵活性,也要换取确定性和可验证性。动态内存代表"灵活但不确定",与这个价值观根本冲突。理解这一点,你就理解了嵌入式实时开发和普通应用开发在内存观念上的根本差异——这也是从"应用工程师"跨向"系统/实时工程师"的一个重要认知转变。

24.11 面试常问问题

Q1:FreeRTOS 为什么不用标准 malloc,自己实现?
标准 malloc 时间不确定(伤实时)、默认非线程安全、易碎片、占用大依赖 C 库、缺乏统计。FreeRTOS 提供 5 种可选实现,统一接口,可控。

Q2:什么是内存碎片?外部和内部碎片的区别?
外部:反复分配释放后空闲内存分散成小段,总量够但没连续大块,大分配失败;内部:按对齐/固定块分配,给多了,浪费在已分配块内部。

Q3:heap_1 到 heap_5 的区别和选型?
heap_1 不能 free(无碎片,对象只建不删用);heap_2 能 free 不合并(弃用);heap_3 包装标准 malloc;heap_4 能 free 且合并相邻块(通用首选);heap_5 = heap_4 + 多块非连续内存。见 24.9。

Q4:heap_4 如何对抗碎片?
free 时合并物理相邻的空闲块成大块,减少外部碎片,适合长期运行、各种大小分配。

Q5:为什么实时/安全系统慎用或禁用动态内存?
时间不确定伤实时、碎片致运行时失败、难验证认证。应对:静态分配、只启动时分配(heap_1)、内存池。

Q6:怎么判断堆够不够用/有没有泄漏?
用 xPortGetMinimumEverFreeHeapSize(历史最低剩余),类似栈的 high water mark;持续下降提示泄漏。

24.12 常见误区

  1. "heap_1 没用,因为不能 free"——恰恰在"对象只建不删"场景最安全(无碎片),很多产品用它。
  2. "碎片就是内存不够"——碎片是"总量够但不连续",和总量不足不同。
  3. "heap_2 和 heap_4 差不多"——heap_2 不合并空闲块,长期严重碎片,已弃用。
  4. "heap_5 可以直接用"——必须先 vPortDefineHeapRegions 登记内存块,否则失败。
  5. "实时系统随便用 malloc"——慎用/禁用,首选静态分配或内存池。
  6. "configTOTAL_HEAP_SIZE 对所有 heap 有效"——heap_3 用标准库堆,这个配置无效。

24.13 本章总结

  • 标准 malloc 时间不确定、非线程安全、易碎片、占用大——不适合嵌入式实时,故 FreeRTOS 自造 5 种 heap。
  • 碎片化:外部(空闲分散不连续,大分配失败)、内部(对齐浪费);是动态内存头号敌人。
  • 统一接口 pvPortMalloc/vPortFree,换 heap 不改业务代码;xPortGetMinimumEverFreeHeapSize 是调优/查泄漏利器。
  • heap_1:只分配不释放,无碎片、最确定,对象只建不删场景首选。
  • heap_2:能 free 不合并,碎片严重,已弃用。
  • heap_4:能 free 且合并相邻空闲块抗碎片,通用首选。
  • heap_3:包装标准 malloc + 挂起调度器保线程安全。
  • heap_5:heap_4 + 支持多块非连续内存(需先 vPortDefineHeapRegions)。
  • 实时/安全系统慎用动态内存(时间不确定、碎片、难验证),用静态分配/heap_1/内存池。

你现在完整地读懂了一个真实 RTOS 内核的每一个核心部件。更重要的是,你看到了它们背后反复出现的统一思想:侵入式链表、按优先级的等待列表、有序列表+tick 驱动+双列表处理溢出、"一切 IPC 皆队列"的复用、"中断/回调只触发、重活交给任务"、"确定性优先于灵活性"。这些思想不是 FreeRTOS 独有的——它们会在第四卷 RT-Thread、第五卷 Linux 内核里以更复杂的形式重现。你在 FreeRTOS 这个"小而透明"的内核里把骨架看清了,接下来读那些"大而复杂"的内核时,就能一眼认出同样的骨架,而不会迷失在细节里。这正是我们"先读 FreeRTOS 再读 Linux"的全部意义。
