本章目标:上一章的驱动怎么"装进"内核、又怎么卸载?答案是内核模块(kernel module)——可动态加载/卸载的内核代码。更重要的是,现代 Linux 驱动不再把"设备信息"写死在驱动里,而是用总线-设备-驱动模型把它们分离:Platform Driver 描述"怎么驱动",设备树(第41章)描述"有哪些硬件",内核负责撮合。这是 RK3588 等 SoC 驱动开发的标准范式。

40.1 内核模块：可插拔的内核代码

内核是一整个大程序。如果每加一个驱动都要重新编译整个内核、重启,那太痛苦了(改一个 LED 驱动要重启整机)。需要一种机制,能在内核运行时动态"插入"或"拔出"一段代码。这就是内核模块(.ko 文件,kernel object)。

模块是可以在系统运行时动态加载进内核、也能卸载的一段内核代码。加载后它就"长在"内核里,拥有内核权限,能注册驱动、响应中断;卸载后干净移除。好处:开发驱动时改一行 insmod 重载即可,不用重编内核、不用重启;发布时按需加载(用到某硬件才加载对应模块,省内存)。宏内核(第28章)本是"全编译进一个大程序",模块机制给了它"运行时可扩展"的灵活性——鱼和熊掌兼得。

40.2 生活例子：乐高扩展块

内核是一座已经搭好的乐高城堡(核心功能)。你想加一个新功能(驱动),不必推倒重搭整个城堡——做一块能插上去的扩展积木(模块),需要时"咔"一声插上(insmod),不用了拔下来(rmmod)。插上后它就是城堡的一部分,和城堡浑然一体(拥有内核权限);拔下来城堡照常运转。

关键是"标准的插口":扩展块和城堡之间有统一的凸点接口(模块的 init/exit 入口、内核导出的符号),所以能自由插拔。这个"标准插口"让模块开发和内核主体解耦——你写模块时只需对接内核提供的 API,不用改内核源码。

40.3 模块的骨架与编译

一个内核模块最小骨架就三部分:

#include <linux/module.h>
#include <linux/init.h>

static int __init my_init(void){        // ★加载时调用(构造)
    printk(KERN_INFO "module loaded\n");
    return 0;   // 返回0表示成功
}
static void __exit my_exit(void){       // ★卸载时调用(析构)
    printk(KERN_INFO "module unloaded\n");
}

module_init(my_init);   // 注册加载入口
module_exit(my_exit);   // 注册卸载入口
MODULE_LICENSE("GPL");  // ★许可证(不写会污染内核, 部分API不可用)
MODULE_AUTHOR("you");

编译用一个特殊 Makefile(借内核构建系统 Kbuild):

# Makefile
obj-m += mymod.o                        # 要编成模块的目标
KDIR := /lib/modules/$(shell uname -r)/build   # 内核源码/头文件路径
all:
	make -C $(KDIR) M=$(PWD) modules   # 借内核的构建系统编译
clean:
	make -C $(KDIR) M=$(PWD) clean

module_init/module_exit 注册加载/卸载入口——这就是上一章驱动里 register_chrdev 所在的 my_init 的由来:模块加载时 my_init 跑,里面注册驱动;卸载时 my_exit 跑,注销驱动。__init 标记的函数用完就释放内存(省内存)。编译产出 .ko 文件。

40.4 模块常用命令

操作模块的命令,驱动开发天天用:

命令作用
insmod mymod.ko加载模块(触发 module_init)
rmmod mymod卸载模块(触发 module_exit)
modprobe mymod加载模块并自动处理依赖(比 insmod 智能)
lsmod列出已加载的模块
modinfo mymod.ko看模块信息(作者、许可证、参数)
dmesg看内核日志(你 printk 的输出在这)

开发循环:改代码 → make → insmod → 测试 → dmesg 看输出 → rmmod → 再改。核心:printk 的输出去 dmesg 看(内核没有 printf 到终端,日志进内核缓冲区)。modprobe 和 insmod 的区别:modprobe 会自动加载依赖的其他模块并从标准路径找,insmod 只加载你指定的单个文件——实际部署用 modprobe。

40.5 设备与驱动分离的思想

现在讲本章的重头戏。上一章的驱动有个隐藏问题:硬件的信息(寄存器地址、中断号、时钟)写死在了驱动代码里。但同一个驱动可能要用在不同板子上(RK3588 和另一块板的同款串口,寄存器地址不同),难道每块板改一次驱动源码?

解决思路:把"设备信息"和"驱动逻辑"分开。驱动只写"怎么操作这类硬件"(通用逻辑),具体的"这块板上这个设备在什么地址、用哪个中断"作为数据单独描述。内核在运行时把"设备信息"喂给"驱动",两者配对。这样同一个驱动配不同的设备信息,就能用在不同板子上,驱动代码完全不用改。这就是"机制与策略分离""代码与数据分离"的经典工程思想。Linux 用总线-设备-驱动模型实现它——下一节。

40.6 总线-设备-驱动模型

Linux 用三个概念组织所有硬件,这是理解现代驱动的框架:

Bus(总线) — Device(设备) — Driver(驱动) 模型:

  ┌─────────── Bus 总线 (如 platform, i2c, spi, usb) ───────────┐
  │  维护两个链表, 负责【撮合】:                                    │
  │                                                              │
  │   Device链表          ←── match() 配对 ──→    Driver链表      │
  │   [设备1: 地址,中断]                          [驱动A: 支持XX]  │
  │   [设备2: ...]                                [驱动B: ...]    │
  └─────────────────────────────────────────────────────────────┘

  规则: 总线不断检查, 若某Device和某Driver能匹配(比如name/兼容串相同)
        → 调用该Driver的 probe(device) → 驱动开始管理这个设备

三个角色:Device(设备,描述"有什么硬件、在哪")、Driver(驱动,描述"怎么操作")、Bus(总线,负责把能配对的设备和驱动撮合起来)。总线像个媒人:一边是设备名单,一边是驱动名单,它按规则(名字/兼容串匹配)牵线。配对成功就调驱动的 probe 函数,把设备交给驱动管理。I2C、SPI、USB、platform 都是"总线"。这个模型让设备和驱动彻底解耦、动态匹配——插入新设备自动找驱动,加载新驱动自动找设备。

40.7 Platform 平台设备

I2C/SPI/USB 是真实的物理总线(有时钟线、数据线)。但 SoC 内部还有大量设备不挂在任何真实总线上——它们直接连在 CPU 的内存地址空间里(如 SoC 内部的 UART 控制器、GPIO 控制器、定时器)。这类设备用一个"虚拟总线"管理,叫 platform bus(平台总线)。

platform 总线: 一个"虚拟总线", 管理【直接挂在SoC内存映射上的设备】

  真实总线:  I2C设备 → I2C总线    SPI设备 → SPI总线
  虚拟总线:  SoC内部UART/GPIO/定时器 → platform总线(它们没有共同的物理总线)

  为什么造个虚拟总线? 为了让这些设备也能套用统一的
  "总线-设备-驱动"模型 → 一致的驱动框架, 不搞特殊化

platform 总线是为"片上设备"造的虚拟总线,好让它们也能纳入统一的"总线-设备-驱动"模型(否则这些设备就得各写各的特殊逻辑)。RK3588 上你写的大部分驱动都是 platform driver——因为 GPIO、UART、I2C 控制器本身、定时器这些 SoC 内部外设,都挂在 platform 总线上。这是嵌入式 Linux 驱动最常见的类型。

40.8 platform_driver 的写法

写一个 platform driver 的标准骨架:

static int my_probe(struct platform_device *pdev){   // ★配对成功时调用
    // 从pdev获取硬件资源(寄存器地址、中断号——来自设备树)
    // 初始化硬件、注册字符设备(第39章)等
    dev_info(&pdev->dev, "probed!\n");
    return 0;
}
static int my_remove(struct platform_device *pdev){  // 设备移除时
    return 0;
}

static const struct of_device_id my_of_match[] = {   // ★匹配表
    { .compatible = "myvendor,mydevice" },  // 和设备树的compatible对应
    { }
};
MODULE_DEVICE_TABLE(of, my_of_match);

static struct platform_driver my_driver = {
    .probe  = my_probe,
    .remove = my_remove,
    .driver = {
        .name = "my_driver",
        .of_match_table = my_of_match,   // 用这个表和设备匹配
    },
};
module_platform_driver(my_driver);   // ★一行搞定注册/注销(替代init/exit)

核心是三样:probe(配对成功后初始化硬件的函数)、of_match_table(靠 compatible 字符串和设备树里的设备匹配)、module_platform_driver(自动生成 init/exit 完成注册)。注意驱动里没有任何硬编码的地址/中断号——这些"设备信息"来自设备树(第41章),在 probe 里动态获取。这就是"驱动逻辑"和"设备信息"分离的落地。

40.9 probe：设备与驱动配对成功

probe 是驱动的灵魂函数,一定要理解它何时被调、干什么:

probe 的触发时机:
  ① 系统启动/加载驱动时, platform总线做匹配
  ② 发现有个设备的 compatible = "myvendor,mydevice"
     正好等于某驱动 of_match_table 里的项 → 匹配成功!
  ③ 总线调用该驱动的 probe(pdev), 把设备传进去
  ④ probe里: 拿到寄存器地址/中断/时钟(从设备树)→ 映射寄存器
             → 初始化硬件 → 注册字符设备/中断处理 → 设备就绪

  一句话: probe = "驱动遇到了它该管的设备, 在这里把它跑起来"

probe 是"设备与驱动成功配对"的回调:总线撮合成功,就调驱动的 probe,并把设备(platform_device)作为参数传进来。probe 里做所有初始化工作:从设备(实际从设备树)读出这个具体设备的寄存器地址、中断号、时钟,把寄存器映射进内核地址空间(ioremap),申请中断(第43章),注册字符设备(第39章),让设备可用。对应地,设备移除或驱动卸载时调 remove 做清理。你写驱动的主要精力,就花在写好 probe(把硬件跑起来)和各操作函数(read/write/ioctl)上。probe 是驱动的"开机启动"入口。

40.10 设备信息从哪来：设备树

反复提到"设备信息来自设备树",这里点明,为第41章铺垫:

驱动里没写死地址, 那"这个设备在0xFE660000、用中断132"从哪来?
  → 来自【设备树 DeviceTree】(第41章)

设备树是一个描述硬件的文本文件(.dts), 例如:
  myuart: serial@fe660000 {
      compatible = "myvendor,mydevice";  ← 用它和驱动匹配!
      reg = <0xfe660000 0x100>;          ← 寄存器地址和大小
      interrupts = <132>;                 ← 中断号
  };

  流程: 内核启动读设备树 → 为每个节点生成一个platform_device
        → platform总线拿它的compatible去匹配驱动
        → 匹配上就probe, probe里用reg/interrupts初始化

设备树是"设备信息"的载体:它用文本描述板子上有哪些硬件、各在什么地址、用什么中断和时钟。内核启动时解析设备树,为每个硬件节点创建一个 platform_device(带上 compatible、reg、interrupts 等信息),然后 platform 总线拿 compatible 去匹配驱动。于是:同一个驱动 + 不同板子的设备树 = 适配不同硬件,驱动代码零修改。这就是设备与驱动分离的完整闭环。设备树的语法和用法,下一章第41章详讲。

40.11 回扣 RT-Thread 与统一模型

把本章概念和你熟悉的对齐:

概念Linux(本章)RT-Thread / 你的经验
可加载代码单元内核模块 .ko(insmod/rmmod)RT-Thread 动态模块 / 组件
设备-驱动分离总线-设备-驱动 + 设备树RT-Thread 设备注册(较简化)
初始化入口probe / module_initRT-Thread INIT_EXPORT 自动初始化(第27章)
片上外设驱动platform driverSTM32 HAL/寄存器直接操作

对比你在 STM32/RT-Thread 的经验,能看出 Linux 驱动模型"重"在哪、为什么重。在 STM32 里,你直接在代码里写死寄存器地址(#define UART0_BASE 0x40011000),因为就一块板、一个固件,简单直接。但 Linux 要一个内核镜像跑在成千上万种硬件上(手机、服务器、RK3588、树莓派……),必须把"设备信息"从驱动里剥离出来(设备树),让驱动通用。总线-设备-驱动模型、probe、设备树,都是为"一套代码适配海量硬件"这个 Linux 特有需求服务的。理解了这个"为什么",你就不会觉得这套框架繁琐,而会明白它的每个部件都在解决真实的工程问题。这也是从 RTOS 迈向 Linux 驱动开发必须完成的思维升级——从"为一块板写死"到"为无数板解耦"。

40.12 面试常问问题

Q1:什么是内核模块?有什么好处?
可在运行时动态加载(insmod)/卸载(rmmod)的内核代码(.ko)。好处:开发时改代码重载即可不用重编内核/重启;发布时按需加载省内存。给宏内核提供了运行时可扩展性。

Q2:module_init/module_exit 的作用?
注册模块的加载/卸载入口函数。加载时执行 init(里面注册驱动),卸载时执行 exit(注销驱动)。

Q3:insmod 和 modprobe 的区别?
insmod 加载指定的单个 .ko;modprobe 从标准路径加载并自动处理模块依赖(先加载依赖的模块)。部署用 modprobe。

Q4:总线-设备-驱动模型是什么?解决什么问题?
用 Bus(总线,撮合)、Device(设备信息)、Driver(驱动逻辑)三者分离,把"设备信息"和"驱动逻辑"解耦。总线按匹配规则(compatible)撮合设备和驱动,配对成功调 probe。让一套驱动适配不同硬件。

Q5:platform 设备是什么?什么时候用 platform driver?
platform 是管理"直接挂在 SoC 内存映射上、无真实总线的片上设备"(UART/GPIO/定时器控制器等)的虚拟总线。SoC 内部外设的驱动多是 platform driver。

Q6:probe 函数何时被调用?做什么?
总线把设备和驱动匹配成功(compatible 相同)时调用,把设备传入。probe 里从设备树获取寄存器地址/中断/时钟,映射寄存器、初始化硬件、注册字符设备/中断,让设备就绪。

Q7:驱动里不写死寄存器地址,那地址从哪来?
来自设备树(.dts):内核解析设备树为每个硬件节点建 platform_device(带 reg/interrupts),probe 里读取。同一驱动配不同设备树适配不同板子,驱动零修改。

40.13 常见误区

  1. "加驱动必须重编内核重启"——用模块可动态 insmod/rmmod,不用重启。
  2. "printk 会打印到终端"——进内核日志缓冲,用 dmesg 看。
  3. "驱动里要写死寄存器地址"——现代做法从设备树获取,驱动保持通用。
  4. "platform 是一种物理总线"——是虚拟总线,为片上无总线设备套用统一模型而设。
  5. "probe 是自己调用的"——由总线在设备-驱动匹配成功时回调。
  6. "模块不写 MODULE_LICENSE 也没事"——会污染内核、部分 GPL-only 的 API 不可用,应写。

40.14 本章总结

  • 内核模块(.ko):运行时可 insmod/rmmod 的内核代码,给宏内核运行时扩展能力;骨架=module_init/module_exit+MODULE_LICENSE,借 Kbuild 编译。
  • 命令:insmod/rmmod/modprobe(处理依赖)/lsmod/modinfo;printk 输出看 dmesg。
  • 核心思想:设备信息与驱动逻辑分离——让一套驱动适配不同硬件。
  • 总线-设备-驱动模型:Bus 撮合、Device(有什么硬件)、Driver(怎么操作),按 compatible 匹配,配对成功调 probe。
  • platform 总线:管片上无真实总线设备(UART/GPIO/定时器),SoC 驱动多是 platform driver;写法=probe+of_match_table+module_platform_driver。
  • probe:设备-驱动配对成功的回调,从设备树取地址/中断/时钟、映射寄存器、初始化硬件、注册设备。
  • 设备信息来自设备树(第41章):内核解析 .dts 建 platform_device;同驱动+不同设备树适配不同板,驱动零改。
  • 回扣:这套"重"框架是为 Linux"一套代码跑海量硬件"服务的,对比 STM32 写死地址是必要的思维升级。

  3. ls /sys/bus/platform/devices 和 /sys/bus/platform/drivers 看系统里的 platform 设备和驱动;ls /sys/bus/i2c 对比。
  4. 写一个 platform driver 骨架(probe 里 printk),用设备树或代码注册一个匹配的 platform_device,观察 probe 被调用。
  5. 思考:为什么 RK3588 的串口驱动是 platform driver 而不是直接写死地址?对比 STM32 你会怎么写。
