本章目标:RK3588 有 8 个核(第56章),Linux 默认把它们当成一个"对称"的核池(SMP)自动调度。但对实时任务,自动调度会带来不确定性(任务被迁来迁去、被别的任务干扰)。本章讲如何手动控制任务在哪个核上跑——CPU 亲和性(绑核)和 CPU 隔离(独占核)。这是在多核 Linux 上保证实时性和性能的关键工程手段,第55章调优清单里的核心一项,做多核实时必用。

57.1 SMP：对称多处理

先明确多核的基本模型——SMP(Symmetric Multi-Processing,对称多处理):

SMP: 多个核【地位对等】, 共享同一份内存和操作系统
  ┌────┬────┬────┬────┐
  │核0 │核1 │核2 │核3 │  ... 所有核跑【同一个】Linux内核
  └────┴────┴────┴────┘  共享同一块内存、同一个任务队列体系
       共享 DDR 内存

  "对称"含义: 每个核都能跑任何任务、访问任何内存,
             操作系统把它们当成一个"核池"统一调度
  → Linux调度器(第30章)自动把任务分配到空闲的核上
  → 你写多线程程序, 内核自动让它们并行在多个核跑

SMP 是"多个对等的核共享一份内存、跑同一个 OS"的模型。"对称"意味着每个核地位平等、都能跑任何任务、访问任何内存,操作系统把它们当成一个统一的"核池"来调度。这是最常见的多核模型——你在 RK3588 上跑 Linux,默认就是 SMP:8 个核由一个 Linux 内核统一管理,调度器(第30章)自动把任务分派到各核并行。好处是简单透明——你写多线程程序,不用管它跑在哪个核,内核自动并行化。但"自动"对实时是把双刃剑:方便,却带来不确定性(任务可能被迁到任何核、被任何任务干扰)——这正是本章要用亲和/隔离来控制的。SMP 与下一章的 AMP(非对称)形成对比。

57.2 生活例子：多个收银台

SMP 像超市的多个收银台(核),顾客(任务)来了,由一个统一的排队系统(调度器)分配到空闲的收银台。对称=每个收银台都一样、能服务任何顾客;自动调度=你不用指定去哪个台,系统看哪个空就分你去哪。这很高效(充分利用所有收银台)。

但设想一位VIP 顾客(实时任务)要求"必须快速、稳定地结账,不能中途被打断、不能排队等"。在自动系统里,他可能被分到一个正忙的台、或者结到一半系统把他挪到别的台(任务迁移,要重新熟悉)。更好的办法:给 VIP 专门留一个收银台(CPU 隔离),只服务他,别的顾客不许用——这样他随到随结、绝不被干扰。这就是 CPU 亲和(指定台)和隔离(专用台)的价值:对普通顾客用自动调度(高效),对 VIP 用专台专用(稳定确定)。

57.3 Linux 如何管理多核

回顾第30章,看 Linux SMP 调度的关键机制:

Linux SMP 调度的组织:
  ① 每个CPU核有自己的【运行队列 runqueue】(per-CPU, 第30章)
     → 减少多核抢一个全局队列的锁竞争
  ② 调度器在每个核上独立选任务(CFS/实时类, 第30/52章)
  ③ ★负载均衡(load balancing): 定期检查各核忙闲,
     把任务从忙的核【迁移】到闲的核, 让负载均匀
  ④ current、per-CPU变量等(第28章): 每个核有自己的一份

  → 结果: 任务默认可以在【任意核】上跑, 且【会被迁移】
     好处: 负载均衡, 充分利用所有核
     坏处(对实时): 迁移有代价、被别的任务干扰、行为不确定

Linux SMP 调度的核心:每个核有自己的运行队列(per-CPU,减少锁竞争,第30章),调度器在每个核独立选任务,并通过负载均衡定期在核间迁移任务以均衡负载。这套机制追求吞吐和公平(第50章)——让所有核都忙起来、负载均匀。但对实时任务,"任务默认可在任意核跑且会被迁移"是问题:①迁移有代价(下一节);②任务可能被调度到正被别的任务占用的核,受干扰;③行为不确定(不知道下次跑哪个核)。这就是为什么实时任务需要"绑核"和"隔离"来摆脱自动调度的不确定性——把关键任务从"随波逐流的核池"里拎出来,固定到可控的核上。

57.4 负载均衡与迁移的代价

任务在核间迁移看似免费,实则有隐性代价——这是绑核的主要动机之一:

任务从 核A 迁移到 核B 的代价:
  ① ★Cache失效: 任务在核A跑时, 它的数据在核A的L1/L2 Cache里(第8章)
     迁到核B后, 核B的Cache里没有这些数据 → 全部Cache未命中
     → 要重新从内存/L3加载 → 慢! (Cache亲和性丢失)
  ② TLB失效: 类似, 地址翻译缓存也要重建(第9章)
  ③ 迁移本身的调度开销、跨核同步

  → 频繁迁移 = 频繁Cache/TLB重建 = 性能损失 + 时间不确定(伤实时)

  ★这就是"Cache亲和性"的价值: 让任务尽量留在同一个核,
    复用它在该核Cache里的热数据 → 快且确定

任务迁移最大的隐性代价是"Cache 亲和性丢失":任务在一个核上跑时,它频繁访问的数据都缓存在该核的 L1/L2 Cache 里(第八章);一旦被迁到另一个核,新核的 Cache 里没有这些数据,全部变成 Cache 未命中,要重新从内存加载——又慢又造成时间波动。TLB(第九章)同理。所以频繁迁移不仅慢(性能损失),更让执行时间不确定(伤实时,第54章 WCET)。反过来,让任务尽量留在同一个核(Cache 亲和),就能复用该核 Cache 里的"热数据",又快又稳定。这是绑核(CPU 亲和)提升性能和实时性的根本原理——把第八章 Cache 的知识用到了多核任务放置上。对实时任务,把它固定在一个核,避免迁移带来的 Cache 抖动,是保证 WCET 可控的重要手段。

57.5 CPU 亲和性 affinity

CPU 亲和性(CPU affinity)就是"指定一个任务只能在哪些核上跑",即绑核:

CPU 亲和性: 限制一个任务(线程/进程)只在指定的核上运行

  命令行:
   taskset -c 2 ./myprogram      # 让程序只在核2上跑
   taskset -cp 2,3          # 把已运行的进程绑到核2,3
   taskset -cp              # 查看某进程的亲和性

  代码(pthread):
   cpu_set_t set;
   CPU_ZERO(&set);
   CPU_SET(2, &set);             // 只允许核2
   pthread_setaffinity_np(thread, sizeof(set), &set);
   // 或 sched_setaffinity(pid, ...)

  效果: 任务被"钉"在指定核, 不再被负载均衡迁走
   → Cache亲和保持、行为确定 → 性能稳、实时性好

CPU 亲和性让你把任务"钉"在指定的核上,不再被负载均衡迁移。命令行用 taskset,代码里用 sched_setaffinity/pthread_setaffinity_np(设置一个 CPU 掩码表示允许哪些核)。绑核后,任务始终在固定核跑,保持 Cache 亲和(第57.4)、行为确定——性能更稳、实时性更好。典型用法:把实时控制线程绑到某个专用核。但注意:光绑核还不够——你把实时任务绑到核 3,但别的任务(和内核的中断、后台线程)可能也被调度到核 3 干扰它。要彻底独占,还需要"隔离"这个核(57.7),把别人都赶走。绑核是第一步,隔离是第二步。

57.6 为什么绑核能提升性能与实时性

汇总绑核的好处(面试常问"绑核有什么用"):

好处原理
保持 Cache 亲和任务不迁移,复用该核 Cache 热数据,减少未命中(第8章)
减少迁移开销没有 Cache/TLB 重建、跨核同步的代价
行为确定始终在固定核,执行时间稳定(利于 WCET,第54章)
隔离干扰(配合隔离)专用核不被别的任务抢占
NUMA/大小核优化绑到就近内存的核 / 绑到大核保证性能

绑核对性能和实时性都有好处,但要区分两种收益:①性能收益(Cache 亲和)——即使不隔离,把一个计算密集的线程绑到固定核,让它复用 Cache 热数据、避免迁移抖动,就能提升性能(如高吞吐服务、密集计算);②实时收益(确定性)——绑核让执行时间稳定(利于 WCET),配合隔离(57.7)让实时任务独占核、不被干扰。对 :计算密集的视觉线程绑到大核(保证性能+Cache 亲和),实时控制线程绑到隔离核(保证确定性)。需要强调:绑核不是万能药,乱绑可能更差——比如把两个都很忙的线程绑到同一个核会互相拖累,或绑核后破坏了负载均衡导致某些核闲置。绑核是"精细控制"的工具,要基于对任务特性的理解来用,不是无脑绑。下一步是用隔离让绑核的实时任务真正独占核。

57.7 CPU 隔离 isolcpus

光绑核不够,要让实时任务独占一个核,需要 CPU 隔离——把某些核从 Linux 的"通用核池"里拿走,专门留给实时任务:

CPU 隔离: 告诉Linux "这几个核不要用于普通调度", 留给专门任务

  内核启动参数(cmdline):
   isolcpus=6,7            # 把核6,7隔离出通用调度
   → 普通任务默认不会被调度到核6,7
   → 核6,7"空着", 只有你【显式绑核】过去的任务才在上面跑

  配合使用:
   isolcpus=6,7 (隔离核6,7)
   + taskset -c 6 ./realtime_task (把实时任务绑到核6)
   → 实时任务独占核6, 没有任何普通任务来抢/干扰 ★

  区别:
   绑核(affinity): 限制"我"只在某核 —— 但别人还能来这核
   隔离(isolcpus): 把某核从通用池拿走 —— 别人不来这核
   两者配合 = 实时任务真正独占一个干净的核

CPU 隔离(isolcpus 启动参数)把指定的核从 Linux 通用调度中"拿走",这些核默认不跑任何普通任务,空出来专门留给你显式绑核过去的实时任务。关键区分绑核和隔离:绑核是"限制我只在某核跑"(但别人还能来这核干扰我);隔离是"把某核从通用池拿走"(别人不来这核)。两者配合才完整:isolcpus=6,7 隔离出核 6/7,再 taskset -c 6 把实时任务绑上去——这样实时任务独占一个干净的核,没有任何普通任务来抢占或干扰,执行时间高度确定。这是在多核 Linux 上给实时任务创造"类 RTOS 环境"的核心手段(第55章调优清单的关键项),车企在 RK3588 上跑实时控制必用。

57.8 隔离核的完整净化

isolcpus 只隔离了"普通任务调度",但一个核上还有别的干扰源(中断、内核线程、时钟)要一并清理,才能得到"真正干净"的核:

让一个核彻底"干净"(给实时任务)的完整净化:
  □ isolcpus=6         隔离普通任务调度
  □ nohz_full=6        关闭该核的周期时钟中断(tickless)
                       → 减少时钟中断周期性打扰实时任务
  □ rcu_nocbs=6        RCU回调不在该核处理(第38章)
                       → RCU的宽限期处理不干扰该核
  □ irqaffinity=0-5    把中断都导向其他核(57.9)
                       → 该核不处理外设中断
  □ 迁移kernel线程、关watchdog等

  → 核6变成一个几乎"什么系统杂务都不做"的纯净核
  → 绑上去的实时任务几乎独占, 延迟极低且确定
  → 接近RTOS的实时环境, 但仍在Linux里(能用Linux功能)

要给实时任务一个"真正干净"的核,光 isolcpus 不够,还要清除该核上的其他系统干扰源:关闭周期时钟中断(nohz_full,让核不被每毫秒的时钟中断打扰)、把 RCU 回调赶走(rcu_nocbs,第38章)、把外设中断导向别的核(irqaffinity,57.9)、迁走内核后台线程。做完这一整套"净化",这个核就几乎不做任何系统杂务,绑上去的实时任务几乎完全独占它,延迟极低且确定。这本质上是在一颗跑 Linux 的多核 SoC 上,"划出"一个核营造接近 RTOS 的实时环境——既有 RTOS 级的确定性(在这个净化核上),又保留 Linux 的全部功能(在其他核上)。这是 PREEMPT_RT(第55章)之外,另一个在 Linux 上做实时的强力手段,而且两者常结合用。它也预示了下一章 AMP 的思想:与其在 SMP 里费力净化一个核,不如干脆让那个核跑独立的 RTOS——那就是 AMP(第58章)。可以说,"净化的隔离核"是 SMP 和 AMP 之间的过渡形态。

57.9 中断亲和性

净化里提到把中断导向别的核——这是中断亲和性(IRQ affinity),单独讲(实时重要):

中断亲和性: 指定某个中断由哪个(些)核处理

  默认: 中断可能被送到任意核处理(包括你的实时核)
   → 中断在实时核上触发ISR, 打断实时任务 → 延迟!

  设置(把中断irq号N导向核0-5, 不用核6):
   echo 3f > /proc/irq/N/smp_affinity   # 位掩码0x3f=核0-5
   # 或 /proc/irq/N/smp_affinity_list 写 "0-5"

  策略:
   ① 实时核(核6): 不处理任何外设中断 → 不被中断打扰
   ② 把网络/存储等高频中断集中到某几个普通核
   ③ 中断和它服务的任务放同一核(Cache亲和) —— 看场景

  → 中断亲和性 + CPU隔离 = 实时核既无任务干扰、也无中断干扰

中断亲和性控制"某个中断由哪个核处理"。默认中断可能落到任意核(包括你的实时核),中断触发时 ISR 会打断实时核上的任务,造成延迟(第51章)。通过 /proc/irq/N/smp_affinity 把中断导向别的核,让实时核不处理任何外设中断,就消除了中断对实时任务的干扰。反过来,对普通任务,可以把中断和它服务的处理任务放同一核(Cache 亲和)提升效率。中断亲和性 + CPU 隔离 + nohz 一起,才能让实时核既无任务干扰、也无中断干扰、也无时钟打扰——真正纯净。这些 /proc 接口(又是第32章"一切皆文件")是多核实时调优的实际操作面。

57.10 大小核的绑核策略

RK3588 是大小核(第56章),绑核时要考虑大小核差异,这是 RK3588 特有的实战:

任务类型绑到原因
高性能计算(视觉/AI 前后处理)大核 A76要性能,大核强
硬实时控制(高频、要 WCET 稳)隔离的核(大核或小核均可,要净化)要确定性,独占+净化
后台/常驻/低负载服务小核 A55省电,小核够用
普通应用不绑,让调度器自动分自动负载均衡即可
RK3588典型分配(视觉机器人):
  核0-3 (A55小核): 系统服务、后台、低负载任务
  核4-5 (A76大核): 视觉/AI计算 (高性能)
  核6-7 (A76大核): isolcpus隔离+净化 → 实时控制任务独占

  → 大核性能给计算, 隔离核确定性给实时, 小核省电给杂务
  各得其所, 兼顾性能/实时/功耗

RK3588 的绑核要利用大小核差异:高性能计算绑大核(A76)、省电常驻绑小核(A55)、硬实时控制绑隔离并净化的核(独占,大小核看需求)、普通应用交给自动调度。一个典型的视觉机器人分配:小核跑系统杂务(省电)、部分大核跑视觉 AI(性能)、隔离的大核跑实时控制(确定性)。这样每类任务都在最合适的核上:计算要性能给大核、实时要确定给隔离核、杂务要省电给小核——兼顾性能、实时、功耗三个目标。这就是在 RK3588 这样的异构多核 SoC 上做系统设计的核心决策:不是让 8 个核平均分担,而是根据任务特性精细分配。机器人的多核部署方案,本质就是这套"任务-核"映射设计。

本章是"多核 + 实时"的工程落地,把第30章的多核调度和第七卷的实时需求结合起来。核心思想:Linux 默认的 SMP 自动调度追求吞吐/公平(第50章),但实时任务需要的是确定性——所以要用亲和(绑核)和隔离(独占+净化)把实时任务从"随波逐流的核池"里拎出来,给它一个可控、纯净的核。这套手段(taskset 绑核、isolcpus 隔离、nohz_full/rcu_nocbs 净化、irq affinity 导流中断)是第55章实时调优清单的核心,也是车企在 RK3588 上做实时控制的标准操作。它还揭示了一个深层逻辑:在多核时代,实时性不只靠"调度算法"(第52章)和"内核可抢占"(第55章),更靠"空间隔离"——把关键任务在物理核层面隔离开,从源头消除干扰。这比在共享核上费力保证实时更简单直接。而当你把一个核"净化"到极致(什么系统杂务都不做),自然会想:何不干脆让这个核跑一个独立的 RTOS?那就是下一章 AMP。本章的"净化隔离核"正是通向 AMP 的桥梁——两者都是"用空间换确定性"的思路,AMP 是它的极致形态。

57.12 面试常问问题

Q1:什么是 SMP?Linux 怎么管多核?
对称多处理:多个对等的核共享内存、跑同一个 OS,当成核池统一调度。Linux 每核一个运行队列(per-CPU),调度器各核独立选任务,负载均衡定期在核间迁移任务均衡负载。

Q2:任务在核间迁移有什么代价?★
主要是 Cache 亲和性丢失:任务的热数据在原核 Cache 里,迁到新核后全部 Cache/TLB 未命中要重建,又慢又造成时间波动(伤性能和实时)。

Q3:什么是 CPU 亲和性(绑核)?有什么用?
限制任务只在指定核跑(taskset/sched_setaffinity)。好处:保持 Cache 亲和、减少迁移开销、执行时间确定(利于 WCET)。用于把实时/计算密集任务钉在固定核。

Q4:绑核和 CPU 隔离的区别?为什么要一起用?★
绑核:限制"我"只在某核(但别人还能来这核干扰);隔离(isolcpus):把某核从通用池拿走(别人不来)。两者配合=实时任务独占一个干净核,没有普通任务干扰。

Q5:怎么让一个核"彻底干净"给实时任务?
isolcpus 隔离普通任务 + nohz_full 关时钟中断 + rcu_nocbs 赶走 RCU 回调 + irq affinity 把中断导向别的核 + 迁走内核线程。让该核不做系统杂务,实时任务近乎独占。

Q6:中断亲和性是什么?对实时有什么用?
指定中断由哪个核处理(/proc/irq/N/smp_affinity)。把中断导向非实时核,让实时核不被中断 ISR 打扰,降低延迟。

Q7:RK3588 上怎么分配任务到大小核?
计算密集(视觉/AI)绑大核 A76(性能);硬实时控制绑隔离净化的核(确定性);后台常驻绑小核 A55(省电);普通应用交自动调度。兼顾性能/实时/功耗。

57.13 常见误区

  1. "绑核一定提升性能"——乱绑可能更差(俩忙线程绑一核、破坏均衡);要基于任务特性。
  2. "绑核就等于独占核"——绑核只限制自己,别人和中断还能来;要配合 isolcpus 隔离。
  3. "isolcpus 就够干净了"——还有时钟中断、RCU、外设中断,要 nohz/rcu_nocbs/irq affinity 一起净化。
  4. "任务迁移没代价"——有 Cache/TLB 重建代价,伤性能和实时。
  5. "多核就是核越多越快"——要合理分配任务、处理迁移和干扰,盲目并行未必快。
  6. "中断落哪个核无所谓"——中断打断实时核会增加延迟,要用亲和性导流。

57.14 本章总结

  • SMP:多个对等核共享内存跑同一 OS;Linux 每核 per-CPU 队列 + 负载均衡自动迁移任务(追求吞吐/公平)。
  • 迁移代价:Cache/TLB 亲和性丢失(热数据没了要重建),又慢又不确定(伤实时)。
  • CPU 亲和性(绑核):taskset/sched_setaffinity 把任务钉在指定核 → 保持 Cache 亲和、行为确定。
  • CPU 隔离(isolcpus):把核从通用池拿走,留给显式绑核的任务。绑核(我只在某核)+ 隔离(别人不来)= 实时任务独占干净核。
  • 完整净化:isolcpus + nohz_full(关时钟中断)+ rcu_nocbs + 中断亲和性(导流中断)→ 近乎纯净核,类 RTOS 环境。
  • 中断亲和性(/proc/irq/N/smp_affinity):把中断导向非实时核,实时核不被打扰。
  • RK3588 大小核策略:计算绑大核、实时绑隔离核、常驻绑小核、普通交自动;兼顾性能/实时/功耗。
  • 核心:实时靠空间隔离(把任务隔到物理核)从源头消除干扰;净化隔离核是通向 AMP(第58章)的桥梁。
