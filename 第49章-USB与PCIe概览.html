本章目标:USB 和 PCIe 是两大高速、复杂总线——USB 接摄像头/存储/各种外设,PCIe 接高速网卡/NVMe SSD/GPU/AI 加速卡。它们的驱动栈庞大,这一章做概览:抓住核心思想(USB 的枚举/描述符/端点、PCIe 的配置空间/BAR/枚举),看清它们如何再次套用第40章的总线模型。作为第六卷收官,本章也会回顾整卷,让你形成完整的驱动知识地图。RK3588 上这两者都有(USB3.0、PCIe3.0)。

49.1 为什么单讲这两个

前面的 GPIO/UART/I2C/SPI/CAN 是简单/中速总线,驱动相对轻量。USB 和 PCIe 不同——它们是高速、可扩展、支持热插拔/自动发现的复杂总线,驱动栈非常庞大(USB 核心+主机控制器+各类设备类驱动,PCIe 枚举+各设备驱动)。

但对嵌入式工程师,你极少从零写 USB/PCIe 主机驱动(那是 SoC 厂商和内核社区的事),更多是:①用现成的 USB/PCIe 设备(内核已有驱动,即插即用);②偶尔为特定 USB/PCIe 设备写设备类驱动;③理解它们的工作原理以便调试。所以本章重点是"理解原理和框架",而非教你写完整驱动。抓住两个核心概念——USB 的枚举/描述符、PCIe 的配置空间/BAR,再看它们如何复用第40章总线模型,你就够用了。RK3588 的 USB3.0(接相机/存储)、PCIe(接 NVMe/网卡/AI 卡)是实际工程常见配置。

49.2 USB：热插拔的分层总线

USB = Universal Serial Bus(通用串行总线)。它的核心特点:

  • 热插拔:随时插拔,系统自动识别、加载驱动(不用重启,不用手动配置)。
  • 主从 + 树形:一个主机(host,SoC),下面通过 Hub 接多个设备,构成树。主机主动发起所有通信。
  • 高速可扩展:USB2.0 480Mbps、USB3.0 5Gbps+,一个主机可接上百设备。
  • 自描述:设备内置"描述符",告诉主机"我是什么、有什么功能"——这是自动识别的基础。

USB 最了不起的是"即插即用":插上任何 USB 设备,系统自动认出它是什么、自动加载对应驱动、自动可用。这背后是枚举(49.4)和描述符机制。对比前面的 I2C/SPI(设备信息要写在设备树里),USB 设备自己带着"身份信息",插上就能被发现——这是热插拔总线和板载固定总线的根本区别。

49.3 USB 拓扑与端点

USB 的组织结构和通信单位:

USB 树形拓扑:
   Host(主机/SoC) ── Root Hub
                       ├── 设备1(摄像头)
                       ├── Hub ── 设备2(键盘)
                       │        └ 设备3(U盘)
                       └── 设备4(...)
   主机是唯一主动方, 所有通信由主机发起(轮询/调度)

设备内部结构:
   设备 → 若干【配置】→ 若干【接口(interface)】→ 若干【端点(endpoint)】
   ★端点(endpoint): USB通信的最小单位, 是一个数据"管道"的一端
     每个端点有方向(IN=设备→主机, OUT=主机→设备)和类型
     端点0特殊: 用于控制/枚举(每个设备都有)

关键概念是"端点(endpoint)"——USB 通信的最小单位,相当于设备里一个数据通道。一个设备有多个端点(如摄像头有控制端点、视频数据端点),主机通过"管道(pipe)"连到端点收发数据。接口(interface)是一组端点,代表设备的一个功能(一个复合设备如 USB 摄像头可能有"视频接口"+"音频接口")。端点 0 特殊,专用于控制和枚举——每个设备都有,主机靠它来"询问设备是谁"(49.4)。理解"设备→接口→端点"的层次,是理解 USB 通信的基础。

49.4 USB 枚举：插上自动认设备

USB 即插即用的核心过程——枚举(enumeration),面试常问:

插上USB设备时, 主机做的枚举过程:
  ① 检测到端口电平变化 → "有设备插入了"
  ② 给设备分配一个地址(总线上唯一)
  ③ 通过【端点0】读设备的【描述符】:★
       - 设备描述符: VID(厂商ID)/PID(产品ID)/设备类别
       - 配置描述符: 有几个接口、端点、供电需求
       - 接口/端点描述符: 每个功能、每个端点的详情
  ④ 主机根据 VID/PID/类别 → 匹配驱动(第40章!)
       - 有专用驱动(按VID/PID) → 用它
       - 无专用但符合标准类(如U盘=大容量存储类) → 用通用类驱动
  ⑤ 加载驱动, 配置设备 → 设备可用

  → 全自动! 这就是"插上就能用"的原理

枚举是 USB"即插即用"的魔法核心:插上设备,主机自动读取设备的描述符(自我介绍),据此匹配并加载驱动。关键是描述符——设备内置的"身份档案",包含 VID(厂商)、PID(产品)、设备类别、有哪些接口和端点。主机拿这些信息去匹配驱动(又是第40章总线-设备-驱动模型!):有厂商专用驱动(按 VID/PID 匹配)就用专用的;没有但符合标准设备类(如所有 U 盘都遵循"大容量存储类"、所有 USB 摄像头遵循 UVC 类)就用通用类驱动。这就是为什么绝大多数 U 盘、鼠标、键盘、摄像头插上即用——它们符合标准类,内核有通用类驱动,不需要厂商单独提供。对比 I2C/SPI 设备信息写死在设备树,USB 设备自带描述符、运行时动态发现——这是热插拔的关键。理解枚举和描述符,你就理解了 USB 的灵魂。

49.5 USB 四种传输类型

USB 针对不同数据需求设计了四种传输类型(不同端点用不同类型):

类型特点用途
控制(Control)可靠、双向,用于配置/命令枚举、设置参数(端点0)
中断(Interrupt)小数据、低延迟、周期轮询(非真中断)鼠标、键盘
批量(Bulk)大数据、可靠(有重传)、不保证时间U 盘、打印机
等时(Isochronous)大数据、保证带宽/实时、不重传(丢了就丢)摄像头、音频

四种类型对应不同的"可靠性 vs 实时性"权衡(和第34章 TCP/UDP 的取舍异曲同工):批量传输要可靠(U 盘数据不能错,有重传),等时传输要实时(摄像头/音频要保证连续带宽,丢帧可容忍所以不重传——像 UDP)。USB 摄像头用等时传输:每秒稳定送图像帧,偶尔丢帧无所谓,但不能卡顿等重传。用 USB 摄像头/图像采集时,底层就是等时传输——理解这点,才明白为什么 USB 摄像头在带宽不足时是"丢帧/花屏"而非"卡住"。中断传输名字有误导:它不是真中断,是主机周期性轮询小数据(鼠标位置)。

49.6 USB 驱动框架

写 USB 设备驱动(如果需要),用内核 USB 框架,结构又是第40章总线模型:

static int my_usb_probe(struct usb_interface *intf,
                        const struct usb_device_id *id){ // ★匹配成功
    // 从intf获取端点信息、初始化设备...
    return 0;
}
static struct usb_device_id my_ids[] = {
    { USB_DEVICE(0x1234, 0x5678) },   // ★按VID/PID匹配
    { USB_INTERFACE_INFO(USB_CLASS_VIDEO, ...) }, // 或按设备类匹配
    { }
};
MODULE_DEVICE_TABLE(usb, my_ids);
static struct usb_driver my_usb_driver = {
    .name = "mydev",
    .probe = my_usb_probe,
    .disconnect = my_usb_disconnect,   // ★拔出时调用(热插拔!)
    .id_table = my_ids,
};
module_usb_driver(my_usb_driver);

// 数据传输用 URB(USB Request Block):
//   urb = usb_alloc_urb(); usb_fill_bulk_urb(...); usb_submit_urb(urb);
//   → 异步提交传输, 完成时回调

和 platform/i2c/spi driver 一个模子:填 usb_driver(probe + id_table)、module_usb_driver 注册。区别:①id_table 按 VID/PID 或设备类匹配(对应枚举读到的描述符,不是设备树 compatible——因为 USB 是运行时发现);②有 disconnect 回调(拔出时调用,体现热插拔——probe 插入时、disconnect 拔出时);③数据传输用 URB(USB Request Block),异步提交、完成回调(适应 USB 主机调度所有传输的模型)。但你大多数时候不用写——标准设备内核已有类驱动。

49.7 PCIe：高速点对点

换到 PCIe = PCI Express,SoC/电脑里最高速的外设总线(接 GPU、NVMe SSD、高速网卡、AI 加速卡):

  • 极高速:每 lane 几 Gbps,多 lane(x1/x4/x8/x16)并行,可达几十上百 Gbps。
  • 点对点 + 交换:不是共享总线,而是每个设备和 Root Complex(或 Switch)点对点直连(用差分串行 lane),靠 Switch 扩展。
  • 内存映射:设备的寄存器/内存映射进 CPU 地址空间(通过 BAR,49.8),CPU 像访问内存一样访问设备。
  • 枚举发现:开机时扫描 PCIe 总线,自动发现所有设备(靠配置空间)。

PCIe 是"内存映射 + 高速串行 + 点对点"的总线,为极高带宽而生。和 USB(树形、主机轮询、热插拔外设)不同,PCIe 是点对点高速直连、设备内存映射进 CPU 空间,适合板内高速设备(不是外接小设备)。RK3588 的 PCIe 常用来接 NVMe SSD(高速存储)、5G 模块、或 AI 加速卡——自动驾驶域控制器接雷达、接算力卡也常走 PCIe。它是通往"高带宽设备"的门户。

49.8 PCIe 配置空间与 BAR

PCIe 两个核心概念,理解它们就理解了 PCIe 驱动:

① 配置空间(Configuration Space):每个PCIe设备有一块标准的"档案"
   - Vendor ID / Device ID  ← 厂商/设备ID(用来匹配驱动, 类比USB的VID/PID)
   - Class Code             ← 设备类别
   - ★BAR (Base Address Registers) ← 关键!
   开机枚举时, 系统读配置空间发现设备、读BAR分配地址

② BAR(基地址寄存器):设备告诉系统"我需要多大的地址空间"
   系统给它分配一段【CPU物理地址】, 把设备的寄存器/内存
   【映射】到这段地址
   → 之后CPU访问这段地址 = 访问设备 (内存映射I/O)
   → 驱动 pci_iomap(BAR) → ioremap → 得到能读写的指针
     readl/writel 操作设备寄存器 (和第41章设备树的reg映射类似!)

PCIe 的核心机制:每个设备有"配置空间"(标准档案,含 Vendor/Device ID 和 BAR),系统枚举时读它发现设备、并通过 BAR 把设备内存映射进 CPU 物理地址空间。BAR(基地址寄存器)是关键:设备通过 BAR 声明"我需要多少地址空间",系统分配一段 CPU 物理地址并建立映射,之后 CPU 访问这段地址就是访问设备(内存映射 I/O)。驱动用 pci_iomap(内部是 ioremap,第41章)拿到指针,readl/writel 操作设备寄存器——和第41章设备树 reg → ioremap 的套路一样,只是地址来自 BAR(运行时枚举分配)而非设备树(静态描述)。这又是"发现设备信息"的两种方式:PCIe/USB 运行时枚举自动发现(设备自带档案),I2C/SPI/platform 靠设备树静态描述(板载固定)。配置空间的 Vendor/Device ID 用来匹配驱动——还是第40章总线模型。

49.9 PCIe 驱动框架

PCIe 设备驱动骨架(又是熟悉的模式):

static int my_pci_probe(struct pci_dev *pdev,
                        const struct pci_device_id *id){  // ★匹配成功
    pci_enable_device(pdev);                  // 使能设备
    pci_request_regions(pdev, "mydev");       // 申请BAR区域
    void __iomem *base = pci_iomap(pdev, 0, 0); // ★映射BAR0 → 得指针
    // readl(base+REG)/writel(v, base+REG) 操作设备寄存器
    // 申请中断(PCIe常用MSI/MSI-X): pci_alloc_irq_vectors...
    return 0;
}
static struct pci_device_id my_ids[] = {
    { PCI_DEVICE(0x8086, 0x1234) },   // ★按Vendor/Device ID匹配
    { }
};
MODULE_DEVICE_TABLE(pci, my_ids);
static struct pci_driver my_pci_driver = {
    .name = "mydev",
    .id_table = my_ids,
    .probe = my_pci_probe,
    .remove = my_pci_remove,
};
module_pci_driver(my_pci_driver);

结构和前面所有驱动一致:填 pci_driver(probe + id_table)、module_pci_driver。id_table 按 Vendor/Device ID 匹配(枚举读配置空间得到)。probe 里:使能设备、申请并 pci_iomap 映射 BAR 得到寄存器指针、申请中断(PCIe 常用 MSI/MSI-X——基于内存写的消息中断,比传统引脚中断更高效,第13章 GIC 也支持 LPI/MSI)。之后就是熟悉的 readl/writel 操作寄存器、处理中断——和其他驱动殊途同归。

49.10 USB/PCIe 的共性：总线模型再现

把 USB、PCIe 和前面所有总线放一起,你会看到惊人的统一:

总线设备发现匹配依据驱动结构
platform设备树静态compatibleprobe + of_match_table
I2C设备树静态compatible/地址probe + of_match_table
SPI设备树静态compatible/片选probe + of_match_table
USB运行时枚举VID/PID/类probe + id_table
PCIe运行时枚举Vendor/Device IDprobe + id_table

所有总线的驱动都是同一个骨架:填一个 xxx_driver(含 probe + 匹配表)、module_xxx_driver 注册、probe 里初始化。差别只在两点:①设备怎么被发现——板载固定总线(platform/i2c/spi)靠设备树静态描述,热插拔/可扩展总线(USB/PCIe)靠运行时枚举(设备自带描述符/配置空间);②匹配依据——设备树用 compatible,USB/PCIe 用 VID/PID。但核心的"总线撮合设备与驱动、配对成功调 probe"(第40章)完全一样。这就是第40章那套抽象模型的终极威力:一套心智模型,通吃 platform/i2c/spi/usb/pci 所有总线。你学会任何一个,其余的都触类旁通——这正是本卷反复强调、现在完全兑现的道理。理解了这个统一,你面对任何新总线(甚至未来的新总线)都能快速上手:先问"设备怎么发现、按什么匹配",再套 probe 骨架。

49.12 面试常问问题

Q1:USB 为什么能即插即用?
靠枚举和描述符:插上时主机自动读设备描述符(VID/PID/类别/端点),据此匹配加载驱动。设备自带"身份信息",运行时动态发现(vs I2C/SPI 写死设备树)。

Q2:USB 枚举过程?
检测插入→分配地址→通过端点0读描述符(设备/配置/接口/端点)→按 VID/PID 或设备类匹配驱动→加载配置→可用。全自动。

Q3:USB 四种传输类型?摄像头用哪种?
控制(枚举/配置)、中断(鼠标键盘小数据)、批量(U盘可靠)、等时(摄像头/音频,保证带宽不重传)。摄像头用等时——保证实时,丢帧可容忍不卡顿(类似 UDP)。

Q4:PCIe 的特点?和 USB 比?
PCIe:点对点高速串行、设备内存映射进 CPU 空间、接板内高速设备(GPU/NVMe/网卡)。USB:树形、主机轮询、热插拔外接设备。PCIe 更快、内存映射;USB 更通用、热插拔。

Q5:PCIe 的 BAR 是什么?
基地址寄存器。设备声明需要多大地址空间,系统分配 CPU 物理地址并映射设备寄存器/内存,CPU 访问该地址即访问设备(内存映射 I/O)。驱动 pci_iomap 得指针操作。

Q6:USB/PCIe 驱动和 I2C/SPI 驱动结构一样吗?
一样:填 xxx_driver(probe+匹配表)+module_xxx_driver。区别:USB/PCIe 运行时枚举发现、按 VID/PID 匹配;I2C/SPI 设备树静态描述、按 compatible 匹配。都基于总线-设备-驱动模型。

Q7:设备信息被发现的两种方式?
①静态(设备树):板载固定总线 platform/i2c/spi,设备信息写在 .dts。②运行时枚举:热插拔/可扩展总线 USB/PCIe,设备自带描述符/配置空间,插上/开机自动发现。

49.13 常见误区

  1. "USB 中断传输是硬件中断"——不是,是主机周期轮询小数据的传输类型。
  2. "USB 摄像头用批量传输"——用等时传输(保证实时带宽,不重传)。
  3. "USB 设备信息要写设备树"——USB 运行时枚举读描述符发现,不写设备树。
  4. "PCIe 是共享总线"——点对点直连,靠 Switch 扩展,不是共享。
  5. "PCIe 设备地址固定"——开机枚举时通过 BAR 动态分配。
  6. "USB/PCIe 驱动完全不同于 I2C/SPI"——同一总线模型骨架,只是发现方式和匹配依据不同。

49.14 本章总结

  • USB/PCIe 是高速、复杂、支持自动发现的总线;嵌入式多用现成驱动,重点理解原理。
  • USB:热插拔、树形、主机主动、自描述。核心=枚举(读描述符:VID/PID/类/端点)→匹配驱动。通信单位=端点(设备→接口→端点)。
  • USB 四传输类型:控制/中断(轮询小数据)/批量(可靠)/等时(实时不重传,摄像头/音频)。
  • USB 驱动:usb_driver(probe+id_table 按 VID/PID/类)、disconnect(热插拔)、URB 传输。
  • PCIe:点对点高速串行、设备内存映射进 CPU 空间、开机枚举。核心=配置空间(Vendor/Device ID + BAR)。BAR→分配 CPU 地址→pci_iomap 映射→readl/writel。中断用 MSI/MSI-X。
  • PCIe 驱动:pci_driver(probe+id_table 按 Vendor/Device ID)。
  • 总线模型再现★:所有总线驱动同一骨架(probe+匹配表+module_xxx_driver);差别只在发现方式(设备树静态 vs 运行时枚举)和匹配依据(compatible vs VID/PID)。
  • 第六卷三主线:总线-设备-驱动骨架、设备与驱动分离、子系统统一抽象;处处回扣前五卷内功。
