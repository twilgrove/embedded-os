第卅五章 Epoll 与 IO 多路复用

一个线程如何同时盯住上万个连接

本章目标:一个服务器要同时处理上万个连接(想想 IoT 网关、机器人多传感器汇聚),怎么用一个(或少数几个)线程盯住这么多 fd?答案是 IO 多路复用,而 Linux 上的王者是 epoll。这章你会看到 epoll 如何用第32章的 fd、第34章的 socket、第30章的进程唤醒、红黑树(第30章 CFS 见过)组合出一个高性能事件通知机制。这是高性能网络编程的核心,面试必问。

35.1 为什么需要 IO 多路复用

假设你写一个服务器,要同时和 10000 个客户端通信。最朴素的想法:每个连接开一个线程,线程里 read() 阻塞等数据。问题来了:10000 个线程——每个线程要栈(第五章,几 MB)、要被调度(第30章,上下文切换开销),10000 个线程光内存和调度开销就压垮系统。

换个思路:大部分连接在大部分时间是"闲着的"(没数据)。能不能用一个线程,同时"盯着"这 10000 个 fd,哪个有数据了就处理哪个?这就是 IO 多路复用(IO multiplexing):用一个线程监控多个 IO 通道,只在有事件的通道上干活。"多路"=多个 fd,"复用"=复用一个线程。

35.2 生活例子：餐厅服务员

一个服务员(线程)管 20 桌客人(20 个连接)。笨办法:给每桌配一个专属服务员(每连接一线程),20 个服务员大部分时间站着发呆(连接空闲),浪费。更笨:一个服务员挨桌问"您需要点餐吗?"(轮询),问一圈累死还慢。

聪明办法:一个服务员站在中间,哪桌举手(有事件)就去哪桌。他不用挨个问,也不用一桌一个人——他只在"有人举手"时行动。epoll 就是这个"看谁举手"的机制:你把所有桌(fd)登记给它,它告诉你"3 号桌和 7 号桌举手了",你只处理这两桌。关键是"举手"的通知机制,让服务员不用轮询、不用一对一。

35.3 阻塞 / 非阻塞 / 多路复用

先厘清几种 IO 模型,epoll 属于哪种:

模型做法问题
阻塞 IOread 没数据就睡,直到有数据一个线程只能等一个 fd
非阻塞 IOread 没数据立即返回 EAGAIN要不停轮询所有 fd,忙等浪费 CPU
IO 多路复用一个 syscall(epoll_wait)同时等多个 fd,有就绪才返回✓ 一线程高效等多个 fd
异步 IO(AIO)发起后内核完成再通知Linux 支持有限,较少用

IO 多路复用的精髓:把"等待多个 fd"这件事交给内核用一个系统调用完成。你调 epoll_wait 睡下去(不占 CPU,第30章睡眠),内核帮你盯着所有登记的 fd,任何一个就绪就唤醒你并告诉你是哪个。既不像阻塞 IO 那样一线程只能等一个,也不像非阻塞轮询那样烧 CPU。三代多路复用 API:select → poll → epoll,epoll 是最终答案。

35.4 select/poll 的困境

epoll 之前有 select/poll,它们能用但有致命性能问题(面试爱对比):

select/poll 的工作方式(每次调用):
  1. 你把要监控的fd集合(比如10000个) 整个传给内核
  2. 内核遍历这10000个fd, 逐个检查有没有就绪  ← O(n)
  3. 返回后, 你还要再遍历10000个 找出哪几个就绪  ← O(n)

  问题:
  ① 每次调用都要把整个fd集合从用户拷到内核(大)
  ② 每次都O(n)遍历所有fd(哪怕只有1个就绪)
  ③ select还有FD_SETSIZE=1024上限

核心痛点:select/poll 每次调用都要"重新告诉内核我关心哪些 fd",并且内核每次都 O(n) 遍历全部 fd。当 fd 有上万个而每次只有几个就绪时,绝大部分遍历是白费的。10000 个连接、每次只 2 个活跃,却要遍历 10000 次——效率随连接数线性恶化。epoll 就是来解决这两个"每次都全量"的问题的。

35.5 epoll 登场

epoll 的核心改进,一句话:把"关心哪些 fd"这件事在内核里"存起来"(一次登记,长期有效),内核用回调机制维护一个"就绪列表",你只取就绪的那几个。

epoll 的思路对比 select:
  select: 每次都问"这10000个里谁好了?" → 每次遍历10000
  epoll:  fd登记一次存内核; 谁就绪了内核主动把它放进"就绪表";
          你只需取就绪表 → 只看就绪的那几个, O(就绪数)

  关键区别:
  select 是"你每次拿全部去问"(拉, O(n))
  epoll  是"内核把就绪的推给你"(推, O(就绪数))

从"轮询式拉取"变成"事件式推送"——这是 epoll 高效的根本。fd 登记一次就存在内核的数据结构里(不用每次重传);哪个 fd 就绪由内核在数据到达时(第34章接收流程里)通过回调标记进就绪表;你 epoll_wait 只拿就绪表。无论登记了 1 万还是 10 万个 fd,epoll_wait 的开销只和"当前就绪的数量"有关,与总数无关。这就是 epoll 能撑百万连接的原因。

35.6 epoll 三个系统调用

epoll 就三个 API,对应"创建、增删改、等待":

① epoll_create() → 创建一个epoll实例, 返回一个epfd(又是fd!)
② epoll_ctl(epfd, ADD/MOD/DEL, fd, event)
      → 往epoll里【登记/修改/删除】要监控的fd (一次性登记)
③ epoll_wait(epfd, events, maxevents, timeout)
      → 阻塞等待, 返回【就绪的fd列表】(只返回就绪的!)

典型循环:

  epoll_ctl(epfd, ADD, listen_fd, ...);   // 登记监听socket
  while(1){
     n = epoll_wait(epfd, evs, MAX, -1);   // 睡到有事件

对比 select:select 每次循环都要重建 fd 集合传进去;epoll 用 epoll_ctl 登记一次,之后循环里只 epoll_wait 拿就绪。注意 epoll_create 也返回一个 fd(epoll 实例本身也是"文件",第32章)——所以 epoll 实例还能被另一个 epoll 监控(嵌套)。

35.7 epoll 底层：红黑树 + 就绪链表

epoll 内核里两个关键数据结构,理解它俩就理解了 epoll:

epoll 实例(struct eventpoll)内部:

  ┌─────────────────────────────────────┐
  │  红黑树 (rbr)                          │
  │  存所有【登记】的fd (epoll_ctl ADD进来的) │
  │  → 增删查O(log n), 支持海量fd            │
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │  就绪链表 (rdllist)                     │
  │  存当前【已就绪】的fd                     │
  │  → epoll_wait直接返回这个链表, O(就绪数)  │
  └─────────────────────────────────────┘

两个结构各司其职:红黑树管"所有登记的 fd"(可能上万,用红黑树保证增删查 O(log n)——又是第30章 CFS 用过的红黑树,可见这个"需要海量元素高效增删查找"的场景反复出现);就绪链表管"当前就绪的 fd"(通常很少)。epoll_ctl ADD 往红黑树加节点;epoll_wait 从就绪链表取。关键:epoll_wait 只碰就绪链表,和红黑树里登记了多少个 fd 无关——这就是 O(就绪数) 而非 O(总数) 的来源。那"谁把 fd 放进就绪链表"?回调机制,下一节。

35.8 回调：数据来了怎么变"就绪"

epoll 最精妙的部分:fd 就绪不是靠遍历发现的,而是靠"回调"主动通知的。把它和第34章的接收流程接起来:

epoll_ctl ADD 一个socket fd 时:
  epoll 在这个fd的"等待队列"上挂一个【回调函数】

数据到达时(第34章接收流程):
  ① 网卡中断 → 软中断处理协议栈 → 数据放进socket接收缓冲
  ② 内核唤醒等在这个socket上的东西 → 触发epoll挂的【回调】
  ③ 回调做什么? 把这个fd加进epoll的【就绪链表】
  ④ 并唤醒阻塞在epoll_wait的进程

所以: 数据一来 → 回调自动把fd放进就绪表 → epoll_wait被唤醒返回它
      全程没有"遍历所有fd", 谁来数据谁自己"报到"

这就是 epoll 不用遍历的秘密:每个被监控的 fd 都在自己的等待队列上挂了 epoll 的回调,数据到达时(第34章那条中断→软中断→唤醒的链)顺手触发回调,fd "主动"把自己加进就绪链表。内核根本不需要问"谁就绪了",就绪的 fd 自己报到。这是"被动等待遍历"到"主动回调上报"的范式转变——和第34章"硬中断只标记、数据自己驱动流程"是同一种"事件驱动"思想。epoll 把网络接收的唤醒链,延伸成了对应用的事件通知。

35.9 水平触发 vs 边缘触发

epoll 有两种触发模式,面试必问、用错会 bug:

水平触发 LT(默认)边缘触发 ET
何时通知只要 fd 还有数据没读完,每次 epoll_wait 都通知只在数据从无到有那一刻通知一次
类比只要水位高于线就一直响铃只在水位"越过"线那一下响一次
编程宽松:这次没读完,下次还会提醒严格:必须一次性读干净(循环 read 到 EAGAIN),否则漏数据
性能略多唤醒唤醒更少,更高效

核心区别在"通知的持续性":LT 是"有数据就一直提醒你",ET 是"只在数据到来的边沿提醒一次"。ET 更高效(通知少),但危险:既然只提醒一次,你就必须在这次把数据全部读完(循环 read 直到返回 EAGAIN 表示读空),否则剩下的数据不会再触发通知,就"丢"了(其实是卡住了)。所以 ET 必须配合非阻塞 fd + 循环读到 EAGAIN。LT 宽松容错(这次没读完下次还提醒),是默认模式,新手用 LT 更安全。面试常问"LT 和 ET 区别、ET 为什么要循环读到 EAGAIN",这就是答案。

35.10 epoll 为什么快（总结性)

把 epoll 高效的原因汇总(面试标准答案):

优化点select/pollepoll
fd 集合传递每次调用全量拷进内核epoll_ctl 登记一次,存内核红黑树
找就绪 fd每次 O(n) 遍历全部回调直接放就绪链表,O(就绪数)
返回结果返回全集,用户再遍历找就绪只返回就绪列表
fd 上限select 有 1024 限制无硬上限(看内存)
随连接数扩展O(n) 线性恶化与总连接数无关

一句话:epoll 把"每次全量拉取+遍历"变成"登记一次+回调推送就绪",使性能与"就绪数"挂钩而非"总 fd 数"挂钩。这让单机撑十万、百万连接成为可能(C10K/C10M 问题)。Nginx、Redis、Netty 等高性能服务全靠 epoll。

35.11 回扣全书

epoll 是一个"集大成"的机制,它复用了前面几乎所有内核构件:

epoll 用到的来自
fd / epoll 实例也是 fd / socket 是文件第32章 VFS
红黑树管理海量登记 fd第30章 CFS(同款数据结构)
epoll_wait 睡眠 / 就绪唤醒进程第30章 调度、进程睡眠唤醒
数据到达触发回调(中断→软中断→唤醒)第34章接收、第七/36章中断
事件驱动、不轮询第七章中断思想的延伸

epoll 完美展示了"内核构件如何组合成高级机制":它没发明什么全新的东西,而是把 fd 抽象(VFS)、红黑树、进程睡眠唤醒、等待队列回调这些已有零件巧妙拼装,解决了"一线程盯万连接"的难题。这正是本教材想教给你的能力——不是记住每个机制的孤立细节,而是理解这些机制如何相互咬合、组合出解决实际问题的方案。对机器人工程师,epoll 是写高性能网络服务(多传感器数据汇聚、多机通信中枢、图传服务器)的必备工具;理解它的原理,你才能在遇到"连接数上去后延迟飙升""ET 模式漏数据"等真实问题时诊断到根因。

35.12 面试常问问题

Q1:什么是 IO 多路复用?解决什么问题?
用一个线程同时监控多个 fd,只在有就绪事件的 fd 上干活,避免"每连接一线程"(太多线程)和"非阻塞轮询"(烧 CPU)。适合海量连接大多空闲的场景。

Q2:select/poll 和 epoll 的区别?epoll 为什么快?
select/poll 每次调用全量传 fd 集合、内核 O(n) 遍历、返回后用户再遍历;epoll 用 epoll_ctl 登记一次存内核红黑树,数据到达靠回调把就绪 fd 放就绪链表,epoll_wait 只取就绪的 O(就绪数)。性能与就绪数挂钩而非总数。

Q3:epoll 的底层数据结构?
红黑树(存所有登记 fd,增删查 O(log n))+ 就绪链表(存当前就绪 fd,epoll_wait 直接取)。

Q4:fd 怎么变"就绪"的?
epoll_ctl ADD 时在 fd 等待队列挂回调;数据到达(网卡中断→软中断→放 socket 缓冲→唤醒等待队列)触发回调,回调把 fd 加进就绪链表并唤醒 epoll_wait。不靠遍历,靠回调上报。

Q5:LT 和 ET 的区别?ET 要注意什么?
LT(默认)只要还有数据没读完就一直通知;ET 只在数据从无到有时通知一次。ET 更高效但必须一次性把数据读干净(非阻塞 fd 循环 read 到 EAGAIN),否则漏数据。

Q6:epoll_create 返回的也是 fd,有什么意义?
epoll 实例本身是"文件"(一切皆文件),所以能被 close、能被另一个 epoll 监控(嵌套)、能在 fd 表统一管理。

Q7:为什么不直接每连接开一线程?
线程有栈(几 MB)和调度开销,上万线程内存和上下文切换压垮系统;而连接大多空闲,一线程 epoll 更高效。

35.13 常见误区

  1. "epoll 一定比 select 快"——fd 少且大多活跃时优势不明显;epoll 的优势在"海量 fd、少数活跃"。
  2. "epoll_wait 返回所有登记的 fd"——只返回就绪的那些。
  3. "ET 模式 read 一次就行"——必须循环 read 到 EAGAIN 读干净,否则剩余数据不再通知。
  4. "epoll 靠遍历找就绪 fd"——靠回调主动上报,不遍历。
  5. "epoll 能加速单个 IO"——它不加速单次 IO,是高效"等待多个 fd"。
  6. "ET 用阻塞 fd 也行"——ET 循环读时,读空后阻塞 fd 会卡死,必须用非阻塞 fd。

35.14 本章总结

  • IO 多路复用:一个线程用一个系统调用同时等多个 fd,只处理就绪的,解决"海量连接大多空闲"(避免每连接一线程、避免非阻塞轮询)。
  • select/poll 痛点:每次全量传 fd + O(n) 遍历全部,随连接数线性恶化。
  • epoll:epoll_create(建实例,也是 fd)/epoll_ctl(登记一次)/epoll_wait(只取就绪)。从"拉取遍历"变"回调推送"。
  • 底层:红黑树(存所有登记 fd,O(log n))+ 就绪链表(存就绪 fd,epoll_wait 直接取)。
  • 就绪机制:fd 等待队列挂回调,数据到达(中断→软中断→唤醒)触发回调把 fd 放就绪链表——不遍历,靠上报。
  • LT(默认,有数据就一直通知,宽松)vs ET(边沿通知一次,高效但须非阻塞+循环读到 EAGAIN)。
  • epoll 快在:登记一次、回调推送、只处理就绪、性能与就绪数(非总数)挂钩 → 撑十万百万连接。
  • 回扣:epoll 是 VFS(fd)+红黑树+调度(睡眠唤醒)+中断回调的组合,全书构件的集大成。
