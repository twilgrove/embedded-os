本章目标:上一章把一个核"净化"到极致,你可能已经想到:何不干脆让那个核跑独立的 RTOS?这就是 AMP(Asymmetric Multi-Processing,非对称多处理)——在一颗 SoC 里,不同核跑不同的操作系统(如大核跑 Linux、小核跑 RT-Thread)。这是"既要 Linux 功能、又要 RTOS 硬实时"这个终极矛盾(第55章)的最优解,也是车企产品的主流架构。本章讲 AMP 的原理、资源划分、挑战,为下一章核间通信铺路。

58.1 SMP 的局限：统一 OS 的代价

SMP(第57章)让所有核跑同一个 Linux,方便统一。但对"既要功能又要硬实时"的需求,SMP 有根本局限:

SMP的局限(即使绑核+隔离+净化):
  ① 所有核共享同一个Linux内核 → 内核里的锁、全局状态、
     偶发的长关中断/不可抢占段(第51章), 仍可能波及"隔离核"
     → 隔离核的实时性仍受Linux内核整体影响, 不是完全独立
  ② Linux内核庞大复杂, 极致硬实时(us级、超严格)仍难保证
  ③ 隔离核在跑Linux, 有Linux的开销和不确定尾巴

  → 结论: 隔离核能做到"接近"RTOS, 但终究还在Linux里,
     无法达到独立RTOS/裸机那种极致确定性
  → 如果需要真正RTOS级的硬实时, 怎么办? → AMP

SMP 即使做到极致隔离净化(第57章),隔离核仍在跑同一个 Linux 内核——内核的全局锁、偶发的长关中断/不可抢占段(第51章)仍可能波及它,达不到独立 RTOS 的极致确定性。Linux 内核太庞大复杂,对超严格的硬实时(微秒级、绝对保证)总有难以消除的不确定尾巴。净化的隔离核能"接近"RTOS,但终究还在 Linux 框架内。那如果需要真正 RTOS 级的硬实时(如高频电机 FOC),怎么办?答案是彻底分开:让那个核跑一个完全独立的 RTOS,不受 Linux 任何影响——这就是 AMP。从 SMP 的"隔离核"到 AMP 的"独立 OS",是量变到质变。

58.2 生活例子：一栋楼两套物业

SMP 像一栋楼由一家物业公司统一管理所有楼层(一个 OS 管所有核)——方便统一,但如果某层要求"绝对安静、24 小时不受任何打扰"(硬实时),而物业偶尔要全楼停电检修(内核的全局操作),这层还是会受影响。

AMP 像把这栋楼分成两部分,各请一套独立物业:大部分楼层(大核)交给"综合服务型物业"(Linux,功能全:餐厅、健身、快递),那个要求绝对安静的楼层(小核)交给"专业静音物业"(RTOS,只保证一件事:绝对安静确定)。两套物业各管各的、互不干扰——静音层永远不会因为综合物业的检修而受影响。它们之间只通过一个"前台"(核间通信,第59章)传递必要信息(如"给静音层送个包裹")。这就是 AMP:不同核跑不同 OS,各自独立,按需通信。

58.3 AMP 是什么

明确 AMP 的定义:

AMP (Asymmetric Multi-Processing, 非对称多处理):
  一颗SoC里, 【不同的核运行不同的操作系统/软件】,
  各自独立, 通过专门的核间通信机制交互

  RK3588 典型AMP:
  ┌─────────────────────┬──────────────────┐
  │  核0-5 (A76/A55)     │  核6-7 (A76)      │
  │  运行 Linux          │  运行 RT-Thread   │
  │  (视觉/AI/通信/应用) │  (硬实时控制)     │
  │  功能强大            │  确定性强         │
  └─────────┬───────────┴────────┬─────────┘
            │  核间通信(RPMsg等,第59章)  │
            └──────── 共享内存交换数据 ────┘

  ★关键: 两个OS【完全独立】运行, 有各自的内存区、中断、外设,
     互不干扰(Linux崩了RTOS照跑, 反之亦然)
     只在需要时通过核间通信交换数据/命令

AMP 的核心:一颗 SoC 里,不同的核跑不同的、完全独立的操作系统,各自有独立的内存、中断、外设,互不干扰,只通过专门的核间通信交换数据。"非对称"(相对 SMP 的"对称")就体现在:核不再是同质的核池,而是被分成几组,每组跑不同的软件、担不同的职责。RK3588 典型 AMP:大部分核跑 Linux(功能强,做视觉/AI/应用),留一两个核跑 RT-Thread/FreeRTOS(确定性强,做硬实时控制)。关键是"完全独立"——两个 OS 是两个独立的世界,Linux 崩溃重启不影响 RTOS 继续跑(反之亦然),这对安全至关重要(飞控 RTOS 不能因为 Linux 应用崩溃而失控)。它们只在必要时通过核间通信(RPMsg,第59章)和共享内存(第60章)握手交互。这就实现了"一颗芯片,两个世界,各取所长"。

58.4 AMP vs SMP 对比

两种多核模型的全面对比(面试高频):

SMP(对称)AMP(非对称)
OS 数量一个(所有核共享)多个(每组核一个)
核的角色对等(核池)分组,各担不同职责
调度OS 统一调度到各核各 OS 管自己的核
内存共享统一划分(各 OS 一块 + 共享区)
隔离性弱(共享内核,互相影响)强(独立 OS,互不干扰)
实时性好(PREEMPT_RT/隔离)但有 Linux 尾巴极好(RTOS 核不受 Linux 影响)
通信共享内存直接(同一 OS)需专门核间通信(RPMsg,第59章)
复杂度低(统一)高(两套系统+通信)

SMP 和 AMP 的根本区别:SMP 是"一个 OS 管所有核"(统一、简单、隔离弱),AMP 是"多个 OS 各管一组核"(分立、复杂、隔离强)。核心权衡在"隔离性 vs 复杂度":AMP 给你强隔离(RTOS 核完全不受 Linux 影响,实时性极好、安全性高),代价是系统更复杂(要管两套 OS、要实现核间通信)。SMP 简单统一,但隔离弱(共享内核,实时有 Linux 尾巴)。选择取决于实时严格程度:中等实时(百 Hz~kHz)用 SMP+PREEMPT_RT+隔离就够;极致硬实时(高频电机、安全攸关)必须 AMP。而且很多产品两者结合:大核 SMP 跑 Linux(内部再绑核隔离),加一两个核 AMP 跑 RTOS。理解这个对比,你就能为具体需求选对多核架构——这是系统架构师的核心判断。

58.5 典型 AMP：Linux + RTOS

最经典、最常用的 AMP 组合——Linux + RTOS,各司其职:

Linux 侧(大核)RTOS 侧(专核)
负责复杂、非实时或软实时任务硬实时控制
典型任务视觉/AI、导航、通信、日志、界面、ROS2电机控制、姿态回路、传感器采集、安全监控
要求功能丰富、生态好确定性、低延迟、高可靠
OSLinux(第五卷)RT-Thread/FreeRTOS(第三/四卷)

这个"Linux + RTOS"的 AMP 分工,完美解决了第55章提出的终极矛盾:既要 Linux 的强大功能,又要 RTOS 的硬实时。答案不是"二选一",也不是"强行把 Linux 改实时"(PREEMPT_RT 有极限),而是"两个都要,分开跑":Linux 在大核上尽情发挥功能(视觉、AI、ROS2、通信),RTOS 在专核上保证硬实时(电机、姿态控制)。各用最擅长的 OS 干最合适的活。对无人机:Linux 大核跑图传编码、视觉避障 AI、遥控通信、导航规划;RTOS 专核跑飞控姿态环、电机控制、IMU 高频采集、安全监控——飞控绝对可靠(RTOS 保证),同时享受 Linux 的智能功能。这就是本教材第三/四卷(RTOS)和第五卷(Linux)在一颗芯片上的会师——你学的两类系统不是二选一,而是在 AMP 里协同工作。这也是为什么本教材要同时深入 RTOS 和 Linux:真实产品两者都要用。

58.6 资源如何划分

AMP 里两个 OS 独立运行,前提是把硬件资源划分开,各用各的,不冲突:

AMP 资源划分(谁用什么, 事先分好):
  ① CPU核: 核0-5给Linux, 核6-7给RTOS
  ② 内存: 把DDR分成几块 —— Linux用一块、RTOS用一块、
          + 一块【共享内存】用于核间通信(第60章)
     → 设备树里给Linux的内存范围要【避开】RTOS的区域!
       (否则Linux会用到RTOS的内存 → 崩溃)
  ③ 外设: 每个外设分给一个OS独占
     → 电机CAN给RTOS、摄像头给Linux, 各配各的驱动
     → 不能两个OS同时驱动一个外设(冲突)
  ④ 中断: GIC(第13章)把不同中断路由给不同核/OS
     → 电机中断给RTOS核, 网络中断给Linux核

  ★核心原则: 资源【静态划分、明确归属】, 避免两个OS抢同一资源

AMP 能工作的前提是把硬件资源明确划分给各个 OS,避免冲突。①CPU 核分组(核 0-5 给 Linux、6-7 给 RTOS);②内存分块——Linux 一块、RTOS 一块、加一块共享区(核间通信用),关键是 Linux 的设备树里要"挖掉"RTOS 用的内存区域(reserved-memory),否则 Linux 会占用 RTOS 的内存导致崩溃;③外设各归各(电机 CAN 给 RTOS、摄像头给 Linux),不能俩 OS 抢一个外设;④中断用 GIC(第13章)路由到对应核/OS。核心原则:静态划分、明确归属。这个划分通常在设备树(Linux 侧,第41章)和 RTOS 的配置里协调好。资源划分做不好是 AMP 最常见的故障源(两个 OS 抢内存/外设/中断)——所以它是 AMP 工程的基础和难点。

58.7 谁来协调：主从模式

AMP 里两个 OS 怎么启动、谁管谁?常见主从(master-slave)模式:

AMP 主从模式(RK3588常用):
  ★Linux 作为 Master(主), RTOS 作为 Remote/Slave(从):
   ① 先启动Linux(第56章启动流程)
   ② Linux通过【RemoteProc】框架(第59章):
       - 加载RTOS的固件(.elf)到RTOS核的内存
       - 启动RTOS核(让它开始执行固件)
       - 之后可以停止/重启RTOS核
   ③ RTOS在专核上独立运行, 做实时控制
   ④ 两者通过RPMsg(第59章)通信

  好处: Linux统一管理(加载/启停RTOS核), 便于开发和恢复
       (RTOS挂了, Linux能重新加载它 —— remoteproc)

  另一种: 对称AMP(两个OS对等启动, 无主从) —— 较少用
  或裸机AMP(从核跑裸机程序而非完整RTOS)

RK3588 常用"主从模式":Linux 是主(master),RTOS 是从(remote)。启动时先起 Linux,然后 Linux 通过 RemoteProc 框架(第59章)把 RTOS 的固件加载到 RTOS 核的内存、启动那个核,让 RTOS 跑起来;之后 Linux 还能停止/重启 RTOS 核。好处是 Linux 统一管理协处理器(加载、启停、故障恢复)——RTOS 挂了,Linux 能用 RemoteProc 重新加载它,提高系统鲁棒性。两个 OS 通过 RPMsg(第59章)通信。RemoteProc(管理协处理器生命周期)和 RPMsg(核间消息传递)是 Linux 官方的 AMP 支持框架,下一章详讲——它们让"主 Linux 控制从 RTOS"标准化、易用。这就是 RK3588 上跑 AMP 的标准方式。

58.8 AMP 的挑战

AMP 强大,但也带来 SMP 没有的挑战(体现工程深度):

挑战说明应对
资源冲突两 OS 抢内存/外设/中断严格静态划分(58.6)
核间通信两 OS 要交换数据但内存独立RPMsg + 共享内存(第59/60章)
Cache 一致性共享内存区两核的 Cache 可能不一致软件同步(flush/invalidate,第43章)
启动/调试两套 OS,调试复杂,难统一观测RemoteProc 管理 + 各自日志
共享资源竞争共用的时钟/电源/总线带宽协调配置,带宽 QoS

AMP 的复杂度主要来自"两个独立系统要协作"带来的一系列问题:资源冲突(要严格划分)、核间通信(内存独立,要专门机制)、共享内存的 Cache 一致性(两个核各有 Cache,一个核写了共享内存,另一个核的 Cache 可能是旧的——第43章 DMA 一致性问题在 AMP 里再次出现,要软件 flush/invalidate)、调试困难(两套 OS 难统一观测)、共享的时钟/电源/总线带宽竞争。其中 Cache 一致性尤其容易踩坑:AMP 的共享内存通信里,如果不正确处理 Cache,会出现"一个核写了数据,另一个核读到旧值"——正是第43章讲的问题,在这里以核间通信的形式重现。这些挑战解释了为什么 AMP 比 SMP 复杂得多,也是为什么需要 RPMsg/RemoteProc 这样的标准框架(第59章)来降低难度。理解这些挑战,你才能真正落地一个可靠的 AMP 系统,而不是搭个 demo 就到处是诡异 bug。

58.9 RK3588 上的 AMP 实践

RK3588 上做 AMP 的实际选择和方案:

RK3588 AMP 的几种实现:
  ① Linux + RTOS on A 核: 拿出1-2个A55/A76核给RT-Thread/FreeRTOS
     用RemoteProc加载、RPMsg通信 (最常见)
  ② 利用SoC内的小MCU: 有些SoC带独立的Cortex-M核(如做电源管理),
     可以跑实时固件
  ③ Linux + 裸机: 从核跑裸机控制程序(更极致但难开发)

  RK3588 AMP 工程要点:
   □ 设备树 reserved-memory: 给RTOS划出专用内存, Linux避开
   □ RTOS固件放对内存地址, 配好它的中断/外设
   □ RemoteProc配置: Linux如何找到并启动RTOS核
   □ RPMsg/virtio共享内存: 配好通信通道(第59章)
   □ 处理共享内存Cache一致性

  瑞芯微提供的方案: 有官方的AMP/RPMsg支持和示例

RK3588 上 AMP 最常见的做法:从 8 个 A 核里拿出 1-2 个给 RT-Thread/FreeRTOS,用 RemoteProc 加载启动、RPMsg 通信。工程要点包括:设备树用 reserved-memory 给 RTOS 划专用内存(Linux 避开)、RTOS 固件放对地址并配好它独占的中断/外设、配置 RemoteProc 让 Linux 能启动 RTOS 核、建好 RPMsg 通信通道、处理共享内存 Cache 一致性。瑞芯微官方提供 AMP/RPMsg 的支持和示例,降低了落地难度。这就是机器人在 RK3588 上"Linux 跑智能应用 + RTOS 跑实时控制"的实际实现路径——把第三/四卷的 RTOS、第五卷的 Linux、本卷的多核和通信,综合成一个真实可用的系统。这是第八卷"综合实战"最集中的体现。

58.10 为什么 AMP 是终极实时方案

回到第55章的终极矛盾,看 AMP 如何是最优解:

"既要Linux功能, 又要硬实时" 的方案演进:
  方案A: 纯RTOS
    → 硬实时✓ 但没有Linux的功能(视觉/AI/网络生态)✗
  方案B: 纯Linux + PREEMPT_RT (第55章)
    → Linux功能✓ 实时性好但有极限(不及RTOS), 极致硬实时✗
  方案C: Linux SMP + 隔离净化核 (第57章)
    → 接近RTOS但仍在Linux里, 有尾巴 △
  ★方案D: AMP (Linux + RTOS)
    → Linux功能✓(大核) + RTOS级硬实时✓(专核) + 强隔离✓
    → 两全其美! 各用最擅长的OS
    代价: 系统复杂、要核间通信

  → AMP是"功能与硬实时兼得"的终极方案,
     所以成为车企复杂实时产品的主流架构

AMP 是"既要 Linux 功能、又要 RTOS 硬实时"这个终极矛盾的最优解。回顾方案演进:纯 RTOS(实时但没功能)、纯 Linux+PREEMPT_RT(有功能但实时有极限)、SMP 隔离核(接近但有 Linux 尾巴),都是"顾此失彼"或"折中";只有 AMP 做到了真正两全——Linux 在大核尽情发挥功能,RTOS 在专核提供 RTOS 级的硬实时,两者强隔离互不拖累。代价是系统复杂(两套 OS + 核间通信),但对高价值的复杂实时产品(无人机、机器人、自动驾驶域控),这个复杂度是值得的。所以 AMP 成为理想/小鹏/Momenta 这类公司复杂实时产品的主流架构。这也回答了一个你可能一直有的疑问:为什么这本教材要同时深入 RTOS(三/四卷)和 Linux(五卷)?因为真实的高端产品用 AMP,两者都要、缺一不可。你学的不是"二选一的知识",而是"组合使用的能力"——AMP 就是这种组合的最高形式。

58.11 回扣全书：各 OS 各得其所

AMP 让本教材的各部分知识在一颗芯片上"会师":

AMP 的组成对应全书
RTOS 侧(RT-Thread/FreeRTOS)第三卷、第四卷
Linux 侧第五卷、第六卷驱动
RTOS 保证硬实时第七卷实时系统
多核划分、资源分配第57章 SMP/亲和
核间通信 RPMsg第59章(下一章)
共享内存 Cache 一致性第14/43章

RTOS 固件加载(EL/启动)第12章、第56章启动

58.12 面试常问问题

Q1:什么是 AMP?和 SMP 的区别?★
AMP(非对称多处理):不同核跑不同的独立 OS,各有独立资源,互不干扰,通过核间通信交互。SMP:所有核跑同一个 OS,对等核池统一调度。AMP 隔离强、实时好但复杂;SMP 统一简单但隔离弱。

Q2:为什么用 AMP 而不是 SMP+PREEMPT_RT?
SMP 即使隔离净化,隔离核仍在同一 Linux 内核里,有 Linux 的不确定尾巴,达不到 RTOS 级极致硬实时。AMP 让 RTOS 在独立核上跑,完全不受 Linux 影响,实时性和隔离性都最好。

Q3:典型的 Linux+RTOS AMP 怎么分工?
Linux(大核)做复杂/软实时:视觉 AI、导航、通信、ROS2;RTOS(专核)做硬实时:电机控制、姿态回路、传感器采集、安全监控。各用最擅长的 OS。

Q4:AMP 里资源怎么划分?
静态划分明确归属:CPU 核分组、内存分块(各 OS 一块+共享区,Linux 设备树 reserved-memory 避开 RTOS 内存)、外设各归各 OS 独占、中断用 GIC 路由到对应核。

Q5:AMP 的主从模式?RemoteProc 是什么?
Linux 主、RTOS 从:Linux 先启动,通过 RemoteProc 加载 RTOS 固件到其内存并启动 RTOS 核,可停止/重启它(故障恢复)。RemoteProc 是 Linux 管理协处理器生命周期的框架。

Q6:AMP 有哪些挑战?
资源冲突(严格划分)、核间通信(RPMsg+共享内存)、共享内存 Cache 一致性(软件 flush/invalidate,第43章)、调试复杂、共享时钟/带宽竞争。

Q7:为什么说 AMP 是"功能与硬实时兼得"的终极方案?
纯 RTOS 没功能、纯 Linux 实时有极限、SMP 隔离有尾巴;AMP 让 Linux 大核发挥功能、RTOS 专核保证硬实时、强隔离互不拖累,两全其美。故成车企复杂实时产品主流架构。

58.13 常见误区

  1. "AMP 和 SMP 差不多"——本质不同:AMP 多个独立 OS,SMP 一个 OS 管所有核。
  2. "AMP 里两个 OS 能随便共享内存"——要划分归属+处理 Cache 一致性,共享区要专门管理。
  3. "PREEMPT_RT 能完全替代 AMP"——极致硬实时仍需 AMP,PREEMPT_RT 有 Linux 尾巴。
  4. "AMP 里两个 OS 会互相影响崩溃"——正确划分下强隔离,一个崩不影响另一个(这正是 AMP 优势)。
  5. "AMP 不需要核间通信"——两 OS 独立内存,必须用 RPMsg/共享内存通信。
  6. "AMP 只能 Linux+RTOS"——也可 Linux+裸机、用内嵌 MCU 等,形式多样。

58.14 本章总结

  • SMP 局限:隔离核仍在同一 Linux 内核,有不确定尾巴,达不到 RTOS 级硬实时 → 需 AMP。
  • AMP(非对称多处理):一颗 SoC 里不同核跑不同的独立 OS,各有独立资源、互不干扰,通过核间通信交互。
  • vs SMP:AMP 隔离强、实时极好但复杂;SMP 统一简单但隔离弱。权衡=隔离性 vs 复杂度。
  • 典型:Linux(大核,功能:视觉/AI/ROS2)+ RTOS(专核,硬实时:电机/姿态)——解决"既要功能又要硬实时"的终极矛盾,各用最擅长的 OS。
  • 资源划分:CPU 核分组、内存分块(reserved-memory 避开)、外设各归各、中断 GIC 路由;静态划分明确归属。
  • 主从模式:Linux 主 + RTOS 从,Linux 用 RemoteProc 加载/启停 RTOS 核(故障恢复)。
  • 挑战:资源冲突、核间通信、共享内存 Cache 一致性(第43章)、调试、带宽竞争。
  • AMP 是"功能与硬实时兼得"的终极方案,车企复杂实时产品主流架构;是本书 RTOS+Linux 两条线的会师。
