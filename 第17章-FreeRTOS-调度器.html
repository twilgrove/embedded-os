本章目标：第16章我们看到任务被创建、挂进就绪列表。这一章看"店长"如何工作——调度器怎么从就绪列表选出下一个任务、什么时候触发切换、时间片怎么轮转、任务怎么阻塞和被唤醒。注意区分:本章讲"决定切换给谁"(调度决策,C 代码),第18章讲"怎么真正切过去"(上下文切换,汇编)。两者合起来才是完整的任务切换。这是第一卷"三支柱之调度器"的源码落地。

17.1 调度器要回答的两个问题

回到第一卷 1.6 的比喻:调度器是店长,厨师(CPU)只管埋头炒,店长决定"下一个炒哪道"。落到代码,调度器只需回答两个问题:

  1. 什么时候需要重新决策?(何时调度)——时钟节拍到、任务主动让出、任务阻塞、有更高优先级任务就绪。
  2. 该选哪个任务?(选谁)——从就绪列表选最高优先级的。

FreeRTOS 把这两个问题拆成两个东西:vTaskSwitchContext(回答"选谁",更新 pxCurrentTCB)和各种触发点(回答"何时",最终都调用 vTaskSwitchContext + 触发实际切换)。本章逐一拆开。

17.2 pxCurrentTCB：当前任务的指针

整个调度系统围着一个全局指针转,它是理解一切的锚点:

/* tasks.c 里最重要的全局变量 */
TCB_t * volatile pxCurrentTCB = NULL;

含义: pxCurrentTCB 永远指向"当前正在CPU上运行的那个任务的TCB"。

整个上下文切换的本质,就是改变 pxCurrentTCB 指向谁,然后让 CPU 的现场变成它指向的那个任务的现场。第18章的汇编会:①把当前寄存器存进 pxCurrentTCB 指的任务栈;②pxCurrentTCB 被 vTaskSwitchContext 改成新任务;③从新的 pxCurrentTCB 指的栈恢复寄存器。所以调度器的核心工作,就是正确地更新这一个指针。

为什么它是 volatile?因为它会被中断上下文(SysTick、PendSV)修改,也被普通代码读。volatile 阻止编译器把它缓存进寄存器,保证每次都读真实内存值(第四章讲的 volatile 的用途)。这是 volatile 的正确用法——单核下防编译器优化;注意它不解决多核问题,但 Cortex-M 单核 FreeRTOS 够用。

17.3 选任务：taskSELECT_HIGHEST_PRIORITY_TASK

"选谁"的核心宏。它做两件事:找到最高的有就绪任务的优先级,从那个优先级的链表里取一个。看 CLZ 优化版:

#define taskSELECT_HIGHEST_PRIORITY_TASK()                              \
{                                                                       \
    UBaseType_t uxTopPriority;                                          \
                                                                        \
    /* ① 用 CLZ 从位图找最高就绪优先级(第16章的O(1)技巧) */             \
    portGET_HIGHEST_PRIORITY( uxTopPriority, uxTopReadyPriority );       \
                                                                        \
    /* ② 从该优先级的就绪链表取下一个任务(同级轮转的关键) */             \
    listGET_OWNER_OF_NEXT_ENTRY( pxCurrentTCB,                          \
                                 &( pxReadyTasksLists[ uxTopPriority ] ) ); \
}

/* portGET_HIGHEST_PRIORITY 在 ARM Cortex-M 上就是: */
#define portGET_HIGHEST_PRIORITY( uxTopPriority, uxReadyPriorities ) \
    uxTopPriority = ( 31UL - ( uint32_t ) __clz( ( uxReadyPriorities ) ) )

拆解两步:

  • ① portGET_HIGHEST_PRIORITY:__clz 是 ARM 的 CLZ 指令(数前导 0),31 - clz(位图) = 最高置 1 位的位置 = 最高就绪优先级。一条指令,O(1)(第16章讲过的确定性调度关键)。
  • ② listGET_OWNER_OF_NEXT_ENTRY:这个链表是环形的,每次取"下一个"节点(通过 pxIndex 前进),于是同优先级的多个任务会被轮流取到——这就是时间片轮转的底层(17.7 详讲)。取出的节点的 owner(嵌在 TCB 里,第16章侵入式链表)就是要运行的 TCB,直接赋给 pxCurrentTCB。

这两步就是 FreeRTOS 调度算法的全部:严格最高优先级 + 同级环形轮转。简单、确定、快——完美契合实时系统需求。对比 Linux CFS 的红黑树+虚拟运行时间(第五卷),FreeRTOS 简单得多,因为它追求的是确定性而非公平性。

17.4 vTaskSwitchContext：切换的决策者

taskSELECT_HIGHEST_PRIORITY_TASK 被包在 vTaskSwitchContext 里。这个函数是"调度决策"的总入口,由汇编切换代码(PendSV)调用:

void vTaskSwitchContext( void )
{
    if( uxSchedulerSuspended != ( UBaseType_t ) 0U )
    {
        /* 调度器被挂起(17.11), 现在不能切, 标记"欠一次切换" */
        xYieldPending = pdTRUE;
    }
    else
    {
        xYieldPending = pdFALSE;

        /* 检查栈溢出(如果开启了栈溢出检测) */
        taskCHECK_FOR_STACK_OVERFLOW();

        /* ★ 核心: 选出最高优先级任务, 更新 pxCurrentTCB */
        taskSELECT_HIGHEST_PRIORITY_TASK();

        /* 运行时间统计等... */
    }
}

注意它只更新 pxCurrentTCB,不做实际的寄存器保存/恢复——那是 PendSV 汇编的活(第18章)。这是关键的职责分离:

vTaskSwitchContext = "决定切给谁"(纯 C,改指针);PendSV 汇编 = "真正切过去"(存旧任务寄存器、恢复新任务寄存器)。为什么分开?因为"选任务"是架构无关的逻辑(所有平台一样),而"存恢复寄存器"是架构强相关的(每个 CPU 寄存器不同)。分开后,vTaskSwitchContext 放在架构无关的 tasks.c,寄存器操作放在架构相关的 port。这就是第16章说的"内核=架构无关+移植层"的体现。

17.5 调度触发的三条路径

"何时调度"?FreeRTOS 里,重新调度(可能切换任务)由三类事件触发,但最终都殊途同归到"触发 PendSV 异常 → PendSV 里调 vTaskSwitchContext → 切换":

触发路径场景怎么触发
① 时钟节拍抢占SysTick 中断里发现该换人(时间片到 / 有更高优先级任务就绪)xTaskIncrementTick 返回 pdTRUE → 触发 PendSV
② 任务主动让出任务调 taskYIELD()直接触发 PendSV
③ 阻塞/唤醒引起任务 vTaskDelay、等队列阻塞;或高优先级任务被唤醒相关 API 内部触发 PendSV
三条路径殊途同归:

  SysTick中断 ──┐
  taskYIELD  ──┼──▶ 触发 PendSV 异常(挂起)
  阻塞/唤醒   ──┘         │
                         ▼(等所有中断处理完, 真正要回线程时)
                  PendSV_Handler(汇编, 第18章)
                         │
                         ├─ 保存当前任务寄存器到其栈
                         ├─ 调 vTaskSwitchContext() ← 更新pxCurrentTCB(本章17.4)
                         └─ 从新任务栈恢复寄存器
                         ▼
                    新任务运行

为什么统一用 PendSV?这是 Cortex-M 的一个精妙设计,第18章会详讲:PendSV 是最低优先级异常,把上下文切换推迟到所有其他中断都处理完、真正要返回线程时才做,避免在中断嵌套中途切换导致混乱。这里先记住:所有调度最终都通过触发 PendSV 来完成实际切换。

17.6 xTaskIncrementTick：心跳里发生了什么

第一卷说"时钟中断是抢占引擎"。这个引擎的核心就是 xTaskIncrementTick,每次 SysTick 中断都调它。它干三件大事:

BaseType_t xTaskIncrementTick( void )
{
    BaseType_t xSwitchRequired = pdFALSE;

    /* ① 节拍计数 +1 */
    const TickType_t xConstTickCount = xTickCount + 1;
    xTickCount = xConstTickCount;

    /* ② 检查有没有延时到期的任务, 把它们从"延时列表"移回"就绪列表" */
    if( xConstTickCount >= xNextTaskUnblockTime )
    {
        for( ;; )
        {
            /* 遍历延时列表(按到期时间排序), 到期的唤醒 */
            pxTCB = 取延时列表头任务;
            if( xConstTickCount < 该任务到期时间 ) break;  /* 没到期, 停止 */

            /* 到期了: 从延时列表和事件列表移除, 加入就绪列表 */
            listREMOVE_ITEM( &( pxTCB->xStateListItem ) );
            prvAddTaskToReadyList( pxTCB );

            /* 如果被唤醒的任务优先级 >= 当前任务, 需要切换(抢占!) */
            if( pxTCB->uxPriority >= pxCurrentTCB->uxPriority )
                xSwitchRequired = pdTRUE;
        }
    }

    /* ③ 时间片轮转: 同优先级还有别的就绪任务? 需要切换 */
    #if ( configUSE_TIME_SLICING == 1 )
    {
        if( listCURRENT_LIST_LENGTH( &pxReadyTasksLists[ pxCurrentTCB->uxPriority ] )
            > ( UBaseType_t ) 1 )
        {
            xSwitchRequired = pdTRUE;
        }
    }
    #endif

    return xSwitchRequired;   /* 返回pdTRUE → SysTick_Handler会触发PendSV */
}

三件事对应第一卷的三个概念落地:

  • ①节拍计数:xTickCount 是系统的"时间",vTaskDelay(n) 就是记下"到 xTickCount+n 唤醒我"。
  • ②唤醒到期任务:把睡够了的任务从延时列表移回就绪列表——如果唤醒的任务优先级更高,标记要切换(这就是"高优先级任务就绪→抢占")。
  • ③时间片:如果同优先级还有别的就绪任务,标记要切换(轮转,下节)。

返回 pdTRUE 时,SysTick 中断处理会触发 PendSV,完成切换。这一个函数,就是第一卷"三支柱之时钟中断"的完整源码。

17.7 时间片轮转的实现

同优先级的多个任务怎么公平轮流?靠两个机制配合:

  1. 触发:上面 xTaskIncrementTick 的第③步——每个 tick,如果同优先级就绪任务 >1 个,就要求切换。
  2. 选择:listGET_OWNER_OF_NEXT_ENTRY(17.3)每次取"下一个"节点。就绪链表是环形的,有个 pxIndex 指针每次前进一格,于是同优先级任务被依次取到,循环往复。
优先级3的就绪链表(环形), pxIndex 每次前进:

   ┌─▶ TaskA ──▶ TaskB ──▶ TaskC ──┐
   └───────────────────────────────┘
   pxIndex → 这次取A, 下次取B, 再下次取C, 再回到A...

每个tick切一次(若>1任务) → A跑1个tick, B跑1个tick, C跑1个tick → 公平轮转

注意 FreeRTOS 的时间片是固定 1 个 tick(不像 Linux 可变时间片)。configTICK_RATE_HZ 决定 tick 频率(如 1000Hz=1ms/tick),所以同级任务每 1ms 轮换一次。时间片轮转只在同优先级任务间发生——不同优先级永远是高的先跑(严格优先级)。这再次体现 FreeRTOS "优先级第一,公平第二"的实时哲学。若关掉 configUSE_TIME_SLICING,同级任务就不轮换,谁先就绪谁一直跑到阻塞,更省切换开销但欠公平。

17.8 主动让出:taskYIELD 与 portYIELD

任务不想等时间片,想立刻让出 CPU(比如干完活了),调 taskYIELD():

#define taskYIELD()  portYIELD()

/* ARM Cortex-M 上 portYIELD 就是"触发PendSV异常": */
#define portYIELD()                                     \
{                                                       \
    /* 往中断控制状态寄存器ICSR的PENDSVSET位写1, 挂起PendSV */ \
    portNVIC_INT_CTRL_REG = portNVIC_PENDSVSET_BIT;     \
    __dsb( portSY_FULL_READ_WRITE );  /* 内存屏障(第14章!) */ \
    __isb( portSY_FULL_READ_WRITE );  /* 指令屏障, 确保PendSV立即生效 */ \
}

看,portYIELD 的本质就是软件挂起 PendSV 异常——往一个寄存器写一位,CPU 就会在合适时机进入 PendSV 处理程序去切换。注意这里的 __dsb/__isb 就是第14章讲的内存屏障和指令同步屏障:确保"挂起 PendSV"这个写操作真正生效、后续指令重新取指,让切换立即发生。第14章的屏障知识在这里派上了用场——这不是理论,是每次任务让出都在用。

17.9 阻塞与唤醒:延时列表的巧妙设计

vTaskDelay(100) 怎么实现"睡 100 个 tick"?不是傻等,而是把任务从就绪列表移到延时列表(Delayed Task List),让出 CPU,到期再移回:

void vTaskDelay( const TickType_t xTicksToDelay )
{
    vTaskSuspendAll();   /* 锁调度器 */
    {
        /* 把当前任务从就绪列表移除, 加入延时列表, 记录唤醒时刻 */
        prvAddCurrentTaskToDelayedList( xTicksToDelay, ... );
    }
    xAlreadyYielded = xTaskResumeAll();

    /* 让出CPU(现在当前任务已不在就绪列表, 会切到别的任务) */
    if( xAlreadyYielded == pdFALSE )
        portYIELD_WITHIN_API();
}

延时列表的精妙:两个列表 + 溢出处理

FreeRTOS 用两个延时列表(xDelayedTaskList1 和 xDelayedTaskList2),外加两个指针 pxDelayedTaskList/pxOverflowDelayedTaskList。为什么要两个?因为 xTickCount 是有限位数(如 32 位),会溢出回绕(从 0xFFFFFFFF 加 1 变回 0)。

如果一个任务要睡到"溢出之后"的时刻,它就放进"溢出延时列表"。当 xTickCount 真的溢出回绕时,两个列表指针一交换,原来的"溢出列表"变成"当前列表"。用列表交换优雅地处理了计数器溢出,不用复杂的比较逻辑。

延时列表按唤醒时刻排序(最早到期的在头部),还维护一个 xNextTaskUnblockTime = 最近一个要唤醒的时刻。这样 xTaskIncrementTick(17.6)不用每个 tick 都遍历整个列表,只需比较 xTickCount >= xNextTaskUnblockTime,不到就直接跳过——大部分 tick 什么都不用做,O(1)。这个"排序 + 记录最近到期时间"的优化,是保证 tick 中断本身极短、不拖累实时性的关键。设计得非常精细。

17.10 抢占式 vs 协作式

FreeRTOS 由 configUSE_PREEMPTION 决定调度模式:

抢占式(configUSE_PREEMPTION=1)协作式(=0)
切换时机时钟节拍/高优先级就绪时随时抢占只在任务主动 taskYIELD 或阻塞时切
依赖SysTick 中断(第一卷:抢占的引擎)任务自觉让出
实时性好(高优先级任务能立刻抢到)差(赖别的任务让出)
风险需处理临界区/并发一个任务不让出会饿死别人(第一卷"闷头炒菜")

绝大多数实际项目用抢占式。协作式回到了第一卷 1.7 说的困境:如果一个任务是死循环不让出,协作式下它会永远霸占 CPU,别的任务饿死——这正是"为什么需要时钟中断做抢占"的活教材。抢占式 = 有时钟中断这个"闹钟"强制打断;协作式 = 没闹钟,全靠自觉。

17.11 调度锁:vTaskSuspendAll / xTaskResumeAll

有时一段代码需要"操作内核数据结构时不被调度打断",但又不想关中断(关中断会影响中断响应)。FreeRTOS 提供挂起调度器:

vTaskSuspendAll();    /* 挂起调度器: 之后不会发生任务切换, 但中断照常! */
{
    /* 这里安全地操作, 不会被切到别的任务 */
}
xTaskResumeAll();     /* 恢复调度器: 补上期间欠下的切换 */

关键区别(面试常考):

  • 关中断:连中断都不响应,最强,但影响实时性(第七章:关中断期间中断被推迟)。
  • 挂起调度器:中断照常响应(ISR 还能跑),只是 ISR 里即使要求切换也不切,而是记下"欠一次切换"(xYieldPending,17.4 见过),等 xTaskResumeAll 时补上。

这是精细的并发控制:挂起调度器保护"任务 vs 任务"的竞争,但不牺牲中断响应;关中断保护"任务 vs 中断"的竞争,但牺牲中断响应。用哪个取决于你要防谁。如果共享数据只在任务间访问(ISR 不碰),用挂起调度器更好(不伤实时);如果 ISR 也访问,必须关中断。这个"按竞争对象选保护手段"的思想,第五卷 Linux(自旋锁 vs 关抢占 vs 关中断)会以更丰富的形式重现。

17.12 临界区:关中断的两种粒度

FreeRTOS 的临界区宏,对应第七章"关中断保护临界区":

taskENTER_CRITICAL();     /* 进入临界区: 关中断(到一定优先级) */
{
    /* 极短的、连中断都不能打断的操作 */
}
taskEXIT_CRITICAL();      /* 退出: 恢复中断 */

/* ISR 里用带 FROM_ISR 的版本: */
uxSavedInterruptStatus = taskENTER_CRITICAL_FROM_ISR();
{ ... }
taskEXIT_CRITICAL_FROM_ISR( uxSavedInterruptStatus );

细节:FreeRTOS 的 taskENTER_CRITICAL 在 Cortex-M 上不是关所有中断,而是通过 BASEPRI 寄存器只屏蔽优先级低于 configMAX_SYSCALL_INTERRUPT_PRIORITY 的中断——比它更高优先级的中断(如飞控的紧急中断)仍能响应!

这是个重要设计:关中断也可以是"部分关"的。把中断分成两类:优先级高于阈值的"不受 FreeRTOS 管的超高优先级中断"(它们不能调用 FreeRTOS API,但响应永远不被延迟,用于极端实时需求),和低于阈值的"普通中断"(能调 FreeRTOS API,会被临界区屏蔽)。这样即使内核在临界区,最紧急的中断(如电机过流保护)依然零延迟响应——对无人机/电机控制至关重要。这也是 xxxFromISR 版本 API 存在的原因:只有优先级低于阈值的 ISR 才能安全调用 FreeRTOS API。面试常抠这个 configMAX_SYSCALL_INTERRUPT_PRIORITY。

17.13 回扣与对比 Linux

把 FreeRTOS 调度器和后面要学的 Linux 调度器先对照一下,建立坐标:

FreeRTOS 调度器Linux CFS(第五卷)
选任务数据结构优先级链表数组 + 位图红黑树(按虚拟运行时间)
选任务复杂度O(1)(CLZ)O(log n)
调度哲学严格优先级 + 同级轮转公平分配 CPU 时间
目标实时确定性吞吐 + 交互体验 + 公平
时间片固定 1 tick动态计算
核心变量pxCurrentTCBrq->curr / per-CPU 运行队列

核心差异一句话:FreeRTOS 为"确定性"而生(选任务时间恒定、高优先级绝对优先),Linux CFS 为"公平和吞吐"而生(不让任何任务饿死、照顾交互)。这就是第一卷反复说的"实时系统 vs 分时系统"的分野,现在你在 FreeRTOS 源码里看到了"确定性"那一半的具体实现。学 Linux CFS 时,你会看到"公平"那一半为了照顾通用场景做了多复杂的取舍——两相对照,理解才深。

17.14 面试常问问题

Q1:FreeRTOS 调度器如何选出下一个任务?
taskSELECT_HIGHEST_PRIORITY_TASK:用位图+CLZ 找最高就绪优先级(O(1)),从该优先级的环形就绪链表取下一个(同级轮转)。更新 pxCurrentTCB。

Q2:vTaskSwitchContext 和 PendSV 的分工?
vTaskSwitchContext(C,架构无关)只决定切给谁、更新 pxCurrentTCB;PendSV(汇编,架构相关)真正保存旧任务、恢复新任务寄存器。职责分离对应"内核=架构无关+移植层"。

Q3:调度在什么时候触发?最终怎么切换?
三条路径:SysTick 抢占、taskYIELD、阻塞/唤醒;殊途同归——都触发 PendSV 异常,在 PendSV 里调 vTaskSwitchContext 并完成切换。

Q4:时间片轮转怎么实现?只在什么任务间?
xTaskIncrementTick 每 tick 检查同优先级就绪任务>1 则要求切换;环形就绪链表 pxIndex 前进依次取任务。只在同优先级任务间轮转(固定 1 tick),不同优先级严格高的先跑。

Q5:vTaskDelay 如何实现?延时列表为什么用两个?
把任务移到延时列表并让出 CPU,到期由 tick 移回就绪列表。两个列表 + 交换指针处理 xTickCount 溢出回绕;按到期时间排序 + xNextTaskUnblockTime 让 tick 处理 O(1)。

Q6:挂起调度器和关中断的区别?
挂起调度器不切任务但中断照常响应(欠的切换事后补);关中断连中断都不响应。前者保护任务间竞争不伤实时,后者保护任务与中断竞争但伤实时。按竞争对象选。

Q7:configMAX_SYSCALL_INTERRUPT_PRIORITY 是什么?
临界区用 BASEPRI 只屏蔽低于此阈值的中断;高于它的超高优先级中断永不被延迟(但不能调 FreeRTOS API)。保证极紧急中断零延迟,对电机/飞控关键。

17.15 常见误区

  1. "vTaskSwitchContext 会保存/恢复寄存器"——不,它只更新 pxCurrentTCB;寄存器操作在 PendSV 汇编里。
  2. "时间片轮转发生在所有任务间"——只在同优先级任务间;不同优先级严格按高低。
  3. "vTaskDelay 是忙等"——不是,任务被移出就绪列表让出 CPU,不占用 CPU。
  4. "挂起调度器=关中断"——不同,挂起调度器时中断照常响应。
  5. "taskENTER_CRITICAL 关所有中断"——Cortex-M 上用 BASEPRI 只关低于阈值的,超高优先级中断仍响应。
  6. "FreeRTOS 追求公平"——它追求确定性/优先级,不追求公平(那是 Linux CFS)。
  7. "优先级数值越小越高"——FreeRTOS 任务优先级数值越大越高(再次提醒,和中断优先级相反)。

17.16 本章总结

  • 调度器回答两问:何时调度(三条触发路径)、选谁(最高优先级)。
  • pxCurrentTCB 是核心锚点;切换的本质是正确更新它。
  • taskSELECT_HIGHEST_PRIORITY_TASK:位图+CLZ 找最高优先级(O(1))+ 环形链表取下一个(同级轮转)。
  • vTaskSwitchContext(C,选谁)与 PendSV(汇编,真正切)职责分离,对应架构无关/移植层。
  • 三条触发路径(SysTick 抢占 / taskYIELD / 阻塞唤醒)殊途同归触发 PendSV。
  • xTaskIncrementTick:计数+1、唤醒到期任务(高优先级则抢占)、时间片检查——第一卷"时钟中断=抢占引擎"的源码。
  • 时间片固定 1 tick,仅同优先级间轮转;portYIELD 本质是挂起 PendSV(用了内存/指令屏障)。
  • vTaskDelay 用延时列表(双列表处理溢出、按到期排序 + xNextTaskUnblockTime 使 tick O(1))。
  • 抢占式(靠 SysTick)vs 协作式(靠自觉);挂起调度器(不伤中断)vs 关中断(伤中断)按竞争对象选。
  • configMAX_SYSCALL_INTERRUPT_PRIORITY 让超高优先级中断零延迟,是实时关键。
