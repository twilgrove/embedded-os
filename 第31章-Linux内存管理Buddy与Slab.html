本章目标：第九、十章讲了虚拟内存(给用户进程用的)。这一章讲内核自己怎么管理物理内存——内核创建 task_struct、分配内核栈、给缓冲区,这些内存从哪来?答案是两层分配器:Buddy(伙伴系统)管理"页"级别的大块,Slab 管理"对象"级别的小块。它们分别是第24章 FreeRTOS heap_4"合并相邻块"和第27章 RT-Thread"内存池/SLAB"的正统放大版。你已经见过它们的预习版,这里看全貌。

31.1 内核也要管理自己的内存

容易忽略的一点:虚拟内存(第十章)是给用户进程用的抽象。但内核自己也要用内存——创建 task_struct(第28章,几 KB)、分配内核栈、网络收发缓冲、文件缓存……这些内存内核得自己管理。而且内核的内存管理有特殊要求:

  • 要能分配物理连续的内存(有些硬件如 DMA 要求物理连续,第八卷)。
  • 要高效:内核频繁分配释放小对象(如每个网络包一个结构体),分配器慢会拖累整个系统。
  • 要对抗碎片:内核长期运行(服务器几年不关机),碎片是大敌(第24章)。

Linux 用两层分配器满足这些:底层 Buddy 系统管理物理页(4KB 及其 2 的幂倍数),上层 Slab 在页之上管理小对象。一大一小,配合工作。

31.2 物理内存的组织：page/zone/node

内核怎么看物理内存?分三个层次:

层次是什么为什么
page(页)物理内存的最小管理单位(4KB),每个物理页对应一个 struct page和 MMU 页大小一致(第九章)
zone(区)把物理内存分区:ZONE_DMA(低地址,给老 DMA)、ZONE_NORMAL(常规)、ZONE_HIGHMEM(高端,32 位遗留)不同硬件对内存地址有不同限制
node(节点)NUMA 系统里每个 CPU 就近的一块内存多路服务器,访问本地内存快

关键是 struct page:内核为每一个物理页维护一个 page 结构(记录这页的状态、引用计数、属于谁)。几 GB 内存有上百万个物理页,就有上百万个 struct page(所以这个结构被压缩得很小,几十字节)。page 是内核内存管理的原子,一切分配最终都是"分配若干个 page"。zone/node 是为了处理硬件的地址限制和 NUMA 拓扑,嵌入式(RK3588,UMA 单节点)主要接触 zone。

31.3 Buddy 系统：按 2 的幂管理页

Buddy(伙伴)系统是 Linux 管理物理页的底层分配器。核心思想:把空闲页按"2 的幂"大小分组管理。

Buddy 把空闲内存按 2^order 页分组(order 0~10):

  order 0: 空闲的【1页】块的链表     (4KB)
  order 1: 空闲的【2页】块的链表     (8KB)
  order 2: 空闲的【4页】块的链表     (16KB)
  order 3: 空闲的【8页】块的链表     (32KB)
  ...
  order 10: 空闲的【1024页】块       (4MB)

分配 alloc_pages(order): 从对应order的链表拿一个块
  要3页? → 向上取整到4页(order 2), 从order 2链表拿

为什么按 2 的幂?因为这样"分裂"和"合并"都很规整。如果需要的 order 没有空闲块,就找更大的块对半分裂;释放时,如果它的"伙伴"(相邻的、同大小的那一半)也空闲,就合并成更大的块。2 的幂让"分裂成两半""合并两个伙伴"总是对齐、简单。这就是第24章 FreeRTOS heap_4"合并相邻块"的升级版——Buddy 用 2 的幂结构让合并更系统化。

31.4 Buddy 的分配与合并

看分裂和合并的具体过程:

分配(要1个order-0页, 但order-0/1都空了, order-2有一个4页块):
  order 2: [████]  ← 有一个4页块
  分裂: [██][██]   → 一半放回order 1, 另一半继续分
  order 1: [██]
  再分裂: [█][█]   → 一半放回order 0, 另一半给你
  order 0: [█]
  给你1页 ✓

释放(还回1页, 检查它的"伙伴"):
  你还的页 [█] 和它的伙伴 [█](相邻的同大小块)都空闲?
  → 合并成 order 1 的 [██]
  → 这个[██]的伙伴也空闲? → 继续合并成 order 2 的 [████]
  → 一直合并到伙伴不空闲为止

"伙伴"的定义:两个大小相同、地址相邻、且是"从同一个大块分裂出来的"两半,互为伙伴。释放时不断向上合并伙伴,把小块拼回大块——这就是对抗外部碎片的机制。Buddy 用地址的位运算(伙伴的地址只差一个 bit)快速找到伙伴,合并判断非常快。

31.5 Buddy 解决了外部碎片

回顾第24章:外部碎片是"空闲内存分散成小块,凑不出大块"。Buddy 的合并机制正是对抗它:

Buddy 通过"释放时尽量合并伙伴",让空闲内存尽可能地保持为大块,从而在需要大块时能满足。没有 Buddy(比如简单的空闲链表),反复分配释放后小块会散落各处,凑不出大块。Buddy 的 2 的幂 + 伙伴合并,是几十年验证的经典抗碎片方案。但 Buddy 也有内部碎片:因为按 2 的幂分配,你要 5 页它给你 8 页(order 3),浪费 3 页。所以 Buddy 适合分配"页级别的大块",不适合分配"几十字节的小对象"(那样内部碎片太严重——要 32 字节却给你一整页 4KB)。小对象交给上层的 Slab(下节)。这就是"两层分配器"分工的原因:Buddy 管大块抗外部碎片,Slab 管小对象避免内部碎片。

31.6 Slab：小对象的分配器

内核频繁分配小对象:task_struct、inode、dentry、网络包结构……如果每个都用 Buddy 分一整页,浪费巨大(内部碎片)。而且这些对象大小固定、分配极频繁。Slab 分配器专门优化这种场景,核心思想:为每种常用对象维护一个"缓存池",预先分配好一批,分配时直接取、释放时还回池子。

Slab 思想: 为每种对象建一个专用缓存(kmem_cache)

  task_struct 缓存: [obj][obj][obj][obj]...  预分配一批task_struct大小的槽
  inode 缓存:       [obj][obj][obj]...        预分配一批inode大小的槽
  ...

  分配 task_struct: 从 task_struct 缓存拿一个空闲槽 → O(1), 无碎片
  释放: 还回缓存(不还给Buddy, 留着复用)

  每个缓存底层用 Buddy 分几页, 切成 N 个对象大小的槽

这不就是第27章 RT-Thread 的内存池吗?——是的!Slab 本质是"一堆按对象大小分类的内存池"。第27章说"RT-Thread 的 SLAB 借自 Linux",第24章说"实时系统用内存池",现在你看到了 Linux 的正统 Slab:为每种对象一个池,分配 O(1)、无外部碎片、无内部碎片(槽大小=对象大小)。

31.7 Slab 缓存怎么工作

Slab 的三个额外优化,让它比朴素内存池更强:

  • 对象复用(不重新初始化):释放的对象保持已初始化状态(构造好的)留在缓存里,下次分配直接用,省去重复初始化。比如一个 task_struct 里有些字段(锁、链表头)每次都要初始化,Slab 让释放的对象保留这些初始化状态,再分配时不用重做——这是 Slab 相比内存池的独特优化。
  • Cache 着色(coloring):让不同 slab 里的对象在 Cache 里错开(用第八章的 Cache Line 知识),减少 Cache 冲突,提升命中率。
  • per-CPU 缓存:每个 CPU 核有自己的一小批空闲对象(第28章 per-CPU 手法),分配时先从本核的拿,不用加锁——多核下极快。

per-CPU 缓存是 Slab 在多核下的关键优化:如果所有核都从一个全局对象池分配,就要抢一个锁,多核竞争激烈。让每个核有自己的"本地缓存",大多数分配从本地拿(无锁),只有本地空了才去全局补充。这个"per-CPU 本地缓存 + 全局池兜底"的两级结构,是多核高性能分配器的通用模式(用户态的 tcmalloc/jemalloc 也这么做)。又一次,你看到"从单核到多核,一切共享资源都要考虑 per-CPU 化"(第28章 current、第30章 runqueue,这里 Slab)。

31.8 SLUB：现代默认

Slab 有几个实现变体,了解即可:

实现特点
SLAB(原始)经典实现,元数据管理较复杂,多核扩展性一般
SLUB(现在默认)简化了元数据(信息存在 struct page 里)、更好的多核扩展性、更少内存开销
SLOB极简,给内存极小的嵌入式设备用

现代 Linux(包括 RK3588)默认用 SLUB。它保留了 Slab 的核心思想(按对象缓存、per-CPU),但实现更简洁、多核性能更好。本章讲的"为每种对象建缓存"的思想,SLAB/SLUB/SLOB 都一样,只是实现细节和优化不同——你抓住思想即可,不用纠结变体。它们统称"slab 分配器",接口是 kmem_cache_create/alloc/free。

31.9 两层配合：Buddy + Slab

把两层分配器的配合画出来,这是内核内存分配的完整图景:

内核内存分配的两层架构:

  ┌─────────────────────────────────────┐
  │  内核代码要内存                        │
  └─────────────────────────────────────┘
       │                    │
   要小对象              要大块/整页
   (task_struct等)      (页缓冲/DMA)
       ▼                    ▼
  ┌──────────┐        ┌──────────┐
  │  Slab    │        │  直接找   │
  │ (SLUB)   │        │  Buddy   │
  │ 按对象缓存 │        │          │
  └────┬─────┘        └────┬─────┘
       │ 缓存不够时         │
       │ 向Buddy要几页       │
       ▼                    ▼
  ┌─────────────────────────────────────┐
  │      Buddy 系统 (管理所有物理页)        │
  │      按2^order, 分裂/合并伙伴           │
  └─────────────────────────────────────┘
       │
       ▼
     物理内存 (struct page 数组)

分工清晰:Buddy 是"批发商"(管所有物理页,按大块出货);Slab 是"零售商"(从 Buddy 批发几页,切成小对象零售)。要小对象找 Slab(它已经切好了,拿了就走);要大块/整页直接找 Buddy。Slab 的对象池空了,就向 Buddy 批发更多页来补。这个"批发+零售"两层结构,既让大块分配抗外部碎片(Buddy),又让小对象分配无内部碎片且极快(Slab)——各取所长。

31.10 kmalloc / vmalloc / 用户 malloc

厘清几个容易混的内存分配接口(面试常问):

接口谁用物理连续?底层用途
kmalloc内核✅ 连续Slab(小)/Buddy(大)内核小块内存,可能给 DMA
vmalloc内核❌ 虚拟连续,物理可不连续Buddy 拿零散页 + 页表映射成连续虚拟地址大块内存(物理凑不出连续的)
alloc_pages内核✅ 连续Buddy 直接要整页
malloc用户态虚拟连续C 库(brk/mmap 系统调用)→ 缺页 → Buddy用户程序动态内存

关键区分:kmalloc 保证物理连续(所以能给 DMA 用),但受 Buddy 大块限制(太大的连续物理内存可能因碎片分不出);vmalloc 只保证虚拟连续,物理上是零散页拼的(用页表映射,第九章),能分很大但不能给要物理连续的 DMA 用,且有页表开销。用户态的 malloc 则是 C 库在用户空间管理堆(通过 brk/mmap 系统调用向内核要虚拟内存),真正的物理页在你访问时才由缺页异常从 Buddy 分配(第十章按需分页)。所以一次用户 malloc + 首次访问,会一路穿过:C 库堆管理 → 缺页异常 → 内核 Buddy 分物理页——第十章和本章在这里连起来了。面试常问"kmalloc 和 vmalloc 的区别""为什么 DMA 用 kmalloc",这就是答案。

31.11 回扣 RTOS 与全书内存线

把全书的"内存管理"这条线收束——你会发现从 FreeRTOS 到 Linux 是同一套思想的放大:

概念FreeRTOS(第24章)RT-Thread(第27章)Linux(本章)
大块分配 + 抗碎片(合并)heap_4(合并相邻块)小内存算法Buddy(伙伴合并)
固定对象池(无碎片,快)(概念)内存池内存池 / SLABSlab/SLUB
多块非连续heap_5memheapvmalloc(拼零散页)
抗碎片核心手法合并相邻空闲块同Buddy 伙伴合并

内存管理的核心矛盾从始至终是同一个:碎片化 vs 效率 vs 确定性。解决手法也是同一套:合并相邻空闲块对抗外部碎片(heap_4→Buddy)、固定大小对象池避免内部碎片并加速(内存池→Slab)、per-CPU 本地缓存应对多核。你从 FreeRTOS 那个几百行的 heap_4 出发,现在完全能理解 Linux 那个复杂得多的 Buddy+Slab——因为核心思想你早就懂了,Linux 只是把它做得更大、更精、加了多核和 NUMA 的考虑。这再次证明"先精读小系统,再看大系统"的价值:骨架和核心思想是可迁移的,复杂性只是同一思想的规模化。

31.12 面试常问问题

Q1:Linux 内核内存分配的两层结构?
底层 Buddy 管理物理页(按 2^order,分裂/合并伙伴,抗外部碎片,出大块);上层 Slab 管理小对象(按对象建缓存,无碎片、O(1)、per-CPU)。Buddy 批发、Slab 零售。

Q2:Buddy 系统如何工作?怎么对抗碎片?
空闲页按 2^order 分组;分配时大块对半分裂,释放时和"伙伴"(相邻同大小块)合并成大块。合并机制让空闲内存尽量保持大块,抗外部碎片。

Q3:为什么需要 Slab?它解决 Buddy 的什么问题?
Buddy 按页(4KB)分配,给小对象用内部碎片严重。Slab 为每种常用对象建缓存,预切成对象大小的槽,分配 O(1)、无内部碎片,还有对象复用/Cache 着色/per-CPU 优化。本质是按大小分类的内存池。

Q4:kmalloc 和 vmalloc 的区别?
kmalloc 物理连续(能给 DMA),底层 Slab/Buddy,受大块碎片限制;vmalloc 只虚拟连续(物理零散页拼),能分很大但不能给物理连续 DMA、有页表开销。

Q5:用户 malloc 到内核的完整路径?
C 库在用户堆管理(brk/mmap 向内核要虚拟地址)→ 首次访问触发缺页异常 → 内核从 Buddy 分物理页填页表(按需分页,第十章)。

Q6:Slab 的 per-CPU 缓存有什么用?
每核有本地空闲对象缓存,分配先从本地拿(无锁),本地空了才去全局补。避免多核抢全局锁,极大提升多核分配性能。

Q7:SLAB/SLUB/SLOB 的区别?
都是 slab 分配器,思想同(按对象缓存)。SLUB 是现代默认(元数据简化、多核扩展好);SLAB 是原始版;SLOB 给极小内存设备。

31.13 常见误区

  1. "内核用 malloc 分配内存"——内核用 kmalloc/vmalloc/alloc_pages,malloc 是用户态的。
  2. "Buddy 能高效分配小对象"——Buddy 按页分配,小对象内部碎片严重,该用 Slab。
  3. "kmalloc 和 vmalloc 一样"——kmalloc 物理连续(可 DMA),vmalloc 只虚拟连续。
  4. "Slab 就是普通内存池"——多了对象复用、Cache 着色、per-CPU 缓存等优化。
  5. "malloc 成功就分配了物理内存"——只给虚拟地址,访问时缺页才分物理页(第十章)。
  6. "Buddy 没有碎片"——抗外部碎片,但有内部碎片(2 的幂向上取整)。

31.14 本章总结

  • 内核自己也要管物理内存(task_struct、栈、缓冲),用两层分配器。
  • 物理内存组织:page(4KB,每页一个 struct page)/ zone(地址限制分区)/ node(NUMA)。
  • Buddy 系统:空闲页按 2^order 分组;分配对半分裂,释放合并伙伴→ 抗外部碎片,出大块。有内部碎片(2 的幂取整)。
  • Slab/SLUB:为每种小对象建缓存(=按大小分类的内存池),O(1)、无碎片,加对象复用/Cache 着色/per-CPU 缓存(多核快)。
  • 两层配合:Buddy 批发页,Slab 零售对象;要小对象找 Slab,要大块找 Buddy。
  • 接口:kmalloc(物理连续,可 DMA)、vmalloc(虚拟连续,物理零散)、alloc_pages(整页)、用户 malloc(经缺页到 Buddy)。
  • 回扣:Buddy=heap_4"合并"的放大,Slab=内存池的放大;内存管理核心矛盾(碎片vs效率vs确定性)和手法从 FreeRTOS 到 Linux 一以贯之。
