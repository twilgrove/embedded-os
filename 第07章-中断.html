CPU 唯一能被"强行打断"的机制，抢占、外设、系统调用的共同根基

本章目标：第一章说"抢占靠时钟中断",但中断本身还没讲透。这一章把它讲成体系——中断是 CPU 唯一能被外部/内部事件强行打断的机制,它同时是"外设响应""抢占式调度""系统调用"三件大事的共同底层。学完你要能讲清一次中断从触发到返回的每一步,理解"中断上下文里为什么不能睡眠""ISR 为什么要短""为什么临界区要关中断"。

7.1 为什么需要中断

CPU 埋头执行指令流,可现实世界的事件是随机、异步发生的:用户按了键、网卡收到包、传感器数据就绪、定时到点。CPU 怎么知道这些事发生了?

两条路:要么 CPU 主动一遍遍去问("轮询"),要么让事件发生时主动来通知 CPU("中断")。下一节告诉你为什么前者行不通。中断的存在,是为了让 CPU 能及时、高效地响应异步事件,而不必时刻盯着。

7.2 轮询的困境

轮询(Polling)就是第一章的前后台系统里那个 if(uart_flag)——CPU 在循环里不停检查"事件来了没"。它有两个致命问题:

  1. 浪费:99% 的检查都是"还没来",纯属空转烧 CPU。就像你在等快递,每 10 秒跑下楼看一次,一天啥也别干了。
  2. 不及时:检查有间隔。如果你循环一圈要 100ms,事件在你刚检查完之后 1ms 发生,它得等 99ms 才被发现。对要求微秒级响应的场景(飞控、电机)是灾难。

更糟的是,轮询让"响应延迟"依赖主循环里其他代码的长短——完全不可控。我们需要一种机制:平时 CPU 该干嘛干嘛,事件一发生立刻(硬件级)打断它去处理。这就是中断。

7.3 生活类比：门铃

你在家工作(CPU 执行主任务)。有客人要来(外部事件)。

  • 轮询:你每隔一会儿就跑到门口看有没有人。累死,还可能刚离开门口客人就到了。
  • 中断:装个门铃。你安心工作,客人按铃(触发中断),你听到铃声立刻放下手头的活(保存现场)、去开门处理(执行中断服务程序),处理完回到刚才停下的地方继续工作(恢复现场)。

门铃的精髓:你不用一直盯着门,事件会主动来打断你;而且打断是"可恢复的"——处理完能精确回到被打断的地方。这正是中断的两个核心特性:异步通知 + 可恢复。

7.4 中断的完整流程

一次中断从发生到结束,走这几步(以 Cortex-M 为例):

主任务正在执行第 N 条指令...
        │
        │ ① 外设(如UART)产生中断信号 → 送到中断控制器(NVIC)
        ▼
   ② NVIC 判断: 该中断是否使能? 优先级够不够高(能否打断当前)?
        │  够 → 通知 CPU
        ▼
   ③ CPU 执行完当前这条指令后, 硬件自动:
        - 保存现场: 把 R0-R3,R12,LR,PC,xPSR 压栈(第四章的自动压栈!)
        - 从"中断向量表"查到该中断的处理函数地址
        - PC ← 处理函数地址   (跳过去)
        - 切换到中断状态
        ▼
   ④ 执行中断服务程序 ISR (你写的 XXX_IRQHandler)
        - 处理事件(读数据、清标志...)
        ▼
   ⑤ ISR 执行返回指令
        - 硬件自动: 从栈弹回 R0-R3,R12,LR,PC,xPSR (恢复现场)
        - PC 回到被打断的第 N+1 条指令
        ▼
主任务从被打断处无缝继续, 仿佛什么都没发生过

注意③里"执行完当前指令才响应"——中断不会把一条指令切到一半,保证了指令的完整性。也注意硬件自动压栈那批寄存器(R0-R3,R12,LR,PC,xPSR),正好是第四章 caller-saved 那批——所以 ISR 你可以用 C 写,编译器会自己保存它要用的 callee-saved(R4-R11),两批合起来现场就全了。这就是为什么 Cortex-M 的中断处理函数可以是普通 C 函数。

7.5 中断向量表

③里 CPU 怎么知道"某个中断该跳到哪个处理函数"?靠一张查找表——中断向量表(Vector Table)。

它就是内存里一张表,每一项存着一个中断处理函数的地址,按中断编号排列。CPU 拿中断号当下标去查,取出地址装进 PC。

中断向量表(在内存固定/可配置位置):

  编号  地址项              含义
  ┌────┬──────────────┐
  │ 0  │ 初始SP值       │  ← 上电时的栈指针
  │ 1  │ Reset_Handler │  ← 复位入口(开机第一个执行的)
  │ 2  │ NMI_Handler   │
  │... │ ...           │
  │ 15 │ SysTick_Handler│ ← 时钟中断(第一章的抢占引擎!)
  │ 16+│ UART_Handler  │  ← 各外设中断
  │    │ TIM_Handler   │
  └────┴──────────────┘

你在 STM32 工程里写的 void USART1_IRQHandler(void){...},链接时它的地址就被填进了向量表对应项。中断号 → 查表 → 得到你的函数地址 → 跳过去。整个"派发"过程是硬件+这张表完成的,极快。RTOS 移植时"接管 SysTick 和 PendSV",本质就是把向量表里这两项换成 RTOS 自己的处理函数。

7.6 中断现场的保存与恢复

中断处理和函数调用一样面临"保现场"问题,但更棘手——因为中断是强行打断的,被打断的代码毫不知情(这点和上下文切换一样,回顾 6.12)。

所以进中断必须保存足够的现场,保证 ISR 执行后能让主任务完全无感地继续。Cortex-M 的巧妙分工:

  • 硬件自动压栈 R0-R3, R12, LR, PC, xPSR(caller-saved 那批);
  • ISR 里若用到 R4-R11(callee-saved),由编译器在 ISR 序言里压栈(和普通函数一样)。

两批合起来,主任务的完整现场就都保住了。这和第四章 4.10 的"影子/自动压栈"、第六章的"序言尾声"完全是一套东西的复用。你看,中断响应本质就是一次"硬件帮你起头的、被迫的函数调用"。

7.7 中断优先级与嵌套

多个中断同时来怎么办?更紧急的中断能否打断正在处理的不太紧急的中断?这就是中断优先级和中断嵌套。

  • 每个中断有个优先级数值。高优先级中断可以打断低优先级中断的 ISR——就像你在开门(处理客人),结果厨房着火了(更紧急),你会放下客人先救火。
  • 处理完高优先级的,再回来接着处理被打断的低优先级 ISR。这就是中断嵌套,像函数调用一样一层层压栈、返回。
  • 同优先级的中断不能互相打断,排队处理。

ARM 的 NVIC(嵌套向量中断控制器)专门管这套优先级仲裁和嵌套。注意 Cortex-M 的优先级数值越小优先级越高(0 最高),这是个经典的坑,配置反了会导致该抢的没抢到。中断嵌套太深会消耗中断栈(MSP),估算栈大小时要考虑最坏嵌套层数(呼应第五章)。

7.8 中断上下文 vs 线程上下文

一个极其重要、面试必考的区分。代码运行在两种"上下文"里:

线程上下文(Thread Context)中断上下文(Interrupt Context)
什么时候正常执行任务/进程代码正在处理中断(ISR 里)
能否睡眠/阻塞能(可以等锁、等信号量、sleep)绝对不能!
能否被调度能被切换不能(它不是个任务)
有没有 task_struct有,它属于某个任务没有,不属于任何任务

为什么中断上下文不能睡眠?

这是理解的关键。"睡眠/阻塞"意味着"让出 CPU,让调度器切到别的任务,等条件满足再回来"。但中断上下文不属于任何任务——它只是"借用"了当前被打断任务的栈在跑。如果 ISR 里睡眠:

  • 调度器该切到谁?ISR 不是任务,没法被"挂起等待再恢复"。
  • 它借用的那个被打断的任务,现在卡在中断里回不去了,整个系统状态错乱。

所以铁律:ISR 里不能调用任何可能阻塞/睡眠的函数(如普通的锁、malloc 大内存、vTaskDelay、printf 有时也危险)。在 Linux 内核里同理,中断处理不能调用可能睡眠的函数。这是新手写驱动最常犯的错。

7.9 上半部 / 下半部：ISR 为什么要短

中断响应时,通常会关闭同级或全部中断(防止重入)。这意味着ISR 执行期间,别的中断被推迟响应。如果 ISR 干很久(比如里面做大量计算),别的紧急中断就被拖累,系统实时性崩塌。

所以铁律:ISR 必须尽可能短、尽可能快。但有时确实有很多活要干(比如收到网络包要做复杂协议处理),怎么办?拆成两半:

上半部(Top Half)下半部(Bottom Half)
做什么紧急的、最少的:清中断标志、把数据搬到缓冲区、记个"待办"耗时的:实际的复杂处理
在哪跑ISR 里(关中断,要快)稍后在线程上下文里(开中断,可从容)

思路:ISR(上半部)只做"救急"的最少动作,然后登记一个"待办任务"就赶紧返回,把耗时的活留给稍后在正常上下文里执行的"下半部"。

  • Linux 里下半部机制:softirq、tasklet、workqueue(第五卷详讲)。
  • RTOS 里常见做法:ISR 里只 给信号量/发消息到队列,唤醒一个专门的处理任务去干重活。

这是嵌入式/驱动开发的核心设计模式。你写 STM32 中断时"在 ISR 里只置个标志,主循环里处理"就是这个思想的雏形。面试爱问"中断里能不能做耗时操作,不能的话怎么办"——答上半部/下半部。

7.10 中断与并发：为什么临界区要关中断

中断带来一个隐蔽的并发问题。设想:主任务正在修改一个链表,改到一半(链表处于不一致的中间状态),中断突然来了,ISR 里也去访问这个链表——它看到的是个坏掉的半成品,程序崩溃或数据损坏。

这就是中断引入的"并发":ISR 和主任务共享数据时,ISR 可能在任何指令边界插进来。解决办法:在主任务操作共享数据的那段"临界区"里暂时关闭中断,让这几步不被打断,做完再开中断。

关中断保护临界区:
   disable_irq();      // 关中断:接下来不许打断我
   list_insert(...);   // 安全地修改共享链表(几步原子完成)
   enable_irq();       // 开中断:恢复

要点: 临界区必须极短! 关中断期间所有中断被推迟, 关久了伤实时性。

这是并发与锁这个大主题(第五卷 SpinLock/Mutex/原子操作)的第一次登场。关中断是最原始、最粗暴的"锁"——它保护的是"任务 vs 中断"之间的竞争。但它代价大(推迟所有中断)、且在多核上不够(关一个核的中断,别的核照样能插进来),所以后面会发展出自旋锁、原子操作等更精细的手段。你现在只需理解:中断带来了并发,并发就需要保护,关中断是最初的保护手段。

7.11 异常、中断、系统调用的统一

一个拔高的认识:中断只是一大类机制里的一种。ARM 把所有"打断正常执行流、跳到特定处理程序"的事件统称异常(Exception),分几类:

类型触发源例子
中断 IRQ外部硬件,异步UART、定时器、GPIO
故障 FaultCPU 执行出错,同步访问非法地址、除零、非法指令
系统调用软件主动触发SVC/syscall 指令(第三章)

它们的处理机制惊人地一致:都是"保存现场 → 查向量表 → 跳到处理程序 → 处理 → 恢复现场返回"。区别只在触发源和是否切换特权级。

这个统一非常深刻:操作系统进入内核的所有入口——外设中断、缺页故障、系统调用——底层都是同一套"异常"机制。你调 read()(SVC 异常)、程序访问野指针(Fault 异常)、时钟到点(IRQ 异常),CPU 用的是同一个"保存现场跳处理程序"的动作。第二卷(ARM 异常/EL)、第五卷(缺页处理、系统调用)会把这套机制反复展开。理解了中断,你就理解了操作系统所有"陷入内核"的时刻。

7.12 回扣：中断是抢占的引擎

终于回到第一章的伏笔。第一章说"抢占靠时钟中断",现在你能完全理解这个链条了:

SysTick 定时器每 1ms 产生一次中断(IRQ)
        │
        ▼
CPU 被强行打断 → 保存当前任务A的现场 → 跳进 SysTick_Handler
        │
        ▼
内核在这里做一件事: 调用调度器, 问"该换人吗?"
        │
   ┌────┴────┐
   │         │
 不换       该换(有更高优先级任务就绪 / A时间片用完)
   │         │
 直接返回   触发 PendSV → 在PendSV里做上下文切换(存A全部现场,换到B)
   │         │
   ▼         ▼
 回到A     切到B运行

没有中断,CPU 就永远不会"主动"想起调度器,一个死循环任务就能霸占 CPU 永不撒手。正是时钟中断这个"周期性的强制打断",给了内核夺回控制权、重新调度的机会。所以说:

中断是抢占式调度的引擎。第一章的三支柱——上下文切换(第四五六章)、调度器、时钟中断(本章)——到这里全部拿到了硬件解释。第一卷的操作系统核心机制,就此拼图完整。为什么切换要放在 PendSV(最低优先级异常)而不是直接在 SysTick 里做?因为要等所有中断处理完、真正要回线程时再切,避免切换过程被别的中断搅乱——这个精妙设计留到第三卷 FreeRTOS 移植时逐行剖析。

7.13 面试常问问题

Q1:中断和轮询的区别?各自优缺点?
轮询主动反复查,浪费 CPU 且响应不及时;中断由事件主动通知,及时高效但有响应/切换开销。低频/简单场景可轮询,实时/异步场景用中断。

Q2:一次中断从触发到返回的完整流程?
外设→中断控制器仲裁优先级→CPU 执行完当前指令→硬件自动保存现场+查向量表跳 ISR→ISR 处理→返回时恢复现场→回被打断处。

Q3:中断上下文能睡眠/阻塞吗?为什么?
不能。中断不属于任何任务、无法被调度挂起恢复;睡眠会让借用的任务卡死、系统错乱。ISR 不能调用可能阻塞的函数。

Q4:ISR 为什么要短?活多怎么办?
ISR 期间常关中断,拖长会推迟其他中断、损害实时性。拆上半部(救急、登记)/下半部(耗时处理,放线程上下文):softirq/tasklet/workqueue 或"ISR 发信号量唤醒任务"。

Q5:为什么临界区要关中断?有什么代价?多核够吗?
防止 ISR 在共享数据修改到一半时插入导致不一致。代价是推迟所有中断,须极短。多核上关本核中断不够,别的核仍可访问,需自旋锁。

Q6:中断、异常、系统调用有什么联系?
都是"异常"机制:保存现场→查向量表→跳处理→恢复返回。区别在触发源(外部硬件/CPU出错/软件主动)。是操作系统进入内核的统一入口。

Q7:中断和抢占式调度什么关系?
时钟中断周期性打断 CPU,给内核调用调度器的机会,是抢占的引擎;没有中断,死循环任务无法被抢占。

7.14 常见误区

  1. "中断会把指令切到一半"——不会,CPU 执行完当前指令才响应中断。
  2. "ISR 里可以随便调用函数"——不能调用可能阻塞/睡眠的(锁、delay、大 malloc 等)。
  3. "中断优先级数值越大越优先"——Cortex-M 相反,数值越小越优先(0 最高)。
  4. "ISR 越全能越好,把活都在里面干完"——反了,ISR 要最短,耗时活丢下半部。
  5. "关中断就能解决所有并发"——只解决"任务 vs 中断"且单核有效;多核需自旋锁/原子操作。
  6. "中断和系统调用没关系"——底层都是异常机制,一套流程。
  7. "共享变量不用保护"——中断可在任意指令边界插入,共享数据需临界区/原子操作,变量还常需 volatile(第四章)。

7.15 本章总结

  • 中断解决"CPU 如何及时响应异步事件";轮询浪费且不及时,中断由事件主动通知、可恢复。
  • 流程:外设→控制器仲裁→执行完当前指令→硬件自动保存现场+查向量表→跳 ISR→处理→恢复现场返回。
  • Cortex-M 硬件自动压 caller-saved 那批,ISR 用到的 callee-saved 由编译器压;故 ISR 可用 C 写。
  • 优先级/嵌套由 NVIC 仲裁,高优先级可打断低优先级 ISR(注意数值越小越高)。
  • 中断上下文不能睡眠——它不属于任何任务,无法被调度挂起恢复。
  • ISR 要短:拆上半部(救急+登记)/下半部(耗时处理放线程上下文)。
  • 中断引入并发:共享数据需临界区(关中断)保护;这是锁的雏形,多核不够用。
  • 中断/故障/系统调用统一为异常机制,是操作系统进内核的共同入口。
  • 回扣:时钟中断是抢占式调度的引擎;第一章三支柱的硬件解释全部集齐。

7.16 思维导图

中断
├─ 一、为何需要:及时响应异步事件
├─ 二、轮询困境:浪费CPU + 响应不及时
├─ 三、类比:门铃(异步通知+可恢复)
├─ 四、完整流程
│  └─ 执行完当前指令→硬件保存现场+查向量表→跳ISR→恢复返回
├─ 五、中断向量表(中断号→处理函数地址)
├─ 六、现场保存:硬件压caller-saved+编译器压callee-saved
├─ 七、优先级与嵌套(NVIC;数值越小越高)
├─ 八、★中断上下文不能睡眠
│  └─ 因:不属于任何任务,无法调度挂起恢复
├─ 九、ISR要短:上半部(救急登记)/下半部(耗时,线程上下文)
│  └─ softirq/tasklet/workqueue;信号量唤醒任务
├─ 十、并发:临界区关中断保护(锁雏形,多核不够)
├─ 十一、异常统一:中断/故障/系统调用同一套机制
│  └─ 操作系统进内核的共同入口
└─ 十二、★回扣:时钟中断=抢占引擎(第一章三支柱集齐)
