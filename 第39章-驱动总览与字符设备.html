本章目标:前五卷你理解了 CPU、ARM、RTOS、Linux 内核的"内功"。第六卷开始学"外功"——驱动开发,让操作系统能操控真实硬件(GPIO、串口、传感器、CAN)。这是嵌入式工程师的看家本领,也是车企岗位的核心考察点。本章从"驱动是什么"讲起,写出你的第一个字符设备驱动,并看清它如何接到第32章 VFS 的 read/write 上。

39.1 驱动是什么、为什么要它

你的应用程序调 read(fd, buf, n) 想从一个温度传感器读数据。但传感器是个具体硬件——它挂在 I2C 总线上、有自己的寄存器、有特定的读取时序。应用程序不可能懂每一款硬件的细节(那样每换个传感器就要重写应用)。需要有人把"通用的 read 请求"翻译成"这款具体硬件的操作时序"。这个"翻译者"就是设备驱动(device driver)。

驱动是内核里专门操控某类硬件的代码,它一头对接内核的通用接口(VFS 的 read/write 等,第32章),一头对接硬件的具体寄存器/时序。驱动的价值:向上给应用提供统一接口(应用不用管硬件细节),向下封装硬件差异。这又是全书那句话——加一层抽象。没有驱动,应用就得直接操作硬件寄存器,既不安全(第十二章特权级)也不通用。

39.2 生活例子：翻译官

你(应用)想和一位只会说方言的老工匠(硬件)沟通。你只会说普通话(标准的 read/write)。中间需要一个翻译官(驱动):你说"我要读数据"(普通话),翻译官把它翻成老工匠的方言和规矩("请按第 3 号阀门,等 2 秒,读刻度")。老工匠给出结果,翻译官再翻回普通话给你。

关键:你只需学会和翻译官说普通话(统一接口),不用懂每个工匠的方言(硬件细节);换个工匠,只要换个对应的翻译官(换驱动),你说话的方式不变。这就是驱动让"应用与硬件解耦"的意义——应用代码稳定,硬件可替换。换一款传感器,改的是驱动,上层算法基本不动。

39.3 三大类设备

Linux 把设备分三大类,这决定了驱动的写法:

类型特点访问方式例子
字符设备(char)像"字节流",按顺序一个个读写,通常不能随机寻址read/write 顺序访问串口、键盘、传感器、LED
块设备(block)按"块"(如 512B)访问,可随机寻址,有缓冲经文件系统/块层硬盘、SD 卡、eMMC、Flash
网络设备(net)收发数据包,不走 /dev,用 socketsocket(第34章)网卡、WiFi

本章聚焦字符设备——它是最基础、嵌入式最常见的类型。你接的绝大多数外设(GPIO、串口、I2C 传感器、SPI 屏、CAN)在驱动层面都是字符设备:数据像流一样一个个进出,用 read/write 操作。字符设备驱动是驱动开发的入门和主力,学透它,后面 GPIO/UART/I2C/SPI 驱动都是它的具体化。块设备(存储)和网络设备有各自更复杂的子系统(块层、网络协议栈第34章),但字符设备的模型是理解一切驱动的基础。

39.4 设备号：主设备号与次设备号

内核怎么把一个设备文件和对应的驱动关联起来?靠设备号。每个设备有一对号码:

设备号 = 主设备号(major) + 次设备号(minor)

  主设备号 major: 标识"哪个驱动"       (类别)
  次设备号 minor: 标识"这个驱动管的第几个设备" (个体)

例: 一个串口驱动 major=4, 管理多个串口:
     /dev/ttyS0  → major=4, minor=64
     /dev/ttyS1  → major=4, minor=65
     (同一个驱动 major相同, 用minor区分是哪一个)

主设备号 = 找到哪个驱动;次设备号 = 驱动内部区分是它管的哪个具体设备。就像"某快递公司(major)的第 N 个网点(minor)"。内核维护一张"主设备号→驱动"的表,应用访问一个设备文件时,内核用它的主设备号找到驱动,把请求交给它;驱动再用次设备号知道操作的是哪个具体设备。ls -l /dev 能看到每个设备文件的主次号。

39.5 /dev 下的设备文件

回到第32章"一切皆文件":设备在用户空间表现为 /dev/ 下的设备文件(设备节点)。

$ ls -l /dev/ttyS0 /dev/null /dev/sda
 crw-rw----  ... 4,  64  /dev/ttyS0   ← c=字符设备, major4 minor64
 crw-rw-rw-  ... 1,   3  /dev/null    ← c=字符设备
 brw-rw----  ... 8,   0  /dev/sda     ← b=块设备

  第一个字母: c=字符设备, b=块设备
  两个数字:   主设备号, 次设备号

设备文件是"硬件在文件系统里的化身"。应用 open("/dev/ttyS0") 就像打开普通文件,但内核发现它是设备文件(带主次号),就把后续 read/write 转给对应驱动,而不是读写磁盘数据。这就是第32章说的"设备文件的 f_op 指向驱动的实现"——open 一个 /dev 文件,VFS 会把它的 file→f_op 挂成驱动提供的操作函数集。于是 cat /dev/ttyS0 就能读串口。

39.6 file_operations：驱动的核心

写字符驱动,核心就是填一张表——struct file_operations。它是驱动和 VFS 之间的契约(第32章的 f_op 就是它):

struct file_operations {
    struct module *owner;
    ssize_t (*read)  (struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);
    int     (*open)  (struct inode *, struct file *);
    int     (*release)(struct inode *, struct file *);  // close时调用
    long    (*unlocked_ioctl)(struct file *, unsigned int, unsigned long);
    // ... mmap, poll 等(后续章节)
};

你写驱动,本质就是实现这些函数,填进 file_operations,然后注册给内核。应用调 read(),内核经 VFS 最终调到你填的 .read 函数;应用 ioctl(),调到你的 .unlocked_ioctl(第48章)。这张函数指针表就是第32章 VFS 多态、第26章 RT-Thread rt_device_ops 的同款设计——驱动开发的一切,围绕填好这张表展开。

39.7 一个最小字符驱动

看一个能跑的最小字符驱动骨架(读它返回固定字符串):

#include <linux/module.h>
#include <linux/fs.h>
#include <linux/uaccess.h>

static int major;
static char msg[] = "hello from driver\n";

static ssize_t my_read(struct file *f, char __user *buf,
                       size_t len, loff_t *off) {
    size_t n = sizeof(msg);
    if (*off >= n) return 0;          // 读完了返回0(EOF)
    if (len > n - *off) len = n - *off;
    if (copy_to_user(buf, msg + *off, len))  // ★内核→用户拷贝
        return -EFAULT;
    *off += len;                      // 更新读位置(第32章f_pos)
    return len;
}
static int my_open(struct inode *i, struct file *f){ return 0; }

static struct file_operations fops = {   // ★填表
    .owner = THIS_MODULE,
    .open  = my_open,
    .read  = my_read,
};

static int __init my_init(void){
    major = register_chrdev(0, "mychar", &fops);  // ★注册, 0=自动分配major
    printk("mychar loaded, major=%d\n", major);
    return 0;
}
static void __exit my_exit(void){ unregister_chrdev(major, "mychar"); }

module_init(my_init);
module_exit(my_exit);
MODULE_LICENSE("GPL");

三个要点:①实现 my_read 等函数;②填进 fops;③在 my_init 里 register_chrdev 注册(把驱动登记到内核,拿到主设备号)。加载后 mknod /dev/mychar c <major> 0 建设备节点,cat /dev/mychar 就会触发 my_read 打印字符串。这就是驱动最小闭环。(module_init/exit、__init 等内核模块机制,下一章第40章详解。)

39.8 open/read 如何到达驱动

把应用的 read 一路追到驱动,这是理解驱动运行的关键(串起第32章):

cat /dev/mychar 发生了什么:
  ① cat 调 open("/dev/mychar") → 系统调用进内核
  ② VFS发现是字符设备文件, 取出它的【主设备号】
  ③ 用major查"字符设备表" → 找到你注册的 file_operations
  ④ 把 file->f_op 设成你的 fops → 调你的 .open
  ⑤ cat 调 read(fd,...) → VFS → file->f_op->read → 【你的 my_read】★
  ⑥ my_read 用 copy_to_user 把数据送到用户buf
  ⑦ cat 收到数据打印; 再read返回0(EOF) → cat结束

整条链把第32章和本章焊在一起:应用的 read → VFS 的 file->f_op->read 多态分发(第32章)→ 落到你驱动填的 my_read。你写驱动,就是在这条链的最末端,提供那个"真正操作硬件"的函数。VFS 负责把请求送到你门口,你负责在函数里操作具体硬件(读寄存器、发时序)。理解这条链,你就明白了"应用一个简单 read,如何最终变成对硬件的操作"——这是嵌入式全栈的核心脉络,从应用一直贯通到硬件。

39.9 用户空间与内核空间的数据搬运

39.7 出现了 copy_to_user,这是驱动的一个安全要点,必须讲清:

为什么不能直接 memcpy(buf, msg, len)?

  buf 是【用户空间】指针, 驱动运行在【内核空间】(第十二章特权级、第十章地址空间)
  直接解引用用户指针危险:
   ① 用户指针可能非法/恶意(指向不该访问的地址)→ 内核崩溃/被攻击
   ② 用户页可能没在内存(被换出, 第十章)→ 要触发缺页处理

  必须用专门函数:
   copy_to_user(用户buf, 内核src, n)   内核→用户
   copy_from_user(内核dst, 用户buf, n) 用户→内核
   → 它们会【检查用户指针合法性】+ 【安全处理缺页】

铁律:驱动里绝不能直接解引用用户空间指针,必须用 copy_to_user/copy_from_user。原因是内核和用户是两个地址空间、两个特权级(第十章、第十二章):用户传来的指针不可信(可能非法或恶意),直接用会让内核崩溃或被攻击(用户借内核之手读写任意内存)。这两个函数会校验指针合法性、安全处理缺页。这是驱动安全的第一课,也是面试常考点——"驱动里怎么和用户空间交换数据"。

39.10 现代写法：cdev 与自动创建设备节点

39.7 用的 register_chrdev 是老式简易接口。现代内核驱动用更规范的方式:

现代字符驱动的标准套路:
  ① alloc_chrdev_region(&dev, 0, 1, "name");  // 申请设备号(major+minor)
  ② cdev_init(&cdev, &fops);                   // 初始化cdev结构
     cdev_add(&cdev, dev, 1);                   // 注册cdev到内核
  ③ class_create() + device_create()           // ★让udev自动在/dev建节点
     → 不用手动 mknod, 加载驱动后 /dev/xxx 自动出现

  卸载时反向: device_destroy → class_destroy → cdev_del → unregister_chrdev_region

两点进步:①用 cdev 结构体规范地表示字符设备(更灵活);②用 class_create+device_create 让用户空间的 udev 守护进程自动创建 /dev 节点(不用手动 mknod)。这就是为什么你插上 U 盘、加载一个驱动后,/dev 下会自动冒出对应设备文件——是内核通知 udev 自动建的。实际工程中用这套标准写法,后面 GPIO/I2C 等驱动都基于它。

39.11 回扣 VFS 与 RT-Thread 设备框架

字符驱动的模型,你其实见过两遍了:

概念Linux 字符驱动(本章)RT-Thread 设备(第26章)VFS(第32章)
操作表file_operationsrt_device_opsf_op
注册设备register_chrdev/cdev_addrt_device_register-
统一接口read/write/open/ioctlrt_device_read/write/controlread/write
设备标识主/次设备号设备名inode

Linux 字符驱动 = 第32章 VFS 的"设备侧实现" = 第26章 RT-Thread 设备框架的放大版。你在第26章写 RT-Thread 驱动时填 rt_device_ops,现在写 Linux 驱动填 file_operations——同一个"填函数指针表 + 注册"的模式。区别在于 Linux 多了设备号、用户/内核空间隔离(copy_to_user)、udev 自动建节点等工程化细节,因为 Linux 面对更复杂的多用户、多设备、安全环境。但核心思想你早就掌握了:驱动就是实现一组标准操作函数,注册给内核,让上层通过统一接口调到你。带着这个认识,后面 GPIO、UART、I2C、SPI 各种驱动,你会发现它们都是这个骨架 + 具体硬件操作的填充。

39.12 面试常问问题

Q1:设备驱动的作用是什么?
内核里操控某类硬件的代码,向上给应用提供统一接口(read/write 等),向下封装硬件寄存器/时序细节,实现应用与硬件解耦(换硬件只换驱动)。

Q2:Linux 设备分哪三类?区别?
字符设备(字节流,顺序 read/write,如串口/传感器)、块设备(按块随机访问,有缓冲,如磁盘)、网络设备(收发包,走 socket 不走 /dev)。

Q3:主设备号和次设备号的作用?
主设备号标识哪个驱动(内核用它找驱动),次设备号标识该驱动管理的哪个具体设备。

Q4:写字符驱动的核心是什么?
实现 file_operations 里的 read/write/open/ioctl 等函数,填表后注册给内核(register_chrdev 或 cdev_add)。应用 read 经 VFS 多态分发到你的函数。

Q5:驱动里怎么和用户空间交换数据?为什么不能直接 memcpy?
用 copy_to_user/copy_from_user。因为用户指针在不同地址空间、不可信(可能非法/恶意),直接解引用会让内核崩溃或被攻击;这两个函数会校验合法性、安全处理缺页。

Q6:应用 read 一个 /dev 文件是怎么到达驱动的?
open 时 VFS 用设备文件的主设备号查字符设备表找到驱动的 file_operations,挂到 file->f_op;read 时经 VFS file->f_op->read 分发到驱动的 read 函数。

Q7:/dev 下的设备节点怎么来的?
现代用 class_create+device_create 通知用户空间 udev 自动创建;老式手动 mknod 指定主次号。

39.13 常见误区

  1. "应用能直接操作硬件寄存器"——用户态无权限(第十二章),必须经驱动(内核态)。
  2. "驱动里能直接用用户传的指针"——不能,必须 copy_to/from_user。
  3. "设备文件存了硬件数据"——设备文件只是入口(带主次号),数据 read 时由驱动实时从硬件取。
  4. "主设备号唯一标识一个设备"——主设备号标识驱动/类别,加次设备号才定位具体设备。
  5. "传感器/GPIO 是块设备"——它们是字符设备(字节流);块设备是磁盘那类。
  6. "写驱动很玄"——核心就是"填 file_operations + 注册",和 RT-Thread 填 rt_device_ops 同理。

39.14 本章总结

  • 驱动:内核里操控硬件的代码,向上提供统一接口(read/write)、向下封装硬件细节 → 应用与硬件解耦(加一层抽象)。
  • 三类设备:字符(字节流,顺序,嵌入式主力)、块(按块随机,存储)、网络(走 socket)。
  • 设备号:主设备号=找哪个驱动,次设备号=驱动内定位哪个设备。/dev 下设备文件带主次号(c/b)。
  • 核心是填 file_operations(read/write/open/ioctl…),注册给内核(register_chrdev / cdev_add)——同 VFS f_op、RT-Thread rt_device_ops。
  • 调用链:应用 read → VFS file->f_op->read 多态 → 你的驱动 read 函数 → 操作硬件。
  • 铁律:用户/内核不同地址空间,驱动交换数据必须用 copy_to_user/copy_from_user(校验+安全缺页),绝不直接解引用用户指针。
  • 现代写法:cdev + class_create/device_create(udev 自动建 /dev 节点)。
