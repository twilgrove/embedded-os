本章目标：第一卷讲的都是"通用 CPU"的原理。第二卷要落到你真正在用的芯片——ARM。你做过 STM32(Cortex-M)也做过 RK3588(Cortex-A),但可能没想过:它们都叫 ARM,为什么一个跑 FreeRTOS、一个跑 Linux,差别这么大?这一章把 ARM 的产品线、架构分层、M 与 A 的本质区别讲清,为后面异常等级、GIC、Cache 一致性打地基。

11.1 为什么第二卷要专门讲 ARM

第一卷的寄存器、中断、MMU 都是"泛泛地讲原理"。但你写 RTOS 移植、写 Linux 驱动、调飞控,面对的是具体的 ARM 芯片,它有具体的异常等级、具体的中断控制器、具体的 Cache 结构。原理不落到具体架构,就永远是纸上谈兵。

而且 、地平线这些公司的产品全是 ARM(从飞控的 Cortex-M 到 AI 芯片的 Cortex-A),ARM 架构细节是这类岗位的硬通货。第二卷就是把第一卷的通用原理,在 ARM 上精确化、具体化。

11.2 ARM 是什么：一家不造芯片的芯片公司

先破除一个常见误解:ARM 公司自己不生产任何芯片。它是一家"卖设计图纸"的公司。

类比:ARM 像一个顶级的建筑设计事务所。它不盖楼,而是设计出优秀的"图纸"(处理器核的设计 IP),然后把图纸授权给各家厂商(高通、瑞芯微、意法半导体 STM、苹果……),这些厂商拿图纸去造自己的芯片。

ARM 公司 ──(授权设计IP/架构)──▶ 各芯片厂商
                                    │
        ┌───────────────┬──────────┼──────────┬─────────┐
      意法半导体        瑞芯微      高通        苹果      ...
      STM32(Cortex-M)  RK3588      骁龙        M系列
                       (Cortex-A)  (Cortex-A) (自研核)

这就是为什么同样是 ARM,STM32、RK3588、苹果 M 芯片差别巨大——它们是不同厂商基于 ARM 授权造的不同产品。有的厂商直接用 ARM 设计好的核(如 STM32 用 Cortex-M),有的只买"架构授权"自己设计核(如苹果的 M 系列自研核,但兼容 ARM 指令集)。

11.3 架构 vs 实现：ARMv8 与 Cortex-A76

一个必须分清的概念,面试爱考:架构(Architecture)和实现(微架构 Microarchitecture)是两回事。

架构(合同/规范)实现(具体的核)
是什么指令集+寄存器+异常模型等"规范"按规范造出的具体处理器核
例子ARMv7-M、ARMv8-A、ARMv9-ACortex-M4、Cortex-A76、Cortex-A55
类比"C 语言标准""GCC 编译器"这个具体实现

关系:架构是"该支持哪些指令、有哪些特权级"的规范(第二章说的 ISA 合同的扩展);实现是"用多深的流水线、多大的 Cache、多快的主频去满足这个规范"的具体设计。同一个架构(如 ARMv8-A)可以有很多不同实现(Cortex-A76 高性能、Cortex-A55 省电),它们跑同样的指令,但性能功耗天差地别。

RK3588 用的是 ARMv8.2-A 架构,具体实现是 4 个 Cortex-A76(大核)+ 4 个 Cortex-A55(小核)。它们架构相同(指令兼容,能跑同一个 Linux),但微架构不同(A76 流水线深、乱序执行、性能强;A55 简单省电)。这正是下面 big.LITTLE 的基础。

11.4 三条产品线：A / R / M

Cortex 分三条线,名字最后一个字母就是定位,记忆口诀:A 应用、R 实时、M 微控。

系列字母含义定位典型场景有 MMU 吗
Cortex-AApplication高性能应用处理器手机、RK3588、跑 Linux/安卓有
Cortex-RReal-time高性能实时处理器硬盘控制器、基带、汽车安全无(有 MPU)
Cortex-MMicrocontroller低功耗微控制器STM32、飞控、IoT,跑 RTOS/裸机无(有 MPU)

本书重点是 A 和 M(你最常接触的两条)。它们的分野,恰好对应第一卷的核心分界线——有没有 MMU、跑不跑得动 Linux。R 系列是介于两者之间的"高性能但要确定性"的特殊定位,汽车和存储用得多,本书少涉及。

注意:Cortex-M/R 没有 MMU,但有个简化版的 MPU(内存保护单元)——它不做地址翻译(不虚拟化地址),只做"内存保护"(划几个区域,设权限,越界报错)。这是"要一点保护、但不要虚拟内存那套开销"的折中,第九章讲 MMU 时的"轻量版"。

11.5 Cortex-M：为实时和确定性而生

你做 STM32 用的就是 Cortex-M(M0/M3/M4/M7/M33...)。它的每一个设计决策都服务于"小、省电、实时、确定":

  • 中断响应快且确定:第七章讲的硬件自动压栈、NVIC、确定的中断延迟(固定周期数)——这对飞控这种要求"必须在 N 微秒内响应"的场景是命根子。
  • 无 MMU:没有虚拟内存的不确定延迟(第十章),地址访问是确定的。
  • 低功耗:简单的流水线(2~3 级)、可深度睡眠,适合电池设备。
  • Thumb-2 指令集:16/32 位混合,代码密度高,省 Flash。
  • 位带、MPU、FPU等面向控制的特性。

Cortex-M 的世界里,通常跑裸机或 RTOS(FreeRTOS/RT-Thread),没有"进程"概念,所有任务共享一个地址空间——这正是第一卷说的 RTOS 分界。

11.6 Cortex-A：为性能和 Linux 而生

RK3588、手机 SoC 用的是 Cortex-A。它的设计目标是"高性能、能跑操作系统、能处理复杂任务":

  • 有 MMU:支持虚拟内存,能跑 Linux/安卓,进程隔离(第九、十章)。
  • 多级 Cache + Cache 一致性:L1/L2/L3,多核共享一致性(本卷后面讲)。
  • 深流水线 + 乱序执行 + 分支预测:榨性能(第二章流水线的高级版)。
  • 多核 SMP:多个核对称多处理,真并行(第一章的并行)。
  • 异常等级 EL0~EL3:完整的特权分级 + 虚拟化 + 安全世界(下一章)。

Cortex-A 的世界里跑的是Linux/安卓,有完整的进程、虚拟内存、用户/内核态。你在 RK3588 上跑 ROS2 节点,就是在这套体系上。

11.7 M 和 A 的关键差异对照

把两条线的差异集中对照,这张表几乎就是第一卷所有分界线的汇总:

维度Cortex-M(STM32)Cortex-A(RK3588)
MMU / 虚拟内存无(仅 MPU 保护)有,支持虚拟内存
跑什么裸机 / FreeRTOS / RT-ThreadLinux / 安卓
进程概念无,任务共享地址空间有,进程独立地址空间
特权分级2 态(特权/非特权,Thread/Handler)4 级(EL0/EL1/EL2/EL3)
位宽32 位为主64 位(AArch64)
中断控制器NVIC(核内,简单快)GIC(片上,复杂,支持多核)
Cache可有可无,通常小多级 + 一致性协议
流水线浅(2~3 级),顺序执行深,乱序,分支预测
核心追求实时、确定、省电性能、吞吐、功能

这张表你要能默背——它是"为什么 STM32 和 RK3588 差这么多"的完整答案,也是第一卷十章的浓缩。本质就一句:M 用简单换确定性和低功耗,A 用复杂换性能和功能。而复杂性的核心,就是 MMU/虚拟内存/多核一致性/多级特权这几样——它们是本卷后续章节的主题。

11.8 运行模式与特权：M 的两态 vs A 的四级

第一章讲的"用户态/内核态",在 M 和 A 上是不同的具体形态。

Cortex-M:两种模式 + 两种特权

Cortex-M 的运行状态:
  ┌─ 模式(Mode) ─────────────────┐
  │  Thread 模式:  跑普通代码(任务)  │
  │  Handler 模式: 跑异常/中断处理    │
  └──────────────────────────────┘
  ┌─ 特权(Privilege) ────────────┐
  │  特权级(Privileged)          │  ← 内核/RTOS 通常在这
  │  非特权级(Unprivileged)      │  ← 可选, 给不信任的任务
  └──────────────────────────────┘

RTOS 里最常见:所有代码都跑在特权级(简单)。进中断自动切到 Handler 模式。这套比 A 简单得多,够用且确定。

Cortex-A:四个异常等级 EL0~EL3

A 系列有完整的四级特权(下一章专门讲):EL0 用户态、EL1 内核态、EL2 虚拟化、EL3 安全监控。第一章说的 EL0/EL1 就是这里的前两级。M 的"两态"和 A 的"四级",都是第一章"特权分级"思想的不同规模实现——任务越复杂、越不可信、越需要虚拟化和安全隔离,分级就越多。

11.9 指令集：ARM / Thumb / A64

ARM 芯片支持的指令集也随产品线不同,面试常问"ARM 和 Thumb 区别":

指令集位宽特点用在
ARM(A32)32 位定长功能全,代码大老的 32 位 A 系列
Thumb / Thumb-2(T32)16/32 位混合代码密度高,省 FlashCortex-M 主力
A64(AArch64)32 位定长(但 64 位处理)64 位,寄存器多(X0-X30)Cortex-A 的 64 位模式

关键理解:Thumb 的存在是为了"代码密度"——嵌入式 Flash 很贵很小,16 位指令能装下更多代码。Cortex-M 主要用 Thumb-2(16/32 混合,兼顾密度和功能)。而 Cortex-A 的 64 位模式用 A64,指令定长 4 字节、寄存器多,面向性能。这呼应第二章 RISC 定长指令的讨论。

11.10 流水线与乱序：M 与 A 的性能鸿沟

第二章讲了流水线。M 和 A 在这上面差距巨大,决定了它们的性能档次:

  • Cortex-M:浅流水线(2~3 级)、顺序执行(in-order)。简单、可预测(利于实时),但性能有限。指令严格按顺序一条条来。
  • Cortex-A:深流水线(十几级)、乱序执行(out-of-order)、超标量(一周期发射多条)、激进分支预测。为榨取性能,CPU 会在保证结果正确的前提下,打乱指令执行顺序——比如一条指令在等内存(miss),就先执行后面不依赖它的指令,不让 CPU 空等。

乱序执行是 A 系列高性能的核心,但它给多核编程埋了一个大坑:内存序(Memory Ordering)——一个核乱序执行,别的核看到的内存写入顺序可能和代码顺序不一样!这就是为什么多核同步需要"内存屏障(Memory Barrier)"指令。这个话题在第五卷(Linux 内存屏障、无锁编程)会展开,是高级并发的硬核内容。M 系列顺序执行,基本没这个烦恼——又一次体现"M 简单确定、A 复杂高性能"。

11.11 RK3588 的 big.LITTLE

你手上的 RK3588 是理解现代 A 系列的好样本。它用了 ARM 的 big.LITTLE 架构:把高性能大核和低功耗小核放在一颗芯片里,按需调度。

RK3588 CPU 布局:
  ┌─────────────────────────────────────┐
  │  4× Cortex-A76 (大核, 高性能, 费电)    │  ← 干重活: 视觉/AI
  │  4× Cortex-A55 (小核, 省电, 性能低)    │  ← 干轻活/待机: 后台任务
  └─────────────────────────────────────┘
  全部共享 L3 Cache 和内存, 架构相同(都跑ARMv8), 指令兼容

调度: Linux 内核(EAS能耗感知调度)按任务轻重, 把它放到大核或小核
  → 重任务上大核(快), 轻任务上小核(省电) → 兼顾性能与续航

为什么能这么混?因为大小核架构相同(11.3 说的),同一个程序在大核小核都能跑,只是快慢功耗不同。Linux 调度器(带能耗感知 EAS)负责决定每个任务放哪个核。这是"性能 vs 功耗"权衡在多核时代的解法**,第八卷讲 CPU 亲和性、SMP 调度时会深入。

11.12 面试常问问题

Q1:ARM 公司是做什么的?为什么各家 ARM 芯片差别很大?
ARM 只设计并授权处理器 IP/架构,不造芯片。各厂商基于授权造不同产品(用 ARM 现成核或自研核),故差异大。

Q2:架构和微架构(实现)的区别?
架构是指令集/寄存器/异常模型等规范(如 ARMv8-A);实现是满足规范的具体核(如 Cortex-A76),决定流水线/Cache/主频。同架构可有多种实现。

Q3:Cortex-M 和 Cortex-A 的本质区别?
见 11.7 表。核心:M 无 MMU、跑 RTOS/裸机、实时确定省电;A 有 MMU、跑 Linux、高性能多核。

Q4:ARM 和 Thumb 指令集的区别?为什么要 Thumb?
ARM 32 位定长功能全但代码大;Thumb 16/32 混合代码密度高省 Flash,Cortex-M 主用。为嵌入式小 Flash 而生。

Q5:什么是 big.LITTLE?RK3588 怎么用?
大核(高性能费电)+小核(省电)同芯片,架构相同指令兼容;调度器按任务轻重分配,兼顾性能与功耗。RK3588 是 4×A76+4×A55。

Q6:MPU 和 MMU 的区别?
MMU 做地址翻译+保护(虚拟内存);MPU 只做区域权限保护,不翻译地址。M/R 系列用 MPU,A 用 MMU。

11.13 常见误区

  1. "ARM 公司造芯片"——不造,只授权设计。
  2. "架构和具体的核是一回事"——架构是规范,核是实现,一对多。
  3. "Cortex-M 也能跑 Linux"——标准 Linux 需要 MMU,M 无 MMU 跑不了(有 uClinux 等无 MMU 变体,但受限)。
  4. "M 和 A 指令完全一样"——不一样,M 用 Thumb-2、A 用 A64/A32,指令集不同。
  5. "大小核是不同架构"——大小核架构相同、指令兼容,只是微架构(性能/功耗)不同。
  6. "乱序执行会算错"——单核结果保证正确,但多核可见的内存序会变,需内存屏障。

11.14 本章总结

  • ARM 是授权设计公司,不造芯片;各厂商基于授权造不同芯片,故差异巨大。
  • 架构(规范,如 ARMv8-A)≠ 实现(核,如 Cortex-A76),同架构可多实现。
  • 三条线:A 应用(有 MMU,跑 Linux)、R 实时、M 微控(无 MMU,跑 RTOS);M/R 用 MPU 做轻量保护。
  • Cortex-M:浅流水线顺序执行、无 MMU、NVIC、两态特权,实时确定省电。
  • Cortex-A:深流水线乱序执行、有 MMU、GIC、EL0-3、多核 SMP,高性能功能全。
  • M/A 差异表 = 第一卷所有分界线的汇总:M 用简单换确定,A 用复杂换性能。
  • 指令集:ARM(A32)/Thumb(代码密度,M 主用)/A64(64 位,A 主用)。
  • 乱序执行是 A 高性能核心,但引入内存序问题,多核需内存屏障。
  • RK3588 用 big.LITTLE(4×A76+4×A55),大小核同架构,调度器按任务分配。

11.16 课后练习与面试题

课后练习

  1. 查你的 STM32 型号,确认它是 Cortex-M 几(M3/M4/M7),对应哪个 ARMv 架构;再查 RK3588 的大小核型号和架构版本。
  2. Linux 上 lscpu 或 cat /proc/cpuinfo 看 RK3588 的核数、大小核、架构特性(features),识别出 A76 和 A55。
  3. 用 taskset 把一个计算程序绑到大核 vs 小核,对比运行时间,亲历性能差异(为第八卷 CPU 亲和性铺垫)。
  4. 把同一段 C 分别用 arm-none-eabi-gcc(M,Thumb)和 aarch64-linux-gnu-gcc(A,A64)编译,对比汇编指令的不同。
  5. 查 Cortex-M 的 MPU 和 Cortex-A 的 MMU 文档,列出两者功能差异。

课后面试题

  1. ARM 的商业模式是什么?为什么各家 ARM 芯片差异大?
  2. 架构和微架构的区别?举例。
  3. Cortex-M 和 Cortex-A 有哪些本质区别?为什么一个跑 RTOS 一个跑 Linux?
  4. MPU 和 MMU 的区别?
  5. 为什么 Cortex-M 用 Thumb 指令集?
  6. 什么是 big.LITTLE?大小核为什么能混用?
  7. 乱序执行带来什么好处和什么问题?
