本章目标：第四章讲了寄存器、第五章讲了栈,这一章把它们拧成一股绳——因为函数调用正是"寄存器 + 栈"配合的经典范例。更重要的是,它和上下文切换只有一线之隔:搞懂函数调用怎么保存/恢复现场,你就理解了任务切换在做什么、又比它多做了什么。学完你要能读懂函数的汇编"序言/尾声",讲清参数传递和返回全过程。

6.1 为什么函数调用值得一章

函数调用你写了千百遍,但底层发生的事远比想象复杂。它要同时协调:寄存器(传参、返回值、返回地址)、栈(栈帧、局部变量、保存现场)、PC(跳过去、跳回来)。它是前几章所有概念的第一次"合奏"。

而且它是通往两个重要主题的桥:

  • 往上:上下文切换(第三卷)= 函数调用式的现场保存 + 换栈换任务;
  • 往下:ABI/调用约定是跨语言、跨库互操作的基础(为什么 C 能调汇编、Python 能调 C)。

6.2 一次函数调用要解决的四个问题

A 调用 B,看似一句 B(),硬件层面必须解决四件事:

  1. 传参:A 要把参数交给 B。放哪?(寄存器 or 栈)
  2. 记路:B 干完要回到 A 的调用点。这个"返回地址"存哪?(LR/栈)
  3. 保现场:B 会用寄存器,可能覆盖 A 正在用的值。谁负责备份?(调用约定 4.8)
  4. 给结果:B 算完的返回值怎么交回 A?(寄存器)

这四件事的具体做法,不是硬件写死的,而是一份大家约定好的"协议"——调用约定。

6.3 调用约定：一份君子协议

调用约定(Calling Convention)规定了上面四件事的具体规则。ARM 的这份协议叫 AAPCS(ARM 架构过程调用标准)。它是 ABI(应用二进制接口)的一部分。

为什么需要它?打个比方:A 公司和 B 公司要合作,得先约定好"货放几号仓库、单据放哪、谁验货"。如果 A 用汇编写、B 用 C 写、甚至 B 是别人编译好的库,只要双方都遵守同一份约定,就能无缝对接。调用约定就是让不同来源的代码能互相调用的通用合同。

ARM(AAPCS)的核心规定(呼应第四章):

寄存器用途谁保存
R0~R3传前 4 个参数;R0 兼放返回值调用者(caller-saved)
R4~R11被调用函数的局部变量被调用者(callee-saved)
R12(IP)临时/过程调用暂存调用者
R13(SP)栈指针特殊,始终维护
R14(LR)返回地址被调用者(若还要调别人)
R15(PC)程序计数器—

6.4 参数怎么传

规则:能用寄存器就用寄存器(快),放不下才用栈。

  • 前 4 个整型/指针参数 → R0, R1, R2, R3(ARM64 是 X0~X7,前 8 个)。
  • 第 5 个及以后的参数 → 压到栈上传递。
  • 返回值 → R0(64 位或结构体有额外规则)。
C:  int foo(int a, int b, int c);   foo(10, 20, 30);

汇编(传参):
   MOV R0, #10    ; a → R0
   MOV R1, #20    ; b → R1
   MOV R2, #30    ; c → R2
   BL  foo        ; 调用(下一节讲 BL)
   ; 返回后 R0 里就是 foo 的返回值

这解释了一个性能经验:参数别太多。超过 4 个(ARM)就得走栈,多一次内存读写。也解释了为什么很多接口喜欢"传一个结构体指针"而不是十个参数——一个指针占一个寄存器就够。

6.5 返回地址与 LR

第三章说过 BL(Branch with Link)是"带返回的跳转"。它比普通跳转 B 多做一件事:

BL foo  做两件事:
   ① LR = 下一条指令的地址   ← 记住"我干完回这里"
   ② PC = foo 的地址         ← 跳过去执行 foo

foo 返回时:
   BX LR   →  PC = LR  ← 跳回记住的地方,继续 A 的代码

妙在:叶子函数(自己不再调别的函数)的返回地址一直待在 LR 里,连栈都不用碰,返回极快。这是 RISC 用寄存器传返回地址的优势(CISC 的 x86 是直接压栈)。

但如果 B 还要调 C 呢?

问题来了:B 被调用时,LR 存着"返回 A 的地址"。可 B 里又执行 BL C,这会覆盖 LR(变成"返回 B 的地址")。等 C 回来,B 想返回 A 却发现 LR 已经被自己搞没了!

解法:B 在调用 C 之前,先把 LR 压栈保存,等 C 回来、B 要返回时再从栈里弹出来。这正是下一节"栈帧序言"要做的事之一。"非叶子函数必须把 LR 存栈",这条规则你现在能自己推导出来了。

6.6 栈帧的建立与拆除(序言 / 尾声)

每个非平凡函数的汇编,开头有一段固定的"序言(Prologue)"建栈帧,结尾有"尾声(Epilogue)"拆栈帧。它俩就是第五章栈帧的施工队。

典型 ARM 函数结构:

foo:
  ; ===== 序言 Prologue(进门先安置) =====
  PUSH {R4-R7, LR}   ① 保存要用的callee-saved寄存器 + 返回地址LR
  SUB  SP, SP, #16   ② 在栈上给局部变量腾空间(下移SP)
  ; (可能还会 MOV FP, SP 设置帧指针)

  ; ===== 函数体 =====
  ...  用 R4-R7 干活, 局部变量通过 [SP,#偏移] 访问
  ...  可能 BL 调用别的函数(此时LR已被序言存好, 安全)

  ; ===== 尾声 Epilogue(出门恢复原样) =====
  ADD  SP, SP, #16   ③ 回收局部变量空间(上移SP)
  POP  {R4-R7, PC}   ④ 恢复R4-R7, 并把返回地址直接弹进PC → 返回!

注意最后那个 POP {..., PC} 的骚操作:把当初压入的 LR 直接弹到 PC,一步完成"恢复寄存器 + 返回"。这是 ARM 常见的高效返回方式。

序言/尾声成对出现,像"有借有还"。序言压了什么,尾声就弹什么,SP 减了多少就加回多少——保证函数返回时栈恢复原状,调用者毫无察觉。如果这个平衡被破坏(比如汇编写错、栈溢出踩乱),就会返回到错误地址,程序跑飞。这是很多"莫名跑飞"bug 的根源。

6.7 完整走一遍：嵌套调用全过程

把前几节合起来,看 main → A → B 的完整栈和寄存器变化:

C:  main() { A(); }   A() { B(); }   B() { int x=1; }

① main 执行 BL A
     LR = "main里A之后那条"
     PC → A                栈: [ main帧 ]

② A 的序言: PUSH {LR}     ← 因为A还要调B,先保住"返回main的地址"
     栈: [ main帧 | A帧(存了返回main的LR) ]

③ A 执行 BL B
     LR = "A里B之后那条"    ← LR被覆盖成返回A的地址(没关系,返回main的已在栈上)
     PC → B                栈不变

④ B 的序言: PUSH {LR} + 腾局部变量x的空间
     栈: [ main帧 | A帧 | B帧(x, 返回A的LR) ]

⑤ B 执行完, 尾声: 回收x空间, POP {PC}
     从B帧弹出"返回A的地址"→PC   → 回到A     栈: [ main帧 | A帧 ]

⑥ A 执行完, 尾声: POP {PC}
     从A帧弹出"返回main的地址"→PC → 回到main  栈: [ main帧 ]

看,返回地址像面包屑一样一层层存在各自的栈帧里,返回时一层层弹出来,精确地原路返回。这就是为什么调用能任意嵌套、递归也不乱——每一层的"回家的路"都独立存在自己的栈帧。

6.8 返回值怎么带回来

约定:返回值放 R0(ARM64 放 X0)。调用者在 BL 返回后直接从 R0 取。

int add(int a,int b){ return a+b; }

add:
   ADD R0, R0, R1   ; a(R0)+b(R1) → R0   (结果直接放R0)
   BX  LR           ; 返回
调用方:
   BL add
   ; 现在 R0 = 结果, 直接用
  • 返回值超过一个寄存器宽度(如 64 位在 32 位机、或返回结构体):用 R0+R1,或"调用者传一个隐藏指针,被调用者把结果写进去"(结构体返回的常见做法)。
  • 这解释了为什么"返回大结构体"可能有性能开销——涉及内存拷贝,不如返回指针。

6.9 递归的真相

递归为什么能工作?现在答案很清晰:每次递归调用,都在栈上新建一个独立的栈帧,拥有自己的局部变量和返回地址。所以 factorial(3) 调 factorial(2) 调 factorial(1),栈上就有三层各自独立的帧,互不干扰,返回时逐层弹回。

factorial(3):  栈上三帧, 各有自己的 n
  ┌────────────┐
  │ f(3) n=3   │
  ├────────────┤
  │ f(2) n=2   │
  ├────────────┤
  │ f(1) n=1   │ ← 触底, 开始逐层返回相乘
  └────────────┘

这也解释了递归的两个代价:①栈空间:每层一个帧,递归太深会栈溢出(第五章);②开销:每层都有序言/尾声的压栈弹栈。所以性能敏感或栈有限(嵌入式)的场景,常把递归改成循环。

6.10 调用链与栈回溯(backtrace)

你在 gdb 里敲 bt,或程序崩溃时打印的那串"调用栈",是怎么来的?就是靠栈帧里存的返回地址。

调试器从当前 SP/FP 出发,顺着"每个栈帧里保存的上一层返回地址和帧指针",一帧一帧往回走,就还原出了 main → A → B → 崩溃点 的完整链条。栈帧的链式结构 = 崩溃现场的"行车记录仪"。

这就是为什么栈被踩坏后,backtrace 会显示乱码或断裂——面包屑被破坏了,追不回去。也是为什么调试嵌入式崩溃时,第一件事常是"看栈有没有溢出"。编译时加 -fno-omit-frame-pointer 保留帧指针,能让回溯更可靠。

6.11 内联、尾调用等优化

编译器会想尽办法减少调用开销:

  • 内联(inline):把小函数的代码直接"贴"到调用处,彻底省掉调用的序言/尾声/传参。代价是代码变大。inline 关键字是建议,-O2 下编译器自主决定。
  • 尾调用优化(TCO):如果函数最后一步是"调用另一个函数并直接返回它的结果",可以复用当前栈帧、用跳转代替调用,不增加栈深度。让某些递归不再栈溢出。
  • 叶子函数优化:不调别人的叶子函数,连 LR 压栈都省了(6.5)。

理解这些,你就明白为什么"看汇编时源码的函数调用有时消失了"——被内联或优化掉了。也明白为什么 debug 版(-O0)和 release 版(-O2)行为看起来不同。

6.12 回扣：函数调用 vs 上下文切换

本章最重要的升华。函数调用和上下文切换,做的事非常像,但有一个关键区别。对比:

函数调用上下文切换
保存现场只存 callee-saved(R4-R11)+LR存全部寄存器(R0-R12,SP,LR,PC,xPSR)
换栈吗不换,同一个栈往下长换!切到另一个任务的栈
谁触发程序主动 BL调度器(常由中断触发)
返回到哪回到调用点,继续同一执行流跳到另一个执行流
知情吗双方代码都知道(有约定)被切的任务毫不知情

为什么切换要存"全部",调用只存"一半"?

因为函数调用是双方约定好的:调用者知道 R0-R3 会被弄脏,自己会提前处理,所以被调方不用管。而上下文切换是强行打断——任务正跑到半路被抢占,它没有任何"我要被切走了,先收拾一下"的机会。所以切换必须把它的完整现场(包括那些函数调用时不用存的 R0-R3、正在用的一切)全部保存,一个都不能漏,否则恢复后就错乱了。

这是本卷的一个高峰认识:上下文切换 = "非自愿的、完整的、跨栈的"函数调用式保存。因为不自愿(被中断打断),所以要存全部;因为跨栈,所以要换 SP。你把第四(寄存器)、五(栈)、六(调用)三章连起来,就完全看懂了第三卷 FreeRTOS 那段 PendSV 切换汇编——它无非就是"手动把完整现场压进 A 的栈、换 SP、从 B 的栈恢复完整现场"。第一卷到这里,操作系统的"心脏"已经在你眼前拆解完毕。

6.13 面试常问问题

Q1:函数调用时参数、返回值、返回地址分别怎么传?
ARM:前 4 参数走 R0-R3,多余走栈;返回值 R0;返回地址 BL 存入 LR,非叶子函数再压栈保存。

Q2:什么是调用约定/ABI?为什么需要?
规定传参、返回、寄存器保存责任的合同,使不同语言/不同编译单元/汇编能互相调用。ARM 是 AAPCS。

Q3:叶子函数和非叶子函数返回地址处理有何不同?
叶子函数返回地址常驻 LR,不压栈,返回快;非叶子函数因还要 BL 会覆盖 LR,必须先把 LR 压栈。

Q4:栈帧里有什么?序言/尾声做什么?
返回地址、保存的 callee-saved 寄存器、局部变量、传出参数。序言建帧(压寄存器/LR、腾局部空间),尾声拆帧(恢复、返回)。

Q5:递归为什么可行?代价是什么?
每层独立栈帧,互不干扰。代价:栈空间(深了溢出)、每层压栈弹栈开销。

Q6:函数调用和上下文切换的区别?
见 6.12 表。核心:调用只存 callee-saved 不换栈、双方知情;切换存全部现场且换栈、被切者不知情——因为切换是强行打断。

6.14 常见误区

  1. "参数都是压栈传的"——ARM 优先用寄存器,超出才压栈;x86-64 也是寄存器优先。
  2. "返回地址一定在栈上"——ARM 先放 LR 寄存器,叶子函数可能根本不压栈。
  3. "inline 一定会内联"——只是建议,编译器可能忽略;-O0 常不内联。
  4. "递归比循环高级/高效"——递归有栈和调用开销,嵌入式常需改循环。
  5. "上下文切换就是个函数调用"——像但不同:切换存全部现场、换栈、被切者不知情。
  6. "返回结构体没成本"——大结构体返回涉及拷贝或隐藏指针,有开销。

6.15 本章总结

  • 一次调用要解决四件事:传参、记返回地址、保现场、给返回值,靠调用约定(AAPCS/ABI)统一。
  • 参数:寄存器优先(R0-R3),多余压栈;返回值放 R0。
  • 返回地址:BL 存入 LR;非叶子函数必须把 LR 压栈(否则被下一次 BL 覆盖)。
  • 序言建栈帧(压 callee-saved+LR、腾局部空间),尾声拆栈帧并返回,成对平衡。
  • 嵌套调用:返回地址像面包屑存在各层栈帧,逐层弹回,精确原路返回。
  • 递归靠每层独立栈帧;代价是栈空间和调用开销。
  • backtrace 靠栈帧链;栈被踩坏则回溯失效。
  • 优化:内联、尾调用、叶子函数优化减少开销。
  • 升华:上下文切换 = 非自愿的、完整的、跨栈的函数调用式保存——存全部现场、换 SP、被切者不知情。第四五六章合流,操作系统心脏拆解完毕。
