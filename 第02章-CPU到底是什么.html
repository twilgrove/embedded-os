本章目标：上一章我们说"CPU 是厨师"，但厨师只是个类比。这一章我们要真正把 CPU 这块芯片拆开——它内部有什么、凭什么会"算"、一条 C 语言语句是怎么变成它能懂的动作、为什么它需要寄存器和时钟。学完你要能回答："CPU 执行一条 a = b + c 到底发生了什么？"

2.1 为什么要先搞懂 CPU

你可能会想：我是做操作系统的，直接学调度、学内核不就行了，为什么要花一整章抠 CPU？

因为操作系统的每一个核心机制，最终都落在 CPU 的某个硬件特性上。不理解 CPU，后面全是空中楼阁。举几个例子，你现在不用懂，只需感受"它们都指向 CPU"：

  • 上一章说的上下文切换，切的就是 CPU 里的寄存器——不懂寄存器，不可能懂切换。
  • 抢占靠中断，而中断是 CPU 的一种硬件行为——CPU 执行完当前指令后去查"有没有中断"。
  • 用户态/内核态是 CPU 内部一个状态位决定的，不是软件想象出来的。
  • 原子操作、锁依赖 CPU 提供的特殊指令(如 ARM 的 LDREX/STREX)。
  • 虚拟内存靠 CPU 里的 MMU 单元把地址翻译一遍。

一句话：操作系统是软件，但它能干的每件事，都是在"使唤 CPU 的硬件能力"。所以我们必须先认识这个被使唤的对象。

2.2 生活类比：CPU 是个只会照单做菜的厨师

上一章把 CPU 比作厨师。现在我们把这个厨师的形象精确化，因为真实的 CPU 是个"能力很怪"的厨师：

  • 他极其快：一秒能做几十亿个动作（GHz 级）。
  • 他极其笨：只会做几百种最基础的动作——"把 A 和 B 相加"、"把这个数搬到那里"、"比较两个数谁大"。仅此而已。
  • 他没有记忆：做完一个动作立刻忘，靠"菜谱"(程序)一步步告诉他下一步干什么。
  • 他手边只有几十个小碗(寄存器)：正在操作的东西必须先放进这几十个小碗，他够不着远处的大仓库(内存)里的东西，除非先"搬"过来。

请牢牢记住这个形象：一个快到离谱、笨到只会几百个基础动作、手边只有几十个小碗、必须靠菜谱一步步指挥的厨师。整个计算机体系，都是围绕"如何伺候好这样一个厨师"设计出来的。

          菜谱(程序,放在大仓库/内存里)
                     │ 一条一条读进来
                     ▼
        ┌────────────────────────────┐
        │            CPU(厨师)         │
        │   ┌─────────┐  ┌──────────┐ │
        │   │ 控制器   │  │  几十个   │ │
        │   │(看菜谱)  │  │  小碗    │ │  ← 寄存器
        │   └────┬────┘  │(寄存器)  │ │
        │        │       └────┬─────┘ │
        │   ┌────▼────────────▼─────┐ │
        │   │   ALU(会算的那双手)     │ │
        │   └───────────────────────┘ │
        └────────────────────────────┘

2.3 CPU 到底在"算"什么：一切都是 0 和 1

我们说 CPU 会"算"，但它算的到底是什么？答案会让你意外：CPU 眼里没有数字、没有字母、没有图片，只有"高电压"和"低电压"两种状态。我们把高电压记作 1，低电压记作 0。

为什么偏偏是两个状态，不是十个？

因为"有电/没电"最容易做、最不容易出错。如果要用电压高低表示 0~9 十个数，你得精确区分 10 个电压档位，稍微有点干扰就认错了。而只区分"有/没有"两档，抗干扰能力极强——这就是为什么全世界的计算机都用二进制。这不是数学偏好，是工程可靠性逼出来的。

一个位、一个字节

  • 一根线上的一个 0 或 1，叫一个 比特(bit，位)——信息的最小单位。
  • 8 个 bit 绑在一起，叫一个 字节(Byte)。8 位能表示 28=256 种组合(0~255)。
  • 为什么是 8 位一组而不是 7 或 10？历史选择 + 刚好够表示一个英文字符(ASCII)，后来成了全行业标准。

数字、文字、图片……全是 0 和 1 的"约定"

关键理解：同一串二进制，是什么意思，完全取决于我们约定怎么解释它。

二进制:  0100 0001
解释为整数  → 65
解释为字符  → 'A'   (ASCII 约定 65 代表 A)
解释为指令  → 某条 CPU 指令的一部分
解释为颜色  → 某个灰度值

这一点极其重要，是后面理解"指令和数据都是二进制、放在同一个内存里"(冯·诺依曼结构)的基础。CPU 本身分不清一段二进制是"数据"还是"指令"，全靠程序流程的约定。

2.4 从开关到加法器：CPU 是怎么"会算"的

现在最神奇的问题来了：CPU 只是一块硅片，凭什么"会做加法"？它又没上过学。

第一层：晶体管就是一个电控开关

CPU 由几十亿个 晶体管(Transistor) 组成。一个晶体管你就理解成一个电控开关：给它的控制脚通电，开关就导通；不通电，就断开。用电来控制电的通断——这是全部魔法的起点。

第二层：几个开关拼成"逻辑门"

把开关按不同方式接线，就能实现"逻辑判断"。这些小电路叫 逻辑门(Logic Gate)：

门生活理解规则
与门 AND两个开关串联，都合上灯才亮1 且 1 = 1，否则 0
或门 OR两个开关并联，任一合上灯就亮有一个 1 就 1
非门 NOT取反，输入 1 输出 00→1，1→0
异或 XOR两个不一样才亮相同为 0，不同为 1

第三层：逻辑门拼成加法器

关键洞察：二进制加法可以完全用逻辑门实现。看一位加法：

  0+0=0(不进位)    0+1=1    1+0=1    1+1=10(本位0,进位1)

观察规律:
  本位结果 = A XOR B    ← 异或门!
  进位     = A AND B    ← 与门!

把这两个门组合起来 = "半加器"
再把多个半加器串起来处理进位 = "全加器" = "多位加法器"

这就是"硅片会算加法"的真相：不是它聪明，而是加法的规则恰好能用几个开关(逻辑门)搭出来。减法、比较、逻辑运算，同理，全是开关的组合。CPU 里那个专门干这些事的电路，就叫 ALU(下一节)。你从"通电/断电"一路推到了"会做数学"——中间没有任何魔法，全是开关的堆叠。

2.5 CPU 的三大件：ALU、寄存器、控制器

把厨师拆开看，他干活靠三样东西。这三样就是 CPU 最核心的三大部件。

① ALU —— 会算的那双手(算术逻辑单元)

ALU(Arithmetic Logic Unit) 就是上一节我们用逻辑门搭出来的那个电路，负责一切实际运算：加、减、与、或、比较大小、移位。厨师的"双手"就是它。

注意 ALU 有个重要特点：它只管算，不管数据从哪来、算完放哪去。你把两个数喂给它的两个输入口，告诉它"做加法"，它立刻从输出口吐出结果。数据的搬运是别人的事。

② 寄存器 —— 手边的小碗(Register)

这是本章、也是整本书最重要的概念之一，第四章会专门深讲，这里先建立直觉。

为什么需要寄存器？没有会怎样？

ALU 要算 b + c，那 b 和 c 这两个数当下得放在哪儿？放在内存(大仓库)里行不行？不行，因为内存太远、太慢。

厨师站在灶台前，食材仓库在几十米外的地下室。如果每切一刀都要跑去仓库拿一次菜、再跑回来，效率低到没法看。正确做法是：把当前要用的几样食材先拿到灶台边的小碗里，用的时候一伸手就够到。

寄存器就是这些"灶台边的小碗"——它直接长在 CPU 内部，紧挨着 ALU，存取速度极快(1 个时钟周期就能拿到)。ALU 运算的输入和输出，几乎都在寄存器之间进行。

为什么寄存器只有几十个，不做几千个？

三个原因，一步步想：

  1. 越快越贵越占地方：寄存器用的是最快的电路，一个寄存器占的芯片面积远大于一格内存。做几千个，芯片面积和成本爆炸。
  2. 指令里要"点名"寄存器，位数有限：指令中要写清"用哪个寄存器"。若有 32 个寄存器，只需 5 个 bit 编号(25=32)；若有 1024 个，就要 10 个 bit，指令会变得又长又臃肿。
  3. 够用就行：编译器有能力把最常用的变量安排进这几十个寄存器，剩下的放内存。几十个已能覆盖绝大多数场景。

ARM Cortex-M 有 16 个核心寄存器(R0~R15)；ARM64(Cortex-A) 有 31 个通用寄存器(X0~X30)。记住这个数量级：几十个，不是几千个。这也解释了上一章为什么"上下文"能被快速保存——因为要存的寄存器就那么几十个。

③ 控制器 —— 看菜谱的大脑(Control Unit)

ALU 会算，寄存器能暂存，但谁来指挥"下一步做什么"？控制器(Control Unit, CU)。

它干的事：从内存里取出下一条指令 → 看懂这条指令要干嘛(译码) → 指挥 ALU、寄存器去完成它。厨师那个"看菜谱、理解这一步、然后动手"的大脑，就是控制器。

控制器里有个关键寄存器叫 PC(Program Counter，程序计数器)，它永远记着"下一条要执行的指令在内存的哪个地址"。控制器取完一条指令，PC 自动指向下一条。PC 就是厨师的手指，一直点在菜谱的当前行上。（还记得吗？上一章说上下文切换要保存 PC——因为它标记了"任务执行到哪一行"。）

2.6 时钟：CPU 的心跳与节拍

前面说厨师"一步一步"做动作。但"一步"到底由什么划分？谁喊"开始下一步"？答案是 时钟(Clock)。

为什么需要时钟？

CPU 内部几十亿个开关同时在动。如果没有一个统一的"节拍"，各部件各干各的，信号还没稳定就被下一级读走，结果全乱。必须有一个总指挥喊节拍，让所有部件"踩着同一个鼓点"齐步走。

这个鼓点就是时钟信号——一个不停在 0 和 1 之间跳变的方波：

时钟信号:  ┌──┐  ┌──┐  ┌──┐  ┌──┐
          │  │  │  │  │  │  │  │
       ───┘  └──┘  └──┘  └──┘  └──
          ↑一个周期↑
       每一次上升沿(0→1),所有部件"走一步"

主频是什么意思

时钟一秒钟跳变多少次，就是 主频(时钟频率)。单位 Hz。

  • 1 GHz = 每秒 10 亿个节拍。
  • RK3588 的 A76 大核主频约 2.4 GHz = 每秒 24 亿个节拍。
  • STM32F4 约 168 MHz = 每秒 1.68 亿个节拍。

所以"CPU 快"到底快在哪？快在它的节拍密——每秒能推进十几亿步。但注意：一步不等于一条指令。有的指令一个周期搞定，有的要好几个周期（比如访问内存的指令）。这是新手最大的误解之一。

面试延伸：为什么主频不能无限提高？因为频率越高，① 每个节拍留给电信号稳定的时间越短，物理上稳不住；② 功耗和发热随频率急剧上升(大致成正比甚至更高)。这就是为什么 2005 年后 CPU 不再拼主频，而是转向"多核"——与其让一个厨师跑更快，不如多请几个厨师。这正好接上了第一章的"并行"。

2.7 一条指令的一生：取指 → 译码 → 执行

现在把三大件和时钟合起来，看 CPU 干活的最核心循环。厨师做菜谱上的每一行，都重复三个动作：

       ┌───────────────────────────────────────┐
       │                                        │
       ▼                                        │
  ① 取指(Fetch)                                 │
     控制器按 PC 指的地址,从内存取出一条指令        │
     PC 自动 +1(指向下一条)                       │
       │                                        │
       ▼                                        │
  ② 译码(Decode)                                │
     控制器看懂这条指令:要做什么操作?用哪些寄存器?   │
       │                                        │
       ▼                                        │
  ③ 执行(Execute)                               │
     指挥 ALU 运算 / 从寄存器取数 / 把结果写回      │
       │                                        │
       └────────── 回到 ① 取下一条 ──────────────┘
              (这个循环永不停止,直到关机)

这个"取指-译码-执行"的无限循环，就是 CPU 的全部工作本质。你电脑里发生的一切——打游戏、放视频、跑 AI——归根结底都是这个循环转了亿万次。

用一行 C 代码走一遍：a = b + c;

假设 b、c 在内存里，编译器会把这一行翻译成几条 CPU 指令(以 ARM 风格示意)：

C 代码:   a = b + c;

编译成指令(简化):
  LDR  R0, [b的地址]   ① 把 b 从内存搬进小碗 R0   (Load)
  LDR  R1, [c的地址]   ② 把 c 从内存搬进小碗 R1
  ADD  R2, R0, R1     ③ 让 ALU 算 R0+R1,结果放 R2 (Add)
  STR  R2, [a的地址]   ④ 把 R2 的结果搬回内存给 a   (Store)

看出门道了吗？一句简单的加法，CPU 要分四步：搬进来、搬进来、算、搬回去。因为 ALU 只能操作寄存器(小碗)里的数，内存(仓库)里的数必须先 Load 进来、算完再 Store 回去。这种"算之前先搬进寄存器"的风格，正是 RISC/ARM 的特点(2.10 节详述)。

这就是"CPU 执行 a=b+c 发生了什么"的完整答案——面试常问，你现在能一步步讲出来了。它也解释了为什么"访问内存慢"会拖累性能：四步里有三步在搬数据。这为第八章讲 Cache 埋下伏笔。

2.8 冯·诺依曼结构与总线

一个根本设计决策：指令和数据放一起

2.3 节说过：指令和数据都是二进制。那它们存哪儿？1945 年冯·诺依曼提出一个影响至今的设计：把指令和数据放在同一个内存里，CPU 通过统一的通道去取。这就是冯·诺依曼结构。

好处是通用、灵活(程序也能被当数据加载、修改)；代价是指令和数据挤同一条通道，形成瓶颈(叫"冯·诺依曼瓶颈")。

与之相对的是哈佛结构：指令和数据分两个内存、两条通道，能同时取指令和取数据，更快。很多单片机(包括你用的 STM32 内核)其实是"改进型哈佛结构"。这就是为什么你会在 STM32 手册里看到 I-Bus(取指令)和 D-Bus(取数据)分开。

总线：厨房与仓库之间的传送带

CPU(厨师)和内存(仓库)是分开的两块芯片，它们之间靠总线(Bus)连接——你就理解成几条传送带。主要三条：

总线作用类比
地址总线CPU 告诉内存"我要第几号格子"喊出仓库货架编号
数据总线真正传输那格子里的数据传送带上运货
控制总线CPU 说"我是要读还是要写"喊"取货"还是"放货"

地址总线的宽度决定了 CPU 最多能寻址多大内存。这是个经典考点：32 根地址线 → 能表示 232 个地址 = 4GB。这就是为什么 32 位系统最多认 4GB 内存！64 位则是 264，大到用不完。你现在明白"32 位/64 位"这个说法一半就指地址/寄存器的宽度了。

2.9 指令集(ISA)：CPU 的"菜谱语言"

厨师笨，只会几百个基础动作。这套"厨师听得懂的所有动作的清单"，就是指令集(Instruction Set Architecture, ISA)。

为什么这东西这么重要？

因为 ISA 是硬件和软件之间的合同。CPU 设计者承诺："我认识这些指令、每条这么做"；编译器和程序员则按这份清单写程序。只要合同不变，同一个程序就能在所有遵守该合同的 CPU 上跑。

这解释了一个你一定困惑过的现象：为什么 x86 编译出来的程序不能直接在 ARM 上跑？因为它们是两份不同的合同、两套不同的"厨师语言"。一个厨师说中文、一个说法语，同一份中文菜谱，法国厨师看不懂。你在 RK3588(ARM)上装软件要找 arm64 版本，就是这个原因。

指令的分类(万变不离其宗)

不管哪种 ISA，指令无非几大类，全部呼应前面讲的三大件：

  • 数据搬运类：Load(内存→寄存器)、Store(寄存器→内存)、Move(寄存器→寄存器)。伺候那几十个小碗。
  • 运算类：ADD、SUB、AND、OR、比较、移位。喂给 ALU 干的。
  • 跳转/分支类：改变 PC 的值，实现 if、for、函数调用。让厨师"跳到菜谱另一行"。
  • 特殊/系统类：触发系统调用、开关中断、操作特权寄存器。这些和操作系统关系最密切。

一个洞察：你写的所有高级语言(C/C++/Python)，最终都被翻译成这四类指令的组合。再复杂的程序，CPU 眼里也不过是"搬、算、跳、系统操作"的排列。

汇编语言：给人看的指令

指令在 CPU 眼里是二进制(如 1110 0000 1000 ...)，人读不了。于是给每条指令起个好记的名字，如 ADD、LDR——这就是汇编语言。汇编和机器码基本一一对应。学内核和 RTOS 移植层，必须能读汇编，因为上下文切换那段代码只能用汇编写(要直接操作寄存器，C 语言够不着)。这也是第一章说"每个 RTOS 都有汇编移植层"的原因。

2.10 RISC vs CISC，ARM vs x86

厨师语言(ISA)有两大流派，理念相反：

CISC(复杂指令集)RISC(精简指令集)
代表x86(Intel/AMD)ARM、RISC-V
理念指令又多又复杂,一条能干很多事指令少而简单,每条只干一件小事
类比菜谱里有"红烧肉"这种一步完成的大动作菜谱只有切、炒、加盐等小动作,大菜靠组合
访问内存运算指令可以直接操作内存里的数运算只能对寄存器,内存必须先 Load/Store
指令长度不定长(1~15 字节),难解析定长(如 4 字节),好解析、好流水线

为什么嵌入式几乎都用 ARM(RISC)？

  1. 省电：指令简单，硬件电路简单，功耗低。手机、无人机、电池设备命根子就是省电。
  2. 好做流水线：定长指令让 CPU 容易"流水作业"(下一节)，性能反而不差。
  3. 芯片小、便宜：适合海量嵌入式设备。

这就是为什么从 STM32 到手机到 RK3588 到的飞控，清一色 ARM。x86 强在桌面/服务器的高性能和生态，但功耗大，不适合嵌入式。

补一个"打脸"细节，面试加分：现代 x86 内部其实也把复杂指令拆成类似 RISC 的"微操作(micro-ops)"来执行。所以 RISC/CISC 的界限如今在实现层面已经模糊，主要区别体现在对外的"指令集合同"和生态上。能说出这句，说明你不是死背书。

2.11 大小端(Endianness)：一个坑了无数人的细节

一个多字节的数(比如 4 字节的整数 0x12345678)存进内存，是"高位字节"先放，还是"低位字节"先放？两种流派：

要存的数: 0x12345678  (12 是最高字节, 78 是最低字节)
内存地址:  低地址 ────────────▶ 高地址

大端(Big-Endian):    12  34  56  78   ← 高位在前,像人读数一样
小端(Little-Endian): 78  56  34  12   ← 低位在前,反着放

生活类比

想象门牌号 1、2、3、4 号房。大端是"把数字最重要的一位放 1 号房"，小端是"把最不重要的一位放 1 号房"。没有对错，纯属约定，但两边不统一就会读错。

为什么你必须知道它

  • ARM 通常用小端(可配置)，网络传输规定用大端(叫"网络字节序")。
  • 做网络通信、CAN 总线、跨设备数据交换、读二进制文件时，如果两边端序不一致，收到的数就是乱的。这类做通信协议的公司极爱考。
  • 处理办法：用 htons/htonl/ntohs/ntohl 这类函数在主机序和网络序之间转换。

面试题原型："怎么用代码判断当前机器是大端还是小端？"答：定义一个 int x=1，取它的第一个字节 *(char*)&x，如果是 1 就是小端(低位放在了低地址)，是 0 就是大端。

2.12 流水线：让厨师流水作业

回到 2.7 的"取指→译码→执行"。假设每步花 1 个时钟周期，一条指令要 3 个周期。问题：执行第 1 条指令时，"取指"部件在干嘛？闲着！这太浪费了。

洗衣店的类比

洗一批衣服要经过：洗 → 烘 → 叠，三台机器。笨办法是：第一批洗完、烘完、叠完，才开始第二批。聪明办法是：第一批去烘的时候，第二批马上开始洗。三台机器同时忙，吞吐量翻几倍。这就是流水线(Pipeline)。

不用流水线(一次一条):
  指令1: 取 译 执
  指令2:       取 译 执
  指令3:             取 译 执     ← 大量部件在空转

用流水线(重叠起来):
  指令1: 取 译 执
  指令2:    取 译 执
  指令3:       取 译 执
  指令4:          取 译 执
        └─每个周期都有一条指令"完工"→吞吐量↑↑

现代 ARM 有十几级甚至更深的流水线。定长指令(RISC)特别适合流水线，这也是 ARM 高效的原因之一。

流水线的麻烦：遇到"跳转"就抓瞎

流水线爽在"提前把后面的指令取进来"。但如果遇到 if(分支跳转)，CPU 在"执行"出结果前，根本不知道下一条该取哪儿的指令——是跳走还是不跳？它只能猜。

  • 猜对了：流水线不断，飞快。
  • 猜错了：已经提前取进来的几条指令全作废，流水线要清空重来，白干好几个周期。这叫流水线冲刷(flush)。

CPU 里有个叫分支预测器的部件专门负责"猜"。这解释了一个著名的性能现象和面试题：为什么处理排好序的数组比乱序的快？因为有序数据让分支预测更容易猜对，减少了流水线冲刷。你写高性能代码时"减少不可预测的分支"，底层原因就在这。

延伸一句(和安全相关)：著名的"幽灵(Spectre)/熔断(Meltdown)"漏洞，就是利用了 CPU"提前猜着执行"(推测执行)会在缓存里留下痕迹，从而偷数据。这说明流水线/推测执行不只是性能话题。

2.13 回扣第一章：CPU 与上下文

现在把这一章和第一章接起来，你会有恍然大悟的感觉。

第一章说"上下文切换就是保存/恢复寄存器"。学完本章你现在知道：

  • 寄存器(小碗)里装着任务当前正在操作的所有数据；
  • PC 记着任务执行到菜谱哪一行；
  • 状态寄存器记着最近比较的结果等。

所以"一个任务此刻的全部现场" = 这几十个寄存器的值。要暂停任务 A 去跑 B，只要把 A 的这几十个寄存器抄到 A 的存档区(栈/TCB)，再把 B 存的值填回这几十个寄存器——B 的 PC 一填回去，厨师的手指就重新点回 B 的菜谱行，无缝续上。

任务A运行中          切换               任务B恢复
CPU寄存器            │                 CPU寄存器
┌────────┐  抄走存档  │  填回B的存档  ┌────────┐
│R0=A的值 │ ─────────▶│◀───────────  │R0=B的值 │
│PC=A的行 │           │               │PC=B的行 │
└────────┘           │               └────────┘
     厨师的"现场"就是这几十个小碗+手指位置

"任务/线程/进程"在硬件层面的真相，就是这几十个寄存器的一组取值。操作系统所谓的"多任务魔法"，物理上不过是反复地抄小碗、填小碗。你把第一章和第二章连起来，就摸到了操作系统最底层的那块基石。

2.14 面试常问问题

Q1：CPU 执行 a = b + c 具体发生了什么？
取指-译码-执行循环；分解成 LDR 把 b、c 从内存装入寄存器、ADD 让 ALU 相加、STR 把结果写回内存。强调 ALU 只能操作寄存器，内存数据必须先 Load。

Q2：为什么需要寄存器？为什么只有几十个？
因为内存太慢太远，运算的操作数必须放在紧邻 ALU 的高速小存储里。数量少是因为：贵、占面积、指令中编号位数有限、几十个够编译器用。

Q3：32 位系统为什么最多认 4GB 内存？
地址总线/地址位宽是 32 位，能表示 2³²=4G 个地址。

Q4：RISC 和 CISC 区别？ARM 属于哪个？为什么嵌入式用 ARM？
RISC 指令少而简单定长、运算只对寄存器；CISC 复杂不定长。ARM 是 RISC。嵌入式选它因为省电、好流水线、芯片小便宜。

Q5：什么是大小端？怎么判断？做什么时要注意？
多字节数在内存中的排列顺序。用 *(char*)&x(x=1) 判断。网络通信、跨设备、二进制协议时必须注意，用 htonl/ntohl 转换。

Q6：流水线是什么？为什么分支会影响性能？
把指令的取/译/执重叠执行提升吞吐；遇到跳转 CPU 需预测下一条，预测错误导致流水线冲刷，白费周期。

2.15 常见误区

  1. "主频越高，一秒执行的指令越多，且成正比"——错。一条指令可能占多个周期(尤其访存指令)，且流水线、Cache 命中率影响巨大。主频只是节拍频率。
  2. "CPU 直接对内存里的变量做运算"——在 RISC/ARM 上不行。必须先 Load 进寄存器。理解这点才懂为什么"访存是性能瓶颈"。
  3. "寄存器是内存的一部分"——不是。寄存器在 CPU 内部，内存是外部芯片，速度差几十上百倍。
  4. "CPU 知道一段二进制是指令还是数据"——不知道。取决于程序流程怎么解释它(冯·诺依曼结构)。PC 指向它就当指令执行。
  5. "大端小端有优劣"——没有，纯约定。只是不统一会出错。
  6. "多核就是主频翻倍"——不是。多核是多个厨师并行，不是单个厨师变快；且很多任务无法完美拆分给多核(见后续并行章节)。

2.16 本章总结

  • CPU 是个"快到离谱、笨到只会几百个基础动作、手边只有几十个小碗、靠菜谱一步步指挥"的厨师。
  • 它只认 0 和 1(有电/没电，为了抗干扰)；数字文字全靠约定解释同一串二进制。
  • "会算"的真相：晶体管=电控开关 → 拼成逻辑门 → 拼成加法器/ALU。没有魔法，全是开关堆叠。
  • 三大件：ALU(会算的手)、寄存器(手边小碗，快但少)、控制器(看菜谱的大脑，含 PC)。
  • 时钟给所有部件统一节拍；主频=每秒节拍数，但一步≠一条指令。
  • 核心循环：取指→译码→执行，永不停止。a=b+c 分解为 Load/Load/Add/Store。
  • 冯·诺依曼把指令数据放一起；总线连接 CPU 与内存；地址总线宽度决定寻址范围(32 位→4GB)。
  • ISA 是软硬件合同，分 RISC(ARM,省电定长,嵌入式首选) 和 CISC(x86)。
  • 大小端是字节排列约定，通信时必须对齐。
  • 流水线让指令重叠执行提吞吐，分支预测失败会冲刷。
  • 回扣：任务的"现场"就是这几十个寄存器的取值，上下文切换=抄小碗、填小碗。
