本章目标:AMP(第58章)里 Linux 和 RTOS 是两个独立世界,但它们必须交互——Linux 要启动/管理 RTOS 核(RemoteProc),两者要交换数据/命令(RPMsg)。本章讲这两个 Linux 官方框架:RemoteProc(管协处理器的生命周期)和 RPMsg(核间消息传递),以及底层的共享内存+中断机制。这是 AMP 的关键技术,做多核异构系统必用,面试常考"两个核怎么通信"。

59.1 AMP 的两个核心需求

AMP 里 Linux(主)和 RTOS(从)要协作,有两个基本需求:

需求问题Linux 框架
管理协处理器Linux 怎么加载 RTOS 固件、启动/停止 RTOS 核、故障恢复?RemoteProc
核间通信两个独立 OS 内存不通,怎么交换数据和命令?RPMsg

两个需求对应两个框架:RemoteProc 解决"生命周期管理"(Linux 如何掌控 RTOS 核的加载和启停),RPMsg 解决"数据通信"(两个 OS 如何互发消息)。它们是配套的——先用 RemoteProc 把 RTOS 核跑起来,再用 RPMsg 和它对话。这两个都是 Linux 内核的标准框架(不是 RK3588 私有),各家 SoC(TI、NXP、瑞芯微、ST)的 AMP 都用它们,所以学会了通用。本章先讲 RemoteProc(启停管理),再讲 RPMsg(通信),最后看底层怎么实现。理解这两个,你就掌握了 AMP 的"骨架"。

59.2 生活例子：总部与分厂

把 Linux 比作总部,RTOS 核比作一个专业分厂(只干一件精密活,如精密加工=实时控制)。

  • RemoteProc = 总部对分厂的管理权:总部负责"建厂"(把生产手册/固件下发给分厂)、"开工"(启动分厂运转)、"停工/重启"(分厂出故障时重新开工)。分厂的开关机由总部掌控。
  • RPMsg = 总部和分厂之间的专线电话/传真:总部发订单(命令)给分厂,分厂回报生产结果(数据)。有固定的收发流程。

关键:总部(Linux)不干预分厂(RTOS)的具体生产(实时控制)——分厂自己独立高效运转;总部只负责"管理分厂的开关机(RemoteProc)"和"和分厂通信下单收货(RPMsg)"。这正是 AMP 的分工:Linux 管理和通信,RTOS 独立干实时的活。两个框架就是"管理权"和"专线电话"。

59.3 RemoteProc：管理协处理器

RemoteProc(Remote Processor,远程处理器框架)是 Linux 里管理"协处理器"(如 AMP 的 RTOS 核)生命周期的框架:

RemoteProc 负责协处理器(RTOS核)的生命周期:
  ① 加载(load):  把RTOS固件(.elf)加载到协处理器的内存
  ② 启动(start): 让协处理器开始执行固件(RTOS跑起来)
  ③ 停止(stop):  停止协处理器
  ④ 恢复(recover): 协处理器崩溃时, 重新加载启动(容错!)

  用户空间接口(sysfs, 第32章一切皆文件):
   echo rtos.elf > /sys/class/remoteproc/remoteproc0/firmware  # 指定固件
   echo start > /sys/class/remoteproc/remoteproc0/state        # 启动
   echo stop  > /sys/class/remoteproc/remoteproc0/state        # 停止

  → Linux像"管理一个设备"一样管理RTOS核
  → RTOS核在RemoteProc眼里就是一个可加载固件、可启停的"处理器设备"

RemoteProc 把协处理器(RTOS 核)抽象成一个"可以加载固件、可以启停"的设备,由 Linux 统一管理其生命周期:加载固件、启动、停止、崩溃后恢复。操作接口是 sysfs(/sys/class/remoteproc/,又是第32章"一切皆文件")——往里 echo 就能指定固件、启停 RTOS 核。关键价值是"可恢复":RTOS 核崩了,Linux 能检测到并用 RemoteProc 重新加载启动它,提高系统鲁棒性(第58章主从模式的容错)。这就是第58章说的"Linux 主控 RTOS 从"的实现——Linux 通过 RemoteProc 掌控 RTOS 核的开关机,像管理一个外设一样管理另一个 OS 的运行。

59.4 固件加载与生命周期

细看 RemoteProc 加载启动 RTOS 的过程(涉及第56章启动、第41章设备树):

RemoteProc 启动一个RTOS核的过程:
  ① Linux读取RTOS固件(ELF格式), 解析它的段(代码/数据在哪个地址)
  ② 把各段拷贝到协处理器的【专用内存】(第58章reserved-memory划的)
  ③ ★解析固件里的【resource table(资源表)】:
      固件告诉Linux "我需要: 一块共享内存做RPMsg、
      用哪个中断通知、要哪些资源"
      → Linux据此配好RPMsg的共享内存和中断(vring, 59.6)
  ④ 释放协处理器的复位(让那个核开始执行固件的入口)
  ⑤ RTOS核从入口开始运行 → RTOS启动、初始化、跑实时任务
  ⑥ 两边的RPMsg通道建立 → 可以通信了

  ★resource table 是固件和Linux的"约定":
    RTOS固件声明自己要什么资源, Linux负责配好 → 自动协商

RemoteProc 启动 RTOS 核的关键是"resource table(资源表)":RTOS 固件里内嵌一张表,声明它需要什么资源(一块共享内存做 RPMsg、用哪个中断通知等),Linux 解析这张表,据此配好通信通道,然后释放协处理器复位让它开始执行。这是一种"固件自描述、Linux 自动配置"的协商机制——类似 USB 描述符(第49章)或设备树(第41章)的思想:让一方声明需求、另一方满足。好处是解耦:RTOS 固件和 Linux 不用硬编码约定,通过资源表动态协商通信资源。整个过程串起了前面多章:加载 ELF 固件(类似第56章加载内核)、放到 reserved-memory(第58章)、配置中断(第13章)、建立 RPMsg 通道(59.5)。理解资源表,你就理解了 Linux 如何"自动地"把一个 RTOS 拉起来并接好通信线。

59.5 RPMsg：核间消息通道

RTOS 核跑起来后,两个 OS 用 RPMsg(Remote Processor Messaging)通信:

RPMsg: 在两个处理器/OS之间传递【消息】的框架
  基于共享内存, 提供类似"网络socket"的消息收发抽象

  概念:
   - endpoint(端点): 通信的一端, 有地址(类似端口, 第34章)
   - channel(通道): 两端之间的逻辑通道, 有名字
   - 发消息: rpmsg_send(endpoint, data, len)
   - 收消息: 注册回调, 消息到达时被调用

  Linux侧: RPMsg设备/驱动, 可暴露成字符设备(/dev/rpmsg_ctrl,
           用户空间也能收发, 又是"一切皆文件")
  RTOS侧: OpenAMP的RPMsg库, 对应的收发API

  → 两个OS像两台网络主机一样【发消息】通信,
     底层是共享内存+中断, 但上层是简洁的消息API

RPMsg 在两个 OS 之间提供"消息收发"的抽象,用起来像网络编程:有 endpoint(端点,类似端口)、channel(通道,有名字),发消息用 rpmsg_send、收消息注册回调。Linux 侧可以把 RPMsg 暴露成字符设备(/dev/rpmsg*),用户空间程序也能收发(又是第32章"一切皆文件");RTOS 侧用 OpenAMP 的 RPMsg 库。本质:两个 OS 像两台网络主机一样互发消息,但底层不是网络,而是共享内存 + 中断(59.6/59.7)。RPMsg 把复杂的"共享内存队列 + 中断通知"封装成简洁的消息 API——又是"加一层抽象",让核间通信像发网络包一样简单。这就是 AMP 里两个 OS "对话"的方式。

59.6 底层机制：virtio + vring

RPMsg 底层怎么实现"共享内存里传消息"?靠 virtio + vring(和虚拟化里的机制同源):

RPMsg底层 = virtio + vring(环形缓冲队列)

  vring: 在共享内存里的一个【环形缓冲区】(第19章队列!)
    ┌──────────────────────────────┐
    │ 一堆buffer + 两个环:              │
    │  available环: 发送方放"有新消息"的buffer │
    │  used环: 接收方放"处理完"的buffer      │
    └──────────────────────────────┘
   两个方向各一组vring(A→B 一组, B→A 一组)

  发消息: 发送方把数据写进一个buffer, 放进available环,
          → 触发中断通知对方 "有消息了"
  收消息: 接收方从available环取buffer, 读数据, 处理完放used环

  ★这不就是第19章FreeRTOS的Queue(环形缓冲+通知)吗?
    只不过这里是【跨OS的共享内存队列】+ 【中断通知】

RPMsg 底层是 virtio/vring,而 vring 本质就是一个"放在共享内存里的环形缓冲队列"——正是第19章 FreeRTOS Queue 的跨核版本!vring 用两个环(available 环放"待处理的消息缓冲",used 环放"已处理的缓冲")实现无锁的生产者-消费者通信(第19章队列思想)。发消息:发送方把数据写进共享内存的 buffer、放进 available 环、触发中断通知对方;收消息:接收方收到中断、从环里取 buffer、读数据。你会发现,核间通信没有任何"新原理"——它就是你早就学过的"环形缓冲队列(第19章)+ 中断通知(第七章)",只不过队列放在了两个 OS 共享的内存里,通知用的是核间中断。这再次印证本教材的核心思想:核心机制(队列、中断)在不同层次反复出现。你在 FreeRTOS 里理解了 Queue,现在看 RPMsg 的 vring 毫无障碍——它就是 Queue 的"跨 OS 放大版"。virtio 是这套共享内存队列的标准化框架(源自虚拟化,虚拟机和宿主机也这么通信)。

59.7 共享内存 + 中断：通信的物理基础

把核间通信的物理基础讲清——归根结底是共享内存 + 核间中断两样:

核间通信的两个物理支柱:
  ① 共享内存: 两个核都能访问的一块DDR区域(第58章划的共享区)
     → 数据放这里, 两个核都能读写 → 数据传递的载体
     → ★但要处理Cache一致性(第43/58章): 一个核写了,
        另一个核的Cache可能是旧的 → 写方要flush、读方要invalidate
        (或用non-cacheable的共享内存, 简单但慢)

  ② 核间中断(IPI / mailbox): 一个核"戳"另一个核
     → "我往共享内存放了消息, 你快来取!" 的通知机制
     → RK3588有专门的mailbox硬件或用GIC的SGI(软件生成中断,第13章)
     → 没有中断通知, 对方就得【轮询】共享内存(浪费, 第43章)

  → 共享内存(传数据) + 中断(通知) = 核间通信的完整物理基础
    RPMsg/vring只是在这之上的软件封装

核间通信归根结底靠两样:共享内存(传数据的载体)+ 核间中断(通知对方"有数据了")。共享内存是两个核都能访问的 DDR 区域(第58章划分);核间中断(IPI/mailbox,或 GIC 的 SGI 软件生成中断,第13章)让一个核能"戳"另一个核——避免对方傻等着轮询(第43章:有中断就别轮询)。关键坑还是 Cache 一致性(第43/58章):共享内存被两个核的 Cache 各自缓存,一个核写了,另一个核可能读到 Cache 里的旧值——所以写方要 flush、读方要 invalidate(或用不可缓存的共享内存,简单但慢)。RPMsg/vring 就是在"共享内存+中断"这个物理基础上的软件封装。理解了这两个支柱,你就理解了所有核间通信的本质——上层的 RPMsg、RemoteProc 都建立在它们之上。

59.8 一条消息的完整旅程

把一条消息从 Linux 到 RTOS 的完整旅程串起来(综合前面):

Linux发一条命令给RTOS(如"设置电机转速500"):
  ① Linux应用: rpmsg_send(ep, "speed=500", ...)
  ② RPMsg把消息写进【共享内存】的vring buffer
  ③ 放进available环 (更新环的索引)
  ④ ★flush Cache: 确保数据真写到内存(RTOS核能读到新值)
  ⑤ 触发【核间中断】通知RTOS核 "有消息"
  ─────────── 跨越到 RTOS 世界 ───────────
  ⑥ RTOS核收到中断 → RPMsg中断处理
  ⑦ ★invalidate Cache: 丢弃旧Cache, 从内存读新数据
  ⑧ 从vring取出buffer, 读到 "speed=500"
  ⑨ RTOS的回调被调用 → 执行: 设置电机转速500 (实时控制)
  ⑩ (可能)RTOS回复结果 → 反向走一遍(B→A的vring)

  → 一条消息 = 共享内存写 + Cache同步 + 中断通知 + 对方读+Cache同步

这条旅程综合了全卷知识:RPMsg API(59.5)→ vring 环形队列(59.6,第19章)→ 共享内存(59.7,第58章)→ Cache 同步(第43章)→ 核间中断(第13章)→ RTOS 回调(第七章中断、第三卷)→ 实时控制。特别注意两次 Cache 同步(第④步 flush、第⑦步 invalidate)——这是 AMP 通信最容易漏、最容易出诡异 bug 的地方(数据发了但对方读到旧值)。一条看似简单的"发命令给电机",背后是共享内存、Cache、中断、队列、两个 OS 的协作——这就是 AMP 核间通信的真实复杂度,也是你在 RK3588 上做 AMP 要处理的实际细节。理解这条完整旅程,你就能调试 AMP 通信问题(消息丢了?查中断;数据错了?查 Cache 一致性)。

59.9 OpenAMP 标准

为了让 AMP 通信标准化、跨平台,有一个开源框架 OpenAMP:

OpenAMP (Open Asymmetric Multi-Processing):
  一个开源框架, 标准化AMP的核间通信和管理:
   - RemoteProc: 生命周期管理(和Linux的对应)
   - RPMsg: 消息传递
   - virtio: 底层传输

  为什么重要:
   ① RTOS侧(RT-Thread/FreeRTOS/裸机)用OpenAMP库,
      就能和Linux的RemoteProc/RPMsg【互通】(两边遵循同一标准)
   ② 跨SoC通用(TI/NXP/瑞芯微/ST都支持)
   ③ 不用自己从零写核间通信协议

  → Linux侧: 内核自带remoteproc/rpmsg
     RTOS侧: 集成OpenAMP库
     两边遵循OpenAMP/virtio标准 → 自动互通

OpenAMP 是 AMP 核间通信的开源标准框架,让 RTOS 侧(RT-Thread/FreeRTOS/裸机)和 Linux 侧能互通。Linux 内核自带 remoteproc/rpmsg 实现,RTOS 侧集成 OpenAMP 库(提供对应的 RPMsg/virtio 实现),两边遵循同一套 virtio/RPMsg 标准,就能自动握手通信——不用自己从零设计核间通信协议。好处是标准化和可移植:换个 SoC(TI、NXP、瑞芯微都支持)、换个 RTOS,通信框架不变。这就是为什么 AMP 开发不像想象中那么难——OpenAMP + Linux remoteproc/rpmsg 提供了现成的、标准的通信基础设施,你主要做配置(资源表、内存划分)和应用逻辑(收发什么消息),不用造轮子。瑞芯微对 RK3588 的 AMP 就基于这套标准。

59.10 用 RPMsg 编程

实际用 RPMsg 收发的代码骨架(Linux 用户空间侧,直观):

// Linux用户空间通过 /dev/rpmsg 收发(需内核rpmsg_char驱动)
int fd = open("/dev/rpmsg0", O_RDWR);   // RPMsg通道字符设备
write(fd, "speed=500", 9);              // 发消息给RTOS
char buf[64];
int n = read(fd, buf, sizeof(buf));     // 收RTOS的回复
// → 就像读写文件/socket一样简单! (RPMsg抽象的价值)

// 内核驱动侧: 注册rpmsg_driver, probe里建endpoint, 收消息回调
static int rpmsg_cb(struct rpmsg_device *rp, void *data, int len,
                    void *priv, u32 src){
    // 处理来自RTOS的消息
    return 0;
}

// RTOS侧(OpenAMP):
//   rpmsg_create_ept(...) 建端点
//   rpmsg_send(ept, data, len) 发
//   收消息回调里处理

RPMsg 编程非常简单——Linux 用户空间可以直接通过 /dev/rpmsg 字符设备 read/write 收发消息(像文件/socket),底层的共享内存、vring、Cache 同步、中断全被封装隐藏。这就是抽象的价值(第32章一切皆文件 + 第59.5 消息抽象):你写 write(fd, "speed=500") 发命令给 RTOS 电机控制,就像写文件一样,不用管底层多复杂。RTOS 侧用 OpenAMP 的 rpmsg_send/回调对应。对应用开发者:在 Linux 上跑视觉/决策,算出控制指令后,通过 RPMsg 发给 RTOS 核执行实时电机控制——就是这么简洁的接口。复杂的 AMP 基础设施(RemoteProc 加载、vring、Cache)由系统层配好,应用层只管收发消息。这体现了良好的分层:底层复杂、上层简单。

核间通信这一章,几乎是全书核心机制的一次大综合:队列(第19章 vring)、IPC/消息(第26/34章)、中断(第7/13章)、Cache 一致性(第14/43章)、字符设备(第39章)、固件加载(第56章)——全都用上了。而它的本质思想你早就懂:RPMsg 就是"跨 OS 的消息队列"——共享内存放队列(第19章 Queue)、中断做通知(第七章)、加一层抽象封装成消息 API(第26/34章 IPC/socket)。没有任何全新的原理,只是把熟悉的机制用在了"两个 OS 之间"这个新场景。这正是本教材反复证明的:掌握了核心机制(队列、中断、共享内存、抽象分层),你就能理解任何看似高深的系统技术——核间通信听起来很"高级",拆开来就是你早就会的几样东西的组合。实战上,RemoteProc(启停 RTOS)+ RPMsg(通信)是 RK3588 AMP 的两大支柱,用它们实现"Linux 智能 + RTOS 实时"的协作。下一章(第60章)深入共享内存和零拷贝——如何让大数据(如图像)在 Linux 和 RTOS/加速器之间高效传递,而不只是 RPMsg 那种小消息。第八卷正一步步把 AMP 系统的每个环节讲透。

59.12 面试常问问题

Q1:AMP 里两个 OS 怎么协作?需要哪两个框架?
RemoteProc(Linux 管理 RTOS 核的生命周期:加载固件、启停、恢复)+ RPMsg(两个 OS 间消息通信)。先用 RemoteProc 启动 RTOS,再用 RPMsg 通信。

Q2:RemoteProc 是什么?怎么用?
Linux 管理协处理器(RTOS 核)生命周期的框架:加载 ELF 固件到其内存、启动、停止、崩溃恢复。通过 sysfs(/sys/class/remoteproc/)操作。关键价值:可恢复(RTOS 崩了 Linux 重新加载)。

Q3:RPMsg 是什么?底层怎么实现?★
核间消息传递框架,用起来像 socket(endpoint/channel/send/回调)。底层=virtio+vring:vring 是共享内存里的环形缓冲队列(本质是第19章 Queue 的跨核版),发消息写队列+触发核间中断通知,收消息取队列。

Q4:核间通信的物理基础是什么?
共享内存(传数据)+ 核间中断(IPI/mailbox/GIC SGI,通知对方"有数据")。没中断就得轮询(浪费)。关键坑:共享内存 Cache 一致性,写方 flush、读方 invalidate。

Q5:核间通信为什么要处理 Cache 一致性?★
共享内存被两核 Cache 各自缓存,一核写了数据,另一核可能读到自己 Cache 里的旧值。所以写方要 flush(刷到内存)、读方要 invalidate(丢弃旧 Cache 重读),或用不可缓存共享内存。这是第43章 DMA 一致性问题在 AMP 的重现。

Q6:OpenAMP 是什么?
AMP 核间通信的开源标准框架(RemoteProc/RPMsg/virtio)。RTOS 侧集成 OpenAMP,和 Linux 内核的 remoteproc/rpmsg 遵循同一标准,自动互通,跨 SoC 通用,不用自己造轮子。

Q7:vring 和 FreeRTOS Queue 什么关系?
vring 本质就是环形缓冲队列(第19章 Queue),只是放在两个 OS 共享的内存里、用核间中断通知。核间通信没有新原理,是"共享内存队列+中断"的组合。

59.13 常见误区

  1. "RemoteProc 和 RPMsg 是一回事"——RemoteProc 管生命周期(启停),RPMsg 管通信(收发),配套但不同。
  2. "核间通信有独特的新原理"——本质是共享内存(队列)+ 中断,是已学机制的组合。
  3. "共享内存写了对方就能读到"——要处理 Cache 一致性(flush/invalidate),否则读到旧值。
  4. "RPMsg 只能内核用"——可暴露成 /dev/rpmsg 字符设备,用户空间也能收发。
  5. "AMP 通信要自己写协议"——用 OpenAMP + Linux remoteproc/rpmsg 标准框架。
  6. "核间通知靠轮询"——用核间中断(IPI/mailbox)通知,别傻轮询。

59.14 本章总结

  • AMP 两需求:RemoteProc(管理协处理器生命周期)+ RPMsg(核间消息通信)。
  • RemoteProc:Linux 把 RTOS 核当"可加载固件、可启停的设备"管理(加载/启动/停止/崩溃恢复),sysfs 操作;靠固件里的 resource table 声明资源、Linux 自动配置。
  • RPMsg:核间消息抽象(endpoint/channel,像 socket),Linux 可暴露 /dev/rpmsg 让用户空间收发。
  • 底层:virtio + vring——vring 是共享内存里的环形缓冲队列(第19章 Queue 跨核版),发消息写队列+触发中断,收消息取队列。
  • 物理基础:共享内存(传数据)+ 核间中断(IPI/mailbox/SGI,通知);关键坑=Cache 一致性(写 flush、读 invalidate)。
  • 一条消息旅程:RPMsg→写 vring→flush→中断→(RTOS)invalidate→读→回调→实时控制。
  • OpenAMP:AMP 通信开源标准(RTOS 侧集成),和 Linux remoteproc/rpmsg 互通,跨 SoC 通用。
  • 本质:核间通信=共享内存队列 + 中断 + 抽象分层的组合,全书核心机制的大综合。
